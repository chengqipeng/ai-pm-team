# CRM 行业上下文压缩场景设计

> 本文档从 CRM 行业的真实业务场景出发，分析每个场景中的上下文膨胀模式和压缩策略。

---

## 一、CRM 行业的 4 类典型 Agent 使用场景

| 场景 | 用户角色 | 典型对话轮次 | 工具调用次数 | 上下文膨胀特征 |
|------|---------|------------|------------|--------------|
| C1: 客户拜访准备 | 销售经理 | 5-8 轮 | 6-10 次 | 多数据源交叉查询，单次结果中等 |
| C2: 客户投诉诊断 | 客服人员 | 8-15 轮 | 5-8 次 | 多轮追问，历史工单数据大 |
| C3: 销售漏斗分析 | 运营分析师 | 3-6 轮 | 4-8 次 | 聚合统计结果大，多维度交叉 |
| C4: 商机批量跟进 | 销售主管 | 10-20 轮 | 15-30 次 | 逐条处理，工具调用密集 |

## 二、C1: 客户拜访准备

### 业务场景

销售经理明天要拜访一个重要客户，让 Agent 准备一份完整的客户背景资料。

### 对话流程

```
用户: "明天要拜访深圳华创科技，帮我准备一下客户背景资料"

Agent 执行计划:
  Step 1: 查询 CRM 中的客户记录（query_data → account）
  Step 2: 查询该客户的所有商机（query_data → opportunity, filters={accountId: xxx}）
  Step 3: 查询该客户的历史工单（query_data → case, filters={accountId: xxx}）
  Step 4: 查询该客户的联系人（query_data → contact, filters={accountId: xxx}）
  Step 5: 查询企业工商信息（company_info → "深圳华创科技有限公司"）
  Step 6: 搜索行业动态（web_search → "华创科技 最新动态"）
  Step 7: 生成拜访准备报告
```

### 上下文膨胀分析

```
消息                                    tokens    累计      占比
─────────────────────────────────────────────────────────────
[user] "明天要拜访深圳华创科技..."         30       30      0.06%
[assistant] tool_use: query_data          50       80      0.16%
[tool_result] 客户基本信息 JSON           800      880     1.7%
[assistant] "华创科技是一家..."           200     1080     2.1%
[assistant] tool_use: query_data(商机)     50     1130     2.2%
[tool_result] 5 条商机记录               2500     3630     7.1%    ← 商机数据较大
[assistant] "该客户有5个商机..."          300     3930     7.7%
[assistant] tool_use: query_data(工单)     50     3980     7.8%
[tool_result] 8 条历史工单               3000     6980    13.7%    ← 工单数据较大
[assistant] "历史上有8个工单..."          250     7230    14.2%
[assistant] tool_use: query_data(联系人)   50     7280    14.3%
[tool_result] 3 个联系人                  600     7880    15.5%
[assistant] "主要联系人有..."            150     8030    15.7%
[assistant] tool_use: company_info         40     8070    15.8%
[tool_result] 工商信息                   1800     9870    19.4%
[assistant] "工商信息显示..."            200    10070    19.7%
[assistant] tool_use: web_search           50    10120    19.8%
[tool_result] 搜索结果                   2100    12220    24.0%
[assistant] "最新动态..."               200    12420    24.4%
[assistant] 生成拜访准备报告             2000    14420    28.3%
─────────────────────────────────────────────────────────────
总计: 14,420 tokens，占 51K 可用空间的 28.3%
```

### 压缩分析

```
结论: 不需要压缩

原因:
  - 7 次工具调用，每次返回 600-3000 tokens
  - 总计 28.3%，远低于 50% 阈值
  - 事前控制有效: query_data 默认 page_size=20，商机和工单各返回 5-8 条，不会爆炸

潜在风险:
  - 如果客户有 50+ 条商机 → query_data 分页返回前 20 条 + total 数
  - 如果客户有 100+ 条工单 → 同上
  - 分页机制确保单次返回不超过 3K tokens

用户追问场景:
  用户: "第 3 个商机的详细情况"
  → Agent 调用 query_data(record_id=xxx) 获取单条详情
  → 新增 ~1K tokens
  → 累计 ~15.5K = 30.4%，仍然安全
```

## 三、C2: 客户投诉诊断

### 业务场景

客服人员收到一个客户投诉"订单状态一直显示处理中"，让 Agent 帮忙诊断问题。

### 对话流程

```
用户: "客户张三投诉说他的订单一直显示处理中，帮我查一下什么情况"

Agent 执行:
  Step 1: 查询客户信息（query_data → contact, filters={name: "张三"}）
  Step 2: 查询该客户的订单（query_data → order, filters={contactId: xxx, status: "processing"}）
  Step 3: 查询订单的审批流程状态（query_data → approval, filters={orderId: xxx}）
  
用户: "审批卡在哪一步了"
  Step 4: 查询审批节点详情（query_data → approval_node）
  
用户: "之前有没有类似的问题"
  Step 5: 搜索历史记忆（search_memories → "订单 处理中 审批 卡住"）
  Step 6: 查询类似工单（query_data → case, filters={category: "订单问题"}）
  
用户: "怎么解决"
  Step 7: Agent 基于历史经验给出解决方案
  
用户: "帮我创建一个工单跟踪这个问题"
  Step 8: 创建工单（modify_data → create case）
  
用户: "通知客户我们在处理了"
  Step 9: 发送通知（send_notification）
  
用户: "把这次的解决方案记下来"
  Step 10: 保存记忆（save_memory）
```

### 上下文膨胀分析

```
总计: 12,130 tokens，占 51K 的 23.8%
10 轮对话 + 10 次工具调用，不需要压缩
```

### 压缩分析

```
结论: 不需要压缩

特点:
  - 客服场景是"诊断型"对话，每轮追问都依赖前面的上下文
  - 不能裁剪早期历史——"订单卡在财务总监审批"这个信息贯穿整个对话
  - 工具结果相对较小（单条订单、单个审批节点）
  - 最大的结果是"5条类似工单"（2000 tokens），但这是诊断的关键参考

如果客服继续处理下一个投诉（同一会话）:
  用户: "还有一个客户李四也投诉了..."
  → 新一轮诊断开始，上下文继续增长
  → 到第 20 轮时可能达到 25K tokens（49%）
  → 接近 50% 阈值
  
  此时 Microcompact 生效:
  → 张三的工具结果（Step 1-6）已经被 Agent 消化到回复中
  → 可以安全清除张三相关的 tool_result
  → 保留 Agent 的回复（包含诊断结论和解决方案）
  → 节省 ~6K tokens
```

## 四、C3: 销售漏斗分析

### 业务场景

运营分析师要生成本月的销售漏斗报告，分析各阶段的转化率。

### 对话流程

```
用户: "帮我分析一下本月的销售漏斗，看看各阶段转化率"

Agent 执行:
  Step 1: 查询商机的阶段定义（query_schema → opportunity 的 stage 字段选项值）
  Step 2: 按阶段统计商机数量（analyze_data → opportunity, group_by=stage）
  Step 3: 按阶段统计商机金额（analyze_data → opportunity, group_by=stage, metrics=sum(amount)）
  Step 4: 计算环比（analyze_data → opportunity, time_field=createdAt, granularity=month）
  Step 5: 查询本月新增商机来源（analyze_data → opportunity, group_by=source, filters={createdAt: >月初}）
  Step 6: 生成漏斗报告

用户: "哪个阶段流失最多"
  Step 7: Agent 基于已有数据分析（不需要新的工具调用）

用户: "流失的商机主要是什么原因"
  Step 8: 查询已关闭商机的关闭原因（analyze_data → opportunity, filters={stage: "closed_lost"}, group_by=lostReason）
```

### 上下文膨胀分析

```
消息                                    tokens    累计      占比
─────────────────────────────────────────────────────────────
[user] "分析本月销售漏斗..."               25       25      0.05%
[tool_result] 阶段定义（6个阶段）          400      475     0.9%
[assistant] "商机有6个阶段..."            150      675     1.3%
[tool_result] 按阶段统计数量              800     1625     3.2%    ← 6个阶段的统计
[assistant] "各阶段商机数量..."           300     2075     4.1%
[tool_result] 按阶段统计金额              800     3025     5.9%
[assistant] "各阶段金额..."              300     3475     6.8%
[tool_result] 月度趋势（3个月）          1200     4825     9.5%    ← 时间序列数据较大
[assistant] "环比趋势..."               400     5375    10.5%
[tool_result] 来源分布                    600     6125    12.0%
[assistant] "新增商机来源..."            200     6475    12.7%
[assistant] 漏斗报告（含表格）           2500     9125    17.9%    ← 报告本身较大

[user] "哪个阶段流失最多"                  15     9140    17.9%
[assistant] "根据数据，方案评估→报价..."   400    9690    19.0%    ← 不需要工具调用

[user] "流失原因"                          10     9700    19.0%
[tool_result] 关闭原因分布                 500   10350    20.3%
[assistant] "主要原因: 价格(35%)..."       300   10800    21.2%
─────────────────────────────────────────────────────────────
总计: 10,800 tokens，占 51K 的 21.2%
```

### 压缩分析

```
结论: 不需要压缩

特点:
  - 分析场景的工具调用以 analyze_data 为主，返回的是聚合统计结果（不是原始记录）
  - 聚合结果天然紧凑: 6 个阶段 × 3 个指标 = 18 个数字，不到 1K tokens
  - 最大的输出是 Agent 生成的报告（2500 tokens），但这是最终交付物，不能压缩
  - query_schema 的阶段定义（400 tokens）全程需要，不能清除

如果用户要求更深入的分析:
  用户: "按销售人员分别看各阶段转化率"
  → analyze_data(group_by=[stage, ownerId]) → 可能 30 个销售 × 6 个阶段 = 180 行
  → ~3K tokens
  → 累计 ~14K = 27.5%，仍然安全

  用户: "再按行业看看"
  → 又一次 analyze_data → ~2K tokens
  → 累计 ~16K = 31.4%，仍然安全

  即使连续做 5 个维度的交叉分析，也只到 ~25K = 49%，刚好不触发压缩
```

## 五、C4: 商机批量跟进（最容易触发压缩的场景）

### 业务场景

销售主管要逐个检查本周需要跟进的商机，更新状态，安排下一步行动。

### 对话流程

```
用户: "帮我看看本周需要跟进的商机，逐个过一遍"

Agent 执行:
  Step 1: 查询本周需跟进的商机列表（query_data → opportunity, filters={nextFollowUp: 本周}）
  → 返回 15 条商机

  对每条商机:
    Step N: 查询商机详情 + 最近活动记录
    Step N+1: 用户决定下一步（更新状态/安排任务/跳过）
    Step N+2: 执行更新（modify_data）
    
  15 条商机 × 3 步 = 45 次工具调用 + 30 轮对话
```

### 上下文膨胀分析

```
初始查询:
  [tool_result] 15 条商机列表              3000 tokens
  [assistant] "本周有15条商机需要跟进..."    500 tokens

第 1 条商机:
  [user] "看第一条"                          10
  [tool_result] 商机详情 + 活动记录         1500
  [assistant] "商机: 华创科技ERP项目..."      400
  [user] "推进到报价阶段"                     15
  [tool_result] 更新成功                      50
  [assistant] "已更新"                        30
  小计: ~2000 tokens

第 2 条商机: ~2000 tokens
第 3 条商机: ~2000 tokens
...

累计:
  3 条后:  3500 + 3×2000 = 9,500 tokens (18.6%)
  5 条后:  3500 + 5×2000 = 13,500 tokens (26.5%)
  8 条后:  3500 + 8×2000 = 19,500 tokens (38.2%)
  10 条后: 3500 + 10×2000 = 23,500 tokens (46.1%)  ← 接近 50%
  12 条后: 3500 + 12×2000 = 27,500 tokens (53.9%)  ← 触发压缩！
  15 条后: 3500 + 15×2000 = 33,500 tokens (65.7%)  ← 如果不压缩会很危险
```

### 压缩策略（关键场景）

```
第 12 条商机处理时触发 Microcompact（53.9% > 50%）:

压缩前的 messages（约 50 条消息）:
  [tool_result] 15条商机列表                3000 tokens
  [assistant] "本周有15条..."               500
  --- 第 1 条商机 ---
  [tool_result] 商机1详情                   1500  ← 可清除
  [assistant] "商机1: 华创科技..."           400  ← 保留（消化结果）
  [tool_result] 更新成功                      50  ← 可清除
  [assistant] "已更新"                        30  ← 保留
  --- 第 2 条商机 ---
  [tool_result] 商机2详情                   1500  ← 可清除
  ...
  --- 第 10 条商机 ---
  [tool_result] 商机10详情                  1500  ← 保留（最近 5 个之内）
  ...
  --- 第 12 条商机 ---
  [tool_result] 商机12详情                  1500  ← 保留（最近的）

Microcompact 执行:
  保留最近 5 个 tool_result（第 10-12 条商机的详情 + 更新结果）
  清除第 1-9 条商机的 tool_result:
    9 条商机 × (1500 + 50) = 13,950 tokens → 替换为 9 × 20 = 180 tokens
    节省: 13,770 tokens

  特殊处理: 初始的"15条商机列表"（3000 tokens）
    这个列表是导航用的（用户说"看第 N 条"时 Agent 需要知道列表）
    → 不清除（标记为受保护）
    → 但如果已经处理了 12 条，可以压缩为:
      "[商机列表已压缩: 共15条，已处理12条(1-12)，待处理3条(13-15)]"
      → 从 3000 tokens 压缩到 50 tokens

压缩后:
  27,500 - 13,770 - 2,950 = 10,780 tokens (21.1%)
  → 回到安全区间，继续处理第 13-15 条

信息不丢失的原因:
  第 1-9 条商机的处理结果在 assistant 消息中:
    "商机1: 华创科技ERP项目 → 已推进到报价阶段"
    "商机2: 中科院设备采购 → 已安排下周拜访"
    ...
  这些 assistant 消息不被 Microcompact 清除
  Agent 仍然知道之前处理了什么
```

### 如果用户回头问已处理的商机

```
用户: "刚才第 3 条商机是什么来着"

情况 1: 第 3 条的 tool_result 还没被清除（在最近 5 个之内）
  → Agent 直接从上下文中找到答案

情况 2: 第 3 条的 tool_result 已被清除
  → Agent 看到 assistant 消息 "商机3: XX公司YY项目 → 已更新为..."
  → 可以回答基本信息
  → 如果用户要详细信息 → Agent 重新调用 query_data(record_id=xxx)
  → 新增 ~1500 tokens，可接受

情况 3: 用户要看所有已处理商机的汇总
  → Agent 从 assistant 消息中提取所有处理结果
  → 不需要重新查询
  → 生成汇总: "已处理12条商机: 5条推进阶段, 3条安排拜访, 2条标记流失, 2条暂缓"
```

## 六、压缩策略与 CRM 场景的匹配

| CRM 场景 | 典型 tokens | 是否触发压缩 | 压缩策略 | 信息保全方式 |
|----------|-----------|------------|---------|------------|
| C1 客户拜访准备 | ~14K (28%) | 否 | 不需要 | — |
| C2 客户投诉诊断 | ~12K (24%) | 否 | 不需要 | — |
| C2 连续处理多个投诉 | ~25K (49%) | 边缘 | Microcompact 清除前一个投诉的 tool_result | 诊断结论在 assistant 消息中 |
| C3 销售漏斗分析 | ~11K (22%) | 否 | 不需要 | — |
| C3 多维度深入分析 | ~25K (49%) | 边缘 | Microcompact 清除早期统计结果 | 分析结论在 assistant 消息中 |
| C4 商机批量跟进 | ~28K+ (55%+) | **是** | Microcompact 清除已处理商机的 tool_result | 处理结果在 assistant 消息中 |
| C4 + 用户回头追问 | +1.5K | 否 | Agent 重新查询单条记录 | 重新调用 query_data |

**关键发现**:
1. CRM 的大部分场景（C1/C2/C3）在事前控制下不需要压缩
2. 唯一容易触发压缩的是 C4（批量逐条处理）——因为工具调用次数多（15×3=45 次）
3. C4 的压缩是安全的——每条商机处理完后，结果已经在 assistant 消息中，tool_result 可以清除
4. 全程不需要 LLM 摘要压缩——Microcompact（纯规则，毫秒级）就够了

## 七、CRM 场景特有的压缩规则

### 7.1 商机列表的特殊处理

```
query_data 返回的列表结果（如"15条待跟进商机"）在批量处理场景中是导航索引。
不能简单清除，但可以在大部分已处理后压缩为进度摘要:

原始: [{id: 1, name: "华创科技ERP", stage: "方案评估", amount: 500万}, ...]  3000 tokens
压缩: "[商机列表: 共15条，已处理12条，待处理: 13.XX公司/14.YY公司/15.ZZ公司]"  100 tokens

触发条件: 列表中 >80% 的记录已被逐条查询过
```

### 7.2 客户 360 视图的保护

```
客户拜访准备（C1）中，Agent 查询了客户的多个维度:
  - 基本信息（account）
  - 商机列表（opportunity）
  - 工单历史（case）
  - 联系人（contact）
  - 工商信息（company_info）

这些数据共同构成"客户 360 视图"。
如果 Microcompact 清除了其中某个维度的 tool_result:
  → Agent 生成的报告可能缺少该维度的信息
  → 但 Agent 在查询每个维度后都生成了 assistant 消息（消化结果）
  → 报告生成时引用的是 assistant 消息，不是 tool_result
  → 所以清除 tool_result 是安全的（前提: Agent 已经消化过）

规则: 只有 Agent 已经生成了包含该数据关键信息的 assistant 消息后，
      对应的 tool_result 才可以被 Microcompact 清除。
```

### 7.3 审批流程数据的保护

```
客服诊断（C2）中，审批流程的状态数据是诊断的核心依据:
  "订单卡在财务总监审批节点，已等待 3 天"

这个信息如果被清除，后续对话中 Agent 可能给出错误的诊断。

规则: 如果 tool_result 中包含"审批"、"流程"、"状态"等关键词，
      且对应的 assistant 消息中没有完整复述该信息，
      则该 tool_result 受保护，不被 Microcompact 清除。

实现: 在 Microcompact 中增加"关键信息检查":
  对于每个待清除的 tool_result:
    1. 找到紧跟其后的 assistant 消息
    2. 检查 assistant 消息是否包含 tool_result 中的关键数值
    3. 如果 assistant 消息遗漏了关键数值 → 不清除该 tool_result
```
