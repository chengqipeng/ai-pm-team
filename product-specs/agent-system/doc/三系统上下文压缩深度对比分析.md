## 一、三系统定位与压缩目标差异

| 维度 | Claude Code | Hermes Agent | neo-apps-ai-agent-service |
|------|------------|--------------|--------------------------|
| 产品形态 | 终端编码助手 | 通用持久 Agent 运行时 | 2B CRM 业务 Agent（LangGraph + FastAPI） |

## 二、架构层次对比

### 2.1 Claude Code：五层级联（Five-Layer Cascade）

Claude Code 的压缩是业界最精细的工程实现，核心思想是**按成本递增逐级触发**：

```
Layer 0: 大结果持久化到磁盘
  ↓ 仍然超限
Layer 1: Cached Microcompact（利用 API cache_edits 删除服务端缓存中的旧工具结果）
  ↓ 仍然超限
Layer 2: Time-Based Microcompact（空闲 >60min 后清理本地消息内容）
  ↓ 仍然超限
Layer 3: Session Memory Compact（利用后台持续维护的结构化笔记作为摘要，零 LLM 成本）
  ↓ 仍然超限
Layer 4: Full Compact（Fork Agent 调用 LLM 生成完整摘要）
```

**关键设计决策：**

1. **Layer 0 — 磁盘持久化**：工具结果超过 50K 字符时写入磁盘，上下文只保留 2KB 预览 + 文件路径。Read 工具豁免（避免循环依赖）。单条消息所有 tool_result 总量上限 200K 字符。

2. **Layer 1 — Cached Microcompact（最精妙的设计）**：利用 Anthropic API 的 `cache_edits` 能力，直接在服务端缓存中删除旧工具结果，**本地消息完全不修改**。prompt cache 前缀不受影响，几乎零成本。

3. **Layer 2 — Time-Based Microcompact**：用户离开超过 60 分钟后（服务端缓存已过期），直接修改本地消息内容清理旧工具结果。与 Layer 1 互斥。

4. **Layer 3 — Session Memory Compact（最创新的设计）**：后台 Fork Agent 持续维护一份结构化 Markdown 笔记（11 个 section）。压缩时直接用这份笔记作为摘要，**零 LLM 调用成本**。

5. **Layer 4 — Full Compact**：最后手段。通过 Fork Agent 共享主会话的 prompt cache 前缀。摘要 prompt 使用两阶段结构：`<analysis>` 思考块 + `<summary>` 正式摘要（9 个 section）。

**触发阈值**：200K 窗口约 167K 触发。熔断器：连续 3 次失败后停止重试。

### 2.2 Hermes Agent：双层压缩 + 可插拔引擎（源码深度分析）

> 源码位置：`agent/context_compressor.py`（53KB）、`agent/context_engine.py`、`agent/prompt_caching.py`、`agent/auxiliary_client.py`

```
Gateway Session Hygiene（85% 阈值，安全网，粗估 token）
  ↓
Agent ContextCompressor（50% 阈值，主压缩器，真实 API token 计数）
  Phase 1: 工具结果智能裁剪（3 遍扫描：去重 → 信息摘要替换 → 参数截断）
  Phase 2: 确定边界（head 保护 + token 预算 tail 保护 + 边界对齐 + 最后用户消息锚定）
  Phase 3: 生成结构化摘要（12 section 模板，迭代更新，focus topic 支持）
  Phase 4: 组装 + 角色交替修复 + 孤立 tool 对清理 + 反抖动保护
```

**精华 1：三遍工具结果裁剪（`_prune_old_tool_results`）**

这是 Hermes 最精细的设计之一——不是简单地用占位符替换旧结果，而是分三遍处理：

- **Pass 1 — 去重**：对所有 tool_result 内容做 MD5 哈希，相同内容只保留最新的一份，旧的替换为 `[Duplicate tool output — same content as a more recent call]`。解决"同一个文件被读 5 次"的问题。
- **Pass 2 — 信息摘要替换**：对保护区外的旧 tool_result（>200 字符），不是用通用占位符，而是调用 `_summarize_tool_result()` 生成**信息丰富的一行摘要**。例如：
  - `[terminal] ran 'npm test' -> exit 0, 47 lines output`
  - `[read_file] read config.py from line 1 (1,200 chars)`
  - `[search_files] content search for 'compress' in agent/ -> 12 matches`
  - `[write_file] wrote to main.py (35 lines)`
  - `[web_search] query='hermes agent context' (3,400 chars result)`
  - 支持 20+ 种工具的专用摘要格式，每种都提取最有价值的信息（命令、退出码、文件路径、匹配数等）
- **Pass 3 — 参数截断**：对保护区外的 assistant 消息中的 tool_call arguments（如 write_file 的 50KB content 参数），截断到 200 字符 + `...[truncated]`。

**保护区边界**使用 token 预算（而非固定消息数）从末尾向前累积，同时保证最少 N 条消息。

**精华 2：Token 预算 Tail 保护（`_find_tail_cut_by_tokens`）**

不是简单的"保留最近 20 条"，而是：
- 从消息列表末尾向前累积 token，直到达到 `tail_token_budget`（默认 = `threshold_tokens × 0.20`，200K 模型约 20K tokens）
- 硬性最少保护 3 条消息
- 允许超出预算 1.5 倍以避免在超大消息中间切割
- **边界对齐**：`_align_boundary_backward()` 确保不拆分 tool_call/tool_result 组——如果切割点落在连续 tool_result 中间，向前回退到父 assistant 消息
- **最后用户消息锚定**（`_ensure_last_user_message_in_tail`，修复 #10896）：确保最近的 user 消息始终在 tail 中，否则 LLM 会丢失当前任务

**精华 3：12-Section 结构化摘要模板**

摘要模板比之前分析的 7 section 更丰富，实际有 12 个 section：

```
## Active Task          ← 最重要的字段：逐字复制用户最近的未完成请求
## Goal                 ← 用户的整体目标
## Constraints & Preferences
## Completed Actions    ← 编号列表，格式：N. ACTION target — outcome [tool: name]
## Active State         ← 工作目录、分支、修改的文件、测试状态、运行中的进程
## In Progress          ← 压缩触发时正在进行的工作
## Blocked              ← 未解决的阻塞/错误，包含精确错误消息
## Key Decisions        ← 重要技术决策及原因
## Resolved Questions   ← 已回答的用户问题（含答案，防止下一个 assistant 重复回答）
## Pending User Asks    ← 未回答的用户请求
## Relevant Files       ← 读取/修改/创建的文件
## Remaining Work       ← 剩余工作（作为上下文，不是指令）
## Critical Context     ← 具体值、错误消息、配置细节
```

关键设计：
- `Active Task` 被标注为"THE SINGLE MOST IMPORTANT FIELD"——要求逐字复制用户最近的未完成请求
- `Completed Actions` 要求精确格式：`N. ACTION target — outcome [tool: name]`，包含文件路径、命令、行号
- `Resolved Questions` 防止下一个 assistant 重复回答已解决的问题
- `Remaining Work` 刻意用"remaining"而非"next steps"，避免被 LLM 误读为主动指令

**精华 4：迭代摘要更新**

后续压缩时，将上次摘要 + 新增对话一起发给 LLM，要求：
- PRESERVE 所有仍然相关的信息
- ADD 新完成的 action 到编号列表（继续编号）
- MOVE 已完成的项从 "In Progress" 到 "Completed Actions"
- MOVE 已回答的问题到 "Resolved Questions"
- UPDATE "Active State" 反映当前状态
- CRITICAL: 更新 "Active Task" 为用户最近的未完成请求

存储在 `self._previous_summary` 中，跨压缩周期保持。

**精华 5：摘要 Prompt 的"隔离"设计**

```python
SUMMARY_PREFIX = (
    "[CONTEXT COMPACTION — REFERENCE ONLY] Earlier turns were compacted "
    "into the summary below. This is a handoff from a previous context "
    "window — treat it as background reference, NOT as active instructions. "
    "Do NOT answer questions or fulfill requests mentioned in this summary; "
    "they were already addressed. "
    "Your current task is identified in the '## Active Task' section..."
)
```

摘要器 preamble 借鉴了 OpenCode 的"do not respond to any questions"和 Codex 的"different assistant"设计：
- 告诉摘要器它是一个"summarization agent creating a context checkpoint"
- 输出将注入给一个"DIFFERENT assistant"
- 不要回答任何问题，只输出结构化摘要

**精华 6：Focus Topic 引导压缩**

支持 `/compress <topic>` 命令（借鉴 Claude Code 的 `/compact`）：
- 当用户指定 focus topic 时，摘要器将 60-70% 的 token 预算分配给相关内容
- 不相关的内容更激进地压缩（一行摘要或省略）

**精华 7：反抖动保护（Anti-Thrashing）**

```python
def should_compress(self, prompt_tokens=None):
    if self._ineffective_compression_count >= 2:
        logger.warning("Compression skipped — last %d compressions saved <10%% each.")
        return False
```

如果连续 2 次压缩每次节省 <10%，跳过后续压缩。防止"每次只删 1-2 条消息"的无限循环。

**精华 8：摘要失败冷却**

- 无 provider 可用：冷却 600 秒（10 分钟）
- 瞬态错误（超时/限流）：冷却 60 秒
- 摘要模型不可用（404/503）：自动降级到主模型重试
- 冷却期间跳过摘要，middle 段直接丢弃（有损但不阻塞）

**精华 9：角色交替修复**

压缩后摘要消息的 role 选择避免与前后消息同 role（API 要求交替）。如果无论选 user 还是 assistant 都会冲突，则将摘要合并到 tail 的第一条消息中，用分隔符隔开。

**精华 10：可插拔引擎 ABC**

`ContextEngine` 定义了完整的生命周期接口：
- `should_compress()` / `compress()` — 核心压缩
- `on_session_start()` / `on_session_end()` / `on_session_reset()` — 会话生命周期
- `get_tool_schemas()` / `handle_tool_call()` — 引擎可以暴露工具（如 LCM 的 `lcm_grep`）
- `update_model()` — 模型切换时重新计算预算
- `get_status()` — 状态展示

**精华 11：Prompt Caching（`system_and_3` 策略）**

4 个 `cache_control` 断点：system prompt + 最近 3 条非 system 消息的滚动窗口。支持 5m / 1h TTL。处理 string/list/None/tool 四种 content 格式。

**精华 12：辅助 LLM 路由（`auxiliary_client.py`）**

摘要调用通过 `call_llm(task="compression")` 路由，支持：
- 7 级 provider 自动检测链（OpenRouter → Nous → Custom → Codex → API-key providers）
- 按 task 配置独立的 provider/model/timeout（`auxiliary.compression.model`）
- 402 支付错误自动降级到下一个 provider
- 连接错误自动降级
- 客户端缓存 + 事件循环感知（防止 async 客户端跨 loop 使用）


### 2.3 neo-apps-ai-agent-service：源头隔离 + 虚拟文件 + 摘要回写

> 源码位置：`repos/apass_old_projects/neo-apps-ai-agent-service/service/neo_agent/`（新版 Agent，2025-12-10 起）

neo-apps 的上下文管理与 Claude Code / Hermes **有本质区别**——它不是在"已经膨胀的上下文中做压缩"，而是**从源头控制什么内容进入 LLM 上下文**。

```
┌─────────────────────────────────────────────────────────────┐
│ 核心架构：工具结果走前端组件，不进入 LLM 历史上下文            │
│                                                              │
│ 当前轮次（LangGraph State 内）:                               │
│   工具结果 → process_sub_agent_result()                      │
│     ├→ 结果 ≤500 字符: 原文作为 ToolMessage 回传 reasoning   │
│     └→ 结果 >500 字符: 调用 LLM 生成摘要 → 摘要回传 reasoning│
│                         原文存入 FileInfo（虚拟文件）          │
│   工具结果 → 前端 UI 组件渲染（CustomContent/TextContent）    │
│     └→ 按标准组件 schema 组合概要数据，推送给前端展示          │
│                                                              │
│ 历史上下文（跨轮次）:                                         │
│   数据库存储: query + answer + answerSummary                  │
│   加载历史时（_init_neo_agent_state）:                        │
│     conversation_history = history_summary_message            │
│       → user: query / assistant: answerSummary（优先）或 answer│
│     file_list 虚拟文件:                                       │
│       → content = 完整 answer / summary = answerSummary       │
│   工具调用过程: ❌ 完全不进入历史上下文                        │
└─────────────────────────────────────────────────────────────┘
```

**关键设计决策（基于新版 Agent 源码分析）：**

1. **历史消息加载（`graph.py:_init_neo_agent_state` + `main.py:_init_history`）**：
   - 从数据库 `query_message` 加载最近 5 轮对话（`pageSize=5`）
   - 构建两套历史：
     - `history_summary_message`：user 的 query + assistant 的 `answerSummary`（优先）或完整 `answer` → 传入 `NeoAgentState.conversation_history`
     - `real_messages`：user 的 query + assistant 的**完整 answer** → 用于构建虚拟文件
   - `_init_neo_agent_state` 将 `real_messages` 转为 `FileInfo` 虚拟文件列表：
     - `file_path="/conversation_history"`
     - `content` = 完整 answer（供需要时引用原文）
     - `summary` = answerSummary（供意图识别等场景使用摘要）
     - `extend.query` = 用户问题，`extend.files` = 关联文件
   - **工具调用过程（tool_call / tool_result）完全不进入历史上下文**

2. **当前轮次的工具结果处理（`node.py:process_sub_agent_result`）**：
   - 子 Agent 执行完后，遍历结果中的 `TextContent` 和 `CustomContent`
   - `CustomContent`（UI 组件）：尝试 `template2str()` 或 `schema2str()` 转为文本
   - 当结果 >500 字符且非 `return_direct` 时，调用 `agent_summary_model()` 生成摘要
   - 摘要作为 `ToolMessage` 回传给 reasoning 循环
   - 原文存入 `TaskResult.original_text`

3. **虚拟文件系统（`state.py:FileInfo` + `node.py:call_sub_agent_after_node`）**：
   - 子 Agent 执行结果被包装为 `FileInfo` 对象存入 `NeoAgentState.file_list`
   - `FileInfo` 包含 `content`（原文）、`summary`（摘要）、`file_path`（如 `/action_result`）
   - 历史对话也被转为虚拟文件（`file_path="/conversation_history"`），用于意图识别
   - 虚拟文件在当前会话内跨步骤引用，但**不持久化到磁盘**，会话结束即消失

4. **回复摘要回写（`node.py:answer_question_node`）**：
   - `update_answer_summary`：当最终回复 >500 字符时，调用 `agent_summary_model()` 生成摘要
   - 摘要写入 `message_.answerSummary` 字段，持久化到数据库
   - 下次加载历史时，`history_summary_message` 优先使用 `answerSummary`
   - 这是 neo-apps 的**核心压缩机制**——不是压缩已有上下文，而是在存储时就生成摘要

5. **前端 UI 组件与上下文的关系**：
   - 工具结果通过 `neo_ai_emit_custom`（标准 UI 组件）和 `neo_ai_emit_message`（文本）推送给前端
   - 组件按 `schema` + `propsData` 渲染（如 BANT 分析卡片、客户画像卡片、音频文件卡片）
   - **组件数据不进入 LLM 上下文**——前端展示和 LLM 上下文是两条独立的数据通道

6. **意图识别中的历史组装（`node.py:get_task_list_impl`）**：
   - 从 `file_list` 中提取 `file_path="/conversation_history"` 的虚拟文件
   - 组装为 `user:{query}\nassistant:{summary}` 格式
   - 只使用摘要，不使用完整回复

7. **reasoning 循环中的历史组装（`node.py:reasoning_node`）**：
   - `messages_ = [SystemMessage, *history, *messages]`
   - `history` = `state.conversation_history`（即 `history_summary_message`，answerSummary 优先）
   - `messages` = 当前轮次的消息链（包含 ToolMessage 摘要）

**实际代码中不存在的东西：**
- ❌ 没有 SQLite/DuckDB/内存数据库
- ❌ 没有长期记忆存储系统（记忆模块 2026-04 开始开发，仍在迭代中）
- ❌ 没有 Microcompact / snip_history 等规则压缩
- ❌ 没有 prompt cache 优化
- ❌ 没有迭代摘要
- ❌ 没有可插拔压缩引擎

## 三、核心机制逐项对比

### 3.1 工具结果管理

| 机制 | Claude Code | Hermes Agent | neo-apps |
|------|------------|--------------|----------|
| 工具结果在上下文中的存在形式 | 完整保留在消息链中，通过多层压缩逐步清理 | 完整保留在消息链中，通过四阶段压缩清理 | **当前轮次**：>500 字符的结果由 LLM 生成摘要后作为 ToolMessage 回传；**历史轮次**：工具结果完全不进入历史上下文 |
| 大结果处理 | 超 50K 写磁盘，保留 2KB 预览 | 无专门处理 | >500 字符调用 LLM 摘要；原文存入 FileInfo 虚拟文件供当前会话引用 |
| 旧结果清理 | cache_edits 删服务端缓存 | >200 字符替换为占位符 | 不需要清理——工具结果本身不进入历史上下文 |
| 前端展示 | 终端文本输出 | 终端文本输出 | 标准 UI 组件（CustomContent）渲染，组件数据与 LLM 上下文分离 |

### 3.2 历史上下文管理

| 机制 | Claude Code | Hermes Agent | neo-apps |
|------|------------|--------------|----------|
| 历史存储内容 | 完整消息链（user + assistant + tool_call + tool_result） | 完整消息链 | **只存 query + answer + answerSummary**，工具过程不存储 |
| 历史加载方式 | 从内存/checkpoint 加载完整消息链 | 从内存加载完整消息链 | 从数据库加载最近 5 轮；构建两套：`conversation_history`（answerSummary）+ 虚拟文件（完整 answer + answerSummary） |
| assistant 回复处理 | 完整保留 | 完整保留 | 虚拟文件 content 保留完整 answer；`conversation_history` 和意图识别使用 answerSummary |
| 压缩时机 | 上下文接近窗口限制时 | 上下文超过 50% 阈值时 | **存储时就压缩**——answer 写入数据库时同步生成 answerSummary |

### 3.3 摘要生成策略

| 机制 | Claude Code | Hermes Agent | neo-apps |
|------|------------|--------------|----------|
| 零成本摘要 | Session Memory（后台持续维护笔记） | 无 | 无 |
| LLM 摘要触发 | Full Compact（上下文超限时） | Phase 3（上下文超 50% 时） | **两个触发点**：① 子 Agent 结果 >500 字符时摘要（`process_sub_agent_result`）；② 最终回复 >500 字符时生成 answerSummary（`answer_question_node`） |
| 摘要用途 | 替换被压缩的消息段 | 替换被压缩的 middle 段 | ① 工具结果摘要回传 reasoning 循环；② answerSummary 存入数据库供下次历史加载 |
| 迭代更新 | 无（Session Memory 是增量的） | 有（传入上次摘要要求更新） | 无 |

### 3.4 缓存策略

| 机制 | Claude Code | Hermes Agent | neo-apps |
|------|------------|--------------|----------|
| Prompt Cache | 极致优化（Cached MC + Fork 共享缓存） | system_and_3 策略 | 无 |

## 五、设计哲学对比

### 5.1 核心设计理念

| 理念 | Claude Code | Hermes Agent | neo-apps |
|------|------------|--------------|----------|
| 压缩范式 | **事后压缩**：完整消息链膨胀后逐级压缩 | **事后压缩**：完整消息链膨胀后四阶段压缩 | **源头隔离**：工具结果不进入历史上下文，存储时就生成摘要 |
| 工具结果的归宿 | 留在消息链中，逐步清理 | 留在消息链中，逐步清理 | 走前端 UI 组件渲染，LLM 上下文只存摘要 |
| 对 prompt cache 的态度 | 极致尊重 | 有意识优化 | 不涉及 |
| 对信息丢失的态度 | 接受有损，通过恢复机制补偿 | 接受有损，通过迭代更新减缓衰减 | 工具原文存虚拟文件（当前会话可引用），历史只存摘要 |
| LLM 摘要成本 | Session Memory 分摊；Full Compact 共享缓存 | 每次压缩调用辅助 LLM | 每次工具结果 >500 字符时调用；每次回复 >500 字符时调用 |
| 复杂度 | 极高（5 层级联 + 恢复 + 熔断） | 中等（双层 + 可插拔） | 低（源头隔离 + 按需摘要） |

### 5.2 三种范式的本质区别

```
Claude Code / Hermes 的思路:
  所有消息（user + assistant + tool_call + tool_result）→ 完整进入上下文
  → 上下文膨胀 → 触发压缩 → 裁剪/摘要/清理

neo-apps 的思路:
  工具结果 → 前端 UI 组件渲染（不进入历史上下文）
  工具结果 → 当前轮次 LLM 摘要 → ToolMessage 回传
  最终回复 → answerSummary 写入 DB
  历史加载 → 只取 query + answerSummary
  → 上下文从源头就是"瘦"的，不需要事后压缩
```

## 六、关键差异深度分析

### 6.1 Claude Code 独有：Session Memory（零成本摘要）

后台 Fork Agent 持续维护结构化笔记（11 section），压缩时直接用笔记作为摘要，零额外 LLM 调用。本质是**增量备份**。

### 6.2 Claude Code 独有：Cached Microcompact（缓存编辑）

利用 Anthropic API `cache_edits` 在服务端缓存中删除旧工具结果，本地消息不修改，prompt cache 不受影响。Anthropic 平台独有能力。

### 6.3 Hermes 独有：可插拔上下文引擎 + 迭代摘要 + 信息摘要替换

Hermes 的精华不仅在架构层面（可插拔引擎），更在工程细节：

1. **信息摘要替换**（vs 通用占位符）：`_summarize_tool_result()` 为 20+ 种工具生成专用的一行摘要，保留最有价值的信息（命令、退出码、文件路径、匹配数）。这比 Claude Code 的 `cache_edits`（直接删除）和通用 `[Old tool result cleared]` 都保留了更多信息。

2. **三遍裁剪**：去重 → 信息摘要 → 参数截断，每一遍解决不同的膨胀源。特别是 Pass 1 的去重（同一文件读 5 次只保留最新）和 Pass 3 的参数截断（write_file 的 50KB content 参数）是其他系统没有的。

3. **最后用户消息锚定**（#10896 修复）：确保最近的 user 消息始终在 tail 保护区中。这解决了一个微妙的 bug——边界对齐可能把最后的 user 消息推入 middle 区被摘要，导致 LLM 丢失当前任务。

4. **Resolved Questions section**：摘要模板中专门有一个 section 记录"已回答的问题及答案"，防止压缩后 LLM 重复回答已解决的问题。这是其他系统都没有的设计。

5. **反抖动保护**：连续 2 次压缩节省 <10% 时自动跳过，防止无限循环。Claude Code 用熔断器（3 次失败停止），但 Hermes 的方案更精细——它检测的是"压缩无效"而非"压缩失败"。

### 6.4 neo-apps 独有：源头隔离 + UI 组件分流 + answerSummary 回写

这是三个系统中**唯一不依赖"事后压缩"的方案**：

1. **UI 组件分流**：工具结果通过 `CustomContent`（BANT 卡片、客户画像、音频文件卡片等）推送给前端渲染，组件数据不进入 LLM 上下文。这是 2B 产品的天然优势——前端有标准组件体系，工具结果的"展示"和"理解"可以分离。

2. **工具结果摘要**：`process_sub_agent_result` 对 >500 字符的结果调用 LLM 生成摘要，摘要（而非原文）作为 ToolMessage 回传 reasoning 循环。原文存入 `FileInfo` 虚拟文件供当前会话内引用。

3. **answerSummary 回写**：`answer_question_node` 在生成最终回复后，对 >500 字符的回复调用 LLM 生成摘要，写入 `message_.answerSummary`。下次加载历史时，`conversation_history` 使用 answerSummary，虚拟文件中 content 保留完整 answer、summary 保留 answerSummary。

4. **双套历史的精妙设计**：新版 Agent 同时维护两套历史视图——`conversation_history`（answerSummary，用于 reasoning 循环，保持上下文精简）和虚拟文件 `file_list`（完整 answer + answerSummary，用于意图识别和需要引用原文时）。这比单纯的"只存摘要"或"只存原文"更灵活。

**这种方案的优势**：上下文从源头就是"瘦"的，不需要复杂的事后压缩机制。在 CRM 场景中，工具返回的是结构化业务数据（客户列表、商机统计、BANT 分析），这些数据的"展示"由前端组件负责，LLM 只需要理解摘要即可做出下一步决策。

**这种方案的劣势**：LLM 在 reasoning 循环中看不到工具结果的完整细节，可能影响多步推理的准确性。如果用户追问"刚才查到的第 3 个客户的详细信息"，LLM 无法从历史上下文中找到（因为只有摘要）。

| 场景 | 最佳方案 | 原因 |
|------|---------|------|
| 编码助手（频繁读文件、运行命令） | Claude Code | 5 层级联 + prompt cache 极致优化 + 压缩后文件恢复 |
| 通用长对话 Agent | Hermes | 可插拔引擎 + 迭代摘要 + 多 LLM 后端支持 |

## 八、对 neo-apps 的改进建议

neo-apps 的"源头隔离"方案在 CRM 场景下是合理的设计选择，但仍有改进空间：

### 8.1 虚拟文件跨轮次持久化

**优先级：高**

当前 `FileInfo` 虚拟文件只在当前会话的 LangGraph State 中存在，会话结束即消失。如果将虚拟文件持久化（如存入 Redis 或数据库），用户追问历史工具结果时可以回读。

### 8.2 answerSummary 质量优化

**优先级：高**

当前 `ANSWER_QUESTION_SUMMARY_PROMPT` 和 `SUMMARY_PROMPT` 是通用模板。可以按业务场景（客户分析、Pipeline 分析、Coaching）使用不同的摘要模板，保留更多业务关键信息（精确数字、客户名称、商机阶段等）。

### 8.3 工具结果摘要的阈值优化

**优先级：中**

当前固定 500 字符阈值。可以按工具类型动态调整——查询类工具（返回列表数据）可以用更低阈值触发摘要，分析类工具（返回洞察结论）可以用更高阈值保留更多原文。

### 8.4 借鉴 Hermes 的迭代摘要

**优先级：中**

当前每次 `answerSummary` 都是独立生成的。如果在生成摘要时传入上一轮的摘要，要求"更新而非重新摘要"，可以在多轮对话中保持更好的信息连贯性。

### 8.5 借鉴 Claude Code 的 Session Memory

**优先级：低（但价值高）**

在 Agent 执行间隙，后台维护一份结构化的会话笔记。这份笔记可以替代当前的 `conversation_history` 虚拟文件，提供更高质量的历史上下文。

## 九、总结

三个系统代表了上下文管理的**三种根本不同的范式**：

1. **Claude Code — 事后压缩的工程极致**：完整消息链 → 五层级联压缩。适合编码场景（工具结果就是代码/日志，LLM 必须看到完整内容才能推理）。

2. **Hermes Agent — 事后压缩的架构优雅**：完整消息链 → 可插拔四阶段压缩。适合通用场景，扩展性最强。

3. **neo-apps — 源头隔离的业务驱动**：工具结果走前端组件渲染 + LLM 上下文只存 query + answerSummary。适合 2B 业务场景（有标准 UI 组件体系，工具结果的"展示"和"理解"可以分离）。

**核心洞察**：neo-apps 的方案不是"压缩做得不够好"，而是**根本不需要 Claude Code / Hermes 那种事后压缩**——因为它从架构层面就把工具结果隔离在了 LLM 上下文之外。这是 2B 产品（有前端组件体系）相对于 CLI 工具（Claude Code）和通用 Agent 运行时（Hermes）的天然优势。

但这种方案的代价是：LLM 在多步推理中只能看到工具结果的摘要，无法引用完整细节。如果未来需要支持更复杂的多步推理（如"基于刚才查到的客户数据做交叉分析"），可能需要引入类似 Claude Code 的虚拟文件磁盘持久化 + 按需回读机制。
