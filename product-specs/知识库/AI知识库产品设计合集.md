## 一、AI知识库功能演进总览

AI知识库是NeoAgent平台的核心基础设施，为百事通"问知识库"能力提供RAG检索支撑。

**核心架构：** AI知识库 → 数据集 → 文档 → 切片 → 向量索引

**演进时间线：**

| 时间 | 版本 | 核心变更 |
|:---|:---|:---|
| 2025-06 | 618版本 | 知识库-数据集两层架构改造，合并智能文档搜索和智能问答 |
| 2025-09 | 知识库同步 | 支持实体字段文档、实体相关文档、公司/合作伙伴知识库的自动同步 |
| 2026-01 | Agentic RAG设计 | 知识库检索优化方案概览，元数据Schema+智能入库+层级索引+Self-Querying |
| 2026-04 | 知识库检索优化V2 | 完整的需求澄清+方案设计，AI全自动元数据管理 |
| 2026-04 | Agent架构重构P0 | knowledge_search从百事通中独立为通用Tool |

## 二、知识库-数据集改造（618版本）

> 来源：Wiki [ID:152093904] AI知识库测试用例（156条）

### 核心改动

1. AI知识库管理入口：NeoAgent > AI知识库管理
2. 两层架构：AI知识库 → 数据集
3. 合并智能文档搜索和智能问答为统一的知识库问答Action
4. 历史数据自动映射到新架构

### AI知识库实体

| 字段 | 说明 |
|:---|:---|
| 名称 | 知识库名称，最大50字符，租户内唯一 |
| API名称 | 系统标识，创建后不可改，自动加_c后缀 |
| 最大召回数量 | 范围[1,10]整数，默认3 |
| 最小匹配度 | 范围(0,1]，1位小数，默认0.5 |
| 描述 | 知识库描述 |
| 引用数据集 | 最多引用10个数据集 |

### 预置知识库

| 名称 | API名称 | 最大召回数量 | 最小匹配度 | 说明 |
|:---|:---|:---|:---|:---|
| 智能文档检索知识库 | file_answer | 5 | 0.5 | 不可删除，可编辑 |
| 财报知识库 | （隐藏） | - | - | 隐藏 |

### 数据集管理

- 数据集列表字段：名称、描述、同步方式、最近同步时间
- 数据集支持新建、编辑、删除
- 数据集可被多个AI知识库引用
- 删除数据集时校验是否被引用

### 知识库同步

支持的同步场景：
- 公司知识库/合作伙伴知识库目录同步
- 实体字段文档同步（自动/手动）
- 实体相关文档同步（自动/手动）
- 删除/恢复操作的同步处理

## 三、知识库检索优化（Agentic RAG）- Wiki设计

> 来源：Wiki [ID:149858163] 知识库检索优化

### 背景与目标

现有知识库RAG检索比较"朴素"，核心逻辑依赖于切片+向量+TopK模式。优化目标是实现从"朴素RAG"到"Agentic RAG"的架构升级。

### 四大优化点

| # | 优化点 | 说明 |
|:--|:-------|:-----|
| 1 | 知识库元数据Schema | 定义可扩展的元数据字段体系，支持租户个性化配置 |
| 2 | 智能入库流水线 | 解析→自动打标→语义切分→混合索引 |
| 3 | 层级索引与摘要增强 | Parent-Child模式+Summary索引 |
| 4 | Self-Querying检索工具 | 支持Agent自动生成过滤条件 |

### 元数据Schema设计

**两层体系：系统预置 + 租户自定义**

**业务元数据（支持租户自定义）：**

| 字段 | apiName | 类型 | 系统预置枚举值 |
|:---|:---|:---|:---|
| 文档类别 | docCategory | Enum | 产品手册/销售话术/成功案例/内部政策/竞品分析/解决方案/合同模板/技术白皮书/培训材料/FAQ |
| 适用行业 | industryVertical | Enum | 制造业/医疗健康/金融服务/零售消费/能源化工/高科技SaaS/教育培训/政府公共/通用 |
| 业务阶段 | businessStage | Enum | 售前咨询/需求调研/方案演示/商务谈判/签约实施/售后服务 |
| 相关产品/服务 | productService | Enum | （空，由租户配置） |
| 目标受众 | targetAudience | Enum | 高管决策者/业务负责人/IT技术人员/采购法务/一线员工 |

**治理元数据（系统固定）：**

| 字段 | apiName | 类型 | 说明 |
|:---|:---|:---|:---|
| 发布日期 | datePublished | Date | 文档创建日期 |

**结构化索引元数据（系统自动生成）：**

| 字段 | apiName | 存储位置 | 说明 |
|:---|:---|:---|:---|
| 文档ID | docId | 文档级 | 唯一标识 |
| 文档摘要 | docSummary | 文档级 | LLM生成，200-300字 |
| 切片ID | chunkId | 切片级 | 唯一标识 |
| 父文档ID | parentDocId | 切片级 | 指向原始文档 |
| 切片序号 | chunkIndex | 切片级 | 在文档中的顺序 |
| 切片类型 | chunkType | 切片级 | Text/Table/Image_Description/Title/Summary |
| 所属章节 | sectionTitle | 切片级 | 切片所属章节标题 |
| 父文档摘要 | parentSummary | 切片级（冗余） | 用于Rerank |

### 元数据配置层级

**决策：租户级统一配置**

| 层级 | 配置内容 |
|:---|:---|
| 租户级别 | 元数据Schema定义（字段、类型、枚举值） |
| AI知识库级别 | 检索配置（最大召回数量、最小匹配度、Rerank策略） |
| 数据集级别 | 数据来源、同步方式、文档管理（不涉及元数据配置） |

### 文档入库智能流水线

**四阶段：解析→自动打标→语义切分→混合索引**

**阶段一：多模态解析**
- 支持PDF/DOCX/PPTX/XLSX/TXT
- 图片调用Vision LLM生成描述
- 音频调用语音转文字

**阶段二：LLM自动打标**
- 输入：文档全文（或前4000 tokens）+ 租户元数据Schema
- 输出：元数据JSON + 文档摘要
- 后处理：归一化到受控词表（精确匹配→包含匹配→向量相似度匹配）

**阶段三：语义切分**
- 策略一：基于标题树切分（按H1/H2章节边界，超800 tokens按段落二次切分）
- 策略二：滑动窗口切分（400 tokens切分，100 tokens重叠）
- 特殊处理：表格整体一个切片、图片描述单独切片、摘要作为Summary切片

**阶段四：混合索引存储**
- 向量索引：Embedding + Metadata Payload
- 倒排索引：BM25关键词检索
- 文档存储：完整文档内容（用于Parent-Child上下文召回）

### 层级索引

**Parent-Child Indexing：**
- 切片检索命中后，通过parentDocId关联父文档
- 自动扩展前后各N个切片（默认N=1）

**Summary-Based Indexing：**
- 每个文档入库时生成chunkType=Summary的摘要切片
- 宏观问题命中摘要切片，微观问题命中内容切片

### Self-Querying检索工具

search_kb工具Schema扩展，支持filters参数：

```json
{
  "query": "语义搜索关键词",
  "filters": {
    "docCategory": "成功案例",
    "industryVertical": ["制造业"],
    "productService": ["CT设备"],
    "timeRange": "Latest_Year"
  },
  "topK": 5
}
```

检索流程：Pre-filtering(元数据过滤) → Semantic Search(向量检索) → Hybrid Search(BM25融合) → Rerank(Cross-Encoder) → 上下文扩展 → 权限过滤

### 后台配置界面

**文档元数据管理：** NeoAgent > AI知识库管理 > 文档元数据管理
- 字段列表（查看/编辑枚举值）
- 添加自定义字段
- 停用/启用字段
- 刷新全部元数据（二次确认，提示token消耗）

**入库流水线配置：** NeoAgent > AI知识库管理 > 文档入库自动打标配置
- 自动打标开关、打标模型
- 摘要生成开关、摘要长度
- 切分策略（优先策略、兜底策略、参数配置）
- 多模态处理（音频转写、图片描述）

**文档元数据查看与编辑：** 知识库 > 文档列表 > 点击文档 > 元数据标签页
- 业务元数据编辑（从枚举值中选择）
- 治理元数据查看
- 文档摘要编辑

## 四、知识库检索优化（Agentic RAG）- 需求澄清

> 来源：TCHub 文档 [ID:714ebdfc] | 作者：fangty | 2026-04-15

### 功能定位

百事通是销售人员的"全能百科全书"，本次优化聚焦「问知识库」能力，从"朴素RAG"升级为"Agentic RAG"。不涉及其他三个数据源。

### 核心痛点

| # | 痛点 | 根因 |
|:--|:-----|:-----|
| P1 | 语义相关但业务无效的噪音召回 | 无元数据过滤，纯靠向量相似度 |
| P2 | Agent无法表达过滤意图 | 工具仅有queryContent一个参数 |
| P3 | 文档无业务标签 | 缺乏元数据体系和自动打标能力 |
| P4 | 切片上下文丢失 | 切片策略单一，无层级索引 |
| P5 | 宏观问题匹配困难 | 无文档摘要索引 |

### 技术现状关键发现

- 知识库工具inputSchema仅有queryContent一个String参数
- searchByApp接口无filters/metadata参数
- 关键词处理依赖jieba分词+硬编码FILTER_KEY_WORDS列表
- 检索结果无Rerank，直接按score排序
- 文档入库由NEO-AI-DATA-PROCESS-SERVICE负责

### 优化目标

| 指标 | 目标 |
|:---|:---|
| 检索准确率（Top5命中率） | 提升30%+ |
| 噪音召回率 | 降低50%+ |
| 复杂问题回答完整性 | 显著提升 |
| 文档自动打标准确率 | ≥85% |

### 关键约束

- 元数据为租户级统一配置，所有数据集共享同一套Schema
- Schema由系统预置开箱即用+AI全自动补充，管理员仅在异常时可选介入
- Schema自动演化只做"加法"，不自动删除已有值
- 权限模型不变：先召回，后权限过滤
- searchByApp接口扩展filters参数时，不传filters应等同于当前行为

## 五、知识库检索优化（Agentic RAG）- 方案设计

> 来源：TCHub 文档 [ID:cdb36299] | 作者：fangty | 2026-04-15

### 一句话方案

为百事通「问知识库」构建"元数据治理→智能入库→层级索引→Self-Querying检索"四层改造链路。

### 元数据Schema核心原则

**AI全自动维护，人工仅在异常时介入**

Schema全自动生命周期：
1. **开箱即用**：系统预置字段+默认枚举值立即可用
2. **AI补充初始化**：累计≥20篇文档时基于summary自动补充租户特有标签
3. **持续自动演化**：偏差信号达阈值时自动更新（只做加法不做减法）

偏差信号类型：
- 枚举值未命中：打标归一化阶段LLM输出值无法匹配
- 新维度出现：打标LLM建议新增分类维度
- 枚举值分布严重不均：某值命中>60%文档

自动更新安全约束：
- 单次最多新增3个枚举值/1个新字段
- 两次更新间隔≥24小时
- 所有变更写入演化历史日志

### Prompt设计

**Prompt 1：Schema智能生成**
- 用途：累计≥20篇文档时自动补充租户特有标签
- 输入：文档summary列表（≤200篇×~200 tokens）+ 当前Schema
- Token预估：~41000 tokens

**Prompt 2：Schema自动演化**
- 用途：偏差信号达阈值时自动更新Schema
- 输入：当前Schema + 偏差信号 + 最近文档summary
- Token预估：~41500 tokens

**Prompt 3：自动打标**
- 用途：文档入库时自动提取元数据+生成摘要
- 输入：文档内容（前4000 tokens）+ 租户Schema
- Token预估：~4800 tokens/文档

### SDK/API扩展

**知识库工具Schema扩展：**
- 从仅有queryContent扩展为含filters+topK
- filters中的enum值从租户Schema动态注入
- 不同租户看到的工具Schema不同

**searchByApp接口扩展：**
- 新增filters（元数据过滤）、enableRerank（是否启用Rerank）、expandContext（Parent-Child上下文扩展范围）
- 所有新增参数可选，向后兼容

### 数据需求

新增实体：
- 租户元数据Schema配置（tenant_metadata_schema）
- Schema演化日志（schema_evolution_log）
- Schema演化历史（schema_evolution_history）

扩展实体：
- 文档级元数据（扩展现有文档存储）
- 切片级元数据（扩展现有切片存储）

## 六、知识库表关系与数据模型

> 来源：Wiki [ID:144511975] 知识库-表关系

### 核心表结构

**知识库文档操作：**

| 表 | 上传文档 | 上传新版本 | 删除旧版本 | 删除文档 |
|:---|:---|:---|:---|:---|
| a_twitter_file | 新增记录（filemeta_id指向p_filemeta_*） | 更新file_*字段 | 无变更 | del_flag=1 |
| a_twitter_file_version | 新增记录 | 新增记录（file_version变更） | del_flag=1 | 关联记录del_flag=1 |
| p_filemeta_* | 新增记录 | 新增记录 | 无变更 | 无变更 |

**关键字段：**
- filetype：0=文档，1=图片，2=音频，3=视频
- repository_flag：0=不加入知识库，1=加入知识库
- entity_belong_id：70=用户，其他=关联实体ID

## 七、产品知识问答助手（文档中心Agent）

> 来源：TCHub 文档 [ID:9983094b] | 作者：张鹏飞 | 2026-04-09

### 方案概述

系统标准Agent方案：每个租户默认包含"文档中心Agent"，每个POD部署公用向量库。

### Agent配置

| 配置项 | 值 |
|:---|:---|
| Agent名称 | 文档中心Agent |
| apiName | docCenterAgent |
| Agent类型 | 系统标准（SYSTEM_STANDARD） |
| 可见性 | Agent列表中不显示 |
| AI License校验 | 跳过 |
| 资源点扣除 | 不扣除 |

### 知识库问答Action配置

| 配置项 | 值 |
|:---|:---|
| 关联知识库 | 文档中心知识库 |
| 最大召回数量 | 10 |
| 最小匹配度 | 0.7 |

### 文档向量化流程

镭内容变更通知 → 获取文档信息 → HTML解析/去标签/分块(≤500字) → 向量化 → 同步到各POD公用向量库

## 八、Agent架构重构中的知识库Tool设计

> 来源：TCHub 文档 [ID:80813e1b] P0-PRD + [ID:75f4dc1b] 预置Skills定义

### knowledge_search Tool

在Agent架构重构P0中，knowledge_search从百事通Topic中独立为通用Tool：

| Tool | apiKey | 说明 |
|:---|:---|:---|
| 知识库检索 | knowledge_search | 检索AI知识库文档（RAG），简单的输入问题→返回匹配文档 |

### smart-search Skill中的知识库使用

百事通Skill（smart_search）的数据源选择策略中，knowledge_search用于：
- 操作指南/方法论类问题（"BANT方法论怎么用"/"怎么配置审批流"）
- 内部经验/案例类问题的补充检索
- 不确定类型问题的兜底策略
