## 三、字段映射验证

### 3.1 formulaCompute 实际字段映射

| 设计字段 | 实际 db_column | 验证结果 |
|---|---|---|
| itemApiKey | dbc_varchar1 | ✅ 与 seed SQL 一致 |
| computeExpression | dbc_textarea1 | ✅ 与 seed SQL 一致 |
| nullTreatment | dbc_int1 | ✅ 与 seed SQL 一致 |
| resultType | dbc_int2 | ✅ 与 seed SQL 一致 |
| entityApiKey | entity_api_key（固定列） | ✅ |

**⚠️ 发现问题 1：metadata_api_key 为空**

生产数据中 `metadata_api_key` 字段全部为空字符串，`api_key` 也为空。这意味着：
- 无法通过 `getByApiKeyMerged(metamodelApiKey, apiKey)` 查询单条公式
- 无法做 apiKey 唯一性校验
- 无法做 Tenant 层覆盖（Tenant 覆盖依赖 apiKey 匹配）

**根因**：迁移时未生成 apiKey，公式定义没有业务唯一标识。

**影响**：FormulaMetaLoader 只能用 `listMergedByEntityApiKey` 批量加载，不能按 apiKey 单条查询。

### 3.2 formulaComputeItem 实际字段映射

| 设计字段 | 实际 db_column | 验证结果 |
|---|---|---|
| computeApiKey | dbc_varchar1 | ✅ |
| referItemApiKey | dbc_varchar2 | ✅ |
| referEntityApiKey | dbc_varchar3 | ✅ |
| itemOrder | dbc_int1 | ⚠️ 当前数据全为 NULL |

**⚠️ 发现问题 2：itemOrder 全为 NULL**

所有 formulaComputeItem 的 `itemOrder`（dbc_int1）均为 NULL，参数无顺序信息。

**影响**：ParamValueResolver 无法按顺序取参数，但实际上参数取值不依赖顺序（每个参数独立取值），不影响计算正确性。

**⚠️ 发现问题 3：computeApiKey 使用负数 ID**

account 实体的 computeApiKey 为 `-11120000100001` 这类负数 ID，而非 apiKey 字符串（如 `isCustomer`）。

这说明 formulaComputeItem 中的 `computeApiKey` 字段存的是**老系统的 itemId**，而非新系统的 `itemApiKey`。

**影响**：FormulaMetaLoader 在关联 formulaCompute 和 formulaComputeItem 时，需要用 `computeApiKey` 做 JOIN，但：
- formulaCompute 的 `dbc_varchar1`（itemApiKey）存的是 `isCustomer`（apiKey 字符串）
- formulaComputeItem 的 `dbc_varchar1`（computeApiKey）存的是 `-11120000100001`（老系统 ID）

两者无法直接关联，需要通过 `entity_api_key` + 业务逻辑匹配。

### 3.3 aggregationCompute 实际字段映射

| 设计字段 | 实际 db_column | 实际含义（验证后） | 验证结果 |
|---|---|---|---|
| itemApiKey | dbc_varchar1 | 存的是**子实体 apiKey**（如 `order`），不是父实体汇总字段 | ⚠️ 语义与设计不符 |
| aggregateEntityApiKey | dbc_varchar2 | 存的是**父实体汇总字段 apiKey**（如 `actualInvoicedAmount`） | ⚠️ 语义与设计不符 |
| aggregateItemApiKey | dbc_varchar3 | 存的是**子实体关联字段 apiKey**（如 `accountId`） | ⚠️ 语义与设计不符 |
| aggregateLinkItemApiKey | dbc_varchar4 | NULL | — |
| aggregateType | dbc_int1 | 0=COUNT, 1=SUM | ✅ |

**⚠️ 发现问题 4：aggregationCompute 字段语义与设计文档不符**

实际数据示例（account 实体）：
```
dbc_varchar1 = "order"              → 子实体 apiKey
dbc_varchar2 = "actualInvoicedAmount" → 父实体汇总字段 apiKey
dbc_varchar3 = "accountId"          → 子实体关联父实体的字段 apiKey
dbc_int1     = 1                    → SUM
```

设计文档中的字段语义：
```
itemApiKey              → 父实体汇总字段
aggregateEntityApiKey   → 子实体
aggregateItemApiKey     → 子实体被汇总字段
aggregateLinkItemApiKey → 子实体关联父实体的字段
```

实际数据的语义是：**"对 {dbc_varchar1} 实体中 {dbc_varchar3} 字段等于父实体 ID 的记录，对 {dbc_varchar2} 字段做 {dbc_int1} 汇总"**。

## 四、公式表达式格式验证

### 4.1 实际表达式样本

```
# 本实体字段引用（简单参数）
account.actualInvoicedAmount - account.paidAmount

# 条件公式
IF(account.level='A(重点客户)', 100, IF(account.level='B(普通客户)', 80, 0))

# 跨实体关联（复杂参数，多级）
campaignInfluence.opportunityId.opportunity.money / campaignInfluence.opportunityId.opportunity.roiCiC

serviceVisitHistory.relatedServiceEmailDetail.serviceEmailDetail.emailTime
```

### 4.2 表达式格式规律

- 参数格式：`{entityApiKey}.{itemApiKey}` 或 `{entityApiKey}.{referItemApiKey}.{targetEntityApiKey}.{targetItemApiKey}...`
- 与老系统格式**完全一致**，无需修改表达式解析逻辑
- 跨实体参数最深达 4 级（`a.b.c.d.e.f`）

## 五、设计方案修正点

### 修正 1：FormulaMetaLoader 加载策略

由于 `metadata_api_key` 为空，加载公式时只能按 `entity_api_key` 批量加载，不支持按 apiKey 单条查询：

```java
// 修正前（设计文档）
FormulaCompute formula = metaLoader.getByApiKey(tenantId, entityApiKey, itemApiKey);

// 修正后（适配实际数据）
List<FormulaCompute> formulas = metaLoader.listByEntityApiKey(tenantId, entityApiKey);
// 在内存中按 itemApiKey 过滤
```

### 修正 2：formulaCompute 与 formulaComputeItem 关联方式

由于 computeApiKey 存的是老系统 ID（负数），无法直接与 formulaCompute.itemApiKey 关联。

两种修正方案：

**方案 A（推荐）**：数据修复，将 formulaComputeItem.computeApiKey 更新为对应的 itemApiKey 字符串。
```sql
UPDATE p_common_metadata fci
SET dbc_varchar1 = fc.dbc_varchar1  -- 用 formulaCompute 的 itemApiKey 替换负数 ID
FROM p_common_metadata fc
WHERE fci.metamodel_api_key = 'formulaComputeItem'
  AND fc.metamodel_api_key = 'formulaCompute'
  AND fci.entity_api_key = fc.entity_api_key
  AND fci.dbc_varchar1 = fc.metadata_api_key  -- 通过老系统 ID 关联
```

**方案 B（临时兼容）**：FormulaMetaLoader 加载时，通过 `entity_api_key` 将所有 formulaComputeItem 加载，再按表达式解析参数匹配（不依赖 computeApiKey 关联）。

### 修正 3：aggregationCompute 字段语义重新映射

根据实际数据，aggregationCompute 的字段语义重新定义：

| db_column | 实际语义 | Java 字段名（修正） |
|---|---|---|
| dbc_varchar1 | 子实体 apiKey | `childEntityApiKey` |
| dbc_varchar2 | 父实体汇总字段 apiKey | `itemApiKey` |
| dbc_varchar3 | 子实体关联父实体的字段 apiKey | `linkItemApiKey` |
| dbc_varchar4 | 子实体被汇总字段 apiKey（NULL 时用 COUNT） | `aggregateItemApiKey` |
| dbc_int1 | 汇总类型（0=COUNT, 1=SUM） | `aggregateType` |

对应的汇总 SQL 模板修正：
```sql
-- 修正后的汇总 SQL
SELECT {aggType_func}({dbc_varchar4 or '*'})
FROM {shardTable}
WHERE entity_api_key = {dbc_varchar1}   -- 子实体
  AND {dbc_varchar3_dbcol} = {parentDataId}  -- 关联父实体
  AND delete_flg = 0
  AND {filterConditions from aggregationComputeDetail}
```

### 修正 4：nullTreatment 枚举值

需确认 nullTreatment=2 的含义，在 ComputeErrorCodeEnum 中补充说明，或在 FormulaEngine 中做兼容处理：

```java
// 临时兼容：nullTreatment=2 按 null 参与计算结果为 null 处理
private Object resolveNullValue(Integer nullTreatment) {
    if (nullTreatment == null || nullTreatment == 0 || nullTreatment == 2) {
        return null;  // null 参与计算结果为 null
    }
    return 0;  // nullTreatment=1：null 当 0
}
```
