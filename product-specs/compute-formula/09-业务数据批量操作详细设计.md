# 业务数据批量操作详细设计（无强事务版）

## 一、核心约束

- **禁止数据库强事务**（`@Transactional`）
- **ID 在业务代码中提前生成**（雪花算法），不依赖数据库自增
- 一致性保障由应用层 operate_log + 回滚机制实现（对齐元数据批量接口）

---

## 二、与元数据批量接口的对比

元数据批量接口（`MetadataMergeWriteServiceImpl`）本身也没有 `@Transactional`，一致性完全依赖 processId + p_meta_operate_log 的应用层机制。业务数据批量接口完全对齐这套设计。

| 维度 | 元数据批量接口 | 业务数据批量接口 |
|---|---|---|
| 数据库事务 | 无 | 无 |
| 一致性机制 | processId + p_meta_operate_log | processId + p_entity_operate_log |
| ID 生成 | IdGenerator.nextId()（雪花） | IdGenerator.nextId()（雪花，提前生成） |
| 分组 key | metamodelApiKey | entityApiKey |
| 数据标识 | apiKey（String） | id（Long，提前生成） |
| 路由 | 固定表 | TableRouteService 分片路由 |
| 前置计算 | 无 | ComputePipeline 公式计算 |
| 后置异步 | 无 | MQ 触发汇总/子实体公式 |

## 三、ID 提前生成策略

### 3.1 为什么要提前生成

跨实体批量操作中，同一批次可能存在依赖关系：
```
createMap:
  account: [{ name: "客户A" }]           ← 新建客户，id 未知
  opportunity: [{ accountId: ???  }]     ← 新建商机，需要引用上面客户的 id
```

如果依赖数据库自增或 MyBatis-Plus 的 `ASSIGN_ID`（insert 后才有 id），则无法在同一批次内建立引用关系。

**解决方案**：在请求进入 Service 层时，对所有 createMap 中没有 id 的数据，统一用雪花算法预生成 id，之后整个批次内都可以引用这个 id。

### 3.2 ID 生成时机

```
API 层收到请求
  ↓
EntityDataBatchService.batchSave()
  ↓
Step 0: assignIds(createMap)   ← 对所有 createMap 数据预生成 id
  ↓
Step 1: executeDelete
Step 2: executeCreate          ← 使用预生成的 id 写库
Step 3: executeUpdate
```

### 3.3 assignIds 实现

```java
private void assignIds(Map<String, List<Map<String, Object>>> createMap) {
    if (createMap == null) return;
    for (List<Map<String, Object>> dataList : createMap.values()) {
        for (Map<String, Object> data : dataList) {
            if (data.get("id") == null) {
                data.put("id", idGenerator.nextId(null));
            }
        }
    }
}
```

## 四、接口设计

**扩展现有 `EntityDataApiService`，新增 `POST /entity/data/batch-save`**，不新增独立接口。

### 请求结构

```json
{
  "createMap": {
    "account": [
      { "name": "客户A", "ownerId": 123, "dbc_varchar1": "上海" }
    ],
    "opportunity": [
      { "name": "商机A", "dbc_bigint1": 456, "dbc_decimal1": 10000 }
    ]
  },
  "updateMap": {
    "account": [
      { "id": 789, "name": "客户A（更新）", "dbc_varchar1": "北京" }
    ]
  },
  "deleteMap": {
    "account": [
      { "id": 101 }
    ]
  }
}
```

- createMap：id 可不传，Service 层统一预生成
- updateMap：必须包含 id
- deleteMap：只需包含 id

### 响应结构

```json
{
  "code": 200,
  "data": {
    "createCount": 2,
    "updateCount": 1,
    "deleteCount": 1,
    "processId": "1234567890123456789",
    "createdIds": {
      "account": [1234567890123456001],
      "opportunity": [1234567890123456002]
    }
  }
}
```

`createdIds` 返回预生成的 id，调用方可用于后续引用。

## 五、核心组件设计

### 5.1 p_entity_operate_log（新建表）

```sql
CREATE TABLE p_entity_operate_log (
    id              BIGINT PRIMARY KEY,
    process_id      BIGINT       NOT NULL,
    tenant_id       BIGINT       NOT NULL,
    entity_api_key  VARCHAR(128) NOT NULL,
    data_id         BIGINT,                 -- create/update/delete 的数据 id
    dml_type        SMALLINT     NOT NULL,  -- 1=create, 2=update, 3=delete
    before_value    TEXT,                   -- 操作前完整行数据（JSON），create 时为 null
    rollback_status SMALLINT     DEFAULT 0, -- 0=未回滚, 1=已回滚
    created_at      BIGINT       NOT NULL
);
CREATE INDEX idx_entity_operate_log_process  ON p_entity_operate_log(process_id);
CREATE INDEX idx_entity_operate_log_pending  ON p_entity_operate_log(created_at)
    WHERE rollback_status = 0;
```

### 5.2 EntityBatchSaveContext（ThreadLocal）

```java
public final class EntityBatchSaveContext {
    private static final ThreadLocal<Long> PROCESS_ID = new ThreadLocal<>();

    public static void setProcessId(Long processId) { PROCESS_ID.set(processId); }
    public static Long getProcessId()               { return PROCESS_ID.get(); }
    public static boolean isInBatchSave()           { return PROCESS_ID.get() != null; }
    public static void clear()                      { PROCESS_ID.remove(); }
}
```

### 5.3 IEntityOperateLogService

### 5.4 EntityDataBatchService（核心）

```java
@Slf4j
@Service
@RequiredArgsConstructor
public class EntityDataBatchService {

    private final IdGenerator              idGenerator;
    private final TableRouteService        routeService;
    private final TenantDataServiceImpl    dataDao;
    private final IEntityOperateLogService operateLogService;
    private final ComputePipeline          computePipeline;
    private final FormulaMetaLoader        metaLoader;
    // AsyncComputeProducer 后续补充

    public EntityBatchSaveResponse batchSave(
            long tenantId, Long operatorId,
            Map<String, List<Map<String, Object>>> createMap,
            Map<String, List<Map<String, Object>>> updateMap,
            Map<String, List<Map<String, Object>>> deleteMap) {

        Long processId = operateLogService.newProcessId();
        log.info("[EntityBatch] 开始 processId={}, tenant={}", processId, tenantId);

        // Step 0: 对 createMap 中所有数据预生成 id
        Map<String, List<Long>> createdIds = assignIds(createMap);

        try {
            int deleteCount = executeDelete(tenantId, operatorId, processId, deleteMap);
            int createCount = executeCreate(tenantId, operatorId, processId, createMap);
            int updateCount = executeUpdate(tenantId, operatorId, processId, updateMap);

            operateLogService.commitByProcessId(processId);
            log.info("[EntityBatch] 成功 processId={}, create={}, update={}, delete={}",
                    processId, createCount, updateCount, deleteCount);

            return EntityBatchSaveResponse.of(createCount, updateCount, deleteCount,
                    processId, createdIds);

        } catch (Exception e) {
            log.error("[EntityBatch] 失败，开始回滚 processId={}", processId, e);
            try {
                operateLogService.rollbackByProcessId(processId);
            } catch (Exception rollbackEx) {
                log.error("[EntityBatch] 回滚失败 processId={}", processId, rollbackEx);
            }
            throw e;
        } finally {
            EntityBatchSaveContext.clear();
        }
    }
}
```

## 六、详细执行流程

### 6.1 executeCreate

```
for each entityApiKey in createMap:
  dataList = createMap[entityApiKey]

  1. 加载字段元数据
     itemDefs = metaLoader.loadItems(tenantId, entityApiKey)

  2. 前置公式计算（批量）
     beforeResult = computePipeline.computeForCreate(tenantId, entityApiKey, dataList)
     → 汇总字段初始化为 0
     → 分离特殊公式
     → 拓扑排序 + 批量求值
     → 将公式结果合并到每条 data

  3. for each data in dataList:
     a. 写 operate_log（dmlType=CREATE, beforeValue=null, dataId=data["id"]）
     b. 路由到分片表，INSERT（使用预生成的 id）
        routeService.executeWithRoute(tenantId, entityApiKey, () -> {
            fixedData.setId(data["id"])   ← 使用预生成的 id
            dataDao.save(fixedData)
            if extColumns: dataDao.update(extColumns, WHERE id=data["id"])
        })

  4. 特殊公式计算（入库后，无事务）
     if specialFormulas not empty:
       try:
         computePipeline.computeSpecialFormulas(...)
         writeBackSpecialResults(...)
       catch: log error, 不抛出

  5. 发 MQ（无事务，失败写 compute_error_log）
     try:
       asyncComputeProducer.sendAggregateFilter(tenantId, entityApiKey, createdIds)
     catch: log + computeErrorLogService.save(...)
```

### 6.2 executeUpdate

```
for each entityApiKey in updateMap:
  dataList = updateMap[entityApiKey]
  ids = dataList.map(d -> d["id"])

  1. 批量查旧数据（用于 beforeValue 快照 + 变更检测）
     oldDataMap = batchGetByIds(tenantId, entityApiKey, ids)

  2. 前置公式计算（批量增量更新）
     changedApiKeysList = dataList.map(d -> d.keySet() - {"id"})
     resultMap = computePipeline.computeForIncrementUpdateBatch(
         tenantId, entityApiKey, dataList, oldDataMap, changedApiKeysList)
     → 相同变更字段集合的数据合并为一次计算
     → 将公式结果合并到每条 data

  3. for each data in dataList:
     a. 写 operate_log（dmlType=UPDATE, beforeValue=oldDataMap[data["id"]]）
     b. 路由到分片表，UPDATE
        routeService.executeWithRoute(tenantId, entityApiKey, () -> {
            dataDao.update(data, WHERE id=data["id"])
        })

  4. 特殊公式计算（同 create）

  5. 发 MQ
     try:
       asyncComputeProducer.sendItemFilter(...)
       asyncComputeProducer.sendAggregateParentFilter(...)
     catch: log + computeErrorLogService.save(...)
```

### 6.3 executeDelete

```
for each entityApiKey in deleteMap:
  ids = deleteMap[entityApiKey].map(d -> d["id"])

  1. 批量查旧数据（用于 beforeValue 快照）
     oldDataMap = batchGetByIds(tenantId, entityApiKey, ids)

  2. for each id in ids:
     a. 写 operate_log（dmlType=DELETE, beforeValue=oldDataMap[id]）
     b. 路由到分片表，软删除（delete_flg=1）
        routeService.executeWithRoute(tenantId, entityApiKey, () -> {
            dataDao.softDelete(id, tenantId)
        })

  3. 发 MQ
     try:
       asyncComputeProducer.sendAggregateFilter(tenantId, entityApiKey, ids)
     catch: log + computeErrorLogService.save(...)
```

### 6.4 rollbackByProcessId

```
1. 查 p_entity_operate_log WHERE process_id=? AND rollback_status=0 ORDER BY created_at DESC
2. EntityBatchSaveContext.clear()（防止回滚操作再次写 operate_log）
3. for each log in DESC order:
   - dmlType=CREATE:
       routeService.executeWithRoute(tenantId, entityApiKey, () ->
           dataDao.softDelete(log.dataId, tenantId))
   - dmlType=UPDATE:
       beforeValue = JSON.parse(log.beforeValue)
       routeService.executeWithRoute(tenantId, entityApiKey, () ->
           dataDao.update(beforeValue, WHERE id=log.dataId))
   - dmlType=DELETE:
       beforeValue = JSON.parse(log.beforeValue)
       routeService.executeWithRoute(tenantId, entityApiKey, () ->
           dataDao.save(beforeValue))   ← 用 beforeValue 重新插入（含原 id）
   - 更新 rollback_status=1
4. 回滚期间跳过公式计算和 MQ 发送
```

## 七、跨实体引用场景

### 场景：同批次新建客户 + 商机，商机引用客户 id

```json
{
  "createMap": {
    "account": [{ "name": "客户A" }],
    "opportunity": [{ "name": "商机A", "dbc_bigint1": null }]
  }
}
```

**调用方处理方式**：
1. 调用方先调一次 `GET /entity/data/next-ids?count=2` 获取预生成的 id
2. 在请求体中直接填入 id 和引用关系：
```json
{
  "createMap": {
    "account": [{ "id": 1001, "name": "客户A" }],
    "opportunity": [{ "id": 1002, "name": "商机A", "dbc_bigint1": 1001 }]
  }
}
```

**或者**：Service 层 assignIds 后，在 API 层将 createdIds 返回给调用方，调用方在下一次请求中使用。

### next-ids 接口

```
GET /entity/data/next-ids?count=5
Response: { "code": 200, "data": { "ids": [1001, 1002, 1003, 1004, 1005] } }
```

实现：
```java
@GetMapping("/next-ids")
public Map<String, Object> nextIds(@RequestParam(defaultValue = "1") int count) {
    List<Long> ids = new ArrayList<>();
    for (int i = 0; i < Math.min(count, 100); i++) {
        ids.add((Long) idGenerator.nextId(null));
    }
    return Map.of("code", 200, "data", Map.of("ids", ids));
}
```

## 八、启动恢复

```java
@PostConstruct
public void recoverPendingTransactions() {
    long threshold = System.currentTimeMillis() - pendingTimeoutMs;
    List<Long> pendingProcessIds = operateLogService.findPendingProcessIds(threshold);
    for (Long processId : pendingProcessIds) {
        try {
            operateLogService.rollbackByProcessId(processId);
        } catch (Exception e) {
            log.error("[EntityPendingRecovery] 回滚失败 processId={}", processId, e);
        }
    }
}
```

## 九、待确认事项

| 问题 | 建议 |
|---|---|
| 批量上限 | 单次请求每个实体最多 500 条，超出返回 400 |
| 公式计算失败策略 | 阻断整批（与元数据接口一致），返回错误信息 |
| 特殊公式失败 | 不阻断，写 p_compute_error_log，由补偿任务重试 |
| MQ 发送失败 | 不阻断，写 p_compute_error_log，由补偿任务重发 |
| next-ids 接口 | 是否需要独立接口，还是由调用方自行生成雪花 ID |
