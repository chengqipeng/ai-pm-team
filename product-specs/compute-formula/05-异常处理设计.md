# 计算公式引擎设计方案 · 05 异常处理设计

## 一、新系统异常规范

### 异常体系

```
BaseException（抽象基类）
  ├── BaseKnownException    业务异常（预期内，HTTP 400）
  │     用于：参数校验失败、业务规则违反、数据不存在等
  └── BaseUnKnownException  系统异常（预期外，HTTP 500）
        用于：数据库超时、缓存不可用、第三方服务异常等
```

**关键字段：**
- `errorCode`（int）：8 位数字，前 3 位项目编码 + 后 5 位业务编码
- `errorMsg`（String）：错误描述，支持 `{0}` 占位符
- `data`（Object）：附加错误上下文，供前端回显（通过 `.withData()` 链式设置）
- `tenantId / userId / traceId`：自动从 GlobalContext 注入

### 错误码规范

```
编码规则：8 位数字
  前 3 位 = 项目编码
  后 5 位 = 业务编码

已分配项目编码：
  100 = metarepo（元数据仓库）
  102 = entity（实体数据）      ← 计算公式引擎使用此编码
  500 = 通用系统错误
  900 = gateway（网关）
```

### 抛出方式

```java
// 1. 直接抛出（推荐）
throw new BaseKnownException(ComputeErrorCodeEnum.FORMULA_ITEM_NOT_FOUND, itemApiKey);

// 2. 带附加数据
throw new BaseKnownException(ComputeErrorCodeEnum.FORMULA_EXPRESSION_ERROR, itemApiKey)
    .withData(Map.of("expression", expr, "params", paramValues));

// 3. 包装底层异常（系统异常）
throw new BaseUnKnownException(ComputeErrorCodeEnum.PARAM_RESOLVE_DB_ERROR, null, cause);

// 4. 工具类校验（参数校验场景）
BaseCheckUtils.throwBaseKnownExceptionIfEmpty(entityApiKey, ComputeErrorCodeEnum.PARAM_INVALID, "entityApiKey");
```

## 三、各场景异常处理策略

### 3.1 公式元数据加载

| 场景 | 异常类型 | 错误码 | 处理方式 |
|---|---|---|---|
| entityApiKey 为空 | BaseKnownException | ENTITY_API_KEY_EMPTY | 直接抛出，阻断写入 |
| 公式字段在 EntityItem 中不存在 | BaseKnownException | FORMULA_ITEM_NOT_FOUND | 直接抛出，阻断写入 |
| 公式引用的实体不存在 | BaseKnownException | FORMULA_ENTITY_NOT_FOUND | 直接抛出，阻断写入 |
| 公式引用的字段不存在 | BaseKnownException | FORMULA_REF_ITEM_NOT_FOUND | 直接抛出，阻断写入 |
| 参数字段不是关联字段（跨实体参数） | BaseKnownException | FORMULA_PARAM_NOT_REFER | 直接抛出，阻断写入 |
| 缓存读取失败 | BaseUnKnownException | CACHE_ERROR | 降级：跳过缓存直接查库，记录 warn 日志 |

```java
// 示例：公式字段校验
EntityItem formulaItem = metaLoader.getItem(tenantId, entityApiKey, formula.getItemApiKey());
if (formulaItem == null) {
    throw new BaseKnownException(ComputeErrorCodeEnum.FORMULA_ITEM_NOT_FOUND,
        entityApiKey + "." + formula.getItemApiKey());
}
```

### 3.2 参数取值（ParamValueResolver）

| 场景 | 异常类型 | 错误码 | 处理方式 |
|---|---|---|---|
| 跨实体查库失败（DB 异常） | BaseUnKnownException | PARAM_RESOLVE_DB_ERROR | 包装原始异常抛出，阻断写入 |
| 批量查询结果数量与输入不一致 | BaseKnownException | FORMULA_BATCH_SIZE_MISMATCH | 降级为逐条查询；若仍不一致则抛出 |
| 关联 ID 为 null（正常业务场景） | 不抛异常 | — | 参数值设为 null，按 nullTreatment 处理 |

```java
// 示例：批量查询降级
try {
    results = batchQueryByIds(tenantId, referEntityApiKey, referIds);
    if (results.size() != expectedSize) {
        log.warn("batch size mismatch, fallback to single query. entity={}, expected={}, actual={}",
            referEntityApiKey, expectedSize, results.size());
        results = singleQueryFallback(tenantId, referEntityApiKey, referIds);
    }
} catch (Exception e) {
    throw new BaseUnKnownException(ComputeErrorCodeEnum.PARAM_RESOLVE_DB_ERROR,
        referEntityApiKey, e);
}
```

### 3.3 公式求值（ExpressionEvaluator）

| 场景 | 异常类型 | 错误码 | 处理方式 |
|---|---|---|---|
| 表达式语法错误（配置问题） | BaseKnownException | FORMULA_EXPRESSION_ERROR | 附加 expression + paramValues 到 data，抛出，阻断写入 |
| 类型不匹配（如字符串与数字比较） | BaseKnownException | FORMULA_EXPRESSION_ERROR | 同上，记录 warn 日志 |
| 求值结果为 null（正常场景） | 不抛异常 | — | 按 nullTreatment 处理 |

```java
// 示例：求值异常处理
try {
    result = ComputeUtils.evaluate(formula.getComputeExpression(), paramValues, context);
} catch (Exception e) {
    log.warn("formula evaluate error, entity={}, item={}, expression={}, params={}",
        entityApiKey, formula.getItemApiKey(), formula.getComputeExpression(), paramValues);
    throw new BaseKnownException(ComputeErrorCodeEnum.FORMULA_EXPRESSION_ERROR,
        formula.getItemApiKey())
        .withData(Map.of(
            "expression", formula.getComputeExpression(),
            "params", paramValues,
            "error", e.getMessage()
        ));
}
```

### 3.4 汇总计算（AggregationEngine，异步消费者）

异步消费者中的异常**不能向上抛出**（会导致 MQ 消息无限重试），需本地处理：

| 场景 | 处理方式 |
|---|---|
| 汇总规则不存在 | 记录 warn 日志，跳过该条消息，ACK |
| 聚合 SQL 执行失败 | 记录 error 日志 + 错误详情，重试 3 次后 ACK（避免死信堆积） |
| 写回父实体失败 | 记录 error 日志，重试 3 次后 ACK，发告警 |
| 消息幂等重复 | 检查 transactionId，已处理则直接 ACK |

```java
// 示例：异步消费者异常处理
@MQConsumer
public void onAggregateFilter(AggregateParentFilterMessage msg) {
    // 幂等检查
    if (idempotentChecker.isProcessed(msg.getTransactionId())) {
        return;
    }
    try {
        aggregationEngine.aggregate(msg.getTenantId(),
            msg.getParentEntityApiKey(),
            msg.getParentDataIds(),
            msg.getAggregateApiKey());
        idempotentChecker.markProcessed(msg.getTransactionId());
    } catch (BaseKnownException e) {
        // 业务异常：规则不存在等，不重试
        log.warn("aggregate skip, code={}, msg={}", e.getErrorCode(), e.getErrorMsg());
    } catch (Exception e) {
        // 系统异常：重试
        log.error("aggregate error, entity={}, parentIds={}", 
            msg.getParentEntityApiKey(), msg.getParentDataIds(), e);
        throw e;  // 抛出触发 MQ 重试
    }
}
```

### 3.5 MQ 发送失败

MQ 发送失败**不阻断主流程**，记录到 ComputeErrorLog 表，由补偿任务重发：

```java
// 示例：MQ 发送失败处理
try {
    asyncComputeProducer.sendItemFilter(tenantId, entityApiKey, dataId, changedItems);
} catch (Exception e) {
    log.error("mq send failed, entity={}, dataId={}", entityApiKey, dataId, e);
    // 写入错误日志表，由补偿任务重发
    computeErrorLogService.save(ComputeErrorLog.builder()
        .tenantId(tenantId)
        .entityApiKey(entityApiKey)
        .dataId(dataId)
        .errorType(ComputeErrorType.MQ_SEND_FAIL)
        .payload(JSON.toJSONString(changedItems))
        .build());
    // 不抛出，不阻断主流程
}
```

### 3.6 特殊公式计算失败（入库后）

特殊公式在数据入库后计算，失败时**不回滚主事务**，记录错误日志：

```java
// 示例：特殊公式异常处理
try {
    computePipeline.computeSpecialFormulas(tenantId, entityApiKey,
        savedDataList, specialFormulas, allItems);
} catch (BaseKnownException e) {
    // 业务异常（如公式配置错误），记录日志，不影响已入库数据
    log.warn("special formula compute failed, entity={}, code={}, msg={}",
        entityApiKey, e.getErrorCode(), e.getErrorMsg());
    computeErrorLogService.save(buildErrorLog(tenantId, entityApiKey, e));
} catch (Exception e) {
    // 系统异常，同上处理
    log.error("special formula compute error, entity={}", entityApiKey, e);
    computeErrorLogService.save(buildErrorLog(tenantId, entityApiKey, e));
}
```

## 四、异常传播边界

```
┌─────────────────────────────────────────────────────────┐
│  EntityDataService.create / update（事务边界）            │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  ComputePipeline（同步前置）                      │    │
│  │  BaseKnownException  → 向上抛出，事务回滚         │    │
│  │  BaseUnKnownException → 向上抛出，事务回滚        │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  持久化（写库）                                           │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  computeSpecialFormulas（入库后，事务外）          │    │
│  │  任何异常 → 捕获，记录 ComputeErrorLog，不抛出    │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  asyncComputeProducer.send（MQ 发送）             │    │
│  │  任何异常 → 捕获，记录 ComputeErrorLog，不抛出    │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  AsyncComputeConsumer（MQ 消费，独立事务）                │
│  BaseKnownException → 捕获，ACK，记录 warn               │
│  其他异常 → 抛出，触发 MQ 重试（最多 3 次后 ACK）         │
└─────────────────────────────────────────────────────────┘
```

## 五、ComputeErrorLog 表设计

```sql
CREATE TABLE p_compute_error_log (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    tenant_id       BIGINT NOT NULL,
    entity_api_key  VARCHAR(128) NOT NULL,
    data_id         BIGINT,
    error_type      TINYINT NOT NULL COMMENT '1=MQ发送失败 2=特殊公式失败 3=汇总计算失败',
    error_code      INT,
    error_msg       VARCHAR(512),
    payload         TEXT COMMENT '触发计算的原始数据（JSON）',
    retry_count     INT DEFAULT 0,
    resolved_flg    TINYINT DEFAULT 0 COMMENT '0=未处理 1=已处理',
    created_at      BIGINT NOT NULL,
    updated_at      BIGINT NOT NULL
);
```

补偿任务每 5 分钟扫描 `resolved_flg=0 AND retry_count < 3` 的记录，重新触发计算。
