# 计算公式引擎设计方案 · 01 现状分析

## 老系统（paas-customize-service）

### 架构分层

```
XObjectDataOperate（写操作流水线）
  ├── [前置同步] XComputeDataBeforeServiceImpl   保存前计算公式字段
  ├── [持久化]   commonDataOperate               写库
  ├── [前置同步] computeAndUpdateSpecialItemValue 特殊公式（自增/本币）写库
  └── [后置异步] XComputeDataAfterServiceImpl     发 MQ，触发子实体公式 + 父实体汇总
```

实时求值引擎 XComputeDataServiceImpl 被前置计算和外部接口共同调用。

### 元数据模型

| 对象 | 关键字段 |
|---|---|
| XObject | apiKey、dbTable |
| XItem | itemType、computeType、dbColumn、decimal、realTimeCompute |
| XComputeFormula | formula（表达式字符串）、nullValue、itemId |
| AggregateRule | argEntityId、argLinkItemId、aggregateType |

公式表达式格式：`{obj.item}` 或跨实体 `{obj.referItem.targetObj.targetItem}`

### 计算触发时机

| 场景 | 方式 | 范围 |
|---|---|---|
| 新建 | 同步前置 | 本实体所有公式字段；汇总字段初始化为 0 |
| 全量更新 | 同步前置 | 对比新旧数据，只算受变更字段影响的公式 |
| 增量更新（单条） | 同步前置 | 按 changedItemIdList 精确触发 |
| 增量更新（批量） | 同步前置 | 相同变更字段集合的数据合并批量计算 |
| 删除 | 异步 MQ | 父实体汇总重算 |
| 父实体汇总 | 异步 MQ | 子实体数据变更后触发父实体汇总字段重算 |
| 子实体公式 | 异步 MQ | 父实体字段变更后触发子实体公式重算 |
| 实时计算字段 | 查询时 | realTimeCompute=true，每次查询重算 |

### 参数取值逻辑

- **简单参数**（`apiKeys.length == 2`，本实体字段）：直接从 internalData 取，不查库
- **复杂参数**（`apiKeys.length > 2`，跨实体）：构建 LEFT JOIN SQL 查库；批量时合并为 IN 查询

### 特殊公式（后置处理）

含自增字段（AUTONUMBER）或本币字段（currencyPart=2）的公式，在数据入库后单独计算，支持最多 19 层嵌套引用。

### 公式依赖排序

拓扑排序：公式 A 依赖公式 B 的结果字段 → B 先算。队列轮询，无依赖先出队，有依赖放回队尾，最多循环 `size * 10` 次防死循环。

### 核心问题

| 问题 | 描述 |
|---|---|
| 表结构耦合 | 参数解析依赖 dbColumn/dbTable，元数据与存储强绑定 |
| 缓存粒度粗 | Redis 以 tenantId 为 key，任何字段变更清空整个租户缓存 |
| 批量不彻底 | 复选框、文件字段仍逐条查库 |
| 归档实体降级 | 归档实体强制逐条查询 |
| 汇总短暂不一致 | 新建时汇总字段置 0，异步补算，存在数据不一致窗口 |
| 实时字段无缓存 | 高频查询场景每次重算，性能差 |

## 新系统（paas-platform-service）

### 元数据模型

**FormulaCompute（公式定义）**
```
entityApiKey, itemApiKey, computeExpression, nullTreatment, resultType
```

**FormulaComputeItem（公式引用字段，显式记录每个参数）**
```
entityApiKey, computeApiKey, referItemApiKey, referEntityApiKey, itemOrder
```

**AggregationCompute（汇总定义）**
```
entityApiKey, itemApiKey, aggregateEntityApiKey, aggregateItemApiKey,
aggregateLinkItemApiKey, aggregateType, filterFormula
```

**AggregationComputeDetail（汇总筛选条件，结构化）**
```
entityApiKey, aggregateApiKey, filterItemApiKey, filterOperator, filterValue, itemOrder
```

**ComputeFactor（计算因子）**
```
entityApiKey, computeApiKey, factorType, factorValue, factorOrder
```

### 数据存储架构

```
p_tenant_data_route   → 路由表：(tenantId, entityApiKey) → tableIndex
p_tenant_data_stats   → 统计表：tableIndex → rowCount（负载均衡）
p_tenant_data_{0~1999} → 2000 张分片表
```

分片路由：Redis 缓存（TTL=3600s）→ p_tenant_data_route → 负载最低的 10 张中随机分配。

TenantData 固定字段：id、entity_api_key、name、owner_id、depart_id、busitype_api_key、approval_status、lock_status、currency_unit、currency_rate 等。扩展字段（`dbc_` 前缀）动态写入同一分片表。
