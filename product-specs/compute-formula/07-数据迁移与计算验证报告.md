3. 将计算结果写回 p_tenant_data 对应的 dbc_ 列
4. 重新查询验证写回值与计算值一致

## 六、发现的问题

### 问题 1：level 字段不在 item 元数据中

`accountScore` 公式依赖 `account.level`，但 `level` 字段在 `p_common_metadata`（metamodel_api_key='item'）中**没有记录**。

**根因**：`level` 是老系统标准实体的固定字段（SELECT 类型），在新系统元数据迁移时未包含在 item 元模型中。

**影响**：
- FormulaMetaLoader 无法通过元数据找到 `level` 的 dbColumn
- ParamValueResolver 无法确定从哪个 dbc_ 列取值
- 验证时通过硬编码 `dbc_varchar2` 绕过，但实际实现需要元数据支持

**修复方案**：在 `p_common_metadata` 中补充 `level` 字段的 item 元数据记录，指定 dbColumn。

### 问题 2：isCustomer 写回覆盖了 level 的测试数据

`isCustomer` 的 dbColumn 是 `dbc_varchar2`，迁移脚本中 `level` 也临时写入了 `dbc_varchar2`，导致 isCustomer 写回后覆盖了 level 值。

**根因**：level 字段没有独立的 dbColumn，测试时复用了 dbc_varchar2。

**修复方案**：同问题 1，为 level 分配独立的 dbColumn（如 `dbc_varchar3`）。

### 问题 3：公式表达式中 `=` 需转换为 `==`

老系统公式表达式使用 `=` 做字符串比较（如 `account.level='A(重点客户)'`），Python eval 需要 `==`。

**影响**：新系统 ExpressionEvaluator 在调用 ComputeUtils.evaluate 前需要做表达式预处理，或者 ComputeUtils 本身已支持（老系统 Java 实现中已处理）。

**结论**：这是 Python 模拟脚本的问题，Java 实现中 ComputeUtils 已正确处理，不影响设计方案。
