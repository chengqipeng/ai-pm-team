## 四、NUMBERSTRING 遗漏分析

### 老系统实现

```java
// paas-compute-core/functions/text/NumberStringFunction.java
// 将数值转换为大写金额字符串（如 123.45 → "壹佰贰拾叁元肆角伍分"）
```

### 新系统状态

FunctionRegistry.java 中**未注册** `numberstring`。TextFunctions.java 中**无** NumberStringFunction 实现。

### 影响评估

- 如果有公式表达式使用了 `NUMBERSTRING()` 函数，新系统求值时会返回 null（函数未找到）
- 需要检查生产数据中是否有公式使用此函数

### 修复建议

在 TextFunctions.java 中补充 NumberStringFunction 实现，并在 FunctionRegistry 中注册。

## 五、架构差异

| 维度 | 老系统 | 新系统 |
|:---|:---|:---|
| 函数接口 | `Function { evaluate(), validate() }` | 同接口，完全兼容 |
| 注册方式 | `FunctionRepository` 静态 Map + `FunctionDescription` 树 | `FunctionRegistry` 静态 Map（更简洁） |
| 函数组织 | 每个函数一个独立 .java 文件（70 个文件） | 按分类合并到 4 个文件（LogicFunctions/MathFunctions/TextFunctions/DateTimeFunctions） |
| 求值引擎 | ANTLR4 `EvaluateVisitor` → `FunctionRepository.get()` | 同 ANTLR4 引擎，`FunctionRegistry.get()` |
| 时区处理 | `FunctionCommonUtil.getDateTimeZone(evalContext)` | `DateTimeUtil.getDateTimeZone(ctx)` |
| 特殊函数 | `FunctionCommonUtil.SPECIAL_FORMULA` | `FunctionRegistry.SPECIAL_FORMULA` |
| 前端函数列表 | `FUNCTION_LIST`（树形结构，含 id/label/value/desc） | 未实现（新系统暂不需要前端函数选择器） |

### 新系统改进

1. **代码组织**：70 个独立文件 → 4 个分类文件，减少文件数量，同类函数集中维护
2. **类名修正**：`RightFuncation` → `RightFunction`（修正拼写错误）
3. **OPTIONAPINAME**：老系统通过 `OptionApiNameFunctionUtils` 工具类处理，新系统直接在函数内通过反射访问 PickOption
4. **FILECOUNT**：老系统依赖 `FileMetaService` 查文件数量，新系统通过 `FileFieldStrategy` 在参数取值阶段处理
