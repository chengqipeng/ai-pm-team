# 计算公式引擎详细设计 · 新老系统对比

---

## 一、整体抽象层次对比

### 老系统抽象层次

```
XComputeDataService（对外接口）
  └── XComputeDataInnerService（对内接口，扩展 parseFormulaParam / getParamParseInfo）
        └── XComputeDataServiceImpl（2177行，所有逻辑混在一个类）

XComputeDataBeforeService（前置计算接口）
  └── XComputeDataBeforeServiceImpl（1653行，新建/更新/汇总全在一个类）

XComputeDataAfterService（后置计算接口）
  └── XComputeDataAfterServiceImpl（MQ 发送）

ParamResultHandler<T>（参数结果处理接口）
  ├── FormulaParamResultHandler（返回 FormulaParamResult）
  └── MapParamResultHandler（返回 Map<String,Object>）
```

**问题：**
- XComputeDataServiceImpl 承担了「元数据解析 + 参数取值 + SQL 构建 + 求值 + 类型转换」5 个职责
- XComputeDataBeforeServiceImpl 承担了「变更检测 + 依赖分析 + 公式调度 + 汇总调度 + 货币处理」5 个职责
- 两个类合计 3800 行，无法单独测试任何一个子职责
- ParamResultHandler 是唯一的抽象扩展点，但只用于区分返回类型，不是真正的策略抽象

### 新系统抽象层次（设计目标）

```
ComputePipeline（流水线入口，协调各引擎）
  ├── FormulaMetaLoader（元数据加载，单一职责）
  ├── DependencyAnalyzer（拓扑排序，单一职责）
  ├── ParamValueResolver（参数取值，单一职责）
  │     ├── LocalParamResolver（本实体字段，直接从 dataMap 取）
  │     └── RemoteParamResolver（跨实体字段，查分片表）
  ├── ExpressionEvaluator（表达式求值，单一职责）
  └── ResultTypeConverter（结果类型转换，单一职责）

AggregationEngine（汇总计算，独立引擎）
AsyncComputeProducer（MQ 发送，独立组件）
```

## 二、核心类详细设计

### 2.1 FormulaMetaLoader

**职责**：加载并缓存公式元数据，屏蔽 Common/Tenant 两层合并细节。

**老系统对应**：散落在 XComputeDataServiceImpl 的 `prepareComputeFormula`、`getItemByApiKey`、`getObjectByApiKey` 等方法中，与求值逻辑混在一起。

```java
@Component
public class FormulaMetaLoader {

    // 缓存 Key: formula:meta:{tenantId}:{entityApiKey}
    // TTL: 30 分钟，FormulaCompute 变更时主动删除

    /**
     * 加载实体的所有公式定义
     */
    public List<FormulaComputeDef> loadFormulas(long tenantId, String entityApiKey);

    /**
     * 加载实体的所有字段定义（apiKey → EntityItemDef）
     */
    public Map<String, EntityItemDef> loadItems(long tenantId, String entityApiKey);

    /**
     * 加载汇总规则
     */
    public List<AggregationComputeDef> loadAggregations(long tenantId, String entityApiKey);

    /**
     * 主动失效缓存（元数据变更时调用）
     */
    public void evict(long tenantId, String entityApiKey);
}

// DTO：公式定义
public record FormulaComputeDef(
    String itemApiKey,          // 目标字段
    String expression,          // 表达式
    int nullTreatment,          // null 处理策略
    int resultType,             // 结果类型
    List<FormulaParamDef> params // 参数列表（来自 formulaComputeItem）
) {}

// DTO：公式参数
public record FormulaParamDef(
    String referItemApiKey,     // 引用字段
    String referEntityApiKey,   // 引用实体（null=本实体）
    int itemOrder               // 顺序
) {}

// DTO：字段定义
public record EntityItemDef(
    String apiKey,
    String dbColumn,
    int itemType,
    int resultType,             // 公式字段的结果类型
    boolean realTimeCompute
) {}
```

**新老差异**：
| 维度 | 老系统 | 新系统 |
|---|---|---|
| 元数据来源 | Dubbo RPC 调用 XMetaRepoService | 本地查 p_common_metadata + p_tenant_metadata |
| 参数定义 | 从表达式字符串解析（`ComputeUtils.getAllParams`） | FormulaComputeItem 显式记录，无需解析 |
| 缓存粒度 | Redis hset，key=tenantId，任何字段变更清空整个租户 | key=tenantId:entityApiKey，精确失效 |
| 缓存内容 | FormulaParamParseInfo（含 dbTable/dbColumn） | FormulaComputeDef（只含 apiKey，不含物理列） |

### 2.2 DependencyAnalyzer

**职责**：对公式列表做拓扑排序，确保被依赖的公式先计算。

**老系统对应**：`XComputeDataBeforeServiceImpl.analysisFormulaRelation`，逻辑正确但与调度逻辑耦合。

```java
@Component
public class DependencyAnalyzer {

    /**
     * 拓扑排序
     * @param formulas    当前实体的公式列表
     * @param itemDefs    字段定义（判断参数是否是公式字段）
     * @return 排序后的公式列表（被依赖的排在前面）
     * @throws BaseKnownException FORMULA_CIRCULAR_DEPENDENCY 循环依赖时抛出
     */
    public List<FormulaComputeDef> sort(
        List<FormulaComputeDef> formulas,
        Map<String, EntityItemDef> itemDefs);

    /**
     * 从变更字段集合中找出受影响的公式（递归）
     * @param changedApiKeys  变更的字段 apiKey 集合
     * @param formulas        实体所有公式
     * @param entityApiKey    实体 apiKey（用于构造参数表达式前缀）
     * @return 受影响的公式集合（已拓扑排序）
     */
    public List<FormulaComputeDef> findAffected(
        Set<String> changedApiKeys,
        List<FormulaComputeDef> formulas,
        String entityApiKey,
        Map<String, EntityItemDef> itemDefs);
}
```

**新老差异**：
| 维度 | 老系统 | 新系统 |
|---|---|---|
| 循环依赖处理 | 最多循环 `size*10` 次，残留直接追加（静默处理） | 检测到循环依赖抛 `FORMULA_CIRCULAR_DEPENDENCY` |
| 变更触发判断 | `checkFormulaChange`：字符串前缀匹配（`param.startsWith(changedExpr+".")`） | 同逻辑，但基于 FormulaParamDef 显式参数，不依赖字符串解析 |
| 实时字段排除 | `xItem.isRealTimeCompute()` 时跳过依赖分析 | 同逻辑，通过 `EntityItemDef.realTimeCompute` 判断 |

### 2.3 ParamValueResolver

**职责**：批量取公式参数的实际值，屏蔽本实体/跨实体/特殊类型的差异。

**老系统对应**：`XComputeDataServiceImpl` 中的 `getParamValue`、`getParamValueBatch`、`buildQuerySQL`、`buildQuerySQLBatch` 等方法，共约 600 行。

```java
public interface ParamValueResolver {

    /**
     * 批量取参数值
     * @param dataList  数据列表（含完整字段值）
     * @param params    需要取值的参数列表
     * @param tenantId
     * @return dataId → (paramKey → value)
     */
    Map<Long, Map<String, Object>> resolve(
        long tenantId,
        List<Map<String, Object>> dataList,
        List<FormulaParamDef> params,
        Map<String, EntityItemDef> itemDefs);
}

// 本实体参数（直接从 dataMap 取）
@Component
class LocalParamResolver {
    // 对应老系统：apiKeys.length == 2 && !needQueryDB 分支
    // 特殊处理：CHECK → checkValueMap；FILE/IMAGE → 文件数量
    Map<Long, Object> resolve(List<Map<String, Object>> dataList, String itemApiKey, EntityItemDef itemDef);
}

// 跨实体参数（查分片表）
@Component
class RemoteParamResolver {
    // 对应老系统：buildQuerySQL + getParamValue 分支
    // 新系统：按 referEntityApiKey 路由到分片表，批量 IN 查询
    // 降级：批量结果数量不一致时，抛 FORMULA_BATCH_SIZE_MISMATCH
    Map<Long, Object> resolve(long tenantId, List<Long> referIds,
                              String referEntityApiKey, String referItemApiKey,
                              EntityItemDef targetItemDef);
}
```

**新老差异**：
| 维度 | 老系统 | 新系统 |
|---|---|---|
| 寻址方式 | dbColumn + dbTable → SQL JOIN | apiKey → 分片路由 → SELECT dbc_xxx |
| SQL 构建 | `buildQuerySQL` 动态拼接多级 LEFT JOIN | 单表 SELECT，关联通过多次查询实现 |
| 批量降级 | `IllegalBusinessDataException` → 降级逐条 | 抛 `FORMULA_BATCH_SIZE_MISMATCH`，不静默降级 |
| 归档实体 | `isArchive()` 检测，强制逐条 | 无归档概念，统一批量 |
| 复选框处理 | 查 `p_custom_pickvalue`，返回 `Integer[]` | 查 `dbc_multi` 列（jsonb），直接取 |
| 主子明细 | `$_master4Validate` 传主实体数据，避免查库 | `MasterContext` 传入，同逻辑 |

### 2.4 ExpressionEvaluator

**职责**：执行公式表达式求值，屏蔽表达式语法差异。

**老系统对应**：`ComputeUtils.evaluate` 调用（外部 jar），加上 `computeExpressValue` 中的前后处理逻辑。

```java
@Component
public class ExpressionEvaluator {

    /**
     * 单条求值
     * @param expression    公式表达式
     * @param paramValues   参数值 Map（key = "entityApiKey.itemApiKey"）
     * @param context       上下文（时区、选项列表等）
     * @return 计算结果（原始类型，未做类型转换）
     * @throws BaseKnownException FORMULA_EXPRESSION_ERROR 表达式执行异常
     */
    public Object evaluate(String expression, Map<String, Object> paramValues, EvalContext context);

    /**
     * 批量求值（同一公式，多条数据）
     */
    public List<Object> evaluateBatch(
        String expression,
        List<Map<String, Object>> paramValuesList,
        EvalContext context);
}

// 求值上下文
public record EvalContext(
    TimeZone timeZone,
    List<PickOption> pickOptions,   // 计算单选字段需要
    int nullTreatment               // null 处理策略
) {}
```

**新老差异**：
| 维度 | 老系统 | 新系统 |
|---|---|---|
| 求值引擎 | `ComputeUtils.evaluate`（外部 jar，基于 MVEL/自研） | 复用同一引擎（迁移时保持兼容） |
| 已计算值替换 | `mapComputedParam` 覆盖 `mapParamValue`（公式 A 依赖公式 B 的结果） | 同逻辑，由 `ComputePipeline` 在拓扑排序后按序传入 |
| 单选字段结果 | `OptionApiNameFunctionUtils.convertApiKeyToOptionCode` 后处理 | 同逻辑，在 `ResultTypeConverter` 中处理 |
| 错误处理 | 记录 warn 日志，抛 `PaasException` | 抛 `BaseKnownException(FORMULA_EXPRESSION_ERROR)`，附带 expression + params |

### 2.5 ResultTypeConverter

**职责**：将求值结果转换为目标字段类型，并处理小数位数。

**老系统对应**：`handleDecimal` + `DataTypeControlUtil.computeObject2XObjectData`，散落在 `computeExpressValue` 中。

```java
@Component
public class ResultTypeConverter {

    /**
     * 转换求值结果为目标类型
     * @param raw        原始求值结果
     * @param resultType 目标类型（对应 itemType 枚举）
     * @param decimal    小数位数（-1=不处理）
     * @param context    时区等上下文
     */
    public Object convert(Object raw, int resultType, int decimal, EvalContext context);
}
```

**新老差异**：
| 维度 | 老系统 | 新系统 |
|---|---|---|
| 小数处理 | `ROUND_HALF_UP`，`handleDecimal` 方法 | 同逻辑，独立为 `ResultTypeConverter` |
| 类型映射 | `DataTypeControlUtil.computeObject2XObjectData`（外部 jar） | 复用同一工具类 |
| BOOL 类型 | `"1"` → `true`，`"0"` → `false` | 同逻辑 |
| 百分比类型 | 同 REAL，存储时 ÷100 | 同逻辑 |

### 2.6 ComputePipeline（流水线协调器）

**职责**：协调上述各组件，实现新建/更新/查询三种场景的计算流水线。

**老系统对应**：`XComputeDataBeforeServiceImpl` 中的 `compute4Create`、`compute4FullUpdate`、`compute4IncrementUpdate` 等方法。

```java
@Component
public class ComputePipeline {

    private final FormulaMetaLoader metaLoader;
    private final DependencyAnalyzer dependencyAnalyzer;
    private final ParamValueResolver paramValueResolver;
    private final ExpressionEvaluator expressionEvaluator;
    private final ResultTypeConverter resultTypeConverter;

    /**
     * 新建前置计算
     * 返回 specialFormulas（含自增/本币字段的公式，需入库后单独计算）
     */
    public ComputeBeforeResult computeForCreate(
        long tenantId, String entityApiKey,
        List<Map<String, Object>> dataList);

    /**
     * 增量更新前置计算（单条）
     */
    public ComputeBeforeResult computeForIncrementUpdate(
        long tenantId, String entityApiKey,
        Map<String, Object> newData, Map<String, Object> oldData,
        Set<String> changedApiKeys);

    /**
     * 增量更新前置计算（批量）
     * 相同变更字段集合的数据合并为一次计算
     */
    public Map<Long, ComputeBeforeResult> computeForIncrementUpdateBatch(
        long tenantId, String entityApiKey,
        List<Map<String, Object>> newDataList,
        Map<Long, Map<String, Object>> oldDataMap,
        List<Set<String>> changedApiKeysList);

    /**
     * 特殊公式计算（入库后调用）
     */
    public void computeSpecialFormulas(
        long tenantId, String entityApiKey,
        List<Map<String, Object>> savedDataList,
        List<FormulaComputeDef> specialFormulas);

    /**
     * 查询时实时计算（不写库）
     */
    public void computeForQuery(
        long tenantId, String entityApiKey,
        List<Map<String, Object>> dataList);
}

// 前置计算结果
public record ComputeBeforeResult(
    Map<String, Object> formulaResults,          // apiKey → 计算值
    List<FormulaComputeDef> specialFormulas,     // 需入库后计算的特殊公式
    List<AggregateCondition> aggregateConditions, // 需触发的父实体汇总条件
    List<String> changedItemApiKeys              // 变更字段（用于发 MQ）
) {}
```

**新老差异**：
| 维度 | 老系统 | 新系统 |
|---|---|---|
| 入口方法 | `compute4Create`、`compute4FullUpdate`、`compute4IncrementUpdate`、`compute4IncrementUpdateBatch` | 同样 4 个场景，但内部通过组件组合实现，不重复代码 |
| 批量分组 | 相同变更字段的数据合并（`getChangeItemKey`） | 同逻辑，但提取为独立方法 |
| 货币处理 | `handleCurrencyChanged` 内嵌在 `getChangedItemList` 中 | 提取为独立的 `CurrencyChangeDetector` |
| 特殊公式 | `analysisSpecialFormulas` 检测自增/本币字段 | 同逻辑，通过 `EntityItemDef.itemType` 判断 |
| 依赖分析 | `analysisFormulaRelation` 内嵌在 Before 类中 | 独立为 `DependencyAnalyzer` |

### 2.7 AggregationEngine

**职责**：执行汇总计算，将子实体数据聚合写回父实体。

**老系统对应**：由 `paas-compute-service`（独立服务）消费 MQ 消息后执行，老系统 customize-service 只负责发消息。

```java
@Component
public class AggregationEngine {

    /**
     * 执行汇总计算并写回父实体
     * @param tenantId
     * @param parentEntityApiKey  父实体
     * @param parentDataIds       需要重算的父实体数据 ID 列表
     * @param aggregationDef      汇总规则定义
     */
    public void aggregate(
        long tenantId,
        String parentEntityApiKey,
        List<Long> parentDataIds,
        AggregationComputeDef aggregationDef);
}
```

汇总 SQL 模板（基于验证报告中修正后的字段语义）：
```sql
-- aggregationDef 字段语义（修正后）：
-- childEntityApiKey  = dbc_varchar1（子实体）
-- itemApiKey         = dbc_varchar2（父实体汇总字段）
-- linkItemApiKey     = dbc_varchar3（子实体关联父实体的字段）
-- aggregateItemApiKey= dbc_varchar4（子实体被汇总字段，null 时用 COUNT(*)）
-- aggregateType      = dbc_int1（0=COUNT, 1=SUM）

SELECT
  {linkItemDbColumn} AS parent_id,
  {aggFunc}({aggregateItemDbColumn}) AS agg_value
FROM {childShardTable}
WHERE entity_api_key = {childEntityApiKey}
  AND {linkItemDbColumn} IN ({parentDataIds})
  AND delete_flg = 0
  AND {filterConditions}   -- 来自 AggregationComputeDetail
GROUP BY {linkItemDbColumn}
```

**新老差异**：
| 维度 | 老系统 | 新系统 |
|---|---|---|
| 执行位置 | 独立的 paas-compute-service 消费 MQ | 本服务内 AsyncComputeConsumer 消费 MQ |
| 筛选条件 | `p_aggregate_criteria`（无结构化，build_sql 字段存 SQL 片段） | `AggregationComputeDetail`（结构化：filterItemApiKey + filterOperator + filterValue） |
| 汇总类型 | type 字段（28=SUM 等，枚举值不直观） | aggregateType（0=COUNT, 1=SUM，语义清晰） |
| 写回方式 | 通过 Dubbo 调用 customize-service 写库 | 直接通过 TableRouteService 路由写分片表 |

## 三、关键抽象：新系统新增的能力

### 3.1 ParamValueResolver 策略模式

老系统 `getParamValue` 是一个 500 行的方法，通过 if-else 处理所有字段类型。新系统拆分为策略：

```java
// 策略接口
public interface FieldValueStrategy {
    boolean supports(EntityItemDef itemDef);
    Object resolve(long tenantId, Map<String, Object> data, EntityItemDef itemDef);
}

// 各策略实现
@Component class SimpleFieldStrategy    implements FieldValueStrategy { /* 普通字段，直接取 */ }
@Component class CheckboxFieldStrategy  implements FieldValueStrategy { /* 复选框，取 dbc_multi */ }
@Component class FileFieldStrategy      implements FieldValueStrategy { /* 文件，返回数量 */ }
@Component class ReferFieldStrategy     implements FieldValueStrategy { /* 关联字段，查分片表 */ }
@Component class MetaModelFieldStrategy implements FieldValueStrategy { /* 元模型字段，特殊处理 */ }
```

### 3.2 FormulaComputeDef 显式参数

老系统通过 `ComputeUtils.getAllParams(expression)` 从表达式字符串中解析参数，每次计算都要解析一遍。

新系统 `FormulaComputeDef.params` 直接包含参数列表（来自 `formulaComputeItem`），无需解析：

```java
// 老系统：每次计算都解析
Set<String> params = ComputeUtils.getAllParams(formulaInfo.getFormula());

// 新系统：直接使用
List<FormulaParamDef> params = formulaDef.params();
```

### 3.3 ComputeBeforeResult 统一返回

老系统前置计算的结果通过多个 out 参数传出（`computeConditionList`、`itemFilterConditionList`、`specialFormulas` 都是传入的空 List，方法内填充），调用方需要管理多个集合。

新系统统一为 `ComputeBeforeResult` 值对象，调用方只需处理一个返回值。

## 四、新老系统完整对比表

| 维度 | 老系统 | 新系统 |
|---|---|---|
| **元数据加载** | Dubbo RPC，每次调用有网络开销 | 本地查库 + Redis 缓存，无 RPC |
| **参数定义** | 表达式字符串解析（运行时） | FormulaComputeItem 显式记录（配置时） |
| **参数寻址** | dbColumn + SQL JOIN（依赖物理表结构） | apiKey → 分片路由（逻辑寻址） |
| **批量计算** | 部分场景逐条降级（复选框、归档实体） | 全路径批量，无静默降级 |
| **依赖分析** | 内嵌在 Before 类，无法单独测试 | 独立 DependencyAnalyzer，可单元测试 |
| **类型转换** | 散落在 computeExpressValue 中 | 独立 ResultTypeConverter |
| **错误处理** | 混合 BusinessException/PaasException | 统一 BaseKnownException/BaseUnKnownException |
| **缓存粒度** | tenantId 级（任何变更清空整个租户） | entityApiKey 级（精确失效） |
| **汇总筛选** | build_sql 字段存 SQL 片段（不可移植） | AggregationComputeDetail 结构化条件 |
| **特殊公式** | 最多 19 层嵌套（loopCount=20） | 同逻辑，上限可配置 |
| **实时字段** | 查询时重算，无缓存 | 查询时重算，可选短 TTL 缓存 |
| **可测试性** | 单个类 2000+ 行，难以单元测试 | 每个组件单一职责，可独立测试 |
| **代码量** | Before+ServiceImpl 合计 ~3800 行 | 预计各组件 100~300 行，总量相近但职责清晰 |

## 五、迁移注意事项（来自验证报告）

1. **level 等标准实体字段缺少 item 元数据**：需补充 dbColumn 映射，否则 ParamValueResolver 无法取值
2. **formulaComputeItem.computeApiKey 为负数 ID**：需修复为 itemApiKey 字符串，否则 FormulaMetaLoader 无法关联公式和参数
3. **aggregationCompute 字段语义与 Java Entity 不符**：需按验证报告修正 Entity 字段名
4. **nullTreatment=2 的语义**：需确认后在 ExpressionEvaluator 中做兼容处理
