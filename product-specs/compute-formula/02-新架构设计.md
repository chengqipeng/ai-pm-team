# 计算公式引擎设计方案 · 02 新架构设计

## 设计原则

1. **apiKey 寻址**：不依赖 dbColumn/dbTable，统一用 apiKey + 分片路由取数据
2. **元数据驱动**：公式定义、参数引用、汇总规则全部从 FormulaCompute/FormulaComputeItem 读取
3. **分层计算**：前置同步保证写入一致性，后置异步处理汇总和跨实体级联
4. **批量优先**：所有路径优先批量，单条是批量的特例
5. **缓存分级**：公式定义长 TTL + 参数解析中 TTL + 实时字段结果短 TTL

---

## 整体分层

```
EntityDataService（写操作入口）
  create / update / softDelete
       │
  ComputePipeline（同步前置）
       │
  ┌────┴──────────────────┐
  │                       │
FormulaEngine         DefaultValueEngine
（公式计算）           （默认值计算）
  │
  ├── FormulaMetaLoader      公式元数据加载（带缓存）
  ├── DependencyAnalyzer     依赖拓扑排序
  ├── ParamValueResolver     参数取值（apiKey → 实际值）
  └── ExpressionEvaluator    表达式求值

       │ 异步 MQ
  AsyncComputeConsumer（后置异步）
  ├── 父实体汇总重算（AggregationEngine）
  ├── 子实体公式重算（FormulaEngine）
  └── 跨实体级联重算
```

## 与老系统核心差异

| 维度 | 老系统 | 新系统 |
|---|---|---|
| 参数寻址 | dbColumn + SQL JOIN | apiKey → 分片路由查询 |
| 公式参数定义 | 表达式字符串解析 | FormulaComputeItem 显式记录 |
| 汇总筛选条件 | 无结构化条件 | AggregationComputeDetail 结构化 |
| 数据存储 | 每实体独立表 | 2000 张分片表 |
| 缓存粒度 | tenantId 级 | entityApiKey + computeApiKey 级 |
| 批量计算 | 部分场景逐条降级 | 全路径批量 |

## ComputePipeline 流水线设计

### 新建流水线

```
1. loadFormulaItems(entityApiKey)
   → 从缓存/DB 加载 FormulaCompute 列表 + FormulaComputeItem 列表

2. initAggregateFields(dataList)
   → 汇总字段初始化为 0（日期/时间类型汇总字段初始化为 null）

3. separateSpecialFormulas(formulaList)
   → 提取含自增字段/本币字段的特殊公式，主流程跳过

4. topologicalSort(formulaList)
   → 拓扑排序，处理公式间依赖

5. batchResolveParams(dataList, formulaList)
   → 批量取所有公式的参数值

6. batchEvaluate(dataList, formulaList, paramValues)
   → 批量执行公式求值

7. writeFormulaResults(dataList, results)
   → 将计算结果写入 dataList（后续统一持久化）

8. [入库后] computeSpecialFormulas(dataList, specialFormulas)
   → 计算特殊公式并单独更新
```

### 增量更新流水线

```
1. diffChangedItems(newData, oldData, changedApiKeys)
   → 对比新旧数据，确定实际变更的字段集合
   → 处理货币字段：汇率变更 → 所有本币字段加入变更集合

2. findAffectedFormulas(changedItems, formulaList)
   → 递归查找受影响的公式（变更字段是某公式的参数 → 该公式字段也变更 → 再触发其他公式）

3. topologicalSort(affectedFormulas)

4. batchResolveParams + batchEvaluate（同新建）

5. buildAggregateConditions(changedItems, aggregateRules)
   → 构建父实体汇总条件（新旧父实体 ID 都要重算）

6. buildItemFilterConditions(changedItems)
   → 构建字段变更消息，用于后置异步触发子实体公式
```

### 全量更新流水线

与增量更新相同，区别在于 diffChangedItems 接收完整的新旧数据对比，而非外部传入的 changedApiKeys。

## FormulaMetaLoader 缓存设计

```
缓存 Key：formula:meta:{tenantId}:{entityApiKey}
缓存内容：
  {
    formulas: [{ itemApiKey, computeExpression, nullTreatment, resultType }],
    params:   [{ computeApiKey, referItemApiKey, referEntityApiKey, itemOrder }]
  }
TTL：30 分钟
失效时机：FormulaCompute 或 FormulaComputeItem 发生变更时主动删除
```

## DependencyAnalyzer 拓扑排序

```java
// 输入：formulaList（当前实体所有公式）
// 输出：有序的 formulaList（被依赖的公式排在前面）

Queue<FormulaCompute> queue = new LinkedList<>(formulaList);
Map<String, FormulaCompute> sorted = new LinkedHashMap<>();
int maxLoop = formulaList.size() * 10;

while (!queue.isEmpty() && maxLoop-- > 0) {
    FormulaCompute formula = queue.poll();
    boolean hasUnresolvedDep = formula.params.stream()
        .filter(p -> p.referEntityApiKey == null)  // 只看本实体参数
        .anyMatch(p -> isFormulaItem(p.referItemApiKey) && !sorted.containsKey(p.referItemApiKey));
    if (hasUnresolvedDep) {
        queue.add(formula);  // 放回队尾
    } else {
        sorted.put(formula.itemApiKey, formula);
    }
}
// 残留（循环依赖）直接追加
queue.forEach(f -> sorted.put(f.itemApiKey, f));
```

## ParamValueResolver 参数取值

### 本实体参数（referEntityApiKey == null）

```
直接从 dataMap.get(referItemApiKey) 取值
特殊处理：
  - 复选框（CHECK）：从 checkValueMap 取，转为逗号分隔字符串
  - 文件/图片（FILE/IMAGE）：返回文件数量（仅支持 isnull 函数）
  - 已计算的公式字段：从 computedResultMap 取（优先级高于 dataMap）
```

### 跨实体参数（referEntityApiKey != null）

```
1. 从 dataMap 取关联字段值（referItemApiKey 对应的 ID）
2. 按 referEntityApiKey 路由到对应分片表
3. 批量查询：SELECT {targetDbColumn} FROM {shardTable}
             WHERE id IN ({referIds}) AND entity_api_key = {referEntityApiKey}
4. 结果按 referDataId 映射回各条数据
```

### 批量优化

```
相同 referEntityApiKey + referItemApiKey 的数据合并为一次查询：
  Map<String, List<Long>> groupByEntity = params.groupBy(referEntityApiKey)
  for each entity:
      SELECT id, {targetColumn} FROM {shardTable}
      WHERE id IN ({allReferIds}) AND entity_api_key = ?
```

### null 值处理（nullTreatment）

| nullTreatment | 含义 | 处理 |
|---|---|---|
| 0（NUMBER_NULL） | null 参与计算结果为 null | 参数为 null 时直接返回 null |
| 1（NUMBER_ZERO） | null 当 0 参与计算 | 参数为 null 时替换为 0 |

## ExpressionEvaluator 求值

复用老系统 `ComputeUtils.evaluate(expression, paramValues, context)`，context 中注入时区信息。

结果后处理：
```
1. handleDecimal(result, decimal)
   → BigDecimal/Double 按 decimal 位数 ROUND_HALF_UP

2. DataTypeControlUtil.computeObject2XObjectData(result, resultType, tz)
   → 按 resultType 转换为目标类型（Long/Double/String/LocalDate 等）

3. OptionApiNameFunctionUtils.convertApiKeyToOptionCode(resultType, pickOptions, result)
   → 计算单选字段：结果为 apiKey，转换为 optionCode 存储
```

## 异步计算（AsyncComputeConsumer）

### MQ 消息类型

| Topic | 触发时机 | 消费逻辑 |
|---|---|---|
| FORMULA_FILTER | 父实体字段变更 | 查找引用了该字段的子实体公式，重新计算 |
| AGGREGATE_ITEM_FILTER | 子实体字段变更 | 触发父实体汇总字段重算 |
| AGGREGATE_ITEM_PARENT_FILTER | 关联字段变更（旧父实体） | 旧父实体汇总字段重算 |
| AGGREGATE_OBJECT_FILTER | 新建/删除 | 父实体汇总字段重算 |

### 汇总计算逻辑（AggregationEngine）

```
1. 加载 AggregationCompute 定义（entityApiKey + itemApiKey）
2. 加载 AggregationComputeDetail 筛选条件
3. 构建聚合 SQL：
   SELECT {aggregateType}({aggregateItemDbColumn})
   FROM {shardTable}
   WHERE entity_api_key = {aggregateEntityApiKey}
     AND {aggregateLinkItemDbColumn} = {parentDataId}
     AND {filterConditions}   ← 来自 AggregationComputeDetail
     AND delete_flg = 0
4. 将结果写回父实体分片表对应字段
```

### 汇总类型

| aggregateType | SQL 函数 | 说明 |
|---|---|---|
| SUM | SUM() | 求和 |
| COUNT | COUNT() | 计数（含 null） |
| COUNT_NOT_NULL | COUNT(col) | 计数（不含 null） |
| MAX | MAX() | 最大值 |
| MIN | MIN() | 最小值 |
| AVG | AVG() | 平均值 |
