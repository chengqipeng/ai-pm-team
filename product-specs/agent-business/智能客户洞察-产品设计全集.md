## 一、目标与价值

> 来源：Wiki 143118204

### 1.1 业务场景

销售在日常工作中需要掌握客户的企业背景、市场动态、财务状况、竞争格局等信息，但这些信息分散在 CRM 系统、工商数据库、互联网搜索、企业知识库等不同体系中，手动收集耗时且容易遗漏。

### 1.2 产品目标

整合内外部数据源，自动生成**结构化洞察报告 + 可执行行动建议**，形成完整的"信息 → 分析 → 结论 → 行动"闭环。

### 1.3 核心亮点

- **个性化定制**：管理员可按客户业务类型分配不同洞察模板，每个模板包含不同维度和数据来源组合
- **闭环行动**：洞察结果直接关联行动建议（创建任务 / 复制文案 / 批量创建）
- **一键转任务**：行动建议可一键转为 CRM 任务，分配给相关人员
- **多数据源融合**：工商数据、网络搜索、财报数据、知识库、CRM 相关记录
- **多种触发方式**：手动刷新、业务事件触发（晚间批量）、审批流触发、自动流触发、定时调度触发

## 二、用户故事

> 来源：Wiki 143118204

### 2.1 新客开拓

- **首次接触前 10 分钟快速准备**：销售领取新客户后，系统自动生成客户洞察，涵盖企业背景、业务模式、市场环境、关键动态等，帮助销售在首次拜访前快速建立认知
- **方案推进阶段**：随着商机推进，洞察内容持续更新，补充竞争格局、财务稳定性等深度信息，辅助方案制定

### 2.2 老客维护

- **续约节点评审**：续约前自动刷新洞察，评估客户经营状况和合作健康度，为续约谈判提供数据支撑
- **增购窗口识别**：通过洞察中的战略动态和需求信号，识别客户的增购潜力
- **流失预警与挽回**：洞察差异检测发现客户重大变更（如关键人变动、经营异常），及时预警并制定挽回策略

## 六、触发机制设计

> 来源：Wiki 143118204

### 6.2 业务事件触发

- 晚间批量执行
- 同一客户一天仅触发一次（去重）

### 6.3 其他触发方式

- **审批流触发**：审批通过后自动刷新洞察
- **自动流触发**：自动化流程中作为节点触发
- **定时调度触发**：按 cron 表达式定期执行

## 七、运行时逻辑与数据存储

> 来源：Wiki 143118204

### 7.1 执行流程

1. **匹配模板**：根据客户业务类型匹配对应的洞察模板
2. **分维度 Prompt 调用**：每个维度单独调用大模型，传入该维度的数据来源和关键点
3. **总结 Prompt 调用**：汇总所有维度结果，生成整体结论和行动建议
4. **存储结果**：将洞察结果持久化

### 7.2 存储规则

- 每个客户最多保留 **10 条**洞察记录
- 超出时删除最早的记录

### 7.3 工商数据校验

- 未找到工商数据时跳过该数据源，不阻断整体流程
- 记录提示信息，告知用户该维度缺少工商数据

### 7.4 首次加载

用户首次打开客户详情页的洞察组件时，如无历史洞察结果，自动触发一次洞察生成。

## 八、洞察差异检测与变更提醒

> 来源：Wiki 143118204

### 8.1 数据模型扩展

客户实体增加以下字段：
- **洞察结果重大变更内容**：记录最近一次差异检测的变更摘要
- **重大变更时间**：记录变更发生的时间戳

### 8.2 差异检测逻辑

使用差异 Prompt 对比前后两次洞察结果，输出：
- `has_diff`：是否存在重大差异（boolean）
- `diff_text`：差异描述文本

### 8.3 变更提醒

管理员可配置触发规则，当检测到重大变更时：
- 更新客户实体的变更字段
- 触发配置的通知/提醒规则

## 九、Prompt 模板设计

> 来源：Wiki 143118204

### 9.1 分维度洞察 Prompt

每个维度单独调用大模型，Prompt 结构：
- **角色定义**：你是一个专业的客户分析师
- **技能描述**：擅长从多源数据中提取关键信息
- **输入数据**：该维度的数据来源内容
- **限制条件**：输出格式要求、字数限制
- **输出 JSON Schema**：结构化输出定义

### 9.2 总结和行动建议 Prompt

汇总所有维度结果，生成：
- 一句话整体结论
- 分条行动建议（含优先级和具体行动描述）

### 9.3 差异检测 Prompt

对比前后两次洞察结果，输出 JSON：
```json
{
  "has_diff": true,
  "diff_text": "客户近期发生管理层变动，CTO 已更换..."
}
```

## 十、预置模板内容

> 来源：Wiki 143118204（7 个维度详细定义）

### 10.1 维度一：业务模式 & 市场环境

- **数据来源**：工商数据、网络搜索
- **关键点**：主营业务、商业模式、目标市场、行业地位、市场规模与增长趋势

### 10.2 维度二：战略与关键动态

- **数据来源**：网络搜索、知识库
- **关键点**：近期战略方向、重大事件（融资/并购/上市）、管理层变动、新产品/新业务线发布

### 10.3 维度三：财务与经营稳定性

- **数据来源**：工商数据、财报数据、网络搜索
- **关键点**：营收规模与增长率、利润率、资产负债率、现金流状况、经营风险信号

### 10.4 维度四：竞争格局 & 差异化

- **数据来源**：网络搜索、知识库
- **关键点**：主要竞争对手、市场份额对比、差异化优势、竞争态势变化

### 10.5 维度五：客户关系健康度

- **数据来源**：查询相关记录（CRM）
- **关键点**：合作历史、合同续约情况、客户满意度、关键联系人关系、投诉/服务工单趋势

### 10.6 维度六：互动效能

- **数据来源**：查询相关记录（CRM）
- **关键点**：拜访频次与质量、沟通响应速度、活动记录趋势、关键决策人触达率

### 10.7 维度七：客户风险与需求信号

- **数据来源**：网络搜索、工商数据、查询相关记录
- **关键点**：经营异常信号、法律诉讼风险、采购需求信号、预算周期与采购计划、技术选型动向

## 十一、模板定制化能力提升：支持插入字段变量

> 来源：Wiki 145979199（v4，2025-12-05）

### 11.1 O1：模板配置支持变量定义与插入

- 在模板的 Prompt 配置中支持插入字段变量，语法：`${字段API名称}`
- 变量来源：客户实体字段、商机实体字段、关联实体字段
- 配置界面提供变量选择器，管理员可从字段列表中选择并插入

### 11.2 O2：运行时变量解析与替换

- 洞察执行时，系统自动将 Prompt 中的 `${字段API名称}` 替换为该客户/商机的实际字段值
- 字段值为空时，替换为空字符串或配置的默认值
- 支持多级关联字段（如 `${account.industry}`）

### 11.3 O3：变量校验优化

- 模板保存时校验变量引用的字段是否存在
- 字段被删除或 API 名称变更时，提示管理员更新模板
- 运行时遇到无法解析的变量，记录警告日志但不阻断执行

- 管理员可配置搜索引擎优先级和适用范围

### 12.2 工商数据接入

- **邓白氏临时方案**：通过邓白氏 API 获取海外企业工商信息
- 作为过渡方案，后续可替换为其他数据源

### 12.3 客户洞察模板分配

**方案一：用户选择**
- 用户在前台手动选择使用哪个洞察模板

**方案二：客户级模板分配**
- 管理员在客户实体上配置默认洞察模板
- 按客户所属区域/类型自动匹配

### 12.4 语言调整

- **搜索关键词语种一致**：海外客户使用英文关键词搜索，国内客户使用中文
- **输出语言跟随用户**：洞察结果的输出语言跟随当前用户的系统语言设置

- 取消工商验证限制的租户控制：部分海外租户无法获取工商数据，允许管理员关闭工商数据校验，直接使用网络搜索等其他数据源

## 十四、多视角连带影响补充方案

> 来源：TCHub 文档 1431fc5c（刘超，2026-04-13）

### 14.1 洞察差异检测

多视角下，差异检测按**同一客户 × 同一模板**的前后两次结果对比。

**设计逻辑**：
1. **对比基准**：同一客户 × 同一模板的前后两次洞察结果对比。不同模板之间不做对比（维度和方向不同，跨模板对比无业务意义）
2. **客户实体差异字段**：保持现有单值字段，不改存储格式。每次差异检测产生结果时直接覆盖
3. **批量触发的覆盖**：业务事件批量触发时，多个模板可能先后产生差异，客户实体字段按执行顺序只保留最后一个
4. **兼容性**：未启用按使用者分配时，逻辑完全不变

### 14.2 商机健康度评估引用客户洞察

多视角下，商机健康度评估通过**触发者（商机 Owner）匹配**对应的客户洞察模板。

**设计逻辑**：
1. **取数策略：按触发者匹配**。商机是项目制的，触发健康度评估的人就是在跟这个项目的人，用他匹配到的客户洞察来评估商机
2. **触发者确定规则**：
   - 手动触发 / 首次打开自动触发 / 阶段变更触发 → 当前操作用户
   - 业务事件触发（晚间批量）→ 商机负责人（opportunity.ownerId）
   - 无法区分时统一按商机 Owner 处理
3. **取数流程**：触发者 → 按使用者规则匹配洞察模板 → 取该客户下该模板的最新洞察结果 → 按健康度模板配置取整体结论或具体维度
4. **无洞察结果时**：为触发者匹配的洞察模板自动生成客户洞察
5. **维度不匹配时**：该数据源视为空，健康度评估继续执行（降级逻辑）
6. **兼容性**：未启用按使用者分配时，行为完全不变

> 客户健康度评估也采用相同逻辑，按执行用户处理。

## 十五、在 AI Native CRM 首页中的定位

> 来源：TCHub P0-PRD 和 P2-PRD（方腾宇，2026-04-22）

### 15.1 作为 Skill 的定位（P0）

在 Agent 架构重构中，客户洞察从"独立组件"升级为 **Skill**：

| 属性 | 值 |
|------|-----|
| Skill 名称 | 客户洞察 |
| apiKey | `customer_insight` |
| 来源 | 迁移自现有组件 → 升级 |
| 核心能力 | 整合多源数据生成客户洞察报告（后台配置驱动） |
| 依赖 Tools | web_search, query_crm |
| 期次 | 第一期迁移升级 |

**升级为 Skill 后的增强**：
- 具备 Agentic 能力：洞察不充分时可换方式重来
- 有独立的后台模板配置页面（NeoAgent → 销售智能 → 智能客户洞察）
- 通过内部接口读取后台模板配置，Skill 在执行时获取配置数据，再结合领域知识完成执行

### 15.2 P2 预置自主执行任务配置

客户洞察作为 P2 的三个预置自主执行任务之一：

| 配置项 | 值 |
|--------|-----|
| 绑定 Skill | 客户洞察（customer_insight） |
| 触发方式 | 事件触发 |
| 触发事件 | 客户.负责人变更时（新客户分配） |
| 执行范围 | 触发事件的客户 |
| 单次上限 | 50 条 |
| 允许操作 CRM | 是 |
| 操作审核策略 | 更新字段（洞察结论、洞察更新时间）→ 自动执行；创建任务 → 需确认 |
| 建议生成规则 | 新客户首次洞察时提醒了解客户；洞察发现重大变更时提醒关注 |

### 15.3 在首页场景中的体现

**场景：新领取客户 → 自动洞察并制定跟进策略**

1. **触发**：CRM 业务事件 — 销售领取新客户
2. **AI 后台执行**：客户洞察 Skill 自动触发，按管理员配置的模板多维度分析客户，洞察结果写入 CRM 客户实体字段
3. **分流**：洞察完成 → Summary Panel；新客户首次分析 → 行动建议区
4. **用户看到**：行动建议区显示中优先级建议，如"客户洞察显示该客户为精密制造行业 A 级客户，年营收 ¥2 亿，有 MES 系统采购需求信号"
5. **用户操作**：点击「采纳并建议」→ 跳转对话页面 → AI 基于客户洞察结果给出详细跟进策略

## 十六、POC 效果验证

> 来源：Wiki 143118204 的 POC 部分

### 16.1 伊顿（老客户场景）

- **数据来源**：网络搜索 + CRM 活动记录 + 商机数据 + 服务工单
- **验证重点**：多数据源融合效果，内外部数据交叉分析能力
- **关键结论**：能够综合内部 CRM 数据和外部搜索信息，生成有针对性的客户洞察和行动建议

### 16.2 山推工程（新客户场景）

- **数据来源**：纯网络搜索（无 CRM 历史数据）
- **验证重点**：仅依赖外部数据时的洞察质量
- **关键结论**：即使没有内部数据，仅通过网络搜索也能生成有价值的企业背景分析和初步行动建议

### 16.3 洞察差异对比验证

- 验证前后两次洞察结果的差异检测准确性
- 确认差异 Prompt 能正确识别重大变更并输出结构化差异描述
