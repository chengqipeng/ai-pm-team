# aPaaS 元数据设计体系

本文档描述元数据实例层的设计——数据如何存储、读取、合并和写入。

元模型定义层（Schema 定义）请参考：[元模型设计体系.md](元模型设计体系.md)
数据迁移方案请参考：[数据迁移方案.md](数据迁移方案.md)

## 1. 大宽表设计

### 1.1 物理列布局

大宽表（`p_common_metadata` / `p_tenant_metadata`）采用统一的列结构：

**固定列（所有元模型共享）：**

| 列名 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT | 雪花算法主键 |
| `api_key` | VARCHAR(255) | 元数据唯一标识 |
| `label` | VARCHAR(255) | 显示标签 |
| `label_key` | VARCHAR(255) | 多语言标签 Key |
| `namespace` | VARCHAR(50) | 来源分类（system/product/tenant） |
| `metamodel_api_key` | VARCHAR(255) | 所属元模型类型 |
| `entity_api_key` | VARCHAR(255) | 所属对象 apiKey |
| `parent_metadata_api_key` | VARCHAR(255) | 父元数据 apiKey |
| `custom_flg` | INT | 0=标准 1=自定义 |
| `metadata_order` | INT | 排序序号 |
| `description` | VARCHAR(500) | 描述 |
| `delete_flg` | SMALLINT | 软删除标记 |
| `created_at/created_by/updated_at/updated_by` | BIGINT | 审计字段 |

**扩展列（dbc_xxxN，语义由 p_meta_item.db_column 定义）：**

| 列前缀 | 数量 | 数据库类型 | Java 类型 | 用途 |
|---|---|---|---|---|
| `dbc_varchar` | 1~30 | VARCHAR(255) | String | 短文本、apiKey 引用 |
| `dbc_textarea` | 1~10 | TEXT | String | 长文本、JSON、公式 |
| `dbc_bigint` | 1~20 | BIGINT | Long | 大整数、时间戳、关联 ID |
| `dbc_int` | 1~15 | INTEGER | Integer | 枚举/选择值 |
| `dbc_smallint` | 1~50 | SMALLINT | Integer | 布尔/开关 |
| `dbc_decimal` | 1~5 | DECIMAL(20,4) | BigDecimal | 精确小数 |

总计 130 个扩展列。

### 1.2 列名格式说明

| 场景 | 格式 | 示例 |
|---|---|---|
| 数据库物理列名 | `dbc_xxxN`（无下划线分隔数字） | `dbc_varchar8`, `dbc_int1`, `dbc_smallint13` |
| Java Entity 字段名 | `dbcXxxN`（camelCase） | `dbcVarchar8`, `dbcInt1`, `dbcSmallint13` |
| p_meta_item.db_column | `dbc_xxxN`（与物理列名一致） | `dbc_varchar3`, `dbc_int1` |
| ItemTypeEnum.dbColumnPrefix | `dbc_xxx`（不含数字） | `dbc_varchar`, `dbc_bigint` |

## 2. Common/Tenant 合并机制

### 2.1 数据流

```
读取请求（tenantId + metamodelApiKey + entityApiKey）
    │
    ├─→ 查 MetaModel 定义
    │     获取 enable_common / enable_tenant / db_table
    │
    ├─→ if enable_common=1: 查 Common 层（p_common_metadata）
    │     WHERE metamodel_api_key=? AND entity_api_key=?
    │     通过 CommonMetadataConverter 转为业务 Entity
    │
    ├─→ if enable_tenant=1: 查 Tenant 层（DynamicTableNameHolder 路由到 db_table 指向的独立大宽表）
    │     WHERE tenant_id=? AND metamodel_api_key=? AND entity_api_key=?
    │
    └─→ 合并（仅当 enable_common=1 且 enable_tenant=1 时需要合并）
          Common 有 Tenant 无 → 用 Common
          同 apiKey → Tenant 覆盖 Common
          Tenant delete_flg=1 → 隐藏该条
```

### 2.2 CommonMetadataConverter 转换流程

```
CommonMetadata（大宽表行）
    │
    ├─ Step 1: 固定列同名映射（非 dbc_ 开头、非 SKIP_FIELDS 的字段）
    │    apiKey → apiKey, label → label, namespace → namespace,
    │    entityApiKey → entityApiKey, customFlg → customFlg,
    │    description → description, descriptionKey → descriptionKey ...
    │    SKIP_FIELDS（不映射）: metamodelApiKey, metadataApiKey,
    │    parentMetadataApiKey, metadataOrder, metaVersion
    │
    └─ Step 2: dbc_xxxN → 业务字段（通过 p_meta_item 列映射）
         getColumnMapping(metamodelApiKey) 返回 {dbc_int1 → itemType, dbc_varchar3 → dbColumn, ...}
         snakeToCamel(dbc_int1) → dbcInt1 → 从 CommonMetadata 读值
         写入 EntityItem.itemType
```

> 注意：老系统的 `objectApiKey → entityApiKey` 特殊映射已删除。大宽表固定列已统一为 `entity_api_key`，
> Converter 通过 Step 1 同名映射直接处理，无需特殊逻辑。
> `customFlg` 已从 SKIP_FIELDS 中移除，因为业务 Entity 中有此字段，应正常映射。
    │    objectApiKey → entityApiKey（历史兼容）
    │
    └─ Step 3: dbc_xxxN → 业务字段（通过 p_meta_item 列映射）
         getColumnMapping(metamodelApiKey) 返回 {dbc_int1 → itemType, dbc_varchar3 → dbColumn, ...}
         snakeToCamel(dbc_int1) → dbcInt1 → 从 CommonMetadata 读值
         写入 EntityItem.itemType
```

### 2.3 namespace 分类

| namespace | 含义 | 存储位置 | 可见性 |
|---|---|---|---|
| system | 系统出厂元数据 | p_common_metadata | 所有租户可见 |
| product | 业务产品元数据 | p_common_metadata | 受 license 控制 |
| tenant | 客户自定义元数据 | p_tenant_metadata | 仅该租户可见 |

### 2.4 国际化字段（xxxKey）规则

所有文本类型的显示字段都必须有对应的 `xxxKey` 字段用于国际化。`xxxKey` 存储的是 `p_meta_i18n_resource` 表中的 `resource_key`，运行时根据用户语言环境查找对应的翻译文本。

**命名规则：**

| 文本字段 | 国际化 Key 字段 | 说明 |
|---|---|---|
| `label` | `labelKey` | 显示标签的国际化 Key |
| `description` | `descriptionKey` | 描述信息的国际化 Key |
| `helpText` | `helpTextKey` | 帮助文本的国际化 Key |
| `checkErrorMsg` | `checkErrorMsgKey` | 校验错误提示的国际化 Key |

**Key 格式规范：**

| 元模型 | Key 前缀 | 格式 | 示例 |
|---|---|---|---|
| entity.label | `XdMDObj.` | `XdMDObj.{entityApiKey}` | `XdMDObj.account` |
| item.label | `XdMDItem.` | `XdMDItem.{itemApiKey}` | `XdMDItem.accountName` |
| item.helpText | `XdMDItm.Hlptxt.` | `XdMDItm.Hlptxt.{entityApiKey}.{itemApiKey}` | `XdMDItm.Hlptxt.account.industryId` |
| item.description | `XdMDItm.Desc.` | `XdMDItm.Desc.{entityApiKey}.{itemApiKey}` | `XdMDItm.Desc.account.accountName` |
| checkRule.label | `XdMDRule.` | `XdMDRule.{ruleApiKey}` | `XdMDRule.AccountNameRequired` |
| checkRule.checkErrorMsg | `XdMDRule.Err.` | `XdMDRule.Err.{ruleApiKey}` | `XdMDRule.Err.AccountNameRequired` |
| pickOption.optionLabel | `XdMDPick.` | `XdMDPick.{entityApiKey}.{itemApiKey}.{optionApiKey}` | `XdMDPick.account.industryId.IT` |
| entityLink.label | `XdMDLink.` | `XdMDLink.{linkApiKey}` | `XdMDLink.ContactToAccount` |

**p_meta_i18n_resource 表结构：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT | 主键 |
| `tenant_id` | BIGINT | 租户 ID |
| `resource_key` | VARCHAR(256) | 资源 Key（对应 xxxKey 的值） |
| `lang_code` | VARCHAR(8) | 语言编码（zh/en/ja 等） |
| `resource_value` | TEXT | 翻译文本 |
| `metamodel_id` | BIGINT | 所属元模型 ID |
| `metadata_id` | BIGINT | 所属元数据 ID |
| `object_id` | BIGINT | 所属对象 ID |

## 3. 写入流程

所有写操作通过 `DynamicTableNameHolder` 路由到 `p_meta_model.db_table` 指向的 Tenant 级独立大宽表（结构与 `p_common_metadata` 一致 + `tenant_id`）：

```
创建：Entity → CommonMetadataConverter.toTenantMetadata() → DynamicTableNameHolder.executeWith(dbTable) → INSERT
更新：查旧值 → Entity → toTenantMetadata() → DynamicTableNameHolder.executeWith(dbTable) → UPDATE/INSERT
删除：
  - Tenant 已有记录 → 软删除（delete_flg=1）
  - Common 级数据 → 插入 delete_flg=1 的 Tenant 记录（遮蔽删除）
```

> 注意：Common 级数据（`p_common_metadata`）由平台初始化或 Module 安装写入，业务层不直接写入 Common 表。

## 4. Java 类体系

### 4.1 Entity 继承体系

```
BaseEntity（id, deleteFlg, createdAt/By, updatedAt/By）
  └─ BaseMetaCommonEntity（apiKey, label, labelKey, namespace, description, descriptionKey, customFlg）
       ├─ BaseMetaTenantEntity（tenantId）
       │    ├─ Entity（entityType, dbTable, enableFlg, ...）          ← 业务视图，不直接映射表
       │    ├─ EntityItem（itemType, dataType, dbColumn, ...）        ← 业务视图，不直接映射表
       │    ├─ EntityLink（linkType, parentEntityApiKey, ...）        ← 业务视图，不直接映射表
       │    ├─ CheckRule（checkFormula, checkErrorMsg, ...）          ← 业务视图，不直接映射表
       │    ├─ PickOption（optionLabel, ...）                         ← 业务视图，不直接映射表
       │    ├─ BusiType（enableFlg, specialFlg, ...）                ← 业务视图，不直接映射表
       │    ├─ DuplicateRule（activeFlg, criteriaLogic, ...）        ← 业务视图，不直接映射表
       │    ├─ DuplicateRuleCriteria（ruleApiKey, matchType, ...）   ← 业务视图，不直接映射表
       │    ├─ DuplicateMatchingRule（ruleApiKey, matchType, ...）   ← 业务视图，不直接映射表
       │    └─ TenantMetadata（metamodelApiKey, dbcVarchar1~30, ...） ← @TableName("p_tenant_metadata")
       │         通过 DynamicTableNameHolder 路由到 p_tenant_entity / p_tenant_item 等
       ├─ CommonMetadata（metamodelApiKey, dbcVarchar1~30, ...）      ← @TableName("p_common_metadata")
       ├─ MetaModel（metamodelType, enableCommon, enableTenant, ...）
       └─ MetaItem（metamodelApiKey, itemType, dbColumn, ...）
```

业务 Entity（Entity/EntityItem/EntityLink 等）是 CommonMetadataConverter 转换后的业务视图，不直接映射到数据库表。
所有 Tenant 级独立大宽表（p_tenant_entity、p_tenant_item 等）结构与 p_common_metadata 完全一致（+ tenant_id），
统一使用 TenantMetadata 实体类 + DynamicTableNameHolder 路由操作。

## 7. 字段类型体系（itemType / itemSubType / dataType）

### 7.1 三层类型模型

每个字段由三个维度描述其类型：

```
itemType（字段类型）    → 决定 UI 交互方式和业务语义
  └─ itemSubType（真实类型） → 决定字段实际承载的数据类型
       └─ dataType（存储类型）  → 决定数据库物理列的存储格式
```

示例：

| 字段 | itemType | itemSubType | dataType | 说明 |
|---|---|---|---|---|
| accountScore | FORMULA(6) | CURRENCY(10) | DECIMAL(4) | 公式计算，结果是货币，存储用 decimal |
| isCustomer | FORMULA(6) | TEXT(1) | VARCHAR(1) | 公式计算，结果是文本，存储用 varchar |
| visitTotalCount | FORMULA(6) | NUMBER(2) | BIGINT(3) | 公式计算，结果是数字，存储用 bigint |
| accountName | TEXT(1) | TEXT(1) | VARCHAR(1) | 普通文本，三层一致 |
| level | PICKLIST(4) | PICKLIST(4) | INT(2) | 单选，三层一致 |
| ownerId | LOOKUP(5) | LOOKUP(5) | BIGINT(3) | 关联引用，三层一致 |

对于非计算型字段，itemType = itemSubType（三层一致）。
对于计算型字段（FORMULA/ROLLUP/COMPUTED），itemType 是计算类型，itemSubType 是计算结果的真实类型。

### 7.6 p_meta_item 存储

三层类型存储在 item 元模型的三个 dbc 列中：

| 字段 | db_column | 说明 |
|---|---|---|
| itemType | dbc_int1 | 字段类型（UI 交互类型） |
| itemSubType | dbc_int2 | 真实类型（原 dataType 列，语义升级） |
| dataType | 新增字段 | 数据库存储类型 |

> 注意：当前 dbc_int2 存储的是老系统的 dataType 编码，需要迁移为新的 itemSubType 编码。
> dataType 可以从 itemSubType 通过 §7.5 的映射表自动推导，不需要独立存储，
> 但为了查询便利和前端展示，建议作为冗余字段保留。

## 8. 数据质量规则

### 8.1 同步后自动清理

| 规则 | 条件 | 说明 |
|---|---|---|
| 维度字段清理 | itemType=99 | 老系统维度分析占位字段，dbColumn 为 dim_xxx，新系统 dbc 列体系无法映射 |
| JOIN 虚拟字段评估 | itemType=21(老26) 且 joinItem/joinObj/joinLink 全空 | 无效虚拟引用，需逐对象评估 |
| 数字 ID 引用修复 | referEntityApiKey/referLinkApiKey 为纯数字或 `_orphan_` 前缀 | 所有引用必须使用 apiKey |
| 孤儿子元数据清理 | parent_metadata_api_key 指向不存在的父记录 | 级联删除断裂的子元数据 |

### 8.2 FORMULA 字段完整性

每个 FORMULA/ROLLUP/COMPUTED 字段必须满足：
- 有对应的 formulaCompute 或 aggregationCompute 定义（通过 `parent_metadata_api_key` 关联）
- `itemSubType` 非空，正确反映计算结果类型

### 8.3 PICKLIST 选项值完整性

每个 PICKLIST/MULTIPICKLIST 字段必须满足以下之一：
- 在 pickOption 中有选项值记录（`parent_metadata_api_key = item.api_key`）
- `itemSubType=1`（PICK_GLOBAL）且 `globalPickItemApiKey` 非空
- 选项由前端硬编码提供（如省市区级联数据）

### 8.4 LOOKUP 引用完整性

每个 LOOKUP/MASTER_DETAIL 字段必须满足：
- `referEntityApiKey` 非空且指向存在的 entity
- `referLinkApiKey` 非空且指向存在的 entityLink
- 对应的 entityLink 的 parentEntityApiKey/childEntityApiKey 都指向存在的 entity
