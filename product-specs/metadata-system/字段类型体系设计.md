# 字段类型体系设计

## 1. 三层类型模型

每个字段由三个维度描述：

```
itemType（字段类型）       → UI 交互方式和业务语义
  └─ itemSubType（真实类型）  → 字段实际承载的数据类型（计算型字段为计算结果类型）
       └─ dataType（存储类型）   → 数据库物理列的存储格式
```

规则：
- 非计算型字段：itemSubType = itemType
- 计算型字段（FORMULA/27）：itemSubType 是用户选择的计算结果类型
- 虚拟字段（JOIN/26）：itemSubType = null, dataType = null
- dataType 由 itemSubType 通过固定映射推导

## 3. 计算类型（FORMULA/27）详细设计

### 3.1 概述

计算类型是一个统一的字段类型（itemType=27），创建时用户选择"计算子类型"（itemSubType），决定计算结果的数据类型。计算字段是只读字段，值由公式引擎实时计算，不占用物理列。

### 3.2 计算子类型（itemSubType）

用户创建计算字段时，需要选择以下子类型之一：

| 计算子类型 | itemSubType | dataType | 说明 | 示例 |
|---|---:|---|---|---|
| 数字 | 5 (NUMBER) | BIGINT(3) | 计算数字值 | 报价金额=单价*数量*折扣+运费 |
| 日期 | 7 (DATE) | BIGINT(3) | 计算日期 | 预计结单日期=创建日期+7 |
| 日期+时间 | 38 (DATETIME) | BIGINT(3) | 计算日期时间 | 预计解决时间=创建时间+24 |
| 文本 | 1 (TEXT) | VARCHAR(1) | 串联文本字段 | 报名订单=报名人姓名&报名日期 |
| 单选 | 2 (SELECT) | VARCHAR(1) | 通过公式赋值选项 | 发货城市=客户.城市 |
| 货币 | 6 (CURRENCY) | DECIMAL(4) | 计算货币金额 | Gross Margin=Amount-Cost |
| 布尔 | 31 (BOOLEAN) | SMALLINT(6) | 计算布尔值 | 是否重点=AND(金额>500万,等级=A级) |
| 百分比 | 33 (PERCENT) | DECIMAL(4) | 计算百分比 | Discount=(Amount-Discounted)/Amount |
| 汇总累计 | 由累计类型决定 | 由累计类型决定 | 汇总关联对象数据 | 见 §3.3 |

### 3.3 汇总累计配置

选择"汇总累计"子类型后，需要配置：

| 配置项 | 说明 |
|---|---|
| 已关联对象 | 选择要汇总的子对象（通过 entityLink 关联） |
| 累计类型 | COUNT / SUM / MAX / MIN |
| 汇总字段 | SUM/MAX/MIN 时选择子对象的数值字段 |
| 过滤条件 | 包括所有记录 / 仅符合条件的记录 |
| 小数点位数 | 结果的小数位数 |

累计类型与 itemSubType 的关系：

| 累计类型 | itemSubType | 说明 |
|---|---|---|
| COUNT | 5 (NUMBER) | 计数，结果为整数 |
| SUM | 继承汇总字段类型 | 求和，结果类型与汇总字段一致 |
| MAX | 继承汇总字段类型 | 最大值 |
| MIN | 继承汇总字段类型 | 最小值 |

### 3.4 计算定义元模型

| 元模型 | 说明 | 关联方式 |
|---|---|---|
| formulaCompute | 公式定义（表达式、空值处理、结果类型） | parent_metadata_api_key = item.api_key |
| formulaComputeItem | 公式引用的字段列表 | parent_metadata_api_key = formulaCompute.api_key |
| aggregationCompute | 汇总累计定义（子对象、累计类型、汇总字段） | parent_metadata_api_key = item.api_key |
| aggregationComputeDetail | 汇总过滤条件 | parent_metadata_api_key = aggregationCompute.api_key |
| computeFactor | 计算因子（公式/汇总共享） | parent_metadata_api_key = compute.api_key |

formulaCompute 中的 resultType 使用 itemType 编码：

| resultType | 含义 | itemSubType |
|---:|---|---:|

### 3.6 aggregationCompute.aggregateType → itemSubType 映射

| aggregateType | 含义 | itemSubType | 说明 |
|---:|---|---|---|
| 0 | COUNT(条件) | 5 (NUMBER) | 带过滤条件的计数 |

## 4. db_column 的两层含义

系统中存在两层 db_column，语义不同：

### 4.1 元模型层 db_column（p_meta_item.db_column）

定义元模型字段存储在**元数据大宽表**（p_common_metadata / p_tenant_*）的哪个 dbc 列。

示例：item 元模型的 `itemType` 字段 → p_meta_item.db_column = `dbc_int1` → 存储在 p_common_metadata.dbc_int1 列中。

这一层的 db_column 由平台初始化时分配，每个元模型的字段定义固定不变。

### 4.2 元数据层 db_column（item 记录的 dbColumn 字段值）

定义业务字段存储在**业务数据大宽表**（如 a_account、a_contact 等）的哪个列。

示例：account 的 `level`（客户级别）字段 → item.dbColumn = `dbc_int1` → 存储在 a_account.dbc_int1 列中。

这一层的 db_column 在创建字段时按 itemType 对应的 dbColumnPrefix 自动分配列号。每个 entity 的列号独立分配，互不冲突。

### 4.3 两层关系图

```
p_meta_item（元模型层）
  api_key='dbColumn', db_column='dbc_varchar3'
  → 表示 item 元模型的 dbColumn 字段存储在元数据大宽表的 dbc_varchar3 列

p_common_metadata（元数据层）
  metamodel_api_key='item', api_key='level', dbc_varchar3='dbc_int1'
  → 表示 account.level 字段存储在业务数据表 a_account 的 dbc_int1 列
```

### 4.4 计算字段的 db_column

计算字段（FORMULA/27）的 db_column 规则：
- 包含实时计算函数（如 NOW()、TODAY()）的字段：不分配 db_column，每次读取时实时计算
- 其他计算字段：需要分配 db_column，计算结果持久化到业务数据表中
- db_column 的前缀由 itemSubType 决定（如 subType=5/NUMBER → dbc_bigint，subType=6/CURRENCY → dbc_decimal）

## 7. SELECT dbColumn 前缀迁移

老系统 SELECT(2) 存数字编码（dbc_int），新系统存选项 apiKey 字符串（dbc_varchar）。

| 字段类型 | 老 dbColumn 前缀 | 新 dbColumn 前缀 | 原因 |
|---|---|---|---|
| SELECT(2) | dbc_int | dbc_varchar | 选项值从数字编码改为 apiKey |
| MULTI_SELECT(3) | dbc_varchar | dbc_varchar | 不变 |

> RELATION_SHIP(10)、MASTER_DETAIL(41)、MULTI_RELATION_SHIP(34) 保持 dbc_bigint 不变。

## 9. 数据迁移步骤

1. **itemType 不转换**：新老编码一致，直接使用
2. **itemSubType 回填**：非计算型 = itemType；FORMULA(27) 从 formulaCompute.resultType 按 §3.5 映射
3. **dataType 回填**：从 itemSubType 通过 §4 推导
4. **SELECT dbColumn 迁移**：dbc_int → dbc_varchar，选项值从数字编码→apiKey
