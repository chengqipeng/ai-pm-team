# 业务数据大宽表设计

## 1. 概述

所有 entity（业务对象）的业务数据统一存储在一张大宽表 `p_tenant_data` 中，通过 `entity_api_key` 区分不同对象的数据。

### 1.1 两层大宽表对比

| 维度 | 元数据大宽表 | 业务数据大宽表 |
|---|---|---|
| 表名 | p_common_metadata / p_tenant_* | p_tenant_data |
| 用途 | 存储元数据定义 | 存储业务数据 |
| 行含义 | 一行 = 一个元数据定义 | 一行 = 一条业务记录 |
| 类型区分 | metamodel_api_key | entity_api_key |
| 列映射定义 | p_meta_item.db_column | item 元数据的 dbColumn 字段值 |

### 1.2 两层 db_column 的区别

| 层级 | 所在位置 | 含义 |
|---|---|---|
| 元模型层 | p_meta_item.db_column | 元模型字段存储在**元数据大宽表**的哪个 dbc 列 |
| 元数据层 | item 记录的 dbColumn 字段值 | 业务字段存储在**业务数据大宽表**的哪个 dbc 列 |

## 3. 新系统表结构

### 3.1 设计原则

1. **统一大宽表**：所有 entity 共用 p_tenant_data 一张表，通过 entity_api_key 区分
2. **固定列完整保留**：老系统所有 entity 通用的固定列在新系统中保持为独立固定列，不占用 dbc 扩展列
3. **列前缀合并**：老系统 9 种 dbc 前缀合并为 7 种，列数完全覆盖老系统容量
4. **列号格式统一**：老 `dbc_varchar_1` → 新 `dbc_varchar1`（去下划线）
5. **MySQL 行大小适配**：VARCHAR(300) 行内列总量控制在 65535 字节以内

### 3.2 固定列（15 列）

所有 entity 通用的固定列。老系统的其他固定列（如 territory_id、currency_unit、high_sea_id 等）在新系统中通过 dbc 扩展列存储。

| 列名 | 类型 | 老列名 | 说明 |
|---|---|---|---|
| id | BIGINT NOT NULL | id | 雪花算法主键 |
| tenant_id | BIGINT NOT NULL | tenant_id | 租户 ID |
| entity_api_key | VARCHAR(100) NOT NULL | — | 新增，所属对象类型 |
| name | VARCHAR(300) | {entity}_name | 记录名称（统一列名） |
| owner_id | BIGINT | owner_id | 数据所有人 |
| depart_id | BIGINT | dim_depart | 所属部门 |
| busitype_api_key | VARCHAR(100) | entity_type | 业务类型（老存 ID，新存 apiKey） |
| applicant_id | BIGINT | applicant_id | 审批提交人 |
| delete_flg | SMALLINT DEFAULT 0 | del_flg | 软删除标记 |
| created_at | BIGINT | created_at | 创建时间 |
| created_by | BIGINT | created_by | 创建人 |
| updated_at | BIGINT | updated_at | 修改时间 |
| updated_by | BIGINT | updated_by | 修改人 |
| lock_status | INT DEFAULT 1 | lock_status | 锁定状态 |
| approval_status | INT | approval_status | 审批状态 |

固定列合计：**15 列**

### 3.3 扩展列（670 列）

老系统 9 种 dbc 前缀合并为 6 种新前缀。文本类统一用 dbc_varchar（含单选 apiKey），多选单独用 dbc_multi。

> MySQL utf8mb4 行大小限制 65535 字节。dbc_varchar 采用混合类型策略：
> 前 35 列为 VARCHAR(300)（行内存储，可建索引），第 36~250 列为 TEXT（溢出页，不占行内空间）。
> 对外统一叫 dbc_varchar，物理类型由列号决定，业务层无感知。

| 新前缀 | 列号范围 | 列数 | 数据库类型 | Java 类型 | 行内大小 | 用途 |
|---|---|---|---|---|---|---|
| dbc_varchar | 1~35 | 35 | VARCHAR(300) | String | 1,202 B/列 | 文本 + 单选 apiKey（可建索引，支持排序） |
| dbc_varchar | 36~250 | 215 | TEXT | String | 0（溢出页） | 文本 + 单选 apiKey（溢出区，不可直接建索引） |
| dbc_multi | 1~30 | 30 | TEXT | String | 0（溢出页） | 多选（逗号分隔 apiKey，如 `opt1,opt2,opt3`） |
| dbc_bigint | 1~195 | 195 | BIGINT | Long | 8 B/列 | 整数、日期时间戳、关联 ID |
| dbc_decimal | 1~140 | 140 | DECIMAL(20,4) | BigDecimal | 9 B/列 | 货币、百分比 |
| dbc_smallint | 1~35 | 35 | SMALLINT | Integer | 2 B/列 | 布尔 0/1 |
| dbc_textarea | 1~20 | 20 | TEXT | String | 0（溢出页） | 富文本、JSON |

总计：15 固定列 + 670 扩展列 = **685 列**

**dbc_varchar 的混合类型说明**：

| 列号 | 物理类型 | 可建索引 | 分配优先级 |
|---|---|---|---|
| 1~35 | VARCHAR(300) | ✓ 可直接建索引 | 高频过滤/排序的字段优先分配 |
| 36~250 | TEXT | ✗ 需前缀索引 | 低频或长文本字段 |

列号分配时，需要索引的字段（如 phone、email）优先分配 1~35 号，不需要索引的字段分配 36~250 号。

**老系统覆盖验证**：

| 类别 | 老系统列数 | 新系统列数 | 覆盖 |
|---|---|---|---|
| 文本 + 单选（varchar + svarchar + select） | 145 + 25 + 80 = 250 | dbc_varchar 250 | ✓ |
| 多选（从老 varchar 中拆出） | ~30 | dbc_multi 30 | ✓ |
| 整数类（integer + date + relation） | 120 + 45 + 30 = 195 | dbc_bigint 195 | ✓ |
| 实数类（real） | 140 | dbc_decimal 140 | ✓ |
| 布尔类（tinyint） | 35 | dbc_smallint 35 | ✓ |
| 长文本类（textarea） | 20 | dbc_textarea 20 | ✓ |
| **合计** | **620 + 30** | **670** | ✓ |

**对应 itemType**：

| 新前缀 | 对应 itemType |
|---|---|
| dbc_varchar | TEXT(1), PICKLIST(2), PHONE(13), EMAIL(12), URL(14), IMAGE(19), GEO(18), AUTONUMBER(20), FILE(39), AUDIO(22) |
| dbc_multi | MULTIPICKLIST(16) |
| dbc_bigint | NUMBER(2), DATE(3), DATETIME(15), LOOKUP(5), MASTER_DETAIL(17) |
| dbc_decimal | CURRENCY(10), PERCENT(11) |
| dbc_smallint | BOOLEAN(9) |
| dbc_textarea | TEXTAREA(8), RICHTEXT(40) |

**行大小验证（MySQL utf8mb4, ROW_FORMAT=DYNAMIC）**：

| 列类型 | 列数 | 单列行内大小 | 小计 |
|---|---|---|---|
| 固定列（15 列） | 15 | — | 2,088 B |
| dbc_varchar1~35 × VARCHAR(300) | 35 | 1,202 B | 42,070 B |
| dbc_varchar36~250 × TEXT | 215 | 0 B（溢出页） | 0 B |
| dbc_multi × TEXT | 30 | 0 B（溢出页） | 0 B |
| dbc_bigint × BIGINT | 195 | 8 B | 1,560 B |
| dbc_decimal × DECIMAL(20,4) | 140 | 9 B | 1,260 B |
| dbc_smallint × SMALLINT | 35 | 2 B | 70 B |
| dbc_textarea × TEXT | 20 | 0 B（溢出页） | 0 B |
| **合计** | **685** | | **47,048 B** (< 65,535) ✓ |

余量 18,487 字节。

### 3.4 老→新列映射规则

| 老前缀 | 老列号 | 新前缀 | 新列号 | 说明 |
|---|---|---|---|---|
| dbc_varchar_ | 1~35 | dbc_varchar | 1~35 | 直接映射，VARCHAR(300) 可索引区 |
| dbc_varchar_ | 36~145 | dbc_varchar | 36~145 | 直接映射，TEXT 溢出区 |
| dbc_svarchar_ | 1~25 | dbc_varchar | 146~170 | 合并到 dbc_varchar 尾部 |
| dbc_select_ | 1~80 | dbc_varchar | 171~250 | 老存数字编码，新存 apiKey 字符串 |
| dbc_integer_ | 1~120 | dbc_bigint | 1~120 | 直接映射 |
| dbc_date_ | 1~45 | dbc_bigint | 121~165 | 接在 integer 后面 |
| dbc_relation_ | 1~30 | dbc_bigint | 166~195 | 接在 date 后面 |
| dbc_real_ | 1~140 | dbc_decimal | 1~140 | 直接映射 |
| dbc_tinyint_ | 1~35 | dbc_smallint | 1~35 | 直接映射 |
| dbc_textarea_ | 1~20 | dbc_textarea | 1~20 | 直接映射 |

> 多选字段（MULTIPICKLIST）在老系统中用 dbc_varchar_ 存逗号分隔字符串，新系统迁移到 dbc_multi。
> SELECT 字段迁移时需要值转换：老系统存数字编码（如 `3`），新系统存 pickOption.apiKey（如 `it_industry`）。

### 3.5 与老系统对比

| 维度 | 老系统（PG） | 新系统（MySQL） |
|---|---|---|
| 前缀数 | 9 种 | 6 种（合并文本类 + 新增 dbc_multi） |
| 扩展列总数 | 620 | 670（多 50 列余量，含 30 列多选） |
| 固定列 | ~119（含业务专用） | 15（通用固定列，其余走扩展列） |
| 总列数 | ~720 | 685 |
| SELECT 存储 | 数字编码（INTEGER） | apiKey 字符串（dbc_varchar） |
| MULTISELECT 存储 | 逗号分隔（VARCHAR） | 逗号分隔 apiKey（dbc_multi TEXT） |
| VARCHAR 长度 | 300 | 300（dbc_varchar 1~35）/ TEXT（36~250） |
| 列名格式 | dbc_varchar_1（带下划线） | dbc_varchar1（无下划线） |
| 行大小限制 | 无（PG 无此限制） | 47,048 / 65,535（MySQL） |

### 3.6 列号分配规则

每个 entity 的列号独立分配，不同 entity 复用相同列号：

1. 创建字段时，根据 itemType 确定 dbColumnPrefix
2. 文本类字段（TEXT/PICKLIST/PHONE/EMAIL/URL 等）统一分配 dbc_varchar（1~250）
   - 需要索引的字段优先分配 1~35 号（VARCHAR(300) 区，可建索引）
   - 不需要索引的字段分配 36~250 号（TEXT 区）
3. MULTIPICKLIST 字段分配 dbc_multi（1~30）
4. 查询该 entity 已用的同前缀列号
5. 取最小空闲号，生成 dbColumn = `{prefix}{number}`
6. 计算字段（FORMULA/27）根据 itemSubType 确定前缀（除实时计算函数外）
7. JOIN(26) 不分配列号
5. 取最小空闲号，生成 dbColumn = `{prefix}{number}`
6. 计算字段（FORMULA/27）根据 itemSubType 确定前缀（除实时计算函数外）
7. JOIN(26) 不分配列号

## 6. 分表策略

详见 §8 分表详细设计。核心策略：预建 2000 张表（p_tenant_data_0 ~ p_tenant_data_1999），按 (tenant_id, entity_api_key) 粒度动态分配表槽位，Least-Loaded 算法优先分配到数据量小的表，纯应用层 DynamicTableNameHolder 路由。

## 7. 业务数据 CRUD 详细设计

### 7.1 核心架构

```
API 层（paas-entity-service REST）
  │
  ├─ EntityDataApi          — 对外 CRUD 接口
  └─ EntityMappingApi       — 字段映射查询接口
        │
Service 层
  │
  ├─ EntityDataService      — 业务编排（校验→转换→持久化→事件）
  ├─ FieldMappingService    — 元数据驱动的字段映射（apiKey ↔ dbColumn）
  ├─ ColumnValueConverter   — 字段值类型转换（API 值 → DB 值）
  └─ DynamicSqlBuilder      — 动态 SQL 拼装
        │
DAO 层
  │
  ├─ BizDataMapper          — MyBatis-Plus 通用 Mapper
  └─ DynamicTableNameHolder — 分表路由
```

### 7.2 字段映射机制（FieldMappingService）

业务数据读写的核心是将前端传入的 `apiKey: value` 转换为 `dbColumn: dbValue`，反之亦然。

#### 7.2.1 映射构建流程

```
构建字段映射(tenantId, entityApiKey):
    │
    ├─ 1. 调用 metarepo-service 获取 item 列表
    │     GET /metarepo/read/items?entityApiKey={entityApiKey}
    │     → List<EntityItem>（已合并 Common + Tenant）
    │
    ├─ 2. 构建映射表
    │     Map<String, FieldMapping> mappings = new LinkedHashMap<>()
    │     for item in items:
    │       if item.enableFlg != 1: continue          // 跳过禁用字段
    │       if item.deleteFlg == 1: continue           // 跳过已删除
    │       if item.dbColumn == null: continue          // 跳过虚拟字段（JOIN/实时FORMULA）
    │       mappings.put(item.apiKey, FieldMapping{
    │           apiKey:      item.apiKey,
    │           dbColumn:    item.dbColumn,             // 如 dbc_varchar1
    │           itemType:    item.itemType,             // 如 1(TEXT)
    │           dataType:    item.dataType,             // 如 1(VARCHAR)
    │           requireFlg:  item.requireFlg,
    │           readonlyFlg: item.readonlyStatus,
    │           label:       item.label
    │       })
    │
    ├─ 3. 追加固定列映射
    │     mappings.put("name",           FieldMapping{dbColumn: "name", ...})
    │     mappings.put("ownerId",        FieldMapping{dbColumn: "owner_id", ...})
    │     mappings.put("departId",       FieldMapping{dbColumn: "depart_id", ...})
    │     mappings.put("busitypeApiKey", FieldMapping{dbColumn: "busitype_api_key", ...})
    │     // ... 其他固定列
    │
    └─ 4. 缓存（Redis + 本地二级缓存）
          key: "field_mapping:{tenantId}:{entityApiKey}"
          TTL: 10 分钟
          失效: 元数据变更时通过 MQ 广播清除
```

#### 7.2.2 FieldMapping 数据结构

```java
public class FieldMapping {
    private String apiKey;          // 字段标识，如 accountName
    private String dbColumn;        // 物理列名，如 dbc_varchar1
    private Integer itemType;       // 字段类型编码
    private Integer dataType;       // 存储类型编码
    private Integer requireFlg;     // 是否必填
    private Integer readonlyFlg;    // 是否只读
    private String label;           // 显示名称（用于错误提示）
    private String referEntityApiKey; // LOOKUP 目标对象（仅关联字段）
}
```

#### 7.2.3 固定列与扩展列的统一处理

| 字段来源 | apiKey 示例 | dbColumn | 映射方式 |
|---|---|---|---|
| 固定列 | name, ownerId, createdAt | name, owner_id, created_at | 硬编码映射，不走 item 元数据 |
| 扩展列 | accountName, phone, level | dbc_varchar1, dbc_varchar2, dbc_text1 | 从 item.dbColumn 动态获取 |
| 系统列 | id, tenantId, entityApiKey | id, tenant_id, entity_api_key | 系统自动填充，不对外暴露 |

### 7.3 数据写入流程

#### 7.3.1 新增记录（Create）

```
POST /entity/data/{entityApiKey}
Body: { "name": "客户A", "phone": "13800138000", "industryId": "it", "level": "A" }

处理流程:
    │
    ├─ 1. 参数校验
    │     ├─ entityApiKey 合法性（entity 存在且 enableFlg=1）
    │     ├─ 必填字段校验（requireFlg=1 的字段必须有值）
    │     └─ 字段值类型校验（见 §7.4）
    │
    ├─ 2. 获取字段映射
```

```java
private static final Pattern DBC_COLUMN_PATTERN =

private static final Set<String> FIXED_COLUMNS = Set.of(
```

## 8. 分表详细设计

### 8.1 数据量估算

| 维度 | 数值 | 说明 |
|---|---|---|
| 租户数 | ≥ 10 万 | 持续增长 |
| 每租户用户数 | 平均 1000 | |
| 每租户 entity 数 | 平均 20~50 | 标准对象 + 自定义对象 |
| 每 entity 平均行数 | 5000~50 万 | 小对象几千行，核心对象百万级 |
| 大租户核心对象 | 500 万行/对象 | account/contact/opportunity |
| 总行数估算 | 100~500 亿行 | |
| 单行大小 | 1~2 KB（659 列，大部分 NULL） | |

### 8.2 分表策略

预建 2000 张结构相同的表（p_tenant_data_0 ~ p_tenant_data_1999），按**租户+entity**粒度动态分配表槽位，分配时感知目标表的数据量，优先分配到数据量小的表。

```
表池: p_tenant_data_0 ~ p_tenant_data_1999（2000 张，预建）
分配粒度: 每个 (tenant_id, entity_api_key) 组合分配到一张表
分配策略: 最小数据量优先（Least-Loaded）
路由方式: 查配置表 p_tenant_data_route（Redis 缓存）
```

### 8.3 分配规则

#### 8.3.1 分配时机

当某个租户的某个 entity **首次写入业务数据**时，触发表槽位分配：

```
首次写入(tenantId, entityApiKey):
    │
    ├─ 1. 查路由缓存
    │     route = RouteCache.get(tenantId, entityApiKey)
    │     if route != null → 直接使用，跳到步骤 4
    │
    ├─ 2. 查路由配置表
    │     route = p_tenant_data_route WHERE tenant_id=? AND entity_api_key=?
    │     if route != null → 写入缓存，跳到步骤 4
    │
    ├─ 3. 分配新槽位（加分布式锁）
    │     tableIndex = TableAllocator.allocate(tenantId, entityApiKey)
    │     INSERT INTO p_tenant_data_route (tenant_id, entity_api_key, table_index)
    │     写入缓存
    │
    └─ 4. 路由到目标表
          DynamicTableNameHolder.executeWith("p_tenant_data_" + tableIndex, () -> { ... })
```

#### 8.3.2 分配算法（Least-Loaded）

```
TableAllocator.allocate(tenantId, entityApiKey):
    │
    ├─ 1. 获取表数据量统计（从 p_tenant_data_stats 读取，定时更新）
    │     stats = [
    │       {tableIndex: 0, rowCount: 12340000},
    │       {tableIndex: 1, rowCount: 8920000},
    │       ...
    │       {tableIndex: 1999, rowCount: 5610000}
    │     ]
    │
    ├─ 2. 选择数据量最小的表
    │     候选表 = stats 按 rowCount 升序排列
    │     选取 rowCount 最小的前 10 张表
    │     从中随机选 1 张（避免并发时全部分配到同一张表）
    │
    └─ 3. 返回 tableIndex
```

#### 8.3.3 同一租户的 entity 分配策略

同一租户的不同 entity **可以分配到不同的表**。原因：

- 不同 entity 的数据量差异巨大（account 500 万行 vs 自定义对象 1000 行）
- 如果同一租户所有 entity 绑定到一张表，大 entity 会拖垮小 entity 的查询性能
- 按 (tenant_id, entity_api_key) 粒度分配，每个组合独立选择最优表

但同一租户的**同一 entity 的所有数据**必须在同一张表内（列表查询不跨表）。

### 8.4 路由配置表

```sql
CREATE TABLE p_tenant_data_route (
    id              BIGINT       NOT NULL,
    tenant_id       BIGINT       NOT NULL,
    entity_api_key  VARCHAR(255) NOT NULL,
    table_index     INT          NOT NULL,   -- 0~1999
    created_at      BIGINT       DEFAULT NULL,
    PRIMARY KEY (id),
    UNIQUE INDEX idx_route_tid_eak (tenant_id, entity_api_key),
    INDEX idx_route_table (table_index)
);
```

> 这张表本身数据量 = 租户数 × 平均 entity 数 = 10 万 × 30 = 300 万行，很小。

### 8.5 表数据量统计表

```sql
CREATE TABLE p_tenant_data_stats (
    table_index     INT          NOT NULL,   -- 0~1999
    row_count       BIGINT       NOT NULL DEFAULT 0,
    updated_at      BIGINT       DEFAULT NULL,
    PRIMARY KEY (table_index)
);
```

更新策略：

| 方式 | 频率 | 说明 |
|---|---|---|
| 定时全量统计 | 每日凌晨 | `SELECT table_index, COUNT(*) FROM p_tenant_data_route GROUP BY table_index` 结合各表实际行数 |
| 写入时增量更新 | 实时 | INSERT 时 `UPDATE p_tenant_data_stats SET row_count = row_count + 1 WHERE table_index = ?` |
| 删除时增量更新 | 实时 | DELETE 时 `row_count = row_count - 1` |

> 增量更新保证分配算法拿到的数据量是近实时的，避免新租户持续分配到已经很大的表。

### 8.6 路由流程（完整）

```
业务数据请求(tenantId, entityApiKey):
    │
    ├─ 1. 查 Redis 缓存
    │     key = "td_route:{tenantId}:{entityApiKey}"
    │     tableIndex = Redis.get(key)
    │     if tableIndex != null → 跳到步骤 3
    │
    ├─ 2. 查配置表 + 可能触发分配
    │     route = SELECT table_index FROM p_tenant_data_route
    │               WHERE tenant_id=? AND entity_api_key=?
    │     if route == null:
    │       tableIndex = TableAllocator.allocate(tenantId, entityApiKey)  // 首次分配
    │       INSERT INTO p_tenant_data_route (...)
    │     else:
    │       tableIndex = route.table_index
    │     Redis.set(key, tableIndex, TTL=1h)
    │
    └─ 3. 执行 SQL
          tableName = "p_tenant_data_" + tableIndex
          DynamicTableNameHolder.executeWith(tableName, () -> {
              // SQL 路由到 p_tenant_data_42
          })
```

### 8.7 容量规划

| 指标 | 计算 | 结果 |
|---|---|---|
| 总表数 | 固定 | 2000 张 |
| 路由记录数 | 10 万租户 × 30 entity | ~300 万条 |
| 平均每表租户×entity 组合 | 300 万 / 2000 | ~1500 组合/表 |
| 平均每表数据量 | 总行数 / 2000 | 5000 万 ~ 2.5 亿行/表 |
| 单表存储 | 1 亿行 × 1.5 KB | ~150 GB/表（均值） |
| 总存储 | 2000 × 150 GB | ~300 TB |

> Least-Loaded 分配保证各表数据量均匀，不会出现某些表特别大、某些表特别空的情况。
> MySQL 单表 2~3 亿行在合理索引下可以工作，但建议通过监控在单表超过 5 亿行前扩容表数量。

| idx_tid_eak_name | tenant_id, entity_api_key, name(191) | 按名称 |
| idx_tid_eak_busitype | tenant_id, entity_api_key, busitype_api_key | 按业务类型 |

动态索引：每个 entity 最多 10 个，每张表最多 50 个。

### 8.11 扩容方案

当 2000 张表不够用时（单表平均超过 5 亿行）：

```
扩容流程:
    │
    ├─ 1. 新建表 p_tenant_data_2000 ~ p_tenant_data_2999（扩容 1000 张）
    │
    ├─ 2. 初始化统计表（新表 row_count = 0）
    │
    ├─ 3. 新的 (tenant_id, entity_api_key) 组合自动分配到新表
    │     （因为新表 row_count 最小，Least-Loaded 算法自然优先选择）
    │
    └─ 4. 已有数据不迁移（存量路由不变）
```

> 扩容只需要建新表 + 扩展统计表，不需要迁移任何数据，不需要修改已有路由。
> 这是 Least-Loaded 动态分配的核心优势——扩容对存量零影响。

### 8.12 租户注销

```
注销租户(tenantId):
    │
    ├─ 1. 查询该租户的所有路由
    │     routes = SELECT * FROM p_tenant_data_route WHERE tenant_id = ?
    │
    ├─ 2. 逐表清理数据
    │     for route in routes:
    │       DELETE FROM p_tenant_data_{route.table_index}
    │         WHERE tenant_id = ? AND entity_api_key = route.entity_api_key
    │       UPDATE p_tenant_data_stats SET row_count = row_count - 删除行数
    │
    ├─ 3. 删除路由记录
    │     DELETE FROM p_tenant_data_route WHERE tenant_id = ?
    │
    └─ 4. 清除缓存
          Redis.deletePattern("td_route:{tenantId}:*")
```

### 8.13 跨 entity 查询（ROLLUP）

同一租户的不同 entity 可能在不同的表中，ROLLUP 汇总需要跨表查询：

```
ROLLUP 计算(tenantId, parentEntityApiKey, childEntityApiKey):
    │
    ├─ 1. 查子对象路由
    │     childRoute = RouteCache.get(tenantId, childEntityApiKey)
    │     childTable = "p_tenant_data_" + childRoute.tableIndex
    │
    ├─ 2. 在子对象表中执行汇总
    │     SELECT SUM(dbc_decimal1) FROM {childTable}
    │       WHERE tenant_id=? AND entity_api_key=childEntityApiKey
    │         AND {关联字段}=父记录ID AND delete_flg=0
    │
    ├─ 3. 查父对象路由
    │     parentRoute = RouteCache.get(tenantId, parentEntityApiKey)
    │     parentTable = "p_tenant_data_" + parentRoute.tableIndex
    │
    └─ 4. 回写父记录
          UPDATE {parentTable} SET {rollupDbColumn} = #{result}
            WHERE id=父记录ID AND tenant_id=?
```

> 跨表但不跨库，通过两次路由查询解决，不需要 JOIN。

## 10. 计算字段处理

### 10.1 计算字段分类

| 类别 | 触发时机 | 存储方式 | 示例 |
|---|---|---|---|
| 持久化公式 | 依赖字段变更时 | 写入 dbColumn | `totalAmount = price * quantity` |
| 实时公式 | 每次读取时 | 不占 dbColumn | `daysSinceCreated = NOW() - createdAt` |
| 汇总累计 | 子记录变更时 | 写入 dbColumn | `orderCount = COUNT(子订单)` |

### 10.2 持久化公式计算流程

```
字段变更触发:
    │
    ├─ 1. 查询依赖关系
    │     SELECT * FROM computeFactor
    │       WHERE factorEntityApiKey=? AND factorItemApiKey IN (变更字段列表)
    │     → 找到依赖这些字段的计算字段
    │
    ├─ 2. 获取公式定义
    │     formulaCompute = 从元数据获取公式表达式
    │
    ├─ 3. 执行计算
    │     result = FormulaEngine.evaluate(formula, currentRecord)
    │
    └─ 4. 回写结果
          UPDATE p_tenant_data SET {formulaDbColumn} = #{result}
            WHERE id=? AND tenant_id=?
```

### 10.3 汇总累计计算流程

```
子记录变更触发:
    │
    ├─ 1. 查询汇总定义
    │     aggregationCompute WHERE aggregateEntityApiKey = 子对象entityApiKey
    │
    ├─ 2. 确定父记录
    │     通过 entityLink 找到父记录 ID
    │
    ├─ 3. 执行汇总 SQL
    │     SELECT COUNT(*) / SUM(dbc_xxx) / MAX(dbc_xxx) / MIN(dbc_xxx)
    │       FROM p_tenant_data
    │       WHERE tenant_id=? AND entity_api_key=子对象
    │         AND {关联字段} = 父记录ID
    │         AND delete_flg=0
    │         AND {过滤条件}  -- aggregationComputeDetail
    │
    └─ 4. 回写父记录
          UPDATE p_tenant_data SET {rollupDbColumn} = #{result}
            WHERE id=父记录ID AND tenant_id=?
```

## 12. 业务数据迁移（老→新）

### 12.1 迁移范围

| 老表 | 新表 | 数据量（估算） | 迁移策略 |
|---|---|---|---|
| a_account | p_tenant_data (entity_api_key='account') | ~500 万/租户 | 按租户分批 |
| a_contact | p_tenant_data (entity_api_key='contact') | ~300 万/租户 | 按租户分批 |
| a_opportunity | p_tenant_data (entity_api_key='opportunity') | ~200 万/租户 | 按租户分批 |
| a_{custom_entity} | p_tenant_data (entity_api_key='{apiKey}') | 不定 | 按租户分批 |

### 12.2 列映射转换

迁移时需要将老列名转换为新列名，映射规则：

```
老列 → 新列 转换(entityApiKey, oldDbColumn):
    │
    ├─ 1. 查询 item 元数据（已迁移到新系统）
    │     item = 从新系统元数据中查找 entityApiKey + 老dbColumn 对应的 item
    │     newDbColumn = item.dbColumn  // 新系统分配的列名
    │
    ├─ 2. 固定列映射
    │     id → id（不变）
    │     tenant_id → tenant_id（不变）
    │     entity_type → busitype_api_key（ID→apiKey 转换）
    │     del_flg → delete_flg（列名变更）
    │     dim_depart → depart_id（列名变更）
    │     {entity}_name → name（统一列名）
    │     workflow_stage_name → workflow_stage（列名变更）
    │
    ├─ 3. 扩展列映射
    │     dbc_varchar_N → dbc_varcharM（列号可能变化）
    │     dbc_integer_N → dbc_bigintM（前缀变更 + 列号变化）
    │     dbc_real_N → dbc_decimalM（前缀变更 + 列号变化）
    │     dbc_select_N → dbc_varcharM（SELECT 从数字编码改为 apiKey 字符串）
    │     dbc_tinyint_N → dbc_smallintM（前缀变更 + 列号变化）
    │     dbc_date_N → dbc_bigintM（前缀变更 + 列号变化）
    │     dbc_relation_N → dbc_bigintM（前缀变更 + 列号变化）
    │     dbc_svarchar_N → dbc_varcharM（前缀合并 + 列号变化）
    │     dbc_textarea_N → dbc_textareaM（列号可能变化）
    │
    └─ 4. 值转换
          ├─ SELECT(老 dbc_select_N 存数字编码) → 新 dbc_varcharM 存 pickOption.apiKey
          ├─ entity_type(老存 busiType ID) → busitype_api_key 存 busiType.apiKey
          └─ 其他字段值不变
```

### 12.3 迁移执行步骤

```
Step 1: 准备阶段
  ├─ 确认元数据迁移已完成（item.dbColumn 已分配新列号）
  ├─ 生成列映射表: old_column → new_column（每个 entity 独立）
  └─ 生成值转换表: SELECT 数字编码 → pickOption.apiKey

Step 2: 全量迁移（按租户分批）
  for tenantId in allTenants:
    for entityApiKey in allEntities:
      ├─ 读取老表: SELECT * FROM a_{entity} WHERE tenant_id=?
      ├─ 列映射 + 值转换
      ├─ 写入新表: INSERT INTO p_tenant_data (...)
      └─ 记录迁移进度

Step 3: 增量同步（双写期间）
  ├─ 老系统写入时同步写入新表（通过 MQ 异步）
  └─ 定时对账: 比对新老表数据一致性

Step 4: 切换
  ├─ 停止老系统写入
  ├─ 最终增量同步
  ├─ 切换读写到新表
  └─ 老表保留 30 天后归档
```

## 17. 数据表拓扑设计（核心章节）

> 本章回答核心问题：需要多少张大宽表、如何路由、如何建索引、数据如何分布。

### 17.1 设计决策：预建分表 + 动态分配

预建 2000 张结构相同的表（p_tenant_data_0 ~ p_tenant_data_1999），按 (tenant_id, entity_api_key) 粒度动态分配到数据量最小的表。

### 17.2 表拓扑全景图

```
paas_entity_data 数据库
│
├── p_tenant_data_0          ← 业务数据表（结构相同，685 列）
├── p_tenant_data_1
├── p_tenant_data_2
├── ...
├── p_tenant_data_1999       ← 共 2000 张
│
├── p_tenant_data_route      ← 路由配置（tenant_id + entity_api_key → table_index）
├── p_tenant_data_stats      ← 表数据量统计（table_index → row_count）
│
└── (扩容时新建 p_tenant_data_2000 ~ p_tenant_data_2999)
```

### 17.3 路由规则

```
路由(tenantId, entityApiKey):
  1. 查缓存/配置表 p_tenant_data_route
  2. 命中 → p_tenant_data_{table_index}
  3. 未命中 → Least-Loaded 分配最小数据量的表 → 写入路由 → 返回
```

- 同一 (tenant_id, entity_api_key) 的数据始终在同一张表内
- 同一租户的不同 entity 可以在不同表中
- 路由结果持久化 + Redis 缓存，查询零计算开销

### 17.4 索引规则

每张表 8 个固定索引（见 §8.10），动态索引每 entity 最多 10 个、每表最多 50 个。

### 17.5 与元数据层的关系

| 元数据字段 | 业务数据层的使用方式 |
|---|---|
| entity.apiKey | 路由配置键 + 填充 entity_api_key 列 |
| item.dbColumn | 决定业务字段写入哪个 dbc 列 |
| item.itemType | 决定字段值的类型转换规则 |
| item.indexType | 决定是否创建动态索引 |
| busiType.apiKey | 填充 busitype_api_key 列 |
| entityLink | 决定关联查询和级联删除规则 |

### 17.6 设计决策记录

| 决策 | 选项 | 选择 | 理由 |
|---|---|---|---|
