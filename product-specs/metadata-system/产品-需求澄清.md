# aPaaS 元数据体系 — 需求澄清

## 1. 需求概述

| 维度 | 内容 |
|:---|:---|
| 需求名称 | aPaaS 元数据驱动平台体系建设 |
| 需求来源 | 平台架构升级 |
| 需求类型 | 平台基础设施 |
| 一句话描述 | 构建以元模型为核心的三层元数据架构（元模型定义→元数据实例→业务数据），实现业务对象的声明式配置与动态扩展 |

## 2. 核心痛点

### 2.1 当前问题
1. 老系统业务对象硬编码，新增对象需要建表、写代码、发版，周期长（2-4 周）
2. 字段类型编码不统一（老编码 23/24/29 等与新 ItemTypeEnum 不一致），导致前后端映射混乱
3. 元数据关联使用 ID（entity_id、item_id），跨环境迁移时 ID 不一致导致数据断裂
4. 缺乏 Common/Tenant 分层机制，出厂元数据与租户自定义元数据混存，无法独立升级
5. 缺乏元模型级别的 Schema 定义，字段语义散落在代码中，无法自描述

### 2.2 量化影响
- 新增业务对象：从 2-4 周 → 目标 10 分钟（配置即生效）
- 字段类型映射错误：当前月均 5-8 次前端渲染异常 → 目标 0
- 跨环境迁移失败率：当前 15% → 目标 <1%（api_key 关联）
- 出厂元数据升级影响租户：当前每次升级需全量回归 → 目标增量覆盖

## 3. 目标用户与使用场景

| 用户角色 | 使用场景 | 核心诉求 |
|:---|:---|:---|
| 平台开发者 | 注册新元模型、定义字段映射 | 通过 p_meta_model + p_meta_item 声明式定义，无需建表 |
| 产品配置师 | 配置业务对象、字段、校验规则 | 通过 metarepo-web 可视化管理元数据 |
| 租户管理员 | 自定义对象、字段、选项值 | Tenant 级独立存储，不影响其他租户 |
| 业务开发者 | 基于元数据开发业务功能 | 通过 MetaRepoReadApi 获取元数据，驱动业务逻辑 |

## 4. 涉及现有模块

| 模块 | 影响范围 |
|:---|:---|
| paas-metarepo-service | 核心改造：元模型四表 + 大宽表 + Common/Tenant 合并 |
| paas-metadata-service | 适配新元数据读取接口 |
| paas-entity-service | 基于新元数据驱动实体数据 CRUD |
| paas-layout-service | 布局渲染依赖元数据字段定义 |
| paas-rule-service | 校验规则执行依赖 checkRule 元数据 |
| paas-metarepo-web | 新建：元数据可视化浏览与管理前端 |
