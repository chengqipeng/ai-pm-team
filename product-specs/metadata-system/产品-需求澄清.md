# aPaaS 元数据驱动平台体系建设 — 需求澄清

> 版本：v2.0 | 日期：2026-03-31

## 需求概述
构建以元模型为核心的三层元数据架构（元模型定义→元数据实例→业务数据），实现业务对象的声明式配置与动态扩展，替代硬编码建表模式。

## 核心痛点
| 维度 | 现状 | 痛点 | 量化影响 |
|:---|:---|:---|:---|
| 对象扩展 | 新增业务对象需建表、写代码、发版 | 周期长，无法快速响应业务需求 | 2-4 周/对象 → 目标 10 分钟 |
| 字段类型 | 老编码（23/24/29 等）与新 ItemTypeEnum 不一致 | 前后端映射混乱，渲染异常 | 月均 5-8 次前端异常 |
| 数据关联 | 使用 ID 关联（entity_id、item_id） | 跨环境迁移时 ID 不一致导致数据断裂 | 迁移失败率 15% |
| 租户隔离 | 出厂元数据与租户自定义混存 | 无法独立升级，每次升级需全量回归 | 升级影响全部租户 |
| Schema 定义 | 字段语义散落在代码中 | 无法自描述，新人理解成本高 | 无统一元模型定义 |

## 量化目标
| 指标 | 当前值 | 目标值 | 衡量方式 |
|:---|:---:|:---:|:---|
| 新增业务对象耗时 | 2-4 周 | 10 分钟 | 从配置到可用的端到端时间 |
| 字段类型映射异常 | 月均 5-8 次 | 0 | 前端渲染异常工单数 |
| 跨环境迁移失败率 | 15% | <1% | api_key 关联后迁移成功率 |
| 出厂数据升级影响 | 全量回归 | 增量覆盖 | Common/Tenant 独立升级 |
| 元数据列表查询 | 未统计 | P95 < 200ms | APM 监控 |
| 元数据合并读取 | 未统计 | P95 < 500ms | APM 监控 |

## 目标用户与场景
| 角色 | 使用场景 | 频率 | 核心诉求 |
|:---|:---|:---|:---|
| 平台开发者 | 注册新元模型、定义字段映射 | 低频（新增元模型时） | 声明式定义，无需建表 |
| 产品配置师 | 配置业务对象、字段、校验规则 | 中频（产品迭代时） | 可视化管理元数据 |
| 租户管理员 | 自定义对象、字段、选项值 | 中频（业务调整时） | Tenant 级独立存储，不影响其他租户 |
| 业务开发者 | 基于元数据开发业务功能 | 高频（日常开发） | 通过 API 获取元数据驱动业务逻辑 |

## 范围界定

**做什么：**
- [x] 元模型四表体系（p_meta_model / p_meta_item / p_meta_link / p_meta_option）
- [x] 大宽表存储机制（p_common_metadata + p_tenant_* 独立快捷表）
- [x] Common/Tenant 合并读取引擎（CommonMetadataConverter + MergeReadService）
- [x] 元数据 CRUD API（MetaRepoReadApi + MetaRepoWriteApi）
- [x] 元模型浏览内部 API（MetamodelBrowseApiService）
- [x] 元数据管理前端（metarepo-web）
- [x] 数据迁移（item_type 编码转换 + db_column 重分配 + api_key 关联标准化）
- [x] DynamicTableNameHolder 路由机制

**不做什么：**
- [ ] 计算字段子元模型恢复（formulaCompute 等 5 个）— 原因：P1 优先级，当前迭代不含
- [ ] Delta 增量覆盖机制 — 原因：P2 优先级，字段已预留但逻辑未实现
- [ ] Module 打包分发 — 原因：P2 优先级，依赖 Delta 机制
- [ ] 业务数据层（entity-service 的实体数据 CRUD）— 原因：属于上层消费方，非本需求范围

## 约束条件
- 技术约束：DDL 必须同时兼容 MySQL 和 PostgreSQL；主键统一雪花算法 BIGINT
- 业务约束：Common 级数据由平台初始化写入，业务层不直接写入 Common 表
- 数据约束：老系统 23,819 条 item 数据需平滑迁移，不可中断业务

## 决策建议
**决策**: [x] 推进  [ ] 调整  [ ] 放弃
**理由**: 元数据体系是 aPaaS 平台的"操作系统"，所有上层业务能力（CRM、AI、工作流等）均依赖此基座。核心架构已实现，当前重点是数据迁移、Schema 校验完善和管理前端交付。
