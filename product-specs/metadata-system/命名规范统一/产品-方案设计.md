# 元模型字段命名规范统一 — 方案设计

> 版本：v1.0 | 日期：2026-03-31
> 前置文档：[需求澄清](产品-需求澄清.md)
> 需求类型：`架构重构` | 特征标签：`数据模型变更`

## 一句话方案
制定统一的三层命名规范（p_meta_item api_key → Java 字段名 → API 字段名），全量梳理并修正 6 种元模型中所有不规范字段。

---

## 命名规范定义

### 规则 1：p_meta_item.api_key 统一使用 camelCase

| 规则 | 说明 | 正确示例 | 错误示例 |
|:---|:---|:---|:---|
| 统一 camelCase | 所有 api_key 使用 Java 风格驼峰 | `enableBusinessType` | `enable_busitype` |
| 与 Java 字段名一致 | api_key 值 = Java Entity 字段名 | `entityApiKey` | `entity_api_key`（作为 api_key 值时） |
| 禁止缩写 | 除公认缩写（Api/Id/Url/Svg）外不缩写 | `enableBusinessType` | `enableBusitype` |

> 注意：db_column 仍然使用 snake_case（如 `entity_api_key`、`dbc_varchar1`），这是数据库列名规范，与 api_key 命名无关。

### 规则 2：布尔字段统一使用 `xxxFlg` 后缀（Integer 0/1）

| 规则 | 说明 | 正确示例 | 错误示例 |
|:---|:---|:---|:---|
| 统一 `Flg` 后缀 | 所有 0/1 布尔字段使用 `xxxFlg` | `enableFlg`、`deleteFlg` | `isActive`、`isDefault` |
| 禁止 `is` 前缀 | Java Integer 类型不用 `is` 前缀 | `defaultFlg` | `isDefault` |
| `enable` 前缀用于开关 | 表示"是否启用某能力" | `enableHistoryLog` | `enableHistoryLogFlg`（不重复加 Flg） |
| `enable` 本身就是布尔语义 | `enableXxx` 不需要再加 `Flg` | `enableTeam` | `enableTeamFlg` |

**布尔字段命名决策树：**

```
字段语义是"是否启用某能力"？
  ├─ 是 → enableXxx（不加 Flg）
  │       例：enableTeam, enableHistoryLog, enableCheckRule
  └─ 否 → 字段语义是"某种状态/标记"？
       ├─ 是 → xxxFlg
       │       例：deleteFlg, customFlg, requireFlg, activeFlg, hiddenFlg
       └─ 否 → 使用具体语义名
               例：searchable, creatable, updatable, cascadeDelete
```

### 规则 3：复合词正确驼峰

| 规则 | 正确示例 | 错误示例 |
|:---|:---|:---|
| 每个单词首字母大写 | `enableCheckRule` | `enableCheckrule` |
| 缩写词视为一个单词 | `entityApiKey` | `entityAPIKey` |
| 公认缩写保持 | `svgApiKey`、`dbTable`、`dbColumn` | — |

### 规则 4：关联字段命名

| 规则 | 正确示例 | 错误示例 |
|:---|:---|:---|
| 单个关联用 `xxxApiKey` | `entityApiKey`、`referEntityApiKey` | `entityId` |
| 多个关联用 `xxxApiKeys`（复数） | `referEntityApiKeys` | `referEntityApiKey`（存多个时） |
| 关联字段不用 `refer` 前缀（除非是 LOOKUP 语义） | `parentEntityApiKey` | `referParentEntityApiKey` |

### 规则 5：p_meta_item api_key 与 Java 字段名的对应关系

| 层级 | 命名风格 | 示例 |
|:---|:---|:---|
| p_meta_item.api_key | camelCase | `enableBusinessType` |
| Java Entity 字段名 | camelCase（与 api_key 一致） | `enableBusinessType` |
| Java API 模型（X 前缀）字段名 | camelCase（与 api_key 一致） | `enableBusinessType` |
| p_meta_item.db_column | snake_case 或 dbc_xxxN | `dbc_smallint1` |
| 数据库物理列名 | snake_case | `entity_api_key` |

## 核心流程

### 执行顺序

| 步骤 | 操作 | 影响范围 | 回滚方案 |
|:---|:---|:---|:---|
| 1 | 备份 p_meta_item 表 | — | — |
| 2 | 执行 p_meta_item api_key UPDATE SQL | ~30 条记录 | 恢复备份 |
| 3 | 更新 seed SQL 文件（seed_meta_item_entity.sql 等） | SQL 文件 | Git revert |
| 4 | 重命名 Java Entity 字段（Entity/EntityItem/EntityLink） | 3 个 Java 文件 | Git revert |
| 5 | 重命名 API 模型字段（XEntity/XEntityItem/XLink） | 3 个 Java 文件 | Git revert |
| 6 | 更新 MetaRepoConverter 映射 | 1 个 Java 文件 | Git revert |
| 7 | 更新设计文档（METADATA-SPEC.md、METAREPO-METAMODEL-DESIGN.md） | 2 个 MD 文件 | Git revert |
| 8 | 前端适配（如有直接引用旧字段名） | metarepo-web | Git revert |
| 9 | 验证：启动服务 + 调用 API + 检查字段名 | — | — |

### 前端兼容策略

API 字段名变更会影响前端。建议：
- 方案 A（推荐）：前端同步改名，一次性切换（metarepo-web 是内部管理工具，用户量小）
- 方案 B：后端 API 层做兼容映射（旧名→新名），设置 2 个迭代的过渡期

## 数据需求

本需求不新增实体/字段/表，仅修改现有 p_meta_item 的 api_key 值和 Java 字段名。

影响的数据：
- p_meta_item 表：~30 条记录的 api_key 值更新
- p_common_metadata 表：无影响（dbc_xxxN 列名不变，api_key 列存储的是元数据实例的 apiKey，不是元模型字段的 apiKey）
- p_tenant_* 表：无影响（同上）
