# 元模型字段命名规范统一 — 需求澄清

> 版本：v1.0 | 日期：2026-03-31

## 需求概述
统一 p_meta_item 中所有元模型字段的 api_key 命名风格，以及对应 Java Entity/API 模型类的属性命名，消除当前多种命名风格混用导致的认知负担和维护成本。

## 核心痛点
| 维度 | 现状 | 痛点 | 量化影响 |
|:---|:---|:---|:---|
| 布尔字段后缀 | `enableFlg`/`detailFlg`/`activeFlg`/`maskFlg`/`customFlg` 混用 `Flg` 后缀 | 语义不清：`Flg` 是老系统遗留缩写，新代码应使用完整单词 | 6 种元模型共 40+ 个布尔字段风格不统一 |
| 布尔字段前缀 | `isDefault`/`isGlobal`/`isCustom`/`isActive`/`isDeleted`（pickOption/referFilter）vs 无前缀 | 同一语义在不同元模型中前缀不一致 | pickOption 用 `is` 前缀，其他元模型不用 |
| 缩写不一致 | `enableBusitype`（Busi）vs `batchCreateLinkByBusitype` | 缩写 `Busi` 不标准，应为 `BusinessType` 或 `BizType` | entity + item 各 1 处 |
| 复合词拼写 | `enableCheckrule`/`enableDuplicaterule` 未正确驼峰 | 应为 `enableCheckRule`/`enableDuplicateRule` | entity 2 处 |
| p_meta_item api_key 风格 | entity 用 `snake_case`（enable_busitype），checkRule/pickOption 用 `camelCase`（entityApiKey） | 同一张表内两种风格混用 | 跨元模型不一致 |
| Java 字段与 api_key 不对应 | Java: `enableBusitype` vs SQL api_key: `enable_busitype` | 转换逻辑需要特殊处理 | 增加 Converter 复杂度 |
| `is` 前缀的 Java 陷阱 | `isComputeMultiCurrencyUnit` 类型是 String 却用 `is` 前缀 | Lombok getter 生成 `isIsComputeMultiCurrencyUnit` 或丢失前缀 | 1 处，但模式有扩散风险 |

## 量化目标
| 指标 | 当前值 | 目标值 | 衡量方式 |
|:---|:---:|:---:|:---|
| 命名风格种类 | 5+ 种混用 | 1 种统一规范 | 代码审查 |
| 不规范字段数 | ~60 个 | 0 | 自动化检查脚本 |
| Converter 特殊映射数 | 多处 | 最小化 | 代码行数 |

## 目标用户与场景
| 角色 | 使用场景 | 频率 | 核心诉求 |
|:---|:---|:---|:---|
| 平台开发者 | 新增元模型字段时选择命名 | 中频 | 有明确规范可遵循，不用猜 |
| 代码审查者 | Review 新增字段命名 | 中频 | 有规范可对照，快速判断 |
| 前端开发者 | 使用 API 返回的字段名 | 高频 | 字段名可预测，无需查文档 |

## 范围界定

**做什么：**
- [x] 制定统一的字段命名规范（覆盖 api_key、Java 字段名、db_column）
- [x] 全量梳理 6 种元模型的所有不规范字段，输出改名映射表
- [x] 修改 Java Entity 类字段名
- [x] 修改 API 模型类（X 前缀）字段名
- [x] 修改 MetaRepoConverter 转换映射
- [x] 修改 p_meta_item seed SQL 中的 api_key
- [x] 更新设计文档（METADATA-SPEC.md、METAREPO-METAMODEL-DESIGN.md）

**不做什么：**
- [ ] 修改 p_common_metadata / p_tenant_* 大宽表的物理列名（dbc_xxxN）— 原因：物理列名与业务语义无关，无需改
- [ ] 修改 p_meta_model 的 api_key（entity/item/entityLink 等）— 原因：已规范
- [ ] 修改基类字段（BaseMetaCommonEntity/BaseMetaTenantEntity）— 原因：框架层，影响面过大

## 约束条件
- 技术约束：改名需同步更新 p_common_metadata 中已有的元数据实例（UPDATE api_key）
- 兼容约束：对外 API 字段名变更需评估前端影响，可能需要兼容期
- 数据约束：p_meta_item 的 api_key 是 CommonMetadataConverter 的映射依据，改名必须同步更新

## 决策建议
**决策**: [x] 推进  [ ] 调整  [ ] 放弃
**理由**: 命名不一致会随着元模型数量增长持续恶化。当前 7 种元模型 176 个字段已有 ~60 个不规范，越早统一成本越低。建议与数据迁移迭代合并执行。
