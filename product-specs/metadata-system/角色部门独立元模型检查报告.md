## 二、设计层面检查详情

### 2.1 元模型注册层（p_meta_model）

Role 和 Department 各自作为独立的元模型注册在 `paas_metarepo_common.p_meta_model` 中，与 entity、item、checkRule 等元模型同级：

| 属性 | Role | Department |
|:---|:---|:---|
| api_key | `role` | `department` |
| db_table | `p_tenant_role` | `p_tenant_department` |
| namespace | system | system |
| enable_common | 0（仅 Tenant 级） | 0（仅 Tenant 级） |
| enable_tenant | 1 | 1 |
| entity_dependency | 0（不依赖 entity） | 0（不依赖 entity） |

关键点：`entity_dependency=0` 表明两者不依赖 entity 元模型，是真正独立的顶级元模型。

### 2.2 字段定义层（p_meta_item）

Role 有 6 个业务字段，Department 有 10 个业务字段，均通过 `metamodel_api_key` 关联到各自的元模型（而非关联到 entity）：

**Role 字段（6 个）：**

| api_key | db_column | label | itemType |
|:---|:---|:---|:---|
| roleCode | dbc_varchar1 | 角色编码 | TEXT |
| roleParentId | dbc_bigint1 | 上级角色 | RELATION_SHIP |
| roleLevel | dbc_bigint2 | 层级深度 | NUMBER |
| roleSortOrder | dbc_bigint3 | 排序号 | NUMBER |
| roleStatus | dbc_smallint1 | 状态 | BOOLEAN |
| roleDescriptionKey | dbc_varchar2 | 描述Key | TEXT |

**Department 字段（10 个）：**

| api_key | db_column | label | itemType |
|:---|:---|:---|:---|
| departName | dbc_varchar1 | 部门名称 | TEXT |
| departLevel | dbc_bigint2 | 部门层级 | NUMBER |
| departPath | dbc_varchar2 | 部门路径 | TEXT |
| managerId | dbc_bigint3 | 部门负责人 | RELATION_SHIP |
| sortOrder | dbc_bigint4 | 排序 | NUMBER |
| enableFlg | dbc_smallint1 | 启用状态 | BOOLEAN |
| deptCode | dbc_varchar3 | 部门编码 | TEXT |
| deptParentId | dbc_bigint1 | 上级部门 | RELATION_SHIP |
| deptStatus | dbc_smallint2 | 状态 | BOOLEAN |
| deptDescriptionKey | dbc_varchar4 | 描述Key | TEXT |

### 2.3 数据存储层

两者的数据存储在 `paas_metarepo` 库的独立快捷表中，表结构与 `p_common_metadata` 大宽表一致（固定列 + dbc_xxxN 扩展列 + tenant_id）：

| 表 | 库 | 当前数据量 |
|:---|:---|:---|
| p_tenant_role | paas_metarepo | 7 条（2 租户） |
| p_tenant_department | paas_metarepo | 10 条（2 租户） |

### 2.4 架构决策记录

`context/projects/metadata-system/project-context.md` 中明确记录了架构决策：

> 2026-04-16 | role/department 注册为独立元模型，数据存在 p_tenant_role/p_tenant_department | 角色和部门是元数据（定义），不是业务数据，应与 checkRule/busiType 同级

## 四、迁移前后对比

### 4.1 迁移前（老架构）

```
Role/Department 注册为 entity 类型（metamodel_api_key='entity'）
  ├── 元模型注册：p_common_metadata（metamodel_api_key='entity', api_key='role'/'department'）
  ├── 字段定义：p_common_metadata（metamodel_api_key='item', entity_api_key='role'/'department'）
  ├── 数据存储：paas_entity_data.p_tenant_data_1（role）/ p_tenant_data_132（department）
  └── Java 代码：EntityDataService.listAll("role") / TableRouteService 查分片表
```

### 4.2 迁移后（新架构）

```
Role/Department 注册为独立元模型（metamodel_api_key='role'/'department'）
  ├── 元模型注册：p_meta_model（api_key='role'/'department', entity_dependency=0）
  ├── 字段定义：p_meta_item（metamodel_api_key='role'/'department'）
  ├── 数据存储：paas_metarepo.p_tenant_role / p_tenant_department
  └── Java 代码：RoleMetadataService / DepartmentServiceImpl（DynamicTableNameHolder 路由）
```

## 六、总结

角色和部门在底层元模型和元数据层面已经是完全独立的模型，具体体现在：

1. **元模型注册独立**：各自在 p_meta_model 中有独立的注册记录，entity_dependency=0
2. **字段定义独立**：各自在 p_meta_item 中有独立的字段定义，metamodel_api_key 分别为 'role' 和 'department'
3. **数据存储独立**：各自有独立的 Tenant 级快捷表（p_tenant_role / p_tenant_department）
4. **代码层独立**：各自有独立的 Service 层（RoleMetadataService / DepartmentServiceImpl），通过 DynamicTableNameHolder 路由
5. **老架构已清理**：老的 entity 注册、老表、老代码均已删除或软删除

唯一需要跟进的是第五节列出的 6 处文档不一致问题，已全部修复完成。同时 `role-department-metadata.md` 中第四节的小节标题也一并从老表名更新为新的快捷表名。
