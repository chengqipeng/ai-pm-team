# aPaaS 元数据驱动平台 — 产品方案设计

> 版本：v4.0 | 日期：2026-03-31
> 技术设计：[元模型设计体系](元模型设计体系.md) | [元数据实例设计](元数据实例设计.md) | [元数据设计规范](元数据设计规范.md) | [数据迁移方案](数据迁移方案.md)

## 产品定位
以元模型为核心，构建"元模型定义→元数据实例→业务数据"三层架构，实现业务对象声明式配置与动态扩展。从"硬编码建表"到"配置即生效"。

## 业务规则

### 元模型管理
- BR-01：api_key 全局唯一，camelCase 格式
- BR-02：必须指定 enable_common 和 enable_tenant（至少一个为 1）
- BR-03：db_table 指向 Tenant 级存储表，格式 p_tenant_{name}
- BR-04：新表结构必须与 p_common_metadata 一致（+ tenant_id）
- BR-05：元模型注册后，可通过 p_meta_item 定义字段，无需建新表

### 字段定义
- BR-06：db_column 三种映射——固定列名 / dbc_xxxN 扩展列 / 特殊映射
- BR-07：固定列优先，dbc 列按数据类型选前缀，同元模型内同前缀递增分配
- BR-08：列名格式统一 dbc_xxxN（无下划线分隔数字）
- BR-09：命名规范详见 [元数据设计规范](元数据设计规范.md) §7

### Common 级数据保护
- BR-10：Common 级数据（p_common_metadata）禁止任何业务层修改，仅平台初始化和 Module 安装可写入
- BR-11：租户对 Common 数据的所有"修改"操作，实质是在 Tenant 层创建覆盖记录，Common 原始数据不变
- BR-12：租户对 Common 数据的"删除"操作，实质是在 Tenant 层插入 delete_flg=1 的遮蔽记录，Common 原始数据不变
- BR-13：Common 层的 dbColumn 永远占位——即使字段被遮蔽删除，该列号不会分配给任何新字段

### 元数据 CRUD
- BR-14：合并规则——Common 有 Tenant 无→用 Common，同 apiKey→Tenant 覆盖 Common，Tenant delete_flg=1→隐藏
- BR-15：namespace=product 需检查 license
- BR-16：所有写操作仅写入 Tenant 级表，禁止直接写入 Common 表
- BR-17：写入前校验 p_meta_option 取值范围 + p_meta_item 必填/唯一约束
- BR-18：级联删除遵循 p_meta_link.cascade_delete 配置

### 同 apiKey 字段的删除与重建
- BR-19：遮蔽删除（delete_flg=1）的字段释放 apiKey 命名空间，允许租户用相同 apiKey 新建字段
- BR-20：禁用（enableFlg=0）的字段不释放 apiKey，不允许同名新建——用户须先删除再新建
- BR-21：同名新建采用"就地替换"机制——UPDATE 已有的遮蔽记录（而非 INSERT），避免唯一约束冲突
- BR-22：就地替换时必须分配新 dbColumn，禁止复用老字段的 dbColumn（老列可能有残留业务数据）
- BR-23：就地替换后标记 customFlg=1、namespace='custom'，与常规覆盖记录区分
- BR-24：就地替换后，老字段的子元数据（pickOption/referenceFilter 等）不自动继承，需重新创建
- BR-25：就地替换与恢复系统字段互斥——已替换的 apiKey 须先删除自定义字段，才能恢复原系统字段
- BR-26：恢复系统字段 = 物理删除 Tenant 遮蔽记录 → Common 层原始数据（含老 dbColumn）自动恢复可见

## 核心流程

> 详细数据流见 [元数据实例设计](元数据实例设计.md) §2-§3，API 接口规格见 §5

**读取：** 请求 → 查 MetaModel 配置 → 查 Common → 查 Tenant → 合并 → namespace/license 过滤 → 返回

**写入（新增）：** 请求 → Schema 校验 → apiKey 唯一性校验（见下方） → Converter 转换 → DynamicTableNameHolder 路由 → INSERT/UPDATE Tenant 表 → 变更日志

**写入（修改 Common 数据）：** 请求 → 禁止直接修改 Common → 复制 Common 记录到 Tenant 层 → 在 Tenant 记录上应用修改 → INSERT Tenant 表（覆盖机制）

**遮蔽删除：** 仅 Common 有→插入 delete_flg=1 的 Tenant 记录（Common 原始数据不变）；Tenant 有记录→软删除；级联删除遵循 p_meta_link.cascade_delete

**同名新建（就地替换）：**
```
新建字段 apiKey 与已遮蔽的 Common 字段相同时:
  1. 合并视图中该 apiKey 不可见（已遮蔽）→ 允许使用该 apiKey
  2. Tenant 层已有遮蔽记录（delete_flg=1）→ 不能 INSERT（唯一约束）
  3. UPDATE 遮蔽记录: delete_flg=0, customFlg=1, namespace='custom',
     写入新 itemType/dbColumn/label 等全部字段属性
  4. 分配新 dbColumn（禁止复用老列号）
  5. Common 层原始数据不变，老 dbColumn 永远占位
```

**恢复系统字段：** 物理删除 Tenant 遮蔽记录 → Common 层数据自动恢复可见（含老 dbColumn 和业务数据）。如果遮蔽记录已被就地替换为自定义字段，须先删除自定义字段再恢复。

## 权限设计

| 角色 | 元模型管理 | Common 元数据 | Tenant 元数据 | 管理前端 |
|:---|:---:|:---:|:---:|:---:|
| 平台管理员 | 读写 | 只读（初始化/Module安装时写入，业务层禁止修改） | 读写 | 全部功能 |
| 租户管理员 | 只读 | 只读（通过覆盖/遮蔽机制间接操作，Common 原始数据不变） | 读写 | 浏览+编辑 |
| 业务开发者 | 只读 | 只读（通过 API） | 只读（通过 API） | 元模型浏览 |

namespace 权限：system→所有租户可见，product→受 license 控制，custom→仅该租户可见

## 安全设计

| 安全风险 | 防护措施 | 优先级 |
|:---|:---|:---:|
| 数据越权（租户间） | Tenant 级表强制 tenant_id 过滤 | P0 |
| namespace 越权 | product 检查 license，custom 检查 tenant_id | P0 |
| Common 数据篡改 | 业务层禁止任何直接写入/修改 Common 表，所有"修改"通过 Tenant 覆盖实现 | P0 |
| Common dbColumn 泄漏 | 遮蔽删除后 Common 层 dbColumn 仍占位，禁止分配给新字段 | P0 |
| SQL 注入 | MyBatis-Plus 参数化查询 | P0 |

## 错误处理与降级

| 错误场景 | 处理策略 | 降级方案 |
|:---|:---|:---|
| 元模型不存在 | 返回 404 | — |
| Schema 校验失败 | 返回 400 + 字段错误列表 | — |
| Common 库查询超时 | 超时 3s | 仅返回 Tenant 数据 |
| Tenant 库查询超时 | 超时 3s | 仅返回 Common 数据 |
| 写入失败 | 事务回滚 + 500 | — |
| 级联删除部分失败 | 整体回滚 | — |
| 变更日志写入失败 | 不影响主流程 | 异步重试 |
| ItemTypeEnum API 加载失败 | 前端回退硬编码 | 功能可用 |

## 验收标准

| 编号 | 验收项 | 验证方法 |
|:---|:---|:---|
| AC-01 | 注册新元模型无需建表即可存储元数据 | 注册→创建→读取验证 |
| AC-02 | 读取接口正确合并 Common + Tenant 数据 | 同 apiKey 覆盖、delete_flg=1 隐藏 |
| AC-03 | 写入接口正确路由到 Tenant 级表，禁止写入 Common 表 | 写入后查询验证，Common 数据无变化 |
| AC-04 | 遮蔽删除和级联删除正确，Common 原始数据不变 | 删除 entity→验证子元数据已删除，Common 层数据不变 |
| AC-05 | Schema 校验拦截非法取值 | 写入非法 itemType→返回错误 |
| AC-06 | item_type 编码全部转换为新 ItemTypeEnum | 查询验证无老编码残留 |
| AC-07 | db_column 格式统一为 dbc_xxxN | 查询验证 |
| AC-08 | 关联字段从 ID 转换为 api_key | 查询验证 |
| AC-09 | 元数据列表查询 P95 < 200ms | APM 监控 |
| AC-10 | namespace=product 受 license 控制 | 无 license 租户不可见 |
| AC-11 | 管理前端正确展示元模型和元数据 | UI 验证 |
| AC-12 | 遮蔽删除系统字段后可用同 apiKey 新建自定义字段 | 删除系统字段→新建同名→读取验证新字段属性正确 |
| AC-13 | 同名新建分配新 dbColumn，不复用老列号 | 验证新字段 dbColumn 与老字段不同，Common 层 dbColumn 仍占位 |
| AC-14 | 同名新建后可删除自定义字段并恢复系统字段 | 删除自定义→恢复→验证系统字段属性和老 dbColumn 恢复 |
| AC-15 | 禁用字段不允许同名新建 | 禁用系统字段→尝试同名新建→返回错误 |
| AC-16 | 修改 Common 数据通过覆盖机制实现，Common 原始数据不变 | 修改系统字段 label→验证 Common 层 label 不变，Tenant 层有覆盖记录 |
