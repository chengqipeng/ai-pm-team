# 多选字段数组类型设计方案

> 版本：v1.0 | 日期：2026-04-23
> 关联文档：[元模型设计体系](元模型设计体系.md) | [业务数据大宽表设计](业务数据大宽表设计.md) | [元数据级联查询引擎设计方案](../frontend-framework/元数据级联查询引擎设计方案.md)

## 一句话方案

将元数据层和业务数据层中所有"逗号分隔字符串表示多值"的字段，统一改为 PG 原生数组类型（`VARCHAR(300)[]`），从存储层根治多值字段的解析问题，为 chain-query 引擎消除 `split`/`json` 等应用层值展开逻辑。

---

## 一、问题定义

### 1.1 当前现状

系统中存在两类多值字段：

**业务数据层**（p_tenant_data 大宽表）和**元数据层**（p_common_metadata / p_tenant_* 大宽表）的多选字段已统一使用 `dbc_array`（`VARCHAR(300)[]`）存储：
- 业务数据层 MULTI_SELECT(3) 类型字段使用 `dbc_array`（VARCHAR(300)[] 数组），30 列
- 元数据层多值字段同样使用 `dbc_array`（VARCHAR(300)[] 数组），10 列

**已消除的历史差异**：早期设计中业务数据层曾规划使用 `dbc_multi`（JSONB），元数据层使用 `dbc_array`（VARCHAR(300)[]），现已统一为 `dbc_array` / `VARCHAR(300)[]`，两层规则一致。

### 1.2 受影响的字段清单

全量盘点系统中所有使用"逗号分隔"存储多值的元数据字段：

| 元模型 | 字段 api_key | 当前 db_column | 当前类型 | 语义 | 影响范围 |
|:---|:---|:---|:---|:---|:---|
| globalPickDependencyDetail | dependentOptionApiKeys | dbc_varchar2 | String | 依赖选项编码列表 | 级联查询核心字段 |
| itemDependencyDetail | dependentOptionApiKeys | dbc_varchar2 | String | 依赖选项编码列表 | 字段级联核心字段 |
| busiTypePickOption | optionApiKeys | dbc_varchar3 | String | 可选选项编码列表 | 业务类型选项过滤 |
| busiTypePickOption | defaultOptionApiKeys | dbc_varchar4 | String | 默认选项编码列表 | 业务类型默认值 |
| item | referEntityApiKeys | dbc_varchar16 | String | 多态引用实体列表 | 多态关联配置 |

共 5 个字段，分布在 4 个元模型中。

### 1.3 为什么不能继续用逗号分隔

1. **查询无法利用索引**：`WHERE 'opt_a' = ANY(array_col)` 可以用 GIN 索引，`LIKE '%opt_a%'` 不行
2. **应用层必须做值展开**：chain-query 引擎需要 `split` 处理器，增加了复杂度和出错概率
3. **数据完整性无保障**：逗号分隔字符串无法约束元素格式，可能出现 `"opt_a,,opt_b"` 或 `"opt_a, opt_b"`（带空格）
4. **前后端转换成本**：前端多选组件天然产出数组，写入时要 `join`，读取时要 `split`，JSON 传输也要额外处理
5. **与业务数据层一致**：业务数据层和元数据层的多选字段现已统一使用 `dbc_array`（VARCHAR(300)[]），两层规则一致

## 二、设计方案

### 2.1 核心决策

**元数据层的多选字段统一使用 PG 原生 `VARCHAR(300)[]` 数组类型**，通过大宽表新增 `dbc_array` 列前缀承载。

选择 `VARCHAR(300)[]` 而非 `JSONB` 的理由：
- 元数据层的多值字段都是简单字符串列表（apiKey 列表、编码列表），不需要 JSONB 的嵌套/查询能力
- 与 `dbc_varchar` 列的 `VARCHAR(300)` 类型约束一致，每个数组元素最长 300 字符
- `VARCHAR(300)[]` 的 `ANY` / `&&` / `@>` 运算符语义更直接，SQL 可读性更好
- `VARCHAR(300)[]` 支持 GIN 索引，查询性能与 JSONB 相当
- MyBatis-Plus 对 PG 数组类型有成熟的 TypeHandler 支持

### 2.2 大宽表列前缀扩展

在 `p_common_metadata` 和所有 `p_tenant_*` 大宽表中新增 `dbc_array` 列前缀：

| 列前缀 | PG 类型 | 容量 | 用途 |
|:---|:---|:---|:---|
| dbc_array | VARCHAR(300)[] | 1~10 | 元数据层多选字段（apiKey 列表、编码列表） |

> 注意：`dbc_array` 列前缀同时用于元数据大宽表（p_common_metadata / p_tenant_*，10 列）和业务数据大宽表（p_tenant_data，30 列）。两层统一使用 `VARCHAR(300)[]` 类型。

DDL（需同时在 p_common_metadata 和所有 p_tenant_* 表执行）：

```sql
-- PostgreSQL

ALTER TABLE p_common_metadata ADD COLUMN dbc_array2 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array3 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array4 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array5 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array6 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array7 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array8 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array9 VARCHAR(300)[];
ALTER TABLE p_common_metadata ADD COLUMN dbc_array10 VARCHAR(300)[];

-- MySQL 兼容（MySQL 不支持原生数组，用 JSON 类型替代）
ALTER TABLE p_common_metadata ADD COLUMN dbc_array1 JSON;
ALTER TABLE p_common_metadata ADD COLUMN dbc_array2 JSON;
-- ... 同上
```

> ⚠️ MySQL 兼容说明：MySQL 无原生数组类型，`dbc_array` 列在 MySQL 中使用 `JSON` 类型，存储 JSON 数组格式 `["a","b","c"]`。TypeHandler 层统一封装，Java 层始终操作 `List<String>`，屏蔽数据库差异。

### 2.3 p_meta_item 的 data_type 扩展

在 DataType 枚举中新增数组类型：

| **7** | **TEXT_ARRAY** | **文本数组** | **dbc_array** | **List\<String\>** | **VARCHAR(300)[]** | **JSON** |

### 2.4 受影响字段的 db_column 重新分配

| 元模型 | 字段 api_key | 旧 db_column | 旧 data_type | 新 db_column | 新 data_type |
|:---|:---|:---|:---|:---|:---|
| globalPickDependencyDetail | dependentOptionApiKeys | dbc_varchar2 | VARCHAR(1) | dbc_array1 | TEXT_ARRAY(7) |
| itemDependencyDetail | dependentOptionApiKeys | dbc_varchar2 | VARCHAR(1) | dbc_array1 | TEXT_ARRAY(7) |
| busiTypePickOption | optionApiKeys | dbc_varchar3 | VARCHAR(1) | dbc_array1 | TEXT_ARRAY(7) |

### 2.5 p_meta_item 记录变更

以 globalPickDependencyDetail.dependentOptionApiKeys 为例：

```sql

  db_column = 'dbc_varchar2',
  data_type = 1,  -- VARCHAR
  label = '依赖选项编码列表'

-- 变更后

  db_column = 'dbc_array1',
  data_type = 7,  -- TEXT_ARRAY
  label = '依赖选项编码列表'
WHERE metamodel_api_key = 'globalPickDependencyDetail' AND api_key = 'dependentOptionApiKeys';
```

### 2.6 存储格式对比

| 维度 | 变更前 | 变更后 |
|:---|:---|:---|
| PG 存储 | `'opt_chengdu,opt_mianyang,opt_bindzhou'` (VARCHAR) | `'{opt_chengdu,opt_mianyang,opt_bindzhou}'` (VARCHAR(300)[]) |
| MySQL 存储 | `'opt_chengdu,opt_mianyang,opt_bindzhou'` (VARCHAR) | `'["opt_chengdu","opt_mianyang","opt_bindzhou"]'` (JSON) |
| Java 类型 | `String` | `List<String>` |
| JSON 传输 | `"opt_chengdu,opt_mianyang,opt_bindzhou"` | `["opt_chengdu","opt_mianyang","opt_bindzhou"]` |
| 前端读取 | `value.split(",")` | 直接使用 |
| 前端写入 | `selected.join(",")` | 直接赋值 |

## 三、Java 层适配

### 3.1 MyBatis TypeHandler

```java
/**
 * PG VARCHAR(300)[] ↔ List<String> 类型处理器。
 * MySQL 环境下自动切换为 JSON 数组读写。
 */
@MappedTypes(List.class)
@MappedJdbcTypes(JdbcType.ARRAY)
public class TextArrayTypeHandler extends BaseTypeHandler<List<String>> {

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, 
                                     List<String> parameter, JdbcType jdbcType) throws SQLException {
        if (isPostgreSQL(ps)) {
            Array array = ps.getConnection().createArrayOf("text", parameter.toArray(new String[0]));
            ps.setArray(i, array);
        } else {
            // MySQL: 存为 JSON 数组
            ps.setString(i, JSON.toJSONString(parameter));
        }
    }

    @Override
    public List<String> getNullableResult(ResultSet rs, String columnName) throws SQLException {
        return parseArray(rs.getObject(columnName));
    }

    private List<String> parseArray(Object value) {
        if (value == null) return null;
        if (value instanceof Array) {
            // PG: 原生数组
            String[] arr = (String[]) ((Array) value).getArray();
            return Arrays.asList(arr);
        }
        if (value instanceof String str) {
            // MySQL: JSON 字符串
            return JSON.parseArray(str, String.class);
        }
        return null;
    }
}
```

## 四、对 chain-query 引擎的简化

### 4.1 消除 split 处理器

变更前，省→市级联查询需要 `outputFormat: "split"`：

```json
{
  "steps": [{
    "metamodelApiKey": "globalPickDependencyDetail",
    "matchField": "controlOptionApiKey",
    "outputField": "dependentOptionApiKeys",
    "outputFormat": "split",
    "splitSeparator": ","
  }]
}
```

变更后，字段本身就是数组，直接用 `collect`：

```json
{
  "steps": [{
    "metamodelApiKey": "globalPickDependencyDetail",
    "matchField": "controlOptionApiKey",
    "outputField": "dependentOptionApiKeys",
    "outputFormat": "collect"
  }]
}
```

> `collect` 模式：收集所有记录的 `dependentOptionApiKeys` 字段值。由于字段本身是 `List<String>`，collect 时自动 flatMap 展开，无需 split。

### 4.2 引擎执行逻辑简化

```java
case "collect":
    return rows.stream()
        .map(r -> getFieldValue(r, step.getOutputField()))
        .filter(Objects::nonNull)
        .flatMap(v -> {
            if (v instanceof List<?> list) {
                // 数组字段：直接展开
                return list.stream().map(String::valueOf);
            }
            // 单值字段：包装为单元素
            return Stream.of(String.valueOf(v));
        })
        .distinct()
        .collect(Collectors.toList());
```

`collect` 处理器自动识别字段类型：单值字段收集为列表，数组字段展开后收集。不需要 `split` 处理器了。

### 4.3 outputFormat 精简

| outputFormat | 变更前 | 变更后 | 说明 |
|:---|:---|:---|:---|
| single | 保留 | 保留 | 取第一条记录的字段值 |
| collect | 保留 | 保留（增强） | 收集所有记录的字段值，数组字段自动展开 |
| split | 保留 | **废弃** | 不再需要，数组字段直接用 collect |
| json | 保留 | 保留 | 兜底：解析 JSON 格式的复杂结构 |
| jsonPath | 保留 | 保留 | 兜底：从 JSON 对象中提取特定路径 |

## 五、对 p_meta_link 扩展的简化

### 5.1 原方案回顾

之前讨论的 p_meta_link 扩展需要 `extract_type` + `extract_expr` 来描述"怎么从逗号分隔字符串中提取值"。

### 5.2 简化后的 p_meta_link 扩展

多选字段改为数组类型后，p_meta_link 只需要增加 `link_type` 和 `match_mode`，不需要 `extract_type` / `extract_expr`：

```sql
ALTER TABLE p_meta_link ADD COLUMN match_mode VARCHAR(10) DEFAULT 'EQ'
  COMMENT 'EQ=等于 / IN=包含于 / ANY=数组包含 / OVERLAP=数组交集';
```

> `link_type` 字段已存在（当前值为 1 或 2），扩展 3=级联引用。

四种 match_mode：

| match_mode | SQL 语义 | 适用场景 |
|:---|:---|:---|
| EQ | `WHERE field = value` | 单值等于（默认，绝大多数场景） |
| IN | `WHERE field IN (values)` | 单值字段匹配多个值 |
| ANY | `WHERE value = ANY(field)` | 查询值是否在数组字段中 |
| OVERLAP | `WHERE field && ARRAY[values]` | 数组字段与给定数组有交集 |

### 5.3 省市区链路的 p_meta_link 配置

```
link 1 (已有): globalPickOption → globalPickDependency
  refer_item_api_key: entityApiKey
  link_type: 1, match_mode: EQ

link 2 (已有): globalPickDependency → globalPickDependencyDetail
  refer_item_api_key: dependencyApiKey
  link_type: 1, match_mode: EQ

link 3 (新增): globalPickDependencyDetail → globalPickOption
  refer_item_api_key: dependentOptionApiKeys
  link_type: 3, match_mode: IN
```

chain-query 引擎执行 link 3 时：
1. 从 globalPickDependencyDetail 取出 `dependentOptionApiKeys`（已经是 `List<String>`）
2. 用 `IN` 匹配 globalPickOption.apiKey
3. 不需要任何值展开逻辑

## 六、PG 数组查询与索引

### 6.1 常用查询模式

```sql
-- 查询数组包含某个值的记录
SELECT * FROM p_tenant_global_pick_dependency_detail
WHERE 'opt_chengdu' = ANY(dbc_array1);

-- 查询数组与给定列表有交集的记录
SELECT * FROM p_tenant_global_pick_dependency_detail
WHERE dbc_array1 && ARRAY['opt_chengdu', 'opt_mianyang'];

-- 查询数组包含给定列表所有元素的记录
SELECT * FROM p_tenant_global_pick_dependency_detail
WHERE dbc_array1 @> ARRAY['opt_chengdu', 'opt_mianyang'];

-- 展开数组为行（用于 chain-query 的中间步骤）
SELECT unnest(dbc_array1) AS option_code
FROM p_tenant_global_pick_dependency_detail
WHERE dbc_varchar1 = 'dep_province_city';
```

### 6.2 GIN 索引

```sql
CREATE INDEX idx_dep_detail_options
ON p_tenant_global_pick_dependency_detail
USING GIN (dbc_array1);
```

`ANY`、`&&`、`@>` 运算符均可利用 GIN 索引。

### 6.3 MySQL 兼容查询

MySQL 使用 JSON 类型存储时，等价查询：

```sql
-- ANY 等价
SELECT * FROM p_tenant_global_pick_dependency_detail
WHERE JSON_CONTAINS(dbc_array1, '"opt_chengdu"');

-- OVERLAP 等价
SELECT * FROM p_tenant_global_pick_dependency_detail
WHERE JSON_OVERLAPS(dbc_array1, '["opt_chengdu","opt_mianyang"]');
```

> DAO 层通过条件构建器封装数据库差异，业务代码不直接写 SQL。

## 七、数据迁移

### 7.1 迁移策略

对于已有数据的字段，需要将逗号分隔字符串迁移为数组：

```sql
-- PG: VARCHAR → VARCHAR(300)[]
UPDATE p_common_metadata
SET dbc_array1 = string_to_array(dbc_varchar2, ',')
WHERE metamodel_api_key = 'globalPickDependencyDetail'
  AND dbc_varchar2 IS NOT NULL AND dbc_varchar2 != '';

-- 迁移后清空旧列（可选，保留一段时间作为回退）
-- UPDATE p_common_metadata SET dbc_varchar2 = NULL
-- WHERE metamodel_api_key = 'globalPickDependencyDetail';
```

### 7.2 迁移顺序

1. DDL：在所有大宽表上添加 `dbc_array1~10` 列
2. p_meta_item：更新受影响字段的 `db_column` 和 `data_type`
3. 数据迁移：逐个元模型执行 VARCHAR → VARCHAR(300)[] 转换
4. 验证：对比迁移前后数据一致性
5. 清理：确认无误后清空旧 VARCHAR 列（旧列号不回收，保持占位）

### 7.3 回退方案

迁移期间旧 VARCHAR 列数据保留不删除。如需回退：
1. p_meta_item 的 `db_column` 改回旧值
2. 旧列数据仍在，立即可用
