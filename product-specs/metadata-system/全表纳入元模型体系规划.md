# 全表纳入元模型体系规划

> 版本：v1.0 | 日期：2026-04-16
> 状态：**已修正** — 经深度分析，业务数据表（passport、passportLog、userRole、operateLog）不应注册到 p_meta_model。
> 详见 [业务数据表元模型化与大宽表分析](models/业务数据表元模型化与大宽表分析.md)
>
> 修正结论：
> - 元模型管理的表：通过 `p_meta_model.enable_tenant_intercept` 配置驱动
> - 业务数据表：通过 Java Entity `@IgnoreTenantLine` 注解声明
> - 元模型定义表（p_meta_*）：框架硬编码

## 一、设计原则

1. `p_meta_*` / `p_common_*` 是元数据体系的基石，框架底层硬编码跳过租户拦截（无 tenant_id 列）
2. 其他所有表注册为元模型，通过 `enable_tenant` / `enable_tenant_intercept` 配置驱动
3. 独立表结构（不必是大宽表），跟 role、department 一样作为独立元模型
4. `MetaRepoDataConfig` 中的 `PLATFORM_TABLES` 硬编码忽略列表最终清除，全部由元模型配置接管

### 3.1 enable_tenant_intercept 配置规则

| 场景 | enable_tenant | enable_tenant_intercept | 说明 |
|:---|:---:|:---:|:---|
| 无 tenant_id 列 | 0 | — | p_passport 等全局表 |
| 有 tenant_id，正常拦截 | 1 | 1 | 绝大多数表（默认） |
| 有 tenant_id，不拦截 | 1 | 0 | 操作日志、p_user、p_user_role 等需要跨租户或启动阶段查询的表 |

## 四、统一后的 TenantInterceptor

### 4.1 框架层（framework-dao）

`TenantInterceptor.ignoreTable()` 只保留一条硬编码规则：

```java
@Override
public boolean ignoreTable(String tableName) {
    String simple = tableName.contains(".")
            ? tableName.substring(tableName.lastIndexOf('.') + 1)
            : tableName;
    // 元数据体系基石：元模型定义表 + Common 级存储表，固定无 tenant_id
    return simple.startsWith("p_meta_") || simple.startsWith("p_common_");
}
```

删除 `IGNORE_TABLES`、`ignorePredicate`、`addIgnoreTable()`、`addIgnorePredicate()` 静态方法。

### 4.2 业务层（MetaRepoDataConfig）

删除 `PLATFORM_TABLES` 和 `addIgnorePredicate` 调用。
`DynamicTableNameInterceptor` 的 schema 路由逻辑不变。

### 4.3 Entity 层

所有 `enable_tenant_intercept=0` 的表对应的 Entity 类加 `@IgnoreTenantLine`：

| Entity | 表 | 原因 |
|:---|:---|:---|
| Passport | p_passport | 全局表，无 tenant_id |
| PlatformUser | p_user | 业务代码手动控制 tenant_id |
| UserRole | p_user_role | 业务代码手动控制 tenant_id |
| OperateLog（基类） | p_*_operate_log | 启动恢复需跨租户扫描 |

> 注意：`enable_tenant=0` 的表（如 passport）也需要 `@IgnoreTenantLine`，因为拦截器在 `ignoreTable` 返回 false 后会调用 `getTenantId()`，如果此时无 tenant_id 上下文会抛异常。`@IgnoreTenantLine` 通过 `InterceptorIgnoreHelper` 在更早的入口处跳过，避免进入 `ignoreTable` 判断。

## 五、实施计划

### 阶段 1：元模型注册（纯 DML，不改代码）
- 执行上述 INSERT SQL，将所有表注册为元模型
- 执行 user 的 UPDATE SQL

### 阶段 2：Entity 加注解
- Passport、PlatformUser、UserRole 加 `@IgnoreTenantLine`
- 验证启动和 CRUD 正常

### 阶段 3：清理 TenantInterceptor
- 删除 `IGNORE_TABLES`、`ignorePredicate`、`addIgnoreTable()`、`addIgnorePredicate()`
- `ignoreTable()` 简化为只判断 `p_meta_*` / `p_common_*` 前缀
- 删除 `MetaRepoDataConfig` 中的 `PLATFORM_TABLES` 和 `addIgnorePredicate` 调用

### 阶段 4：验证
- 全链路验证用户/角色/部门 CRUD
- 验证启动恢复（EntityPendingTransactionRecovery）
- 验证登录流程（Passport 查询）

## 六、统一后的全景

```
SQL 经过拦截器
  │
  ├─ DataBaseServiceImpl.executeWithTenantControl()
  │   Entity 标注 @IgnoreTenantLine?
  │   ├─ 是 → InterceptorIgnoreHelper.handle(tenantLine=true)
  │   │       → TenantLineInnerInterceptor.beforeQuery() 第一行跳过
  │   └─ 否 → 直接执行
  │
  └─ TenantLineInnerInterceptor 正常流程
      → 解析 SQL 表名
      → TenantInterceptor.ignoreTable(tableName)
        ├─ p_meta_* / p_common_* → true（跳过）
        └─ 其他 → false → getTenantId() → 追加 AND tenant_id = ?
```

所有忽略租户拦截的行为只有两个来源：
1. **框架硬编码**：`p_meta_*` / `p_common_*`（元数据体系基石，永远不变）
2. **元模型配置驱动**：`enable_tenant_intercept=0` → Entity 加 `@IgnoreTenantLine` → `DataBaseServiceImpl` 自动处理
