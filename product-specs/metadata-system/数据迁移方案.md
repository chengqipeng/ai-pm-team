## 关键迁移决策

### api_key 生成策略

老库中部分元模型的 `api_key` 为空（如 formulaCompute、pickOption、computeFactor 等），新系统要求 `api_key` 非空且唯一。

| 情况 | 策略 | 示例 |
|:---|:---|:---|
| 老库有 api_key | 直接使用 | entity: `account`, item: `name` |
| 老库 api_key 为空 | 使用 metadata_id 转字符串 | formulaCompute: `-11120000100001` |
| 独立表无 api_key | 使用业务标识字段 | entitySvg: svg_class 如 `dashboard` |

### 老→新 api_key 映射

| 老库 api_key | 新库 api_key | 说明 |
|:---|:---|:---|
| xobject | entity | 对象定义 |
| xobjectLink | entityLink | 关联关系 |
| objectBusiType | busiType | 业务类型 |
| duplicateRuleDetail | duplicateRuleCriteria | 查重条件 |
| globalPick | globalPickOption | 全局选项集 |
| customSvg | entitySvg | 实体图标 |

### ID→apiKey 转换

老库使用 `metadata_id`（Long）作为跨模型引用，新库使用 `api_key`（String）。转换映射：

| 引用字段 | 老库格式 | 新库格式 | 转换方式 |
|:---|:---|:---|:---|
| entity_api_key | objectId (Long) | entity.api_key (String) | entity_id_map 查表 |
| parent_metadata_api_key | parent_metadata_id (Long) | parent.api_key (String) | 各模型 id_map 查表 |
| dbc 列中的引用 | xxxId (Long) | xxxApiKey (String) | 对应模型 id_map 查表 |

### Tenant 级迁移策略

1. 结构一致：所有 p_tenant_* 表结构与 p_common_metadata 完全一致 + tenant_id
2. 分批迁移：按 tenant_id 分批，每批 10K 行
3. ID→apiKey：复用 Common 级的 entity_id_map 和 item_id_map
4. 增量同步：支持断点续传，记录已同步的 tenant_id
