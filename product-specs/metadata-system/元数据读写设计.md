# 元数据读写设计

## 1. 数据分层

| 层级 | 存储位置 | 写入方 | 可见性 | 说明 |
|---|---|---|---|---|
| Common | p_common_metadata | 平台/Module | 所有租户 | 出厂元数据，系统标准字段 |
| Tenant | p_tenant_* (各元模型独立表) | 租户管理员 | 仅该租户 | 租户自定义字段、覆盖 |

## 2. 读取逻辑（已实现）

```
读取请求(tenantId, metamodelApiKey, entityApiKey)
    │
    ├─ 查 Common 层: p_common_metadata WHERE metamodel_api_key=? AND entity_api_key=?
    │   → CommonMetadataConverter 转为业务 Entity
    │
    ├─ 查 Tenant 层: p_tenant_{table} WHERE tenant_id=? AND metamodel_api_key=? AND entity_api_key=?
    │   → CommonMetadataConverter 转为业务 Entity
    │
    └─ 合并:
         Common 有 Tenant 无 → 用 Common（标准数据）
         同 apiKey → Tenant 覆盖 Common（租户定制）
         Tenant delete_flg=1 → 隐藏该条（遮蔽删除）
```

## 3. 写入逻辑

### 3.1 核心原则

1. **Common 层只读**：业务层不直接写入 p_common_metadata，Common 数据由平台初始化/Module 安装写入
2. **Tenant 层可写**：所有业务写操作都写入 Tenant 层
3. **覆盖机制**：Tenant 层同 apiKey 的记录覆盖 Common 层
4. **遮蔽删除**：删除 Common 数据时，在 Tenant 层插入 delete_flg=1 的记录

### 3.2 新增元数据

```
新增字段(tenantId, entityApiKey, itemData)
    │
    ├─ 校验:
    │   ├─ apiKey 唯一性（Common + Tenant 中不能重复）
    │   ├─ itemType 合法性（必须在 ItemTypeEnum 中）
    │   ├─ dbColumn 分配（根据 itemType 自动分配下一个空闲列号）
    │   └─ 必填字段校验（apiKey, label, itemType）
    │
    ├─ 设置默认值:
    │   ├─ namespace = 'tenant'
    │   ├─ customFlg = 1（自定义）
    │   ├─ itemSubType = 根据 itemType 推导
    │   ├─ dataType = 根据 itemSubType 推导
    │   └─ dbColumn = 自动分配
    │
    ├─ 转换: Entity → TenantMetadata（通过 CommonMetadataConverter.toTenantMetadata）
    │
    └─ 写入: INSERT INTO p_tenant_{table}
```

### 3.3 修改元数据

```
修改字段(tenantId, apiKey, updateData)
    │
    ├─ 查询现有数据:
    │   ├─ 先查 Tenant 层
    │   └─ 再查 Common 层
    │
    ├─ 判断数据来源:
    │   │
    │   ├─ 来自 Tenant 层 → 直接 UPDATE Tenant 记录
    │   │
    │   └─ 来自 Common 层 → 不能修改 Common，需要:
    │       ├─ 复制 Common 记录到 Tenant 层
    │       ├─ 在 Tenant 记录上应用修改
    │       └─ INSERT INTO p_tenant_{table}（Tenant 覆盖 Common）
    │
    ├─ 校验:
    │   ├─ 不可修改字段: apiKey, itemType, dbColumn（创建后不可变）
    │   └─ 可修改字段: label, description, requireFlg, enableFlg, 等
    │
    └─ 写入: UPDATE 或 INSERT p_tenant_{table}
```

### 3.4 删除元数据

```
删除字段(tenantId, apiKey)
    │
    ├─ 查询数据来源:
    │   │
    │   ├─ 仅 Tenant 层有（租户自定义字段）:
    │   │   └─ 物理删除 或 软删除（delete_flg=1）
    │   │
    │   ├─ 仅 Common 层有（标准字段）:
    │   │   └─ 遮蔽删除: INSERT Tenant 记录 with delete_flg=1
    │   │
    │   └─ Common + Tenant 都有（租户已覆盖的标准字段）:
    │       └─ UPDATE Tenant 记录 set delete_flg=1
    │
    └─ 级联处理:
        ├─ 删除 entity → 级联删除其下所有 item/entityLink/checkRule 等
        └─ 删除 item → 级联删除其下 pickOption/referenceFilter 等
```

## 4. 实体元数据（Entity）写入

### 4.1 新增 Entity

| 字段 | 必填 | 说明 |
|---|---|---|
| apiKey | 是 | 唯一标识，创建后不可变 |
| label | 是 | 显示名称 |
| entityType | 否 | 默认 1（自定义对象） |
| dbTable | 自动 | 自动生成 `a_{apiKey}` |
| namespace | 自动 | 'tenant' |
| customFlg | 自动 | 1 |
| enableFlg | 否 | 默认 1（启用） |

### 4.2 修改 Entity

可修改字段：label, description, enableFlg, svgApiKey, svgColor, historyLogFlg 等
不可修改字段：apiKey, entityType, dbTable

## 5. 字段元数据（Item）写入

### 5.1 新增 Item

| 字段 | 必填 | 说明 |
|---|---|---|
| entityApiKey | 是 | 所属对象 |
| apiKey | 是 | 唯一标识，创建后不可变 |
| label | 是 | 显示名称 |
| itemType | 是 | 字段类型（见字段类型体系设计） |
| itemSubType | 自动 | 非计算型=itemType；计算型由用户选择 |
| dataType | 自动 | 从 itemSubType 推导 |
| dbColumn | 自动 | 根据 itemType 自动分配（见 §5.2） |
| namespace | 自动 | 'tenant' |
| customFlg | 自动 | 1 |

### 5.2 dbColumn 自动分配逻辑

```
分配 dbColumn(entityApiKey, itemType, itemSubType):
    │
    ├─ 确定前缀:
    │   ├─ 非计算型: 由 itemType 决定（如 TEXT→dbc_varchar, NUMBER→dbc_bigint）
    │   ├─ 计算型(FORMULA/27): 由 itemSubType 决定
    │   └─ 虚拟型(JOIN/26): 不分配，返回 null
    │
    ├─ 固定列字段: 直接返回固定列名
    │   ├─ ownerId → 'owner_id'
    │   ├─ createdAt → 'created_at'
    │   └─ ... 等（见业务数据大宽表设计 §3.2）
    │
    ├─ 查询已用列号:
    │   SELECT dbColumn FROM item WHERE entityApiKey=? AND dbColumn LIKE '{prefix}%'
    │   → 提取已用的列号集合
    │
    └─ 分配最小空闲号:
        for i = 1 to MAX:
            if i not in usedSet:
                return '{prefix}{i}'
```

### 5.3 修改 Item

可修改字段：label, description, requireFlg, enableFlg, helpText, defaultValue, itemOrder 等
不可修改字段：apiKey, entityApiKey, itemType, dbColumn

### 5.4 计算字段（FORMULA/27）特殊处理

新增计算字段时，除了 item 记录外，还需要创建关联的计算定义：

```
新增计算字段:
    ├─ 创建 item 记录（itemType=27, itemSubType=用户选择的计算结果类型）
    │
    ├─ 如果是公式计算:
    │   ├─ 创建 formulaCompute 记录（公式表达式、空值处理、结果类型）
    │   └─ 创建 formulaComputeItem 记录（公式引用的字段列表）
    │
    └─ 如果是汇总累计:
        ├─ 创建 aggregationCompute 记录（子对象、累计类型、汇总字段）
        └─ 创建 aggregationComputeDetail 记录（过滤条件，可选）
```

## 9. 同 apiKey 字段的删除与重建机制

### 9.1 问题场景

租户删除（遮蔽）或禁用一个 Common 层系统字段后，希望用相同的 apiKey 创建一个全新的自定义字段。例如：

```
Common 层: phone (apiKey=phone, itemType=PHONE, dbColumn=dbc_varchar2)
租户操作: 遮蔽删除 phone → 新建 phone (itemType=TEXT, dbColumn=?)
```

这会产生三个冲突：
1. Tenant 层已有 `apiKey=phone` 的遮蔽记录（delete_flg=1），唯一约束阻止 INSERT
2. 老字段占用的 `dbc_varchar2` 列中还有业务数据
3. 新字段的 itemType 可能与老字段不同，dbColumn 前缀可能变化

### 9.2 设计方案：就地替换（UPDATE 遮蔽记录）

当租户新建字段的 apiKey 与一个已遮蔽的 Common 字段相同时，不执行 INSERT，而是 UPDATE 已有的遮蔽记录：

```
新建字段(tenantId, entityApiKey, itemData):
    │
    ├─ 1. 常规唯一性校验
    │     合并视图中查找 apiKey（Common + Tenant, delete_flg=0）
    │     if 存在且可见 → 报错"apiKey 已存在"
    │
    ├─ 2. 检查是否命中遮蔽记录
    │     maskedRecord = Tenant 层查找 apiKey WHERE delete_flg=1
    │     commonRecord = Common 层查找同 apiKey
    │
    ├─ 3a. 无遮蔽记录 → 常规新建（INSERT）
    │
    └─ 3b. 有遮蔽记录 → 就地替换（UPDATE）
          │
          ├─ 分配新 dbColumn（不复用老字段的 dbColumn）
          │   newDbColumn = 按 itemType 分配下一个空闲列号
          │
          ├─ UPDATE 遮蔽记录:
          │   SET delete_flg = 0,           -- 取消遮蔽
          │       custom_flg = 1,           -- 标记为自定义
          │       namespace = 'custom',     -- 租户自定义
          │       label = 新 label,
          │       itemType = 新 itemType,
          │       itemSubType = 新 itemSubType,
          │       dataType = 新 dataType,
          │       dbColumn = newDbColumn,    -- 新列号，不复用老列
          │       ... 其他字段用新值覆盖
          │
          └─ 记录变更日志: op_type = REPLACE_MASKED
```

### 9.3 dbColumn 处理规则

| 场景 | dbColumn 处理 | 理由 |
|---|---|---|
| 新字段 itemType 与老字段相同 | 分配新列号，不复用老列 | 老列可能有残留业务数据，复用会导致脏数据 |
| 新字段 itemType 与老字段不同 | 分配新列号（前缀可能不同） | 类型不同，列前缀不同，必须新分配 |
| 老字段的 dbColumn | 标记为"已废弃"，不再分配给新字段 | 老列中的业务数据不清理，避免数据丢失 |

废弃列号的处理：

```
dbColumn 分配时排除废弃列:
    │
    ├─ 查询已用列号（包括 delete_flg=1 的遮蔽记录的 dbColumn）
    │   SELECT dbColumn FROM item
    │     WHERE entityApiKey=? AND dbColumn LIKE '{prefix}%'
    │     AND delete_flg = 0
    │   UNION
    │   SELECT dbColumn FROM commonMetadata
    │     WHERE entityApiKey=? AND dbColumn LIKE '{prefix}%'
    │   → usedSet（包含所有在用 + Common 层占用的列号）
    │
    └─ 注意: 被替换的遮蔽记录的老 dbColumn 已经被新值覆盖，
          但 Common 层的同 apiKey 记录的 dbColumn 仍然在 usedSet 中，
          确保老列号不会被重新分配给其他字段
```

> 关键点：Common 层的 dbColumn 永远占位。即使租户遮蔽了该字段，Common 层的 `dbc_varchar2` 仍然被视为"已用"，不会分配给任何新字段。这保证了：
> 1. 如果租户将来恢复系统字段，老 dbColumn 还在
> 2. 业务数据表中老列的数据不会被其他字段覆盖

### 9.4 禁用字段的处理

禁用（enableFlg=0）与遮蔽删除（delete_flg=1）不同：

| 状态 | 合并视图可见性 | apiKey 占用 | 允许同名新建 |
|---|---|---|---|
| 正常（enableFlg=1, delete_flg=0） | 可见 | 是 | 否 |
| 禁用（enableFlg=0, delete_flg=0） | 可见但禁用 | 是 | 否 |
| 遮蔽删除（delete_flg=1） | 不可见 | 否 | 是（就地替换） |

禁用字段在合并视图中仍然可见（只是 UI 上灰显不可用），apiKey 仍然被占用，不允许新建同名字段。用户必须先删除（遮蔽）该字段，才能用相同 apiKey 新建。

### 9.5 恢复系统字段

租户遮蔽删除系统字段后，有两种后续操作：

**操作 A：恢复系统字段（取消遮蔽）**

```
恢复字段(tenantId, entityApiKey, apiKey):
    │
    ├─ 查询 Tenant 遮蔽记录
    │   record = Tenant WHERE apiKey=? AND delete_flg=1
    │
    ├─ 检查是否已被替换
    │   if record.customFlg == 1 AND record.namespace == 'custom':
    │     → 该记录已被"就地替换"为自定义字段，不是遮蔽记录
    │     → 报错: "该 apiKey 已被自定义字段占用，请先删除自定义字段"
    │
    └─ 执行恢复
        DELETE Tenant 遮蔽记录（物理删除）
        → Common 层的系统字段重新可见
        → 老 dbColumn 和业务数据自动恢复
```

**操作 B：用同 apiKey 新建自定义字段（就地替换，见 §9.2）**

两个操作互斥：一旦执行了 B（就地替换），就不能再执行 A（恢复系统字段），除非先删除自定义字段。

```

                    │  Common 系统字段  │
                    │  (phone, PHONE,  │
                    │   dbc_varchar2)  │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────────┐
        │ 租户覆盖  │  │ 租户禁用  │  │  租户遮蔽删除  │
        │ (修改属性) │  │(enableFlg │  │ (delete_flg  │
        │          │  │   =0)    │  │    =1)       │
        └──────────┘  └─────┬────┘  └──────┬───────┘
                            │              │
                            │         ┌────┴────┐
                            │         │         │
                            ▼         ▼         ▼
                      ┌──────────┐ ┌────────┐ ┌──────────────┐
                      │ 先删除   │ │ 恢复   │ │ 同名新建     │
                      │ 再新建   │ │ 系统字段│ │（就地替换）   │
                      └──────────┘ │        │ │              │
                                   │ DELETE  │ │ UPDATE 遮蔽  │
                                   │ Tenant  │ │ 记录，写入新  │
                                   │ 遮蔽记录│ │ 字段属性     │
                                   └────────┘ └──────────────┘
```

### 9.7 合并读取逻辑的调整

原合并逻辑：
```
同 apiKey → Tenant 覆盖 Common
Tenant delete_flg=1 → 隐藏该条
```

调整后：
```
同 apiKey 的 Tenant 记录:
  if delete_flg=1 → 隐藏（遮蔽删除，未被替换）
  if delete_flg=0 AND customFlg=1 AND namespace='custom'
    → 使用 Tenant 记录（就地替换的自定义字段，完全忽略 Common）
  if delete_flg=0 AND customFlg=0
    → 使用 Tenant 记录（常规覆盖，属性修改）
```

> 就地替换后的记录与常规覆盖记录的区别：customFlg=1 + namespace='custom' + itemType/dbColumn 可能与 Common 不同。合并逻辑不需要特殊分支，因为"Tenant 覆盖 Common"的规则已经天然处理了——Tenant 记录的所有字段值（包括 itemType、dbColumn）都会覆盖 Common。

### 9.8 业务数据的影响

| 场景 | 老列（dbc_varchar2） | 新列（dbc_varchar28） | 说明 |
|---|---|---|---|
| 遮蔽删除后未新建 | 数据保留，不可见 | — | 恢复系统字段后数据自动回来 |
| 就地替换后 | 数据保留，成为孤儿数据 | 新字段写入此列 | 老列数据不清理，不影响新字段 |
| 恢复系统字段 | 数据恢复可见 | — | 无损恢复 |

> 老列的孤儿数据不主动清理。原因：
> 1. 清理需要全表扫描 UPDATE，大表代价高
> 2. 孤儿数据不影响任何查询（没有字段映射指向它）
> 3. 如果将来恢复系统字段，数据还能用

### 9.9 约束与限制

| 约束 | 说明 |
|---|---|
| 仅遮蔽删除的字段允许同名新建 | 禁用字段不允许（apiKey 仍被占用） |
| 新字段必须分配新 dbColumn | 禁止复用老字段的 dbColumn，避免脏数据 |
| Common 层 dbColumn 永远占位 | 即使遮蔽，老列号不会分配给其他字段 |
| 就地替换后不可恢复系统字段 | 必须先删除自定义字段，再恢复 |
| 子元数据不继承 | 就地替换后，老字段的 pickOption/referenceFilter 等子元数据不会自动关联到新字段，需要重新创建 |
