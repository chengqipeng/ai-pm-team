# 元模型 → 元数据 → 数据 关系体系分析

> 版本：1.0 | 日期：2026-04-22
> 目标：梳理三层架构中所有关系类型，识别设计不合理之处，构建完善的关系体系

---

## 二、六种关系类型

### 关系 1：元模型定义层内部关系（p_meta_link，metamodel_layer=1）

**含义**：四表体系的内部结构关系。

```
metaModel ──→ metaItem    "元模型包含字段定义"    referItemApiKey = metamodelApiKey
metaModel ──→ metaLink    "元模型包含关联关系"    referItemApiKey = parentMetamodelApiKey
metaModel ──→ metaOption  "元模型包含选项值"      referItemApiKey = metamodelApiKey
metaItem  ──→ metaOption  "字段包含选项值"        referItemApiKey = itemApiKey
```

**数据来源**：`p_meta_link`（刚补充的 4 条）
**UI 场景**：元模型定义页面，左侧选 metaModel，右侧 Tab 显示 metaItem/metaLink/metaOption

### 关系 2：元数据实例层内部关系（p_meta_link，metamodel_layer=0）

**含义**：元模型类型之间的父子关系，决定了元数据实例的层级结构。

```
entity ──→ item              referItemApiKey = entityApiKey
entity ──→ entityLink        referItemApiKey = entityApiKey
entity ──→ checkRule         referItemApiKey = entityApiKey
entity ──→ busiType          referItemApiKey = entityApiKey
entity ──→ duplicateRule     referItemApiKey = entityApiKey
entity ──→ fieldSet          referItemApiKey = entityApiKey
entity ──→ sharingRule       referItemApiKey = entityApiKey
entity ──→ dataPermission    referItemApiKey = entityApiKey
item   ──→ pickOption        referItemApiKey = itemApiKey
item   ──→ referenceFilter   referItemApiKey = itemApiKey
item   ──→ formulaCompute    referItemApiKey = itemApiKey
item   ──→ aggregationCompute referItemApiKey = itemApiKey
item   ──→ itemDependency    referItemApiKey = controlItemApiKey
...（共 30+ 条）
```

**数据来源**：`p_meta_link`（从远程数据库同步）
**UI 场景**：实体管理页面，左侧选 entity(account)，右侧 Tab 显示 item/checkRule/busiType 等

### 关系 3：跨层关系 — 定义层→实例层（隐含）

**含义**：每个 `p_meta_model` 记录天然对应一组元数据实例。

```
metaModel 记录 'role'     ──→  role 的所有元数据实例（systemAdmin, salesManager, ...）
metaModel 记录 'entity'   ──→  entity 的所有元数据实例（account, opportunity, ...）
metaModel 记录 'item'     ──→  item 的所有元数据实例（3000+ 条字段定义）
```

**数据来源**：隐含关系，通过 `listMetadata(metamodelApiKey)` 查询
**不在 p_meta_link 中**：因为这不是"类型 A 包含类型 B"，而是"类型 A 的实例列表"
**UI 场景**：元模型定义页面选中 metaModel 记录 'role' → 可以查看 role 的所有实例

### 关系 4：跨层关系 — 实例层→数据层（隐含）

**含义**：`entity` 的每个实例对应一张业务数据表。

```
entity 实例 'account'     ──→  p_tenant_data_132（account 的业务数据）
entity 实例 'opportunity' ──→  p_tenant_data_456（opportunity 的业务数据）
entity 实例 'user'        ──→  p_user（用户数据，独立表）
```

**数据来源**：`p_meta_model.enable_data_source=1`（当前仅 entity）+ `p_tenant_data_route` 路由表
**UI 场景**：画布数据源"数据"分类，选择 account → 加载 `/entity/data/account`

### 关系 5：元数据实例之间的关联（entityLink 元数据）

**含义**：具体的元数据实例之间的关联关系（不是类型级别，是实例级别）。

```
account ──→ contact       "客户关联联系人"    linkType=2(关联)
account ──→ opportunity   "客户关联商机"      linkType=2(关联)
role    ──→ role          "角色自关联（上级）" linkType=0(查找)
role    ──→ user          "角色关联用户"      linkType=2(多对多)
department ──→ department "部门自关联（上级）" linkType=0(查找)
department ──→ user       "部门关联用户"      linkType=2(关联)
```

**数据来源**：`entityLink` 元数据实例（存储在 `p_common_metadata` / `p_tenant_entity_link`）
**UI 场景**：画布联动，左侧选 department → 右侧可加载 user 数据

### 关系 6：元数据引用关系（metadataRef，运行时）

**含义**：元数据之间的运行时引用，用于删除保护。

```
formulaCompute(fullName) ──引用──→ item(firstName)    "公式引用字段"
checkRule(phoneRequired) ──引用──→ item(phone)         "校验规则引用字段"
pageSection(canvas)      ──引用──→ item(name)          "画布列引用字段"
item(defaultValue)       ──引用──→ item(ownerName)     "默认值引用字段"
```

**数据来源**：`metadataRef` 元模型（刚实现的 P0 阶段）
**UI 场景**：删除字段时提示"被公式 fullName 引用，无法删除"

## 三、当前设计中的不合理之处


### 问题 1：entity_dependency 与 p_meta_link 语义重叠

**现状**：
- `p_meta_model.entity_dependency=1` 标记"该元模型的实例需要 entity_api_key"
- `p_meta_link` 中 `entity → item` 的记录也表达了"item 是 entity 的子元模型"

**问题**：
- `entity_dependency` 只说"依赖 entity"，不说"依赖谁"。`pickOption` 的 `entity_dependency=1`，但它的直接父是 `item`，不是 `entity`
- 两个数据源表达同一件事，容易不一致
- 前端之前用 `entity_dependency` 做联动约束，但它粒度太粗，无法区分直接父子和间接父子

**建议**：
- `entity_dependency` 保留为**存储层标记**（决定元数据实例是否需要 `entity_api_key` 列），不用于 UI 联动
- UI 联动约束统一从 `p_meta_link` 推导，`p_meta_link` 是唯一的关系权威数据源

### 问题 2：metamodel_layer 只有两个值，无法表达完整层级

**现状**：
- `metamodel_layer=1`：metaModel/metaItem/metaLink/metaOption（定义层）
- `metamodel_layer=0`：其他所有元模型（实例层）

**问题**：
- entity 和 role 都是 `metamodel_layer=0`，但它们的性质完全不同：
  - entity 的实例（account）可以进一步产生业务数据（第三层）
  - role 的实例（systemAdmin）就是最终数据，没有第三层
- 画布数据源的三级分类（元模型/元数据/数据）无法从 `metamodel_layer` 单一字段推导
- 需要 `enable_data_source` 额外字段来标记"实例可作为数据源"

**建议**：
- 当前设计可接受（`metamodel_layer` + `enable_data_source` 组合使用）
- 但应在文档中明确：`metamodel_layer` 区分的是"描述元数据的元数据"和"元数据本身"，不是三级分类

### 问题 3：entityLink 同时承担两种角色

**现状**：
- `entityLink` 既是 `p_meta_link` 中的一个子元模型（entity → entityLink），又是元数据实例之间关联关系的载体
- `entityLink` 的 `parentEntityApiKey` / `childEntityApiKey` 描述的是**元数据实例之间**的关联（如 account → contact）
- `p_meta_link` 的 `parentMetamodelApiKey` / `childMetamodelApiKey` 描述的是**元模型类型之间**的关联（如 entity → item）

**问题**：
- 命名混淆：`entityLink` 的 `parentEntityApiKey` 实际上可以是任何元数据实例（如 role、department），不仅限于 entity
- 当 role/department 独立为元模型后，它们的 entityLink（如 `role_to_user`）的 `entity_api_key` 字段填的是 `role`，但 `role` 不是 entity
- `entityLink` 的 `entity_api_key`（所属实体）字段语义不清：对于 `role_to_user`，它应该填 `role`（所属元模型实例）还是空？

**建议**：
- `entityLink` 的 `entity_api_key` 统一理解为"所属父元数据的 apiKey"，不限于 entity
- 长期考虑将 `entityLink` 重命名为 `metadataLink`（元数据关联），但短期不改（影响面太大）
- 在文档中明确 `entityLink` 的语义：描述任意两个元数据实例之间的关联关系

### 问题 4：独立元模型（role/department）的关联关系存储位置不统一

**现状**：
- entity 体系的关联关系在 `p_meta_link` 中（类型级）+ `entityLink` 元数据中（实例级）
- role/department 的关联关系只在 `entityLink` 元数据中（刚补充的 4 条种子数据）
- role/department 没有在 `p_meta_link` 中注册类型级关联

**问题**：
- `p_meta_link` 中没有 `role → user` 或 `department → user` 的记录
- 前端 `getChildMetamodels('role')` 返回空，无法自动推导 role 可以关联 user
- 但 `entityLink` 中有 `role_to_user` 记录，描述了实例级关联

**根因**：
- `p_meta_link` 描述的是**元模型类型之间**的父子关系（entity 包含 item）
- `role → user` 不是"role 类型包含 user 类型"，而是"role 实例关联 user 实例"
- 这两种关系本质不同，不应该混在同一张表中

**建议**：
- `p_meta_link` 只存类型级父子关系（不变）
- `entityLink` 存实例级关联关系（不变）
- 前端联动约束需要同时查询两个数据源：
  - `p_meta_link`：推导"右侧可选哪些子元模型类型"
  - `entityLink`：推导"右侧可选哪些关联的元数据/数据对象"

### 问题 5：画布三级分类与实际数据层级不完全对应

**现状**：
画布数据源分三级：
1. 元模型（metamodel_layer=1）：metaModel/metaItem/metaLink/metaOption
2. 元数据（metamodel_layer=0）：entity/item/role/department/...
3. 数据（enable_data_source=1 的实例）：account/opportunity/...

**问题**：
- `item` 在画布中归类为"元数据"，但它的实例（如 account.phone）不是独立的数据对象，而是 entity 的子属性
- 用户在画布中选"元数据 → item"时，得到的是所有 entity 的所有字段（3000+ 条），没有按 entity 过滤
- 需要先选 entity，再选 item，但画布的二级选择器没有这种级联逻辑
- `user` 既是元数据（`p_meta_model` 中注册了 `user` 元模型），又是业务数据（`p_user` 表），身份模糊

**建议**：
- 画布数据源选择器中，对于 `entity_dependency=1` 的元模型（item/pickOption/checkRule 等），不应该直接出现在"元数据"分类的顶层列表中
- 它们应该只在"左侧选了 entity 后，右侧可选的子类型"中出现（通过 p_meta_link 约束）
- 独立元模型（role/department/sharingRule 等）才应该出现在"元数据"分类的顶层列表中

### 问题 6：缺少"元数据实例之间的关联关系"的统一查询接口

**现状**：
- `p_meta_link` 有 `GET /meta/meta-links` 接口（类型级关联）
- `entityLink` 通过 `GET /meta/metadata?metamodelApiKey=entityLink` 查询（实例级关联）
- 但没有一个统一的接口能回答"role 可以关联哪些对象？"

**问题**：
- 前端需要分别查两个接口，再合并结果
- `entityLink` 的查询需要知道 `entityApiKey`（如 `role`），但画布编辑器在选择数据源时还不知道具体的 entityApiKey

**建议**：
- 新增后端接口 `GET /meta/related-objects?metamodelApiKey=role`，返回 role 通过 entityLink 关联的所有对象
- 或者在前端 `useCanvasMetadata` 启动时一次性加载所有 entityLink 数据，构建关联图

## 四、完善的关系体系设计

### 4.1 关系数据源矩阵

| 关系类型 | 数据来源 | 查询方式 | UI 用途 |
|---|---|---|---|
| 元模型类型父子 | `p_meta_link` | `GET /meta/meta-links` | 画布右侧约束：可选哪些子元模型 |
| 元数据实例关联 | `entityLink` 元数据 | `GET /meta/metadata?metamodelApiKey=entityLink` | 画布右侧约束：可选哪些关联对象 |
| 定义→实例 | 隐含 | `GET /meta/metadata?metamodelApiKey=X` | 元模型定义页面：查看实例列表 |
| 实例→数据 | `enable_data_source` + 路由表 | `GET /entity/data/{entityApiKey}` | 画布数据分类：加载业务数据 |
| 运行时引用 | `metadataRef` 元模型 | `MetadataRefService.findByTarget()` | 删除保护 |
| 自关联（树形） | 元数据实例的 parentApiKey 字段 | 前端 buildTree | 树形组件渲染 |

### 4.2 画布联动约束的完整推导逻辑

```
用户在画布右侧选择数据源时：

1. 获取左侧组件的 dataSource 配置
   → leftDs = { type: 'metadata', metamodelApiKey: 'entity' }

2. 判断左侧是定义层还是实例层
   → isDefinitionLayer = (leftDs.type 不是 'metadata' 且不是 'entityData')

3. 如果是定义层（如 metaModel）：
   a. 从 p_meta_link 查子元模型 → [metaItem, metaLink, metaOption]
   b. 元模型分类：显示子元模型 + 自身
   c. 元数据分类：显示所有可见元数据（跨层：定义→实例）
   d. 数据分类：不约束

4. 如果是实例层（如 entity）：
   a. 从 p_meta_link 查子元模型 → [item, checkRule, busiType, ...]
   b. 从 entityLink 查关联对象 → [contact, opportunity, ...]（需加载 entityLink 数据）
   c. 元模型分类：不显示（实例层的子元模型不在定义层）
   d. 元数据分类：显示子元模型 + 同模型 + 关联对象
   e. 数据分类：显示关联对象中 enable_data_source=1 的实例

5. 如果是独立元模型（如 role，无子元模型）：
   a. 从 p_meta_link 查子元模型 → []
   b. 从 entityLink 查关联对象 → [user]
   c. 元数据分类：显示同模型（role 自身）
   d. 数据分类：显示关联对象（user）
```

### 4.3 需要修正的元数据分类展示

当前"元数据"分类的下拉列表显示所有 `enable_ui_config=1 AND metamodel_layer=0` 的元模型（约 50 个），包括 item/pickOption/checkRule 等 entity 的子元模型。

**修正方案**：将元数据分类分为两组：

| 组 | 条件 | 数量 | 示例 | 展示方式 |
|---|---|---|---|---|
| 独立元模型 | `entity_dependency=0 AND metamodel_layer=0 AND enable_ui_config=1` | ~15 | entity, role, department, globalPickOption | 直接显示在下拉列表中 |
| 子元模型 | `entity_dependency=1 AND metamodel_layer=0 AND enable_ui_config=1` | ~35 | item, pickOption, checkRule, busiType | 只在左侧选了父元模型后，通过 p_meta_link 约束显示 |

这样用户在"元数据"分类中只看到 15 个独立元模型，而不是 50 个混在一起。

### 4.4 entityLink 数据的前端加载策略

**方案 A**：启动时全量加载
- `useCanvasMetadata` 启动时调用 `listMetadata('entityLink')` 加载所有 entityLink 数据
- 构建 `parentEntityApiKey → [childEntityApiKey]` 的关联图
- 优点：简单，一次加载
- 缺点：数据量可能较大（每个 entity 平均 5 条 entityLink，30 个 entity = 150 条）

**方案 B**：按需加载
- 当用户选择左侧数据源后，按 `entityApiKey` 查询 entityLink
- 优点：按需，数据量小
- 缺点：每次切换左侧数据源都要请求

**建议**：方案 A。150 条 entityLink 数据量很小，启动时一次加载即可。

| P2 | entity_dependency 字段语义收窄（仅存储层标记） | 设计规范 |
| P2 | 考虑 entityLink 重命名为 metadataLink | 长期规划 |
