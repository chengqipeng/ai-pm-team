# 元数据设计规范

本文档定义 aPaaS 平台元数据的存储、命名、关联、唯一性等规范。所有新增元数据必须遵循。

---

## 1. 双库架构

| 数据库 | 用途 | 表前缀 |
|---|---|---|
| paas_metarepo_common | 元模型定义 + Common 级元数据 | p_meta_*、p_common_* |
| paas_metarepo | Tenant 级元数据 + 日志等 | p_tenant_*、p_meta_log 等 |

---

## 2. 表命名规范

| 类型 | 前缀 | tenant_id | 示例 |
|---|---|---|---|
| 元模型定义 | p_meta_ | 无 | p_meta_model、p_meta_item |
| Common 元数据 | p_common_metadata | 无 | 统一大宽表，metamodel_api_key 区分类型 |
| Tenant 元数据 | p_tenant_ | 有 | p_tenant_entity、p_tenant_item |

Common 级元数据不使用独立表，统一存储在 `p_common_metadata` 大宽表中。

### 2.1 db_table 路由规则

`p_meta_model.db_table` 指向该元模型的 Tenant 级写入表，是 `DynamicTableNameHolder` 数据路由的核心依据。

**核心原则：**
- Common 级数据统一存储在 `p_common_metadata` 大宽表中（通过 `metamodel_api_key` 区分类型）
- Tenant 级数据存储在 `db_table` 指向的独立快捷表中
- 每个出厂元模型都有自己的独立 Tenant 快捷表，不共享 `p_tenant_metadata`
- 独立快捷表的结构与 `p_common_metadata` 完全一致（固定列 + dbc_xxxN 扩展列 + tenant_id）

**当前 23 种元模型的 db_table 配置：**

| 元模型 | enable_common | enable_tenant | db_table | Common 存储 | Tenant 存储 |
|---|---|---|---|---|---|
| entity | 1 | 1 | `p_tenant_entity` | p_common_metadata | p_tenant_entity |
| item | 1 | 1 | `p_tenant_item` | p_common_metadata | p_tenant_item |

| busiTypePickOption | 1 | 1 | `p_tenant_busi_type_pick_option` | p_common_metadata | p_tenant_busi_type_pick_option |

**读取路由逻辑：**

| enable_common | enable_tenant | 读取行为 |
|---|---|---|
| 1 | 1 | 先查 `p_common_metadata`（CommonMetadataConverter 转换），再查 `db_table` 指向的 Tenant 表，合并返回 |
| 1 | 0 | 仅查 `p_common_metadata` |
| 0 | 1 | 仅查 `db_table` 指向的 Tenant 表 |

**写入路由逻辑：** 所有写操作通过 `DynamicTableNameHolder.executeWith(dbTable)` 路由到 `db_table` 指向的 Tenant 表。

**新增元模型的 db_table 命名规则：**
- 格式：`p_tenant_{snake_case_name}`
- 新表通过 `CREATE TABLE ... LIKE p_common_metadata` + `ALTER TABLE ADD COLUMN tenant_id BIGINT NOT NULL` 创建
- 禁止多个元模型共享同一张 Tenant 表（`p_tenant_metadata` 仅作为兜底，出厂元模型不使用）

## 3. namespace 分类

所有元数据必须有 `namespace` 字段（VARCHAR(50) NOT NULL）：

| namespace | 含义 | 存储位置 | 可见性 |
|---|---|---|---|
| system | 系统出厂元数据 | p_common_metadata | 所有租户可见，不受 license 控制 |
| product | 业务产品出厂元数据 | p_common_metadata | 受 license 控制，付费可见 |
| custom | 客户自定义元数据 | p_tenant_* | 所有租户管理员可见 |

## 4. 主键与必填字段规范

### 4.1 主键规范（禁止使用自增 ID）

元数据表禁止使用 `id` 列作为主键，统一使用 `api_key` 或联合主键：

| 元数据类型 | 主键（p_common_metadata） | 主键（Tenant 表） |
|---|---|---|
| entity | (metamodel_api_key='entity', api_key) | (tenant_id, api_key) |
| item | (metamodel_api_key='item', api_key) | (tenant_id, entity_api_key, api_key) |
| pickoption | (metamodel_api_key='pick_option', api_key) | (tenant_id, entity_api_key, item_api_key, api_key) |
| entity_link | (metamodel_api_key='entity_link', api_key) | (tenant_id, api_key) |
| check_rule | (metamodel_api_key='check_rule', api_key) | (tenant_id, entity_api_key, api_key) |
| refer_filter | (metamodel_api_key='refer_filter', api_key) | (tenant_id, entity_api_key, item_api_key, api_key) |
| metadata（通用） | (metamodel_api_key, api_key) | (tenant_id, metamodel_api_key, api_key) |
| meta_model | api_key | - |
| meta_item | (metamodel_api_key, api_key) | - |
| meta_option | (metamodel_api_key, item_api_key, option_code) | - |

### 4.2 必填字段

所有元数据表必须包含以下字段：

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| namespace | VARCHAR(50) | NOT NULL | 元数据分类 |
| api_key | VARCHAR(255) | NOT NULL | 唯一标识（参与主键） |
| label | VARCHAR(255) | NOT NULL | 显示标签 |
| label_key | VARCHAR(255) | | 多语言 Key |
| custom_flg | INT | | 自定义标记（0=标准 1=自定义），由基类 BaseMetaCommonEntity 提供 |

所有文本字段必须有对应的 xxxKey 国际化字段：
- name → name_key
- label → label_key
- description → description_key
- help_text → help_text_key

Tenant 级表额外必须有：
- tenant_id BIGINT NOT NULL

Common 级表禁止有 tenant_id 字段。

## 6. Common/Tenant 合并查询规范

Common 级元数据统一存储在 `p_common_metadata` 大宽表中，通过 `metamodel_api_key` 区分类型。
查询时通过 `CommonTenantServiceImpl` 合并两层数据：

1. 查 p_common_metadata（WHERE metamodel_api_key = ? AND delete_flg = 0）
2. 通过 p_meta_item.db_column 映射，将 dbc_xxx_N 列还原为业务 Entity 字段
3. 查 Tenant 快捷表（p_tenant_*，WHERE tenant_id = ?）
4. 按 api_key 合并：
   - Common 有、Tenant 无 → 使用 Common
   - 同 api_key → Tenant 覆盖 Common
   - Tenant 独有 → 纯租户自定义
   - Tenant delete_flg=1 → 对此租户隐藏
5. namespace 过滤：product 类型检查 license

## 7. 字段命名规范

### 7.1 p_meta_item.api_key 命名规则

所有 p_meta_item 的 api_key 统一使用 camelCase，与 Java Entity 字段名保持一致：

| 规则 | 说明 | 正确示例 | 错误示例 |
|---|---|---|---|
| 统一 camelCase | 所有 api_key 使用 Java 风格驼峰 | `enableBusinessType` | `enable_busitype` |
| 与 Java 字段名一致 | api_key 值 = Java Entity 字段名 | `entityApiKey` | `entity_api_key`（作为 api_key 值时） |
| 禁止非标准缩写 | 除公认缩写（Api/Id/Url/Svg/Db）外不缩写 | `enableBusinessType` | `enableBusitype` |
| 复合词正确驼峰 | 每个单词首字母大写 | `enableCheckRule` | `enableCheckrule` |

> 注意：db_column 仍然使用 snake_case（如 `entity_api_key`、`dbc_varchar1`），这是数据库列名规范。

### 7.2 布尔字段命名规则

所有 0/1 布尔字段（Java Integer 类型）统一使用 `xxxFlg` 后缀：

| 规则 | 说明 | 正确示例 | 错误示例 |
|---|---|---|---|
| 统一 `*Flg` 后缀 | 所有布尔属性使用 `xxxFlg` | `historyLogFlg`、`checkRuleFlg` | `enableHistoryLog`、`enableCheckRule` |
| 禁止 `enable*` 前缀 | `enable` 暗示功能开关，但实际都是布尔属性 | `teamFlg` | `enableTeam` |
| 禁止 `is*` 前缀 | Java Integer 类型不用 `is` | `defaultFlg` | `isDefault` |
| 例外：`enableFlg` | 作为通用"是否启用"标记保留 | `enableFlg` | — |
| 例外：位掩码字段 | Long 类型的位掩码不是布尔，保留 `enable` 前缀 | `enableConfig`(Long)、`enableFlow`(Long) | — |

示例：
- ✅ `historyLogFlg`、`checkRuleFlg`、`teamFlg`、`sharingFlg`、`reportFlg`
- ❌ `enableHistoryLog`、`enableCheckRule`、`enableTeam`、`enableSharing`、`enableReport`

### 7.3 关联字段命名规则

| 规则 | 正确示例 | 错误示例 |
|---|---|---|
| 单个关联用 `xxxApiKey` | `entityApiKey`、`referEntityApiKey` | `entityId` |
| 多个关联用 `xxxApiKeys`（复数） | `referEntityApiKeys` | `referEntityApiKey`（存多个时） |

### 7.4 布尔字段存储与使用规范

**存储层：** 数据库 SMALLINT(0/1)，Java Integer，p_meta_item item_type=31, data_type=6。

**不使用 Java Boolean 的原因：**
- 三值语义：null（未配置/继承默认）、0（否）、1（是），Boolean 的 null 和 false 容易混淆
- Lombok getter：Boolean 字段生成 `isXxxFlg()`，引入 `is` 前缀，与命名规范冲突
- 大宽表 Converter：dbc_smallint 列读出 Integer，无需额外转换
- DDL 兼容：MySQL BOOLEAN=TINYINT(1) 与 PostgreSQL 原生 BOOLEAN 不一致

**代码中使用 `MetaBooleanUtils` 工具类：**

```java
import static com.hongyang.platform.metarepo.service.common.utils.MetaBooleanUtils.*;

// 判断
if (isTrue(entity.getEnableFlg())) { ... }
if (isFalse(item.getRequireFlg())) { ... }
if (isNull(item.getConfigFlg())) { /* 未配置，使用默认值 */ }

// 赋值
entity.setEnableFlg(TRUE);
entity.setHiddenFlg(FALSE);
item.setRequireFlg(toFlg(someCondition));
```

**禁止写法：**
```java
// ❌ 直接比较，null 会 NPE
if (entity.getEnableFlg() == 1) { ... }

// ❌ 使用 Boolean 类型
private Boolean enableFlg;  // 禁止

// ❌ 硬编码魔法数字
entity.setEnableFlg(1);  // 应使用 MetaBooleanUtils.TRUE
```

## 8. DDL 兼容规范

所有 DDL 必须同时兼容 MySQL 和 PostgreSQL：
- 主键：BIGINT NOT NULL（雪花算法）
- 字符串：VARCHAR(N)
- 布尔/枚举：SMALLINT
- 时间戳：BIGINT（毫秒）
- 禁止：AUTO_INCREMENT、ENGINE、COMMENT、BOOLEAN、ENUM

## 8. Common/Tenant 合并查询详细规范

### 8.1 合并算法

```
merge(commonList, tenantList):
    tenantMap = tenantList 按 apiKey 建索引

    for common in commonList:
        tenant = tenantMap.remove(common.apiKey)
        if tenant != null:
            if tenant.deleteFlg == 1 → 跳过（对此租户隐藏）
            else → fillNullFields(tenant, common)  // Tenant 覆盖，null 字段从 Common 回填
                   result.add(tenant)
        else:
            result.add(common)

    for tenant in tenantMap.values():  // Tenant 独有
        if tenant.deleteFlg != 1 → result.add(tenant)
```

### 8.2 Tenant 覆盖时的 null 字段回填（fillNullFields）

当 Tenant 数据覆盖 Common 数据时，Tenant 中为 null 的字段自动从 Common 回填。

**原因**：租户自定义时通常只修改部分字段（如 label、itemOrder），其他字段（如 dataType、dbColumn）保持 Common 原值。如果不回填，Tenant 覆盖后这些字段会丢失。

**实现**：`MetadataMergeReadServiceImpl.fillNullFields(target, source)` — 遍历所有字段，target 为 null 时从 source 取值。

### 8.3 字段类型自动推导（CommonMetadataConverter Step 3）

当 `itemSubType` 或 `dataType` 为空时，从 `itemType` 自动推导：

| 条件 | 推导规则 |
|---|---|
| `itemSubType == null && itemType != null` | 非计算型：`itemSubType = itemType`；计算型（27）：`itemSubType = 0` |
| `dataType == null && itemType != null` | `dataType = ItemTypeEnum.fromCode(itemType).defaultDataType.code` |

### 8.4 业务数据公用字段注入（CommonFieldProvider）

业务数据大宽表 `p_tenant_data` 的 19 个固定列是所有 entity 共享的公用字段。

**问题**：`p_common_metadata` 主键 `(metamodel_api_key, api_key)` 不含 `entity_api_key`，无法为每个 entity 分别存储同名公用字段（如每个 entity 都有 `id`、`name`）。

**方案**：通过 `CommonFieldProvider.injectCommonFields()` 在查询 item 列表时动态注入，不依赖数据库存储。仅当 `metamodelApiKey = 'item'` 时注入，去重（已存在同 apiKey 的记录则跳过）。

19 个公用字段详见 [业务数据大宽表设计.md](业务数据大宽表设计.md) §3.2.1。

## 9. 两层 dbTable 与分片路由

系统中有两个不同含义的 `dbTable`：

| 层级 | 所在位置 | 含义 | 示例 |
|---|---|---|---|
| 元模型层 | `p_meta_model.db_table` | **元数据**存储的 Tenant 级大宽表 | entity → `p_tenant_entity` |
| 元数据层 | entity 记录的 `dbTable`（`dbc_varchar2`） | **业务数据**存储的真实分片表名 | account → `p_tenant_data_0` |

**关键规则**：
- 元模型层的 `db_table` 每种元模型不同（`p_tenant_entity`、`p_tenant_item` 等），用于元数据读写路由
- 元数据层的 `dbTable` 存储真实分片表名（`p_tenant_data_{N}`），由 `p_tenant_data_route` 按 `(tenant_id, entity_api_key)` 分配
- 初始化时通过 `init_local_dev.py` 为每个 entity 分配路由并将分片表名写入 `dbTable`
- 运行时 `TableRouteService.resolveTableName()` 也通过 `p_tenant_data_route` 查询，与 `dbTable` 存储的值一致

## 10. 本地开发环境初始化

使用统一脚本 `sql/init_local_dev.py` 一键初始化：

```bash
# 完整初始化（建表 + 从远程同步数据 + 数据修正）
python3 sql/init_local_dev.py

# 仅建表
python3 sql/init_local_dev.py --create

# 仅同步数据
python3 sql/init_local_dev.py --sync

# 仅执行数据修正
python3 sql/init_local_dev.py --fix
```

**数据修正内容**：
1. 补充 `MetamodelApiKeyEnum` 中注册但远程缺失的元模型定义
2. 回填元模型定义表中 null 的 `id` 列
3. 统一所有 entity 的 `dbTable` 为 `p_tenant_data`

**前端配置**：
- axios 请求 header 使用 `x-tenant-id`（非 `tenantId`），与后端 `GlobalContextConstant` 对齐
- 默认 tenantId 为 `292193`（与 `App.tsx` 一致）
