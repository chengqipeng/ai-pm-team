# 默认值管理设计方案

> 版本：1.0 | 日期：2026-04-21
> 目标：defaultFlg 的完整生命周期管理，前后端统一校验

---

## 一、业务规则

### 1.1 单默认模式（defaultFlg 互斥）

适用场景：语言管理、时区管理等，同一分组下只能有一个默认值。

规则：
- 新增记录设为默认 → 自动清除同组其他记录的默认标记
- 编辑记录设为默认 → 同上
- 删除默认记录 → **必须指定新的默认记录**，不允许无默认
- 至少保留一条默认记录

### 1.2 多默认模式（defaultFlg 可多选）

适用场景：某些配置允许多个默认值。

规则：
- 新增/编辑设为默认 → 不清除其他默认
- 删除默认记录 → 如果还有其他默认记录则允许，否则需要指定

### 1.3 模式配置

在元模型级别配置默认值模式，通过 p_meta_model 的扩展字段：

| 字段 | 说明 |
|:---|:---|
| defaultMode | 0=无默认值管理 1=单默认（互斥） 2=多默认 |

当前 globalPickOption 应该是 `defaultMode=1`（单默认互斥）。

---

## 二、后端校验

### 2.1 删除校验

`MetadataMergeWriteServiceImpl.delete` 中：

```
if 被删除记录的 defaultFlg == 1:
    查询同组其他 defaultFlg=1 的记录
    if 没有其他默认记录:
        if 请求中指定了 newDefaultApiKey:
            将 newDefaultApiKey 设为默认
        else:
            抛异常："删除默认记录时必须指定新的默认记录"
```

### 2.2 创建/更新校验

已有的 `clearOtherDefaults` 方法处理单默认互斥。

### 2.3 API 扩展

删除默认记录时，请求体需要额外传 `newDefaultApiKey`：

```json
{
  "delete_map": {
    "globalPickOption": [
      { "api_key": "asiaShanghai", "new_default_api_key": "asiaTokyo" }
    ]
  }
}
```

## 三、前端交互

### 3.1 删除默认记录时

点击删除按钮 → 检测到该记录是默认记录 → 弹出选择框：

```
┌─────────────────────────────────────┐
│  删除默认记录                        │
│                                     │
│  「Asia/Shanghai (UTC+8)」是当前     │
│  默认时区，删除后需要选择新的默认：   │
│                                     │
│  ○ Asia/Hong_Kong (UTC+8)           │
│  ○ Asia/Taipei (UTC+8)              │
│  ● Asia/Tokyo (UTC+9)               │
│  ○ Asia/Seoul (UTC+9)               │
│  ...                                │
│                                     │
│           [取消]  [确认删除]          │
└─────────────────────────────────────┘
```

### 3.2 实现位置

在 ActionContext 的 delete 处理中：
1. 检查被删除记录的 defaultFlg
2. 如果是默认记录 → 弹出选择新默认的弹框
3. 用户选择后 → 先设置新默认 → 再删除旧记录
