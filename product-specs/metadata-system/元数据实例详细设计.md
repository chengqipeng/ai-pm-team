## 1. 总览

### 1.1 元数据实例的本质

元数据实例是"描述业务数据结构的数据"。例如：
- entity 实例 `account`（客户）描述了"客户"这个业务对象的结构
- item 实例 `accountName`（客户名称）描述了"客户名称"这个字段的类型、约束、显示规则
- pickOption 实例 `IT`（IT行业）描述了"行业"字段的一个可选值

### 1.2 数据规模（当前状态）

| 层级 | 存储位置 | 数据量 | 说明 |

| Tenant 级 | p_tenant_* (29张表) | ~220,000 条 | tid=292193 的租户自定义数据 |

## 2. 层级关系与数据流

### 2.1 完整层级树

```
entity（业务对象）— 如 account(客户)、opportunity(商机)
  │
  ├── item（字段）— 如 accountName(客户名称)、phone(电话)
  │     ├── pickOption（选项值）— 如 IT(IT行业)、Finance(金融)
  │     ├── referenceFilter（关联过滤）— LOOKUP 字段的下拉过滤条件
  │     │     └── referenceFilterDetail（过滤条件明细）
  │     ├── formulaCompute（计算公式）— FORMULA 字段的公式定义
  │     │     ├── formulaComputeItem（公式引用字段）
  │     │     └── computeFactor（计算因子）
  │     ├── aggregationCompute（汇总累计）— ROLLUP 字段的汇总定义
  │     │     ├── aggregationComputeDetail（汇总条件）
  │     │     └── computeFactor（计算因子，共享）
  │     └── itemDependency（字段依赖）— PICKLIST 字段级联
  │           └── itemDependencyDetail（依赖明细）
  │
  ├── entityLink（关联关系）— 如 ContactToAccount(联系人→客户)
  ├── checkRule（校验规则）— 如 AccountNameRequired(客户名称必填)
  ├── busiType（业务类型）— 如 Enterprise(企业客户)、Personal(个人客户)
  │     ├── busiTypePickOption（业务类型选项值）
  │     └── busiTypeMapping（业务类型映射）
  │           └── busiTypeMappingDetail（映射明细）
  ├── duplicateRule（查重规则）
  │     ├── duplicateRuleCriteria（查重条件）
  │     └── duplicateMatchingRule（查重匹配规则）
  └── fieldSet（字段集）— 字段分组
        └── fieldSetItem（字段集字段）

globalPickOption（全局选项集）— 如 province(省份)、city(城市)
  └── globalPickDependency（全局选项集依赖）
        └── globalPickDependencyDetail（依赖明细）

entitySvg（实体图标）— 独立，entity.svgApiKey 引用
```

### 2.2 数据流转：创建一个完整的业务对象

以创建"客户(account)"对象为例，展示元数据的完整生命周期：

```
Step 1: 创建 entity
  → p_common_metadata INSERT (metamodel_api_key='entity', api_key='account', label='客户')
  → dbc_int1=0(标准对象), dbc_varchar1='customer'(svgApiKey), dbc_varchar2='t_account'(dbTable)

Step 2: 创建 item（字段）
  → p_common_metadata INSERT (metamodel_api_key='item', entity_api_key='account', api_key='accountName')
  → dbc_int1=1(TEXT), dbc_varchar3='dbc_varchar1'(dbColumn), dbc_smallint1=1(必填)

Step 3: 创建 pickOption（选项值，仅 PICKLIST 类型字段）
  → p_common_metadata INSERT (metamodel_api_key='pickOption', entity_api_key='account')
  → parent_metadata_api_key='industryId'(所属字段), label='IT行业'

Step 4: 创建 entityLink（关联关系）
  → p_common_metadata INSERT (metamodel_api_key='entityLink', entity_api_key='account')
  → dbc_varchar2='contact'(parentEntity), dbc_varchar3='account'(childEntity), dbc_int1=0(LOOKUP)

Step 5: 创建 checkRule（校验规则）
  → p_common_metadata INSERT (metamodel_api_key='checkRule', entity_api_key='account')
  → dbc_textarea1='NOT ISBLANK(accountName)'(公式), dbc_varchar1='客户名称不能为空'(错误提示)

Step 6: 租户自定义（Tenant 级覆盖）
  → p_tenant_entity INSERT (tenant_id=292193, metamodel_api_key='entity', api_key='account')
  → 租户可以修改 label、添加自定义字段、修改校验规则等
  → 读取时 Tenant 覆盖 Common，实现租户个性化
```

### 3.3 entityLink（关联关系）

**业务含义**：定义对象之间的关联关系，如"联系人→客户"(LOOKUP)、"订单明细→订单"(主从)。

**核心字段**：

| 字段 | db_column | 业务语义 | 示例 |
|:---|:---|:---|:---|
| apiKey | api_key | 关联唯一标识 | `contact_account_lookup` |
| parentEntityApiKey | dbc_varchar2 | 父对象（被引用方） | `account` |
| childEntityApiKey | dbc_varchar3 | 子对象（引用方） | `contact` |
| linkType | dbc_int1 | 0=LOOKUP, 1=主从, 2=多对多 | `0` |
| cascadeDelete | dbc_smallint2 | 级联删除策略 | `1` |
| typeProperty | dbc_varchar1 | 扩展属性 JSON | `{}` |

**关联类型说明**：
- LOOKUP(0)：查找关系，子对象通过字段引用父对象，无级联约束
- 主从(1)：父子关系，子对象生命周期依赖父对象，支持级联删除
- 多对多(2)：通过中间表关联，双方独立

## 8. Common/Tenant 合并读取详细流程

### 8.1 合并规则

```
输入：metamodelApiKey, entityApiKey, tenantId
输出：合并后的业务 Entity 列表

1. 查 MetaModel 定义
   metaModel = p_meta_model WHERE api_key = metamodelApiKey
   enableCommon = metaModel.enable_common
   enableTenant = metaModel.enable_tenant
   dbTable = metaModel.db_table

2. 查 Common 层（如果 enableCommon=1）
   commonRows = p_common_metadata
     WHERE metamodel_api_key = metamodelApiKey
       AND entity_api_key = entityApiKey
       AND delete_flg = 0
   → 通过 CommonMetadataConverter 转为业务 Entity

3. 查 Tenant 层（如果 enableTenant=1）
   tenantRows = {dbTable}  // DynamicTableNameHolder 路由
     WHERE tenant_id = tenantId
       AND metamodel_api_key = metamodelApiKey
       AND entity_api_key = entityApiKey

4. 合并（仅当 enableCommon=1 且 enableTenant=1）
   result = {}
   for row in commonRows:
     result[row.apiKey] = row
   for row in tenantRows:
     if row.deleteFlg == 1:
       result.remove(row.apiKey)  // Tenant 遮蔽删除
     else:
       result[row.apiKey] = row   // Tenant 覆盖 Common
   return result.values()
```

### 8.2 CommonMetadataConverter 转换细节

```
CommonMetadata（大宽表一行）→ 业务 Entity（如 EntityItem）

Step 1: 固定列同名映射
  apiKey ← api_key
  label ← label
  labelKey ← label_key
  namespace ← namespace
  entityApiKey ← entity_api_key
  customFlg ← custom_flg
  description ← description
  deleteFlg ← delete_flg
  createdAt/By, updatedAt/By ← 对应列

Step 2: dbc 列映射（通过 p_meta_item.db_column）
  columnMapping = getColumnMapping(metamodelApiKey)
  // 返回 {dbc_int1: itemType, dbc_varchar3: dbColumn, ...}
  for (dbcCol, fieldName) in columnMapping:
    value = commonMetadata.get(dbcCol)  // 如 dbc_int1 的值
    entity.set(fieldName, value)        // 设置 EntityItem.itemType
```

### 8.3 DynamicTableNameHolder 路由机制

```java
// 写入 Tenant 级数据时
DynamicTableNameHolder.executeWith("p_tenant_item", () -> {
    // TenantMetadata 的 @TableName("p_tenant_metadata") 被拦截
    // DynamicTableNameInterceptor 将表名替换为 "p_tenant_item"
    tenantMetadataMapper.insert(tenantMetadata);
});

// 读取 Tenant 级数据时
DynamicTableNameHolder.executeWith("p_tenant_item", () -> {
    return tenantMetadataMapper.selectList(wrapper);
});
```

## 9. 写入流程详细设计

### 9.1 创建元数据

```
输入：metamodelApiKey, 业务 Entity（如 XEntityItem）
处理：
  1. 校验 apiKey 唯一性（同 entity 内不重复）
  2. 校验字段值合法性（通过 p_meta_option 约束）
  3. 校验父元数据存在（通过 p_meta_link 确定父子关系）
  4. 转换为 TenantMetadata（业务字段 → dbc 列）
  5. DynamicTableNameHolder 路由到目标 Tenant 表
  6. INSERT
  7. 记录变更日志（p_tenant_meta_log）
```

### 9.2 更新元数据

```
输入：metamodelApiKey, apiKey, 更新字段
处理：
  1. 查询现有记录（Common + Tenant 合并）
  2. 如果是 Common 级数据且 Tenant 无覆盖 → 创建 Tenant 覆盖记录
  3. 如果已有 Tenant 记录 → 直接更新
  4. 记录变更日志
```

### 9.3 删除元数据

```
输入：metamodelApiKey, apiKey
处理：
  1. 查询现有记录
  2. 如果是 Tenant 独有 → 物理删除或软删除
  3. 如果是 Common 级数据 → 插入 delete_flg=1 的 Tenant 记录（遮蔽删除）
  4. 通过 p_meta_link 级联删除子元数据
  5. 记录变更日志
```
