# 元模型关系体系设计方案

> 版本：1.0 | 日期：2026-04-22
> 目标：构建完善的关系体系，支撑画布 UI 联动约束

---

## 一、两个关系数据源，职责清晰

| 数据源 | 存储 | 记录什么 | 粒度 | 数据量 |
|---|---|---|---|---|
| **p_meta_link** | `paas_metarepo_common.p_meta_link` | 元模型之间的父子包含关系 | 类型级 | 36 条 |
| **entityLink** | `p_common_metadata` (metamodel_api_key='entityLink') | 元数据实例之间的关联关系 | 实例级 | 30+ 条 |

### 1.1 p_meta_link — 元模型之间的关系

回答的问题：**"元模型 A 包含哪些子元模型？"**

```
entity 包含 item          → entity 的每个实例下都有 item 子数据
item 包含 pickOption      → item 的每个实例下都有 pickOption 子数据
metaModel 包含 metaItem   → 每个 metaModel 都有 metaItem 字段定义
```

特点：
- 父子关系，子的生命周期依赖父
- 子数据通过 `referItemApiKey` 指定的字段关联到父
- 所有元模型都在 `p_meta_model` 中注册
- 关系是静态的，不随实例数据变化

### 1.2 entityLink — 元数据实例之间的关联

回答的问题：**"元数据实例 A 可以关联哪些对象的数据？"**

```
role 关联 user            → 角色可以关联用户（通过 p_user_role）
department 关联 user      → 部门包含用户（通过 p_user.departApiKey）
account 关联 contact      → 客户关联联系人（通过 RELATION_SHIP 字段）
role 自关联 role          → 角色有上级角色（树形）
```

特点：
- 关联关系，各自有独立生命周期
- 关联的对象可以是元数据（role）也可以是业务数据（user）
- 关系可以跨层（元数据 → 业务数据）
- 关系可以是动态的（entity 之间的关联由 entityLink 实例数据决定）

## 四、画布联动约束的完整逻辑

### 4.1 前端数据加载

`useCanvasMetadata` 启动时并行加载：

```typescript
const [metamodels, metaLinks, entityLinks] = await Promise.all([
  listMetaModels(),                          // 62 个元模型
  listMetaLinks(),                           // 36 条类型级关系
  listMetadata('entityLink'),                // 30+ 条实例级关联
]);
```

构建三个索引：

```typescript
// 索引 1：元模型父子关系（p_meta_link）
// key = 父元模型 apiKey, value = 子元模型 apiKey 列表
childrenByParent: {
  metaModel: ['metaItem', 'metaLink', 'metaOption'],
  entity: ['item', 'entityLink', 'checkRule', 'busiType', ...],
  item: ['pickOption', 'referenceFilter', 'formulaCompute', ...],
  ...
}

// 索引 2：元数据实例关联（entityLink）
// key = parentEntityApiKey, value = childEntityApiKey 列表
associationsByParent: {
  role: ['role', 'user'],           // role 自关联 + role→user
  department: ['department', 'user'], // department 自关联 + department→user
  account: ['contact', 'opportunity', 'lead'],
  ...
}

// 索引 3：独立元模型列表（无父元模型）
// 用于画布"元数据"分类的顶层显示
independentMetamodels: ['entity', 'role', 'department', 'globalPickOption', ...]
```

### 4.2 约束规则

当用户编辑右侧组件的数据源时：

```
输入：左侧组件的 dataSource 配置
输出：右侧三个分类（元模型/元数据/数据）各自可选的列表

规则：

1. 左侧是定义层（type = metaModel/metaItem/metaLink/metaOption）
   元模型分类：childrenByParent[leftType] + leftType 自身
   元数据分类：所有独立元模型（跨层：定义→实例）
   数据分类：不约束

2. 左侧是实例层（type = metadata, metamodelApiKey = X）
   元模型分类：不显示
   元数据分类：childrenByParent[X] + X 自身（同模型联动）
   数据分类：associationsByParent[X]（entityLink 关联的对象）

3. 左侧是数据层（type = entityData）
   不约束（数据层组件通常不作为左侧）

4. 无左侧组件
   全部不约束
```

### 4.3 具体场景验证

#### 场景 1：元模型定义页面

左侧：`{ type: 'metaModel' }` → 定义层

| 分类 | 可选列表 | 来源 |
|---|---|---|
| 元模型 | metaItem, metaLink, metaOption, metaModel | childrenByParent['metaModel'] + 自身 |
| 元数据 | entity, role, department, globalPickOption, ... | 所有独立元模型 |
| 数据 | 不约束 | — |

#### 场景 2：实体管理页面

左侧：`{ type: 'metadata', metamodelApiKey: 'entity' }`

| 分类 | 可选列表 | 来源 |
|---|---|---|
| 元模型 | 不显示 | — |
| 元数据 | item, entityLink, checkRule, busiType, ..., entity | childrenByParent['entity'] + 自身 |
| 数据 | account, contact, opportunity, ... | associationsByParent['entity']（entity 实例之间的关联） |

注意：数据分类需要特殊处理。`associationsByParent['entity']` 返回的是 entity 实例之间的关联（如 account→contact），但这些关联取决于具体选中的 entity 实例。在画布配置阶段，还不知道用户会选哪个 entity 实例，所以数据分类应该显示所有 `enable_data_source=1` 的 entity 实例。

#### 场景 3：角色管理页面

左侧：`{ type: 'metadata', metamodelApiKey: 'role' }`

| 分类 | 可选列表 | 来源 |
|---|---|---|
| 元模型 | 不显示 | — |
| 元数据 | role | childrenByParent['role']=[] + 自身 |
| 数据 | user | associationsByParent['role']=['role','user']，过滤掉自关联 |

#### 场景 4：部门管理页面

左侧：`{ type: 'metadata', metamodelApiKey: 'department' }`

| 分类 | 可选列表 | 来源 |
|---|---|---|
| 元模型 | 不显示 | — |
| 元数据 | department | childrenByParent['department']=[] + 自身 |
| 数据 | user | associationsByParent['department']=['department','user']，过滤掉自关联 |

#### 场景 5：全局选项集管理页面

左侧：`{ type: 'metadata', metamodelApiKey: 'globalPickOption' }`

| 分类 | 可选列表 | 来源 |
|---|---|---|
| 元模型 | 不显示 | — |
| 元数据 | globalPickDependency, globalPickOption | childrenByParent['globalPickOption'] + 自身 |
| 数据 | 无 | associationsByParent['globalPickOption']=[] |

## 五、数据分类的特殊处理

"数据"分类显示的是业务数据对象（第三层），来源有两个：

### 5.1 来自 entityLink 的关联对象

当左侧是 role/department 等独立元模型时，`associationsByParent` 返回关联的对象（如 user）。

但 user 是 entity 的实例，不是独立的数据源类型。画布中选了 user 后，需要知道用什么 API 加载数据：
- user 的数据在 `p_user` 表 → 走 `/entity/data/user` 接口
- 这个信息从 `p_meta_model(entity).enable_data_source=1` + `p_tenant_data_route` 获取

### 5.2 来自 enable_data_source 的 entity 实例

当左侧是 entity 时，右侧"数据"分类应该显示所有 entity 实例（account, opportunity, ...），因为 entity 实例之间可以互相关联。

这个列表已经在 `useCanvasMetadata` 中通过 `enable_data_source=1` 加载了。

### 5.3 合并逻辑

```typescript
// 右侧"数据"分类的可选列表
function getDataOptions(leftMetamodelApiKey: string): string[] {
  // 从 entityLink 获取关联对象
  const associations = associationsByParent[leftMetamodelApiKey] || [];
  // 过滤掉自关联
  const filtered = associations.filter(a => a !== leftMetamodelApiKey);
  // 如果左侧是 entity，显示所有 entity 实例（enable_data_source=1）
  if (leftMetamodelApiKey === 'entity') {
    return entityDataSubTypes; // 已有的 entity 实例列表
  }
  return filtered;
}
```
