# 业务数据表元模型化与大宽表分析

> 日期：2026-04-16
> 前置：[元模型设计体系](../元模型设计体系.md)、[operate-log-passport-userRole-元模型化分析](operate-log-passport-userRole-元模型化分析.md)

## 一、概念澄清：元模型表 vs 元数据表 vs 业务数据表

### 1.1 三层架构中的表分类

```
第一层：元模型表（定义"有哪些类型"）
  → p_meta_model, p_meta_item, p_meta_link, p_meta_option
  → 存储在 paas_metarepo_common，无 tenant_id

第二层：元数据表（存储"类型的实例数据"）
  → p_common_metadata（Common 级，无 tenant_id）
  → p_tenant_entity, p_tenant_item, p_tenant_role, p_tenant_department ...（Tenant 级快捷表）
  → 走 MetaServiceImpl / CommonTenantServiceImpl 链路
  → 支持 Common/Tenant 合并读取

第三层：业务数据表（存储"最终用户操作的业务记录"）
  → p_tenant_data_*（业务数据分片表）
  → p_user, p_passport, p_user_role, p_*_operate_log ...（独立业务表）
  → 走 DataBaseServiceImpl / EntityDataService 链路
```

### 1.2 关键区分标准

| 判断维度 | 元模型表 | 元数据表 | 业务数据表 |
|:---|:---|:---|:---|
| 存储什么 | 类型定义（"有哪些类型"） | 类型的实例（"某类型的具体配置"） | 最终用户的业务记录 |
| 是否走合并读取 | — | ✅ Common/Tenant 合并 | ❌ 不合并 |
| 是否支持租户自定义字段 | — | ✅ 通过 p_meta_item 扩展 | 取决于是否用大宽表 |
| 典型例子 | p_meta_model, p_meta_item | p_tenant_entity, p_tenant_role | p_tenant_data_*, p_user |

### 1.3 p_passport / p_passport_log / p_user_role / p_*_operate_log 的分类

| 表 | 本质 | 分类 | 理由 |
|:---|:---|:---|:---|
| p_passport | 全局认证凭证 | **业务数据表** | 不定义类型，不是类型实例，不走合并读取 |
| p_passport_log | 密码变更历史 | **业务数据表** | 审计日志，不参与元数据体系 |
| p_user_role | 用户角色关联 | **业务数据表** | 纯关联表，不是类型实例 |
| p_meta_operate_log | 元数据操作回滚日志 | **业务数据表** | 系统补偿机制，不是类型实例 |
| p_entity_operate_log | 业务数据操作回滚日志 | **业务数据表** | 同上 |

**结论：这 5 张表都是业务数据表，不应注册到 p_meta_model。**

`p_meta_model` 的职责是注册元模型（类型定义），不应成为"所有表的注册中心"。
这些表的租户拦截行为通过 Java Entity 上的 `@IgnoreTenantLine` 注解声明，不需要 `p_meta_model.enable_tenant_intercept` 配置。

### 1.4 enable_tenant_intercept 的配置载体

| 表的类型 | 配置载体 | 说明 |
|:---|:---|:---|
| 元模型管理的表 | `p_meta_model.enable_tenant_intercept` | 元模型配置驱动 |
| 业务数据表 | Java Entity `@IgnoreTenantLine` 注解 | 代码声明驱动 |
| 元模型定义表（p_meta_*） | `TenantInterceptor.ignoreTable()` 硬编码 | 框架基石 |

## 二、大宽表必要性深度分析

### 2.1 大宽表解决什么问题

大宽表（dbc_xxxN 扩展列）的核心价值是**零 DDL 扩展**：
- 新增字段不需要 ALTER TABLE，只需在 p_meta_item 中注册一条记录
- 租户可以自定义字段，不同租户的同一张表可以有不同的字段
- 字段类型、校验规则、显示名称全部由元数据驱动

### 2.2 逐表分析

#### p_passport — 全局身份凭证

**当前字段：**
```
id, phone, password, password_salt, valid, pwd_flg
+ BaseEntity 6 字段（delete_flg, created_at/by, updated_at/by）
```

**未来演进可能：**

| 演进方向 | 可能性 | 新增字段 | 是否需要大宽表 |
|:---|:---|:---|:---|
| 多因素认证 | 高 | mfa_enabled, mfa_type, mfa_secret | 字段固定，ALTER TABLE 即可 |
| 社交登录 | 高 | wechat_openid, dingtalk_id, feishu_id | 第三方绑定已有独立表 p_tenant_third_binding |
| 邮箱登录 | 中 | email（作为登录凭证） | 1 个字段，ALTER TABLE |
| 账号合并 | 低 | merged_to_id | 1 个字段 |
| 租户自定义认证字段 | **极低** | — | 认证凭证是平台级，不应由租户自定义 |

**结论：❌ 不需要大宽表。**

理由：
1. Passport 是**平台级全局表**，无 tenant_id，不存在"不同租户不同字段"的场景
2. 认证凭证的字段语义必须严格固定（手机号、密码、有效状态），不能让租户随意扩展
3. 未来新增字段（MFA、邮箱）是平台级统一升级，ALTER TABLE 一次性完成
4. 安全敏感数据不适合放在大宽表中（大宽表的 dbc 列没有列级权限控制）

#### p_passport_log — 密码变更日志

**当前字段：**
```
id, passport_id, password, change_type
+ BaseEntity 6 字段
```

**未来演进可能：**

| 演进方向 | 可能性 | 新增字段 | 是否需要大宽表 |
|:---|:---|:---|:---|
| 记录变更来源 | 高 | source_ip, user_agent | 审计增强，固定字段 |
| 记录操作人 | 中 | operator_id, operator_type | 固定字段 |
| 租户自定义日志字段 | **极低** | — | 密码日志是安全审计，不应自定义 |

**结论：❌ 不需要大宽表。**

理由：
1. 同 Passport，全局表，无租户自定义需求
2. 审计日志的字段必须固定，确保合规性和可追溯性
3. 未来扩展（IP、UA）是平台级统一升级

#### p_meta_operate_log / p_entity_operate_log — 操作回滚日志

**当前字段：**
```
id, tenant_id, process_id, target_type, target_key, target_id,
dml_type, before_value, rollback_status
+ BaseEntity 6 字段
```

**未来演进可能：**

| 演进方向 | 可能性 | 新增字段 | 是否需要大宽表 |
|:---|:---|:---|:---|
| 记录操作人 | 高 | operator_id | 1 个字段，ALTER TABLE |
| 记录操作耗时 | 中 | duration_ms | 1 个字段 |
| 记录回滚原因 | 中 | rollback_reason | 1 个字段 |
| after_value（操作后快照） | 低 | after_value TEXT | 1 个字段 |
| 租户自定义日志字段 | **极低** | — | 系统补偿机制，不应自定义 |

**结论：❌ 不需要大宽表。**

理由：
1. 操作日志是**系统内部补偿机制**，不面向最终用户，不存在租户自定义需求
2. 这些表的生命周期很短（commit 后删除），不是长期存储
3. `before_value` 已经是 TEXT/JSON 格式，可以存储任意结构的快照数据，天然具备扩展性
4. 启动恢复需要跨租户扫描，大宽表的 DynamicTableNameHolder 路由会增加复杂度

#### p_user_role — 用户角色关联

**当前字段：**
```
id, tenant_id, user_id, role_api_key
+ BaseEntity 6 字段
```

**未来演进可能：**

| 演进方向 | 可能性 | 新增字段 | 是否需要大宽表 |
|:---|:---|:---|:---|
| 角色有效期 | 中 | valid_from, valid_to | 2 个字段 |
| 角色来源 | 中 | assign_type（手动/自动/继承） | 1 个字段 |
| 角色优先级 | 低 | priority | 1 个字段 |
| 租户自定义关联属性 | **低** | — | 关联表的语义是"谁有什么角色"，扩展空间有限 |

**结论：❌ 不需要大宽表。**

理由：
1. 关联表的核心语义是"A 关联 B"，字段天然很少
2. 即使加有效期、来源等字段，也是平台级统一升级，不是租户自定义
3. 关联表的查询模式是高频 JOIN/IN，大宽表的 dbc 列会降低查询效率
4. 关联表的数据量可能很大（用户数 × 角色数），大宽表的列数开销不合理

### 2.3 什么样的表需要大宽表？

| 特征 | 需要大宽表 | 不需要大宽表 |
|:---|:---|:---|
| 租户可自定义字段 | ✅ | — |
| 字段数量可能持续增长 | ✅ | — |
| 不同租户字段不同 | ✅ | — |
| 面向最终用户操作 | ✅ | — |
| 系统内部基础设施 | — | ✅ |
| 安全敏感数据 | — | ✅ |
| 字段语义必须固定 | — | ✅ |
| 关联表（字段极少） | — | ✅ |

当前系统中使用大宽表的表：
- `p_tenant_data_*`（业务数据）— 租户自定义字段（Account 的行业、Contact 的职位等）
- `p_user`（用户数据）— 租户自定义字段（jobTitle、workLocation 等）
- `p_tenant_entity/item/...`（元数据快捷表）— 元模型字段定义驱动

这 4 张表（passport、passportLog、operateLog×2）+ userRole 都不满足大宽表的使用条件。

### 2.4 总结

| 表 | 需要大宽表 | 需要元模型注册 | 需要 p_meta_item | 租户拦截配置方式 |
|:---|:---|:---|:---|:---|
| p_passport | ❌ | ❌ | ❌ | @IgnoreTenantLine |
| p_passport_log | ❌ | ❌ | ❌ | @IgnoreTenantLine |
| p_meta_operate_log | ❌ | ❌ | ❌ | @IgnoreTenantLine |
| p_entity_operate_log | ❌ | ❌ | ❌ | @IgnoreTenantLine |
| p_user_role | ❌ | ❌ | ❌ | @IgnoreTenantLine |

这些表保持独立表结构，通过 `@IgnoreTenantLine` 注解声明租户拦截行为，不纳入元模型体系。
未来字段扩展通过 ALTER TABLE 完成（平台级统一升级），不需要零 DDL 的大宽表能力。
