## 一、百事通功能演进总览

百事通是销售助理 Agent 中的核心能力，定位为销售人员的"全能百科全书"，整合了四大数据源：

| 数据源 | API名称 | 说明 |
|:---|:---|:---|
| 查数据 | query_CRM_data | 查询CRM系统内的业务数据（客户/商机/联系人/订单等） |
| 问知识库 | search_documents | 企业内部知识库的检索与结构化问答 |
| 搜全网 | search_web | 查询公开的背景信息、业务情报等 |
| 乐享知识库 | search_lexiang_documents | CRM外部知识库（乐享平台）的检索与问答 |
| 智能问答 | smart_search | 综合多源数据回答用户问题 |

**演进时间线：**

| 时间 | 版本 | 核心变更 |
|:---|:---|:---|
| 2025-05 | 618升级 | 合并文档搜索和回答功能，重组Topic结构 |
| 2025-06 | Topic优化 | 百事通、智能网络搜索、查询CRM数据重组为统一Topic |
| 2025-07 | 融合多源 | 支持多数据源融合回答，新增乐享知识库集成 |
| 2025-08 | 伙伴支持 | 百事通支持PRM伙伴用户 |
| 2025-12 | 现状梳理 | 全面梳理百事通各能力现状和核心流程 |
| 2026-01 | Agentic RAG | 从线性检索升级为代理式检索增强生成架构 |
| 2026-04 | 知识库检索优化 | 元数据治理+智能入库+层级索引+Self-Querying |
| 2026-04 | 乐享问答优化 | 意图分类+版本时效性+回答详略控制 |
| 2026-04 | License控制优化 | Function合并（智能网络搜索、查询CRM数据→百事通） |

## 二、618百事通升级（2025-05）

> 来源：Wiki [ID:138575430]

### 核心变更

- 百事通的两个功能合并为一个，不区分找文档还是回答问题，统一改为：根据知识库检索的知识回答问题+知识来源
- 百事通、智能网络搜索、查询CRM客户相关信息重组topic
- 优化意图识别能力

### 元数据改动

1. 百事通两个功能合并：删除文档回答action，文档检索action描述修改
2. 重组topic：删除客户情报检索topic，智能网络检索和查询CRM客户信息action修改关联topic

### 逻辑改动

- 文档检索prompt修改，可以回答问题，回答时标识引用
- 前端根据接口返回的action渲染百事通下有哪些action
- 前端百事通topic增加参数 `action_api_key`
- 前端百事通topic在 `extend_params` 中增加参数 `deep_search`，默认为false

### 各action检索后事件

| 来源 | 处理规则 |
|:---|:---|
| 网络检索 | 大模型将检索和回答一起执行 |
| 文档回答 | 无权限文档内容为"没有内容查看权限"；返回文档列表中大模型回答中提到的文档需要提到前面 |
| 数据查询 | 查询数据和数据列表不需要回答，问到字段时才需要回答 |

## 三、查询相关Topic优化-百事通升级（2025-06）

> 来源：Wiki [ID:138568890]

### 问题

- 用户从字面意思上不清楚功能背后的含义
- 找文档和回答问题两个区分度太小
- 外部找网络资讯、内部查询CRM不容易触发

### 方案设计

1. 合并「问客户情报」内容到「百事通」主题下
2. 智能文档搜索、通过知识库回答问题两个action合并为"问知识库"
3. 智能网络搜索并入「百事通」，名称为"搜全网"
4. 查询CRM数据并入「百事通」，名称为"查数据"
5. 快捷指令下增加特殊交互：是否深度思考、查CRM/问知识库/搜全网

### 优化后的主题和action信息

| 主题 | action名称 | API名称 | 操作说明 |
|:---|:---|:---|:---|
| 百事通 | 问知识库 | search_documents | 企业内部知识库的检索与结构化问答 |
| 百事通 | 搜全网 | search_web | 查询公开的背景信息、业务情报等 |
| 百事通 | 查数据 | query_CRM_data | 查询系统内的业务数据 |

### Topic Instructions

当查询【公司内部知识】时，调用问知识库 `search_documents`；当查询【外部企业信息】时，使用搜全网 `search_web`；当涉及【内部数据查询】时，使用查CRM数据 `query_CRM_data`。对模糊请求优先匹配最相关操作，优先级：内部数据查询 > 问知识库 > 搜全网。

## 四、百事通-融合多源回答问题（2025-07）

> 来源：Wiki [ID:141543347]

### 目标

打造一站式、智能化的销售知识助手「百事通」，让销售不需要知道"去哪找"，只需表达"想要什么"。

### 核心设计

- 数据源可以灵活选择，既能指定数据源，也能混合多源
- 混合数据源时，答案需要清晰指明来源
- 新增乐享知识库作为第一个外部知识来源

### 运行时交互规则

| 功能点 | 规则 |
|:---|:---|
| 来源选择 | 增加"自动"选项排第一；自动和其他来源互斥 |
| 来源记忆 | 上次选择的来源下次打开继续选择，跨对话全局记忆 |
| 多来源回答 | 全部来源时分tab页签展示；序号全局排序；顺序：CRM > 知识库 > 乐享 > 网络 |
| 中断处理 | 多来源涉及查CRM触发中断时，整体中断等待用户选择 |

### 新增action

| 操作名称 | API名称 | 操作说明 | 受控license-function |
|:---|:---|:---|:---|
| 查数据 | query_CRM_data | 查询系统内的业务数据 | 销售助手Agent-查询CRM数据 |
| 搜全网 | search_web | 查询公开信息 | 销售助手Agent-智能网络搜索 |
| 问知识库 | search_documents | 企业内部知识库检索 | 销售助手Agent-百事通 |
| 乐享知识库 | search_lexiang_documents | CRM外部知识库检索 | 销售助手Agent-乐享知识库 |
| 智能问答 | smart_search | 综合多源回答 | 销售助手Agent-百事通 |

## 五、百事通支持伙伴用户（2025-08）

> 来源：Wiki [ID:143124080]

### 核心改动

- Agent元模型增加 `userType` 字段（1=内部用户, 2=PRM伙伴用户）
- Action元模型增加 `prmFlg` 字段（0=不支持PRM, 1=支持PRM）
- 百事通五个Action的 prmFlg=1，其他action为0
- PRM Agent的查知识库和智能检索使用新的action（reference_key: file_answer_prm, smart_search_prm）
- 内部用户只能看到内部用户的agent，外部用户只能看到外部用户agent

## 六、百事通现状梳理（2025-12）

> 来源：Wiki [ID:148127310]

百事通是销售助理使用最高频的功能。核心流程包含6个步骤：

| 步骤 | 搜全网 | 问知识库 | 乐享知识库 | 查CRM | 融合检索 |
|:---|:---|:---|:---|:---|:---|
| 1. query多轮改写 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 2. 意图识别和路由 | 无 | 无 | 无 | 无 | 已废弃 |
| 3. 检索内容提取 | 提取关键字 | 拆分子问题+jieba提取 | 无 | 提取实体+XOQL | 使用基础action |
| 4. 检索执行 | 搜索引擎 | 多问题召回合并 | 乐享API | XOQL查询 | 使用基础action |
| 5. 结果精选重排 | Rerank | Rerank | Rerank | 无 | 使用基础action |
| 6. 组装Prompt回答 | 结构化输出 | 结构化输出 | 同左 | 格式化输出 | 使用基础action |

## 七、百事通：Agentic RAG改造（2026-01~02）

> 来源：Wiki [ID:149848790]

### 背景与目标

将百事通从"自动化检索工具"升级为"Agentic RAG（代理式检索增强生成）"架构，从"普通Topic"改造为独立的action（内部相当于一个固化流程的专家Agent）。

### 架构分层：快慢双轨

- **单步骤任务 (Single Step)**：事实性、单一意图的简单问题，并行检索+流式输出，首字响应 < 3秒
- **多步骤任务 (Multi Step)**：对比分析、多跳推理的复杂问题，引入 Agent Loop（规划-执行-评估-反思循环）

### 交互升级：默认自动，按需手动

- 自动模式 (Auto Mode)：AI根据问题意图自动筛选数据源
- 手动模式 (Manual Mode)：用户手动修改来源勾选

### 核心模块

| 模块 | 职责 |
|:---|:---|
| 上下文注入 | 解析Config参数，定义运行模式（AUTO/MANUAL） |
| 复杂度路由 | 分流中枢，判定Single/Multi Step |
| 单步骤引擎 | 意图映射→并行检索→内容精选 |
| 多步骤Agent | 规划器→执行循环→结果评估→反思 |
| 中断机制 | 查CRM匹配多个同名客户时挂起等待选择 |
| 内容精选合并 | Rerank→Filter→Deduplicate |
| 全局合成器 | 结构化输出+引用标注+幻觉控制 |

### 复杂度路由判定逻辑

- **单步骤**：意图明确、仅需一次性调用一个或多个工具即可获取完整答案。无依赖性，支持并发检索
- **多步骤**：需要多轮交互、存在逻辑依赖、跨源关联分析或需要自主规划路径

### 百事通Action定义

百事通作为一个action，预置在销售助理Agent的百事通主题下。同时在资产库中预置该action。

| 属性 | 值 |
|:---|:---|
| 操作类型 | 系统标准 |
| 操作名称 | 百事通智能检索 |
| API名称 | retrieve_knowledge |
| 是否直接输出 | 是（开启流式透传） |

### 后台配置

在"NeoAgent > 销售智能"下新增"百事通"子菜单，提供检索工具管理配置界面。

预置4个工具：

| 工具名称 | API名称 | 工具描述 | 默认状态 |
|:---|:---|:---|:---|
| 查数据 | query_CRM_data | 查询系统内的业务数据 | 启用 |
| 问知识库 | search_documents | 企业内部知识库检索 | 启用 |
| 乐享知识库 | search_lexiang_documents | CRM外部知识库检索 | 禁用 |
| 搜全网 | search_web | 查询公开信息 | 启用 |

### 核心Prompt设计

包含5个核心节点Prompt：
1. **复杂度路由**：判断FAST_TRACK/SLOW_TRACK
2. **任务规划**：多步骤任务的拆解和规划
3. **任务执行**：参数提取、API调用、依赖处理
4. **内容精选合并**：Rerank+去重+过滤
5. **答案生成**：结构化输出+引用标注+幻觉控制

### 关键设计决策

| 问题 | 决策 |
|:---|:---|
| 复杂度路由与Agent规划是否合并？ | 不合并，保持分离。路由是轻量级判断，合并会拖慢80%的简单请求 |
| 复杂问题是否需要答案评估与反思？ | 不在生成答案后引入独立反思循环。通过Prompt强约束保证输出质量 |

## 八、知识库检索优化-需求澄清（2026-04）

> 来源：TCHub 文档 [ID:714ebdfc] | 作者：fangty | 关联功能：百事通 > 问知识库

### 功能定位

百事通是销售人员的"全能百科全书"，本次优化聚焦「问知识库」能力，从"朴素RAG"升级为"Agentic RAG"。不涉及其他三个数据源。

### 核心痛点

| # | 痛点 | 具体表现 | 根因 |
|:--|:-----|:---------|:-----|
| P1 | 语义相关但业务无效的噪音召回 | 问"最新的制造业定价策略"，召回旧版策略和其他行业策略 | 无元数据过滤 |
| P2 | Agent无法表达过滤意图 | 工具仅有queryContent一个参数 | 工具Schema过于简单 |
| P3 | 文档无业务标签 | 入库时无行业、类别等元数据 | 缺乏元数据体系 |
| P4 | 切片上下文丢失 | 固定窗口切片导致语义碎片化 | 切片策略单一 |
| P5 | 宏观问题匹配困难 | "有哪些金融行业案例"难以命中 | 无文档摘要索引 |

### 优化目标

| 指标 | 目标 |
|:---|:---|
| 检索准确率（Top5命中率） | 提升30%+ |
| 噪音召回率 | 降低50%+ |
| 复杂问题回答完整性 | 显著提升 |
| 文档自动打标准确率 | ≥85% |

### 需求范围

| # | 优化点 | 涉及服务 |
|:--|:-------|:---------|
| 1 | 知识库元数据Schema | metadata-app-service + DATA-PROCESS-SERVICE |
| 2 | 文档入库智能流水线 | DATA-PROCESS-SERVICE |
| 3 | 层级索引与摘要增强 | DATA-PROCESS-SERVICE |
| 4 | Self-Querying检索工具 | agent-platform-service + ai-agent-service |

## 九、知识库检索优化-方案设计（2026-04）

> 来源：TCHub 文档 [ID:cdb36299] | 作者：fangty

### 一句话方案

为百事通「问知识库」构建"元数据治理→智能入库→层级索引→Self-Querying检索"四层改造链路。

### 元数据Schema设计

**核心原则：AI全自动维护，人工仅在异常时介入**

两层体系：
- **系统预置（开箱即用）**：docCategory、businessStage、targetAudience等通用字段
- **AI自动扩展（租户特有）**：industryVertical、productService等因租户而异的字段

**系统预置业务元数据：**

| 字段 | apiName | 类型 | 说明 |
|:---|:---|:---|:---|
| 文档类别 | docCategory | 单选 | 产品手册/销售话术/成功案例/内部政策等 |
| 适用行业 | industryVertical | 多选 | 制造业/医疗健康/金融服务等 |
| 业务阶段 | businessStage | 多选 | 售前咨询/需求调研/方案演示等 |
| 相关产品/服务 | productService | 多选 | 由AI根据文档自动推荐 |
| 目标受众 | targetAudience | 多选 | 高管决策者/业务负责人/IT技术人员等 |

### Schema全自动生命周期

1. **开箱即用**：租户开通知识库后系统预置字段立即可用
2. **AI补充初始化**：累计≥20篇文档时基于summary自动补充租户特有标签
3. **持续自动演化**：偏差信号达阈值时自动更新Schema（只做加法不做减法）

### 文档入库智能流水线

四阶段：解析→自动打标→语义切分→混合索引

切分策略：
- 基于标题树切分：按H1/H2章节边界，超800 tokens按段落二次切分
- 滑动窗口切分：400 tokens切分，100 tokens重叠

### Self-Querying检索工具Schema扩展

从仅有 `queryContent` 扩展为含 `filters` + `topK`，支持结构化过滤+语义检索的混合检索。

### searchByApp接口扩展

新增参数：`filters`（元数据过滤）、`enableRerank`（是否启用Rerank）、`expandContext`（Parent-Child上下文扩展范围）。所有新增参数可选，向后兼容。

## 十、百事通-乐享知识库问答优化-需求澄清（2026-04）

> 来源：TCHub 文档 [ID:1a5bf1ed] | 作者：方腾宇

### 核心痛点

| 维度 | 现状 | 痛点 |
|:---|:---|:---|
| 检索完整性 | 部分文档在乐享前台能搜到但百事通搜不到 | 用户对百事通信任度下降 |
| 版本时效性 | 多版本文档不区分新旧 | 用户得到过时的答案 |
| 回答详略失当 | 不区分用户意图，所有回答都倾向于全部总结 | 信息过载，核心信息被淹没 |

### 量化目标

| 指标 | 目标 |
|:---|:---|
| 乐享检索召回率 | 与乐享前台搜索结果一致 |
| 版本时效性准确率 | 多版本文档优先引用最新版本 |
| 回答详略适当性 | 检索型简洁（概述+链接），问答型针对性详细 |

## 十一、百事通-乐享知识库问答优化-方案设计（2026-04）

> 来源：TCHub 文档 [ID:326a7fb4] | 作者：方腾宇

### 一句话方案

以Prompt优化为核心手段，解决乐享知识库问答的三个体验问题。

### 核心设计决策

**决策1：以Prompt优化为核心手段，不改API调用链路**

**决策2：引入"用户意图分类"**

| 意图类型 | 判断信号 | 回答策略 |
|:---|:---|:---|
| 检索型 | 名词短语、含"找/有没有" | 概述+文档链接 |
| 问答型 | 疑问句、期望具体答案 | 针对性详细回答+引用来源 |

**决策3：版本时效性——Prompt层启发式推断**

从标题/内容中推断版本新旧，优先引用最新版本。

**决策4：回答详略度控制**

- 检索型：📄标题+概述+🔗链接
- 问答型：聚焦问题本身+来源链接

## 十二、百事通-乐享知识库问答优化-PRD（2026-04）

> 来源：TCHub 文档 [ID:fdcb26f3] | 作者：方腾宇

### 核心逻辑模型

在现有链路中插入三个新处理步骤：
1. **Step 1: 意图分类** - 判断"检索型"还是"问答型"
2. **Step 2: 版本时效性分析** - 识别多版本文档，推断新旧
3. **Step 3: 按意图类型生成回答** - 检索型→概述+链接，问答型→详细回答+版本标注

### 验收标准

| 编号 | 验收项 |
|:---|:---|
| AC-1 | 乐享检索完整性：乐享前台能搜到的文档百事通也能搜到 |
| AC-2 | 检索型意图识别：输入名词短语回答为概述+链接格式 |
| AC-3 | 问答型意图识别：输入疑问句回答为针对性详细解答 |
| AC-4 | 版本优先：多版本文档优先引用最新版本 |
| AC-5 | 版本标注：问答型回答标注版本/时间信息 |
| AC-6 | 多版本提示：检索型回答中标注"最新"/"历史版本" |
| AC-7 | 兜底策略：无法判断版本时给出提示 |
| AC-8 | 回答聚焦：问答型回答不展开问题之外的内容 |
| AC-9 | 文档链接：所有回答都包含乐享原文链接 |
| AC-10 | 错误降级：乐享API不可用时给出友好提示 |

## 二十、Agent架构重构P0中的百事通Skill设计（2026-04）

> 来源：TCHub 文档 [ID:80813e1b] | 作者：fangty | Agent架构重构P0-PRD

### 百事通在新架构中的定位

在Agent架构重构（P0）中，百事通从Topic升级为Skill，并进行了拆分：

| 拆分后 | 类型 | 说明 |
|:---|:---|:---|
| 百事通 | Skill (smart_search) | 检索策略编排：判断数据源优先级、整合多源结果、标注引用来源 |
| CRM数据查询 | Skill (crm_query) | 自然语言→CRM查询：业务语义理解、实体关系、字段映射 |
| knowledge_search | Tool（通用） | 知识库RAG检索 |
| web_search | Tool（通用） | 互联网搜索 |
| lx_search | Tool（通用） | 乐享知识库检索 |

### 百事通Skill的核心领域知识

- **检索策略**：涉及具体业务实体时优先查CRM → 涉及"怎么做"/"什么是"时优先查知识库 → 涉及外部信息/最新动态时搜全网 → 涉及内部经验/案例时查乐享
- **多源结果整合**：去重、排序、冲突处理（多个来源信息矛盾时指出差异）
- **引用溯源**：回答中标注每条信息的来源

### CRM数据查询Skill说明

| 维度 | 说明 |
|:---|:---|
| 做什么 | 理解用户的自然语言查询意图，转化为CRM数据查询并返回结构化结果 |
| 为什么是Skill不是Tool | 涉及大量业务语义理解（实体关系、字段别名、常见查询模式） |
| 核心领域知识 | CRM实体关系图、字段语义映射、常见查询模式、结果格式化规则 |
| 依赖Tools | query_crm |
