# 元数据级联查询引擎设计方案

> 版本：1.0 | 日期：2026-04-23
> 目标：通用的多步级联查询能力，覆盖所有元模型间的关联查询场景，零独立接口开发

---

## 一、问题定义

### 1.1 省市区级联的完整查询链路

```
用户选中省份 guangdong
  ↓
Step 1: globalPickOption WHERE entityApiKey='province'
  → 34 个省份（已在左侧列表显示）

Step 2: globalPickDependencyDetail WHERE dependencyApiKey='provinceToCity' AND controlOptionApiKey='guangdong'
  → 得到 dependentOptionApiKeys = ["guangZhou","shenZhen","zhuHai",...]（TEXT[] 数组）

Step 3: globalPickOption WHERE entityApiKey='city' AND apiKey IN ('guangZhou','shenZhen','zhuHai',...)
  → 得到广东省下 21 个城市的完整数据
```

### 1.2 实体→字段→选项值的查询链路

```
用户选中实体 account，再选中字段 industry
  ↓
Step 1: pickOption WHERE itemApiKey='industry'
  → 得到 industry 字段的所有选项值
```

### 1.3 共性分析

所有级联查询都是同一个模式：

```
从 A 表选中一条记录
  → 通过关系配置找到 B 表的关联记录
  → 从 B 表记录中提取目标值（可能是 apiKey 列表）
  → 用目标值回查 C 表得到最终数据
```

区别只在于：
- 简单场景（entity→item）：一步直接过滤，`referItemApiKey` 就够了
- 复杂场景（省→市）：需要经过中间表，提取 apiKey 列表，再回查

## 二、关系模型扩展

### 2.1 当前 p_meta_link 的局限

当前 p_meta_link 只记录**父子包含关系**：

```
parent_metamodel_api_key → child_metamodel_api_key
refer_item_api_key = 子数据用哪个字段关联父数据
```

缺少的关系类型：
- **级联引用**：通过中间表的字段值，反向引用另一个元模型的记录
- **多步链路**：A → B → C 的完整查询路径

### 2.2 扩展方案：p_meta_link 增加 link_type 语义

当前 `link_type` 字段只有一个值（2=父子包含）。扩展为：

| link_type | 含义 | 查询方式 | 示例 |
|:----------|:-----|:---------|:-----|
| 1 | 父子包含（级联删除） | `{referField} = parent.apiKey` | entity→item |
| 2 | 父子包含（阻止删除） | `{referField} = parent.apiKey` | metaModel→metaItem |
| 3 | **级联引用** | 通过中间表数组字段值反查目标表 | globalPickDependencyDetail.dependentOptionApiKeys（TEXT[]）→ globalPickOption |

### 2.3 级联引用的数据结构

对于 link_type=3（级联引用），需要额外字段描述"从哪个字段取值，回查哪个表"：

在 p_meta_link 中新增记录：

| api_key | parent | child | refer_item_api_key | link_type | 新增字段 |
|:--------|:-------|:------|:-------------------|:----------|:---------|
| depDetail_to_pickOption | globalPickDependencyDetail | globalPickOption | dependentOptionApiKeys | 3 | target_entity_api_key_field=entityApiKey（目标表用哪个字段区分分类） |

**但这样改 p_meta_link 表结构影响太大。**

### 2.4 更轻量的方案：不改表结构，用现有字段表达

p_meta_link 已有的 `cascade_delete` 字段可以复用语义：

| cascade_delete | 当前含义 | 扩展含义 |
|:---------------|:---------|:---------|
| 0 | 不级联 | 查找引用（不级联删除） |
| 1 | 级联删除 | 父子包含（级联删除） |
| 2 | 阻止删除 | 父子包含（阻止删除） |
| 3 | — | **级联引用**（通过字段值反查） |

对于 cascade_delete=3 的记录，`refer_item_api_key` 的含义变为"从 parent 的哪个字段取 apiKey 列表"。

## 三、不改表结构的方案：前端查询管道

### 3.1 核心思路

不改 p_meta_link 表结构，在前端 `DataSourceConfig` 中支持**查询管道（pipeline）**配置：

```typescript
interface DataSourceConfig {
  // ... 现有字段 ...
  
  /**
   * 查询管道 — 多步级联查询。
   * 每一步的结果作为下一步的输入。最后一步的结果作为组件的数据。
   */
  pipeline?: PipelineStep[];
}

interface PipelineStep {
  /** 查询哪个元模型 */
  metamodelApiKey: string;
  /** 过滤条件（支持 $prev 引用上一步结果、$input 引用管道输入） */
  filter: string;
  /** 从结果中提取哪个字段作为下一步的输入（默认 apiKey） */
  outputField?: string;
  /** 提取模式：single=取第一条记录的字段值，collect=收集所有记录的字段值合并 */
  outputMode?: 'single' | 'collect';
}
```

### 3.2 省市区级联的 pipeline 配置

```json
{
  "type": "metadata",
  "metamodelApiKey": "globalPickOption",
  "pipeline": [
    {
      "metamodelApiKey": "globalPickDependencyDetail",
      "filter": "dependencyApiKey = 'provinceToCity' AND controlOptionApiKey = $input",
      "outputField": "dependentOptionApiKeys",
      "outputMode": "collect"
    }
  ],
  "filter": "entityApiKey = 'city' AND apiKey IN $pipeline"
}
```

执行流程：
```
$input = 左侧选中的省份 apiKey（如 guangdong）
  ↓
Pipeline Step 1:
  查 globalPickDependencyDetail WHERE dependencyApiKey='provinceToCity' AND controlOptionApiKey='guangdong'
  取 dependentOptionApiKeys = ["guangZhou","shenZhen",...]（TEXT[] 数组，直接展开）
  ↓
$pipeline = ["guangZhou","shenZhen",...]
  ↓
最终查询:
  查 globalPickOption WHERE entityApiKey='city' AND apiKey IN ('guangZhou','shenZhen',...)
  → 返回 21 个城市
```

### 3.3 实体→字段→选项值的 pipeline 配置

这个场景不需要 pipeline，直接用 parentRef 就够了：

```json
{
  "type": "metadata",
  "metamodelApiKey": "pickOption",
  "parentRef": { "parentStateKey": "selected_item_apiKey", "filterField": "itemApiKey" }
}
```

### 3.4 pipeline 的通用性

| 场景 | 需要 pipeline | 原因 |
|:-----|:-------------|:-----|
| entity→item | ❌ | 直接父子关系，parentRef 够用 |
| entity→item→pickOption | ❌ | 两级 parentRef 串联 |
| 省→市 | ✅ | 需要经过 dependencyDetail 中间表反查 |
| 市→区 | ✅ | 同上 |
| 选项集→依赖定义 | ❌ | 直接父子关系 |
| 选项集→依赖明细 | ❌ | 直接父子关系 |

**只有"通过中间表的字段值反查另一个表"的场景需要 pipeline。** 这类场景在系统中主要是 globalPickOption 的级联。

## 四、后端通用 API 方案（推荐）

### 4.1 为什么推荐后端

前端 pipeline 需要多次 HTTP 请求（查 detail → 查 option），性能差且逻辑复杂。后端一次查询完成更合理。

### 4.2 通用级联查询接口

```
GET /meta/cascade-options?dependencyApiKey=provinceToCity&controlApiKey=guangdong
```

后端逻辑（通用，不针对省市区）：

```java
@GetMapping("/meta/cascade-options")
public List<?> cascadeOptions(
    @RequestParam String dependencyApiKey,
    @RequestParam String controlApiKey
) {
    // Step 1: 查 globalPickDependency 获取依赖定义
    GlobalPickDependency dep = mergeReadService.getByApiKeyMerged("globalPickDependency", dependencyApiKey);
    // dep.dependentItemApiKey = "city"（目标选项集）
    
    // Step 2: 查 globalPickDependencyDetail 获取 apiKey 列表
    List<GlobalPickDependencyDetail> details = mergeReadService.listMergedByEntityApiKey(
        "globalPickDependencyDetail", dep.getEntityApiKey());
    List<String> targetApiKeys = details.stream()
        .filter(d -> controlApiKey.equals(d.getControlOptionApiKey()))
        .map(GlobalPickDependencyDetail::getDependentOptionApiKeys)
        .filter(Objects::nonNull)
        .flatMap(List::stream)  // TEXT[] 数组字段，直接展开
        .collect(Collectors.toList());
    
    // Step 3: 查 globalPickOption 获取目标选项值
    List<GlobalPickOption> allOptions = mergeReadService.listMergedByEntityApiKey(
        "globalPickOption", dep.getDependentItemApiKey());
    Set<String> targetSet = new HashSet<>(targetApiKeys);
    return allOptions.stream()
        .filter(o -> targetSet.contains(o.getApiKey()))
        .collect(Collectors.toList());
}
```

### 4.3 更通用：元数据链式查询接口

不只是 globalPickOption，任何元模型间的链式查询都支持：

```
POST /meta/chain-query
{
  "steps": [
    {
      "metamodelApiKey": "globalPickDependencyDetail",
      "filters": { "dependencyApiKey": "provinceToCity", "controlOptionApiKey": "guangdong" },
      "outputField": "dependentOptionApiKeys"
    }
  ],
  "target": {
    "metamodelApiKey": "globalPickOption",
    "entityApiKey": "city",
    "filterField": "apiKey",
    "filterMode": "IN"
  }
}
```

返回：globalPickOption 中 entityApiKey='city' 且 apiKey 在 dependentOptionApiKeys 列表中的记录。

### 4.4 接口通用性验证

| 场景 | steps | target |
|:-----|:------|:-------|
| 省→市 | detail WHERE dep=provinceToCity AND ctrl=guangdong → dependentOptionApiKeys | globalPickOption WHERE entityApiKey=city AND apiKey IN $steps |
| 市→区 | detail WHERE dep=cityToDistrict AND ctrl=guangZhou → dependentOptionApiKeys | globalPickOption WHERE entityApiKey=district AND apiKey IN $steps |
| 字段→关联选项 | referFilterDetail WHERE filterApiKey=xxx → dependentOptionApiKeys | pickOption WHERE apiKey IN $steps |

## 五、前端配置 UI

### 5.1 不需要用户理解 pipeline

用户在属性面板中看到的是：

```
数据源
  数据分类: [元数据]
  选择对象: 全局选项集 (globalPickOption) ▼

关联上级
  上级组件: [列表面板 - 全局选项集(province)  ▼]
  关联方式: [省份→城市（级联查询）            ▼]
```

"关联方式"下拉的选项来源：

1. **直接关系**（从 p_meta_link 推导）：
   - `entityApiKey = 上级.apiKey`（同模型子数据）
   - `apiKey = 上级.apiKey`（同记录详情）

2. **级联关系**（从 globalPickDependency 推导）：
   - 如果上级是 globalPickOption(province)，当前也是 globalPickOption
   - 系统查 globalPickDependency，发现 province→city 的依赖
   - 自动提供"省份→城市（级联查询）"选项

### 5.2 关联方式的自动推导逻辑

```typescript
function getLinkageOptions(parentMetamodel: string, parentEntityApiKey: string, 
                           currentMetamodel: string): LinkageOption[] {
  const options = [];
  
  // 1. p_meta_link 直接关系
  for (const link of metaLinks) {
    if (link.parentMetamodelApiKey === parentMetamodel && link.childMetamodelApiKey === currentMetamodel) {
      options.push({
        type: 'direct',
        filterField: link.referItemApiKey,
        label: `${link.referItemApiKey} = 上级.apiKey`
      });
    }
  }
  
  // 2. 同模型子数据
  if (parentMetamodel === currentMetamodel) {
    options.push({ type: 'direct', filterField: 'entityApiKey', label: '子数据（entityApiKey = 上级.apiKey）' });
    options.push({ type: 'direct', filterField: 'apiKey', label: '同记录详情（apiKey = 上级.apiKey）' });
  }
  
  // 3. globalPickDependency 级联关系
  if (parentMetamodel === 'globalPickOption' && currentMetamodel === 'globalPickOption' && parentEntityApiKey) {
    for (const dep of globalPickDependencies) {
      if (dep.controlItemApiKey === parentEntityApiKey) {
        options.push({
          type: 'cascade',
          dependencyApiKey: dep.apiKey,
          targetEntityApiKey: dep.dependentItemApiKey,
          label: `${dep.label}（级联查询）`
        });
      }
    }
  }
  
  return options;
}
```

### 5.3 DataSourceConfig 中的存储

用户选择"省份→城市（级联查询）"后，生成的配置：

```json
{
  "type": "metadata",
  "metamodelApiKey": "globalPickOption",
  "parentRef": {
    "parentStateKey": "selected_globalPickOption_apiKey",
    "type": "cascade",
    "dependencyApiKey": "provinceToCity",
    "targetEntityApiKey": "city"
  }
}
```

前端 useDataSource 检测到 `parentRef.type === 'cascade'`，调用后端级联查询接口：

```
GET /meta/cascade-options?dependencyApiKey=provinceToCity&controlApiKey=guangdong
```

### 5.4 parentRef 类型扩展

```typescript
interface ParentRef {
  parentStateKey: string;
  
  // 直接关系（type=direct，默认）
  type?: 'direct' | 'cascade';
  filterField?: string;          // direct 模式：用哪个字段过滤
  parentValueField?: string;     // direct 模式：从上级取哪个字段（默认 apiKey）
  
  // 级联关系（type=cascade）
  dependencyApiKey?: string;     // cascade 模式：依赖定义的 apiKey
  targetEntityApiKey?: string;   // cascade 模式：目标选项集的 entityApiKey
}
```

## 七、实现优先级

| 优先级 | 内容 | 改动 |
|:-------|:-----|:-----|
| **P0** | 后端 `/meta/cascade-options` 接口 | MetamodelBrowseApiService.java 新增方法 |
| **P0** | 前端 parentRef 支持 `type: 'cascade'` | useDataSource.ts |
| **P0** | 属性面板"关联方式"下拉支持级联选项 | PropertyPanel.tsx ParentRefEditor |
| **P1** | 后端通用 `/meta/chain-query` 接口 | 支持任意元模型链式查询 |
| **P1** | 属性面板自动推导 globalPickDependency 级联选项 | ParentRefEditor 加载 dependency 数据 |
| **P2** | itemDependency 级联（字段间联动） | 同样的 cascade 模式 |
