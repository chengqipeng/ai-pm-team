# 画布左右联动完整设计方案

> 版本：2.0 | 日期：2026-04-23
> 目标：统一所有联动场景的配置模型，覆盖主子、多级级联、多对多关联

---

## 一、问题全景

当前系统有三种数据关系，但联动机制只覆盖了前两种：

| 关系类型 | 数据来源 | 当前支持 | 典型场景 |
|:---------|:---------|:---------|:---------|
| 父子包含（1:N） | p_meta_link | ✅ 已支持 | entity→item, metaModel→metaItem |
| 实例关联（M:N） | entityLink | ⚠️ 部分支持 | role→user, department→user |
| 同模型层级（树） | parent_metadata_api_key | ❌ 不支持 | province→city→district, role→子角色 |

缺失的核心能力：
1. **多级级联**：省→市→区，需要三级联动，当前只支持一级
2. **动态过滤值**：过滤条件的值需要来自上级选中项，当前只支持静态值
3. **多对多中间表**：role→user 通过 p_user_role 关联，需要走专用 API
4. **同模型树形联动**：选中父角色后显示子角色列表

---

## 二、关系模型统一抽象

所有联动场景归纳为 **5 种联动模式**：

### 模式 A：主子联动（父子包含）

```
左侧选中 entity(account) → 右侧加载 item(entityApiKey=account)
左侧选中 metaModel(role) → 右侧加载 metaItem(metamodelApiKey=role)
```

**关系来源**：p_meta_link（referItemApiKey 指定子数据的关联字段）
**过滤方式**：子数据的 `referItemApiKey` 字段 = 父数据的 apiKey
**当前状态**：✅ 已实现（模式二、模式三）

### 模式 B：同模型详情联动

```
左侧选中 role(systemAdmin) → 右侧 detailHeader 显示 systemAdmin 的详情
```

**关系来源**：同一个 metamodelApiKey
**过滤方式**：右侧数据的 apiKey = 左侧选中值
**当前状态**：✅ 已实现（模式一）

### 模式 C：跨模型关联联动（多对多）

```
左侧选中 role(systemAdmin) → 右侧加载该角色下的 user 列表
左侧选中 department(techDept) → 右侧加载该部门下的 user 列表
```

**关系来源**：entityLink（parentEntityApiKey→childEntityApiKey）
**过滤方式**：需要通过中间表或关联字段查询
**当前状态**：⚠️ 部分支持（filterField 机制可用，但配置复杂）

### 模式 D：同模型树形联动（dbc 列驱动）

```
左侧选中 department(techDept) → 右侧加载 department(deptParentApiKey=techDept)
左侧选中 role(管理层) → 右侧加载 role(roleParentApiKey=管理层)
```

**关系来源**：各元模型自行注册的 dbc 扩展列（如 department.deptParentApiKey = dbc_varchar5）
**过滤方式**：子数据的 dbc 层级字段 = 父数据的 apiKey
**当前状态**：✅ 已支持（部门树、角色树通过 dbc 列映射实现）

> ⚠️ 注意：省→市→区级联**不属于**此模式。省市区之间没有 parentMetadataApiKey 直接关联，
> 而是通过 globalPickDependency + globalPickDependencyDetail 依赖体系 + chainQuery 链式查询实现，属于模式 E。

### 模式 E：跨中间表链式联动（chainQuery 驱动）

```
省份列表 → [globalPickDependencyDetail] → 城市列表 → [globalPickDependencyDetail] → 区县列表
实体列表 → 字段列表 → 选项值列表（三级，p_meta_link 链式推导）
```

**关系来源**：
- 省→市→区：globalPickDependency 依赖体系，通过 globalPickDependencyDetail 中间表的 `dependentOptionApiKeys`（TEXT[] 数组）反查目标选项
- 实体→字段→选项：p_meta_link 父子关系的链式组合
**过滤方式**：每级的过滤值来自上一级的选中项，复杂场景需要经过中间表提取 apiKey 列表
**当前状态**：✅ 已实现（chainQuery + cascadePanel）

## 三、统一联动配置模型

### 3.1 FilterCondition 扩展：$动态值

```typescript
interface FilterCondition {
  field: string;
  op: 'eq' | 'ne' | 'gt' | 'lt' | 'gte' | 'lte' | 'like' | 'in';
  value: unknown;  // 静态值 或 '$stateKey' 动态引用
}
```

`value` 以 `$` 开头时，运行时从 `PageContext.state` 中取值：

```json
{ "field": "deptParentApiKey", "op": "eq", "value": "$selectedDept" }
```

运行时解析：`$selectedDept` → `PageContext.state.selectedDept` → `"techDept"`

**动态值为空时（上级未选中），该条件自动跳过**，不会过滤掉所有数据。

### 3.2 DataSourceConfig 不变

现有的 `DataSourceConfig` 已经足够表达所有联动场景，不需要新增字段：

```typescript
interface DataSourceConfig {
  type: DataSourceType;
  metamodelApiKey?: string;
  entityApiKey?: string;
  depends?: string[];           // 监听哪些 stateKey 的变化
  stateKey?: string;            // 选中后写入哪个 stateKey
  filters?: FilterCondition[];  // 过滤条件（支持 $动态值）
  filterLogic?: 'and' | 'or';
  filterField?: string;         // 跨模型联动（模式 C）
  filterFrom?: string;
  lazy?: boolean;
}
```

### 3.3 五种模式的配置方式

#### 模式 A：主子联动

```json
// 左侧：实体列表
{ "type": "metadata", "metamodelApiKey": "entity", "stateKey": "selectedEntity" }

// 右侧：字段列表（自动联动，depends 由系统推导）
{ "type": "metadata", "metamodelApiKey": "item", "depends": ["selectedEntity"] }
```

> 系统已有能力，无需改动。

#### 模式 B：同模型详情

```json
// 左侧：角色列表
{ "type": "metadata", "metamodelApiKey": "role", "stateKey": "selectedRole" }

// 右侧：角色详情
{ "type": "metadata", "metamodelApiKey": "role", "depends": ["selectedRole"] }
```

> 系统已有能力，无需改动。

#### 模式 C：跨模型关联（多对多）

```json
// 左侧：角色列表
{ "type": "metadata", "metamodelApiKey": "role", "stateKey": "selectedRole" }

// 右侧：用户列表（通过 filterField 过滤）
{ "type": "entityData", "entityApiKey": "user",
  "depends": ["selectedRole"],
  "filterField": "roleApiKey", "filterFrom": "apiKey" }
```

> 系统已有能力（模式四），但配置复杂。后续可通过 entityLink 自动推导 filterField。

#### 模式 D：同模型树形联动（dbc 列驱动）

```json
// 左侧：部门列表
{ "type": "metadata", "metamodelApiKey": "department",
  "stateKey": "selectedDept" }

// 右侧：子部门列表（deptParentApiKey = 选中的部门）
{ "type": "metadata", "metamodelApiKey": "department",
  "filters": [
    { "field": "deptParentApiKey", "op": "eq", "value": "$selectedDept" }
  ] }
```

> 部门树通过 dbc_varchar5 → deptParentApiKey 映射实现，角色树通过 dbc_varchar5 → roleParentApiKey 映射实现。
> ⚠️ 省→市→区级联不属于此模式，见模式 E。

#### 模式 E：跨中间表链式联动（chainQuery 驱动）

**场景一：省→市→区（通过 globalPickDependency 依赖体系）**

省市区之间的关联通过 `globalPickDependencyDetail` 中间表实现，不是 parentMetadataApiKey 直接关联：

```
省份(guangdong)
  → 查 globalPickDependencyDetail WHERE dependencyApiKey='provinceToCity' AND controlOptionApiKey='guangdong'
  → 取 dependentOptionApiKeys = ["guangZhou","shenZhen",...] (TEXT[] 数组)
  → 查 globalPickOption WHERE entityApiKey='city' AND apiKey IN (...)
  → 得到广东省下的城市
```

推荐使用 chainQueryRef 配置：

```json
// 左侧：省份列表
{ "type": "metadata", "metamodelApiKey": "globalPickOption",
  "filters": [{ "field": "entityApiKey", "op": "eq", "value": "province" }],
  "stateKey": "selectedProvince" }

// 右侧：城市列表（通过 chainQuery 链式查询）
{ "chainQueryRef": "provinceToCity",
  "chainQueryInputKey": "selectedProvince" }
```

**场景二：cascadePanel + $dep() 表达式**

```json
{ "type": "cascadePanel", "props": {
  "layoutZone": "left",
  "levels": [
    { "label": "省份", "metamodelApiKey": "globalPickOption", "entityApiKey": "province",
      "stateKey": "selectedProvince" },
    { "label": "城市", "metamodelApiKey": "globalPickOption",
      "filter": "entityApiKey = 'city' AND apiKey IN $dep('provinceToCity', $prev.apiKey)",
      "stateKey": "selectedCity" },
    { "label": "区县", "metamodelApiKey": "globalPickOption",
      "filter": "entityApiKey = 'district' AND apiKey IN $dep('cityToDistrict', $prev.apiKey)",
      "stateKey": "selectedDistrict" }
  ]
}}
```

**场景三：多个 listPanel + chainQuery（灵活但复杂）**

用 Tab 内放多个列表，每个列表用 chainQuery 引用上级：

```json
// Tab 1: 城市列表（通过 chainQuery 从省份查城市）
{ "chainQueryRef": "provinceToCity",
  "chainQueryInputKey": "selectedProvince",
  "stateKey": "selectedCity" }

// Tab 2: 区县列表（通过 chainQuery 从城市查区县）
{ "chainQueryRef": "cityToDistrict",
  "chainQueryInputKey": "selectedCity" }
```

**场景四：实体→字段→选项值（p_meta_link 链式推导，零配置）**

```json
{ "type": "cascadePanel", "props": {
  "layoutZone": "left",
  "levels": [
    { "label": "实体", "metamodelApiKey": "entity", "stateKey": "selectedEntity" },
    { "label": "字段", "metamodelApiKey": "item", "stateKey": "selectedItem" },
    { "label": "选项值", "metamodelApiKey": "pickOption" }
  ]
}}
```

> p_meta_link 中有 entity→item（referItemApiKey=entityApiKey）和 item→pickOption（referItemApiKey=itemApiKey），
> cascadePanel 自动推导过滤条件，无需手写 filter。

## 四、完整场景矩阵

### 4.1 所有可配置的联动场景

| 场景 | 模式 | 左侧 | 右侧 | 联动字段 | 配置方式 |
|:-----|:-----|:-----|:-----|:---------|:---------|
| 元模型→字段 | A | metaModel | metaItem | metamodelApiKey | depends 自动推导 |
| 元模型→关联 | A | metaModel | metaLink | parentMetamodelApiKey | depends 自动推导 |
| 元模型→选项 | A | metaModel | metaOption | metamodelApiKey | depends 自动推导 |
| 实体→字段 | A | entity | item | entityApiKey | depends 自动推导 |
| 实体→校验规则 | A | entity | checkRule | entityApiKey | depends 自动推导 |
| 实体→业务类型 | A | entity | busiType | entityApiKey | depends 自动推导 |
| 实体→查重规则 | A | entity | duplicateRule | entityApiKey | depends 自动推导 |
| 实体→共享规则 | A | entity | sharingRule | entityApiKey | depends 自动推导 |
| 角色→角色详情 | B | role | role(detail) | apiKey | depends 自动推导 |
| 部门→部门详情 | B | department | department(detail) | apiKey | depends 自动推导 |
| 角色→用户 | C | role | user | roleApiKey | filterField 配置 |
| 部门→用户 | C | department | user | departApiKey | filterField 配置 |
| 省→市 | E | globalPickOption(province) | globalPickOption(city) | chainQuery(provinceToCity) | chainQueryRef 模板引用 |
| 市→区 | E | globalPickOption(city) | globalPickOption(district) | chainQuery(cityToDistrict) | chainQueryRef 模板引用 |
| 省→市→区 | E | cascadePanel | — | chainQuery + $dep() | cascadePanel + $dep 表达式 |
| 实体→字段→选项 | E | cascadePanel | — | entityApiKey/itemApiKey | p_meta_link 自动推导 |
| 角色→子角色 | D | role | role(children) | roleParentApiKey (dbc_varchar5) | $动态值 |
| 部门→子部门 | D | department | department(children) | deptParentApiKey (dbc_varchar5) | $动态值 |
| 选项集→依赖 | A | globalPickOption | globalPickDependency | entityApiKey | depends 自动推导 |
| 选项集→子选项 | A | globalPickOption | globalPickOption | entityApiKey | depends 自动推导（entityApiKey = 选中的选项集标识） |

### 4.2 多对多关联的完整路径

| 关联 | 中间表/字段 | API 路径 | 配置方式 |
|:-----|:-----------|:---------|:---------|
| role → user | p_user_role | `/auth/users?roleApiKey=xxx` | filterField=roleApiKey |
| department → user | p_user.depart_api_key | `/entity/data/user?departApiKey=xxx` | filterField=departApiKey |
| account → contact | entityLink(account→contact) | `/entity/data/contact?accountApiKey=xxx` | filterField=accountApiKey |
| account → opportunity | entityLink(account→opportunity) | `/entity/data/opportunity?accountApiKey=xxx` | filterField=accountApiKey |

## 七、配置示例汇总

### 7.1 省→市→区（cascadePanel + $dep 表达式）

> 省市区之间的关联通过 globalPickDependency 依赖体系实现，使用 $dep() 表达式或 chainQuery。

```json
{
  "components": [
    { "type": "splitPanel", "props": { "leftWidth": 360 } },
    { "type": "cascadePanel", "props": {
      "layoutZone": "left",
      "levels": [
        { "label": "省份", "metamodelApiKey": "globalPickOption",
          "filter": "entityApiKey = 'province'", "stateKey": "selectedProvince" },
        { "label": "城市", "metamodelApiKey": "globalPickOption",
          "filter": "entityApiKey = 'city' AND apiKey IN $dep('provinceToCity', $prev.apiKey)",
          "stateKey": "selectedCity" },
        { "label": "区县", "metamodelApiKey": "globalPickOption",
          "filter": "entityApiKey = 'district' AND apiKey IN $dep('cityToDistrict', $prev.apiKey)",
          "stateKey": "selectedDistrict" }
      ]
    }},
    { "type": "detailHeader", "props": {
      "dataSource": { "type": "metadata", "metamodelApiKey": "globalPickOption",
        "filters": [{ "field": "entityApiKey", "op": "eq", "value": "district" }],
        "depends": ["selectedDistrict"] }
    }}
  ]
}
```
