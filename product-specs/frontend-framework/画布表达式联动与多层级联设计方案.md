# 画布表达式联动与多层级联设计方案

> 版本：1.0 | 日期：2026-04-23

---

## 一、设计目标

用一套通用的表达式机制，替代当前零散的 `depends`/`filterField`/`$动态值` 配置，实现：

1. 任意元模型的左右分栏联动（由 p_meta_link 驱动）
2. 任意深度的多层级联（路径声明 + 自动推导）
3. 可覆盖的过滤表达式（简单场景零配置，复杂场景写表达式）

---

## 二、核心概念

### 2.1 p_meta_link 是联动的唯一数据源

```
p_meta_link 记录：
  parent_metamodel_api_key = entity
  child_metamodel_api_key = item
  refer_item_api_key = entityApiKey    ← 子数据用这个字段关联父数据
```

含义：**item 的 entityApiKey 字段 = 父级 entity 记录的 apiKey**

这条规则适用于所有父子关系，不需要针对具体元模型写特殊逻辑。

### 2.2 自动推导规则

当右侧组件的元模型是左侧元模型的子元模型时（p_meta_link 中有记录），系统自动生成过滤条件：

```
{referItemApiKey} = $prev.apiKey
```

| 左侧元模型 | 右侧元模型 | p_meta_link.refer | 自动生成的过滤 |
|:-----------|:-----------|:-----------------|:--------------|
| entity | item | entityApiKey | `entityApiKey = $prev.apiKey` |
| entity | checkRule | entityApiKey | `entityApiKey = $prev.apiKey` |
| entity | sharingRule | entityApiKey | `entityApiKey = $prev.apiKey` |
| item | pickOption | itemApiKey | `itemApiKey = $prev.apiKey` |
| item | formulaCompute | itemApiKey | `itemApiKey = $prev.apiKey` |
| item | itemDependency | controlItemApiKey | `controlItemApiKey = $prev.apiKey` |
| globalPickOption | globalPickDependency | entityApiKey | `entityApiKey = $prev.apiKey` |
| globalPickDependency | globalPickDependencyDetail | dependencyApiKey | `dependencyApiKey = $prev.apiKey` |
| metaModel | metaItem | metamodelApiKey | `metamodelApiKey = $prev.apiKey` |
| metaModel | metaLink | parentMetamodelApiKey | `parentMetamodelApiKey = $prev.apiKey` |
| sharingRule | sharingRuleCondition | ruleApiKey | `ruleApiKey = $prev.apiKey` |
| page | pageSection | entityApiKey | `entityApiKey = $prev.apiKey` |

**零配置**：只要声明左右两侧的元模型，系统查 p_meta_link 自动知道怎么过滤。

## 三、表达式语法

### 3.1 变量

| 变量 | 含义 | 示例值 |
|:-----|:-----|:-------|
| `$prev.apiKey` | 上一层选中记录的 apiKey | `account` |
| `$prev.{field}` | 上一层选中记录的任意字段 | `$prev.entityApiKey` → `province` |
| `$level1.apiKey` | 第 1 层选中记录的 apiKey | `account` |
| `$level2.apiKey` | 第 2 层选中记录的 apiKey | `amount` |
| `$levelN.{field}` | 第 N 层选中记录的任意字段 | `$level1.label` → `客户` |

### 3.2 操作符

| 操作符 | 含义 | 示例 |
|:-------|:-----|:-----|
| `=` | 等于 | `entityApiKey = $prev.apiKey` |
| `!=` | 不等于 | `deleteFlg != 1` |
| `IN` | 在列表中 | `apiKey IN $prev.dependentOptionApiKeys` |
| `LIKE` | 包含 | `label LIKE '北京'` |
| `IS NULL` | 为空 | `entityApiKey IS NULL` |
| `IS NOT NULL` | 不为空 | `entityApiKey IS NOT NULL` |

### 3.3 组合

```
entityApiKey = $prev.apiKey AND enableFlg = 1
entityApiKey = 'city' AND apiKey IN $prev.dependentOptionApiKeys
```

### 3.4 特殊：数组字段展开

当引用的字段值是 TEXT[] 数组类型（如 `dependentOptionApiKeys = ["beijingCity","guangZhou",...]`），`IN` 操作符自动展开：

```
apiKey IN $prev.dependentOptionApiKeys
→ apiKey IN ('beijingCity', 'guangZhou', ...)
```

> 多选字段统一使用 PG 原生数组类型（TEXT[]），不再使用逗号分隔字符串。详见 [多选字段数组类型设计方案](../product-specs/metadata-system/多选字段数组类型设计方案.md)。

## 四、多层级联配置

### 4.1 cascadePanel levels 配置

```typescript
interface CascadeLevel {
  /** 元模型 apiKey */
  metamodelApiKey: string;
  /** 显示名 */
  label: string;
  /** 过滤表达式（可选，不写则从 p_meta_link 自动推导） */
  filter?: string;
  /** 选中后写入 PageContext 的 stateKey（可选） */
  stateKey?: string;
  /** 主显示字段（默认 label） */
  displayField?: string;
  /** 副显示字段（默认 apiKey） */
  secondaryField?: string;
}
```

### 4.2 自动推导流程

```
levels: [
  { metamodelApiKey: "entity" },
  { metamodelApiKey: "item" },
  { metamodelApiKey: "formulaCompute" }
]

系统处理：
  第1层 entity: 无过滤（根节点）
  第2层 item: 查 p_meta_link(entity → item) → refer=entityApiKey
              自动生成: entityApiKey = $level1.apiKey
  第3层 formulaCompute: 查 p_meta_link(item → formulaCompute) → refer=itemApiKey
              自动生成: itemApiKey = $level2.apiKey
```

### 4.3 显式覆盖

当自动推导不满足需求时，显式写 filter：

```json
{
  "levels": [
    { "metamodelApiKey": "globalPickOption", "filter": "entityApiKey IS NULL", "label": "选项集" },
    { "metamodelApiKey": "globalPickOption", "filter": "entityApiKey = $prev.apiKey", "label": "选项值" },
    { "metamodelApiKey": "globalPickOption",
      "filter": "entityApiKey = 'district' AND apiKey IN $level2.dependentOptionApiKeys",
      "label": "关联区县" }
  ]
}
```

注意第 2 层和第 1 层是同一个元模型（globalPickOption），但过滤条件不同。这种情况 p_meta_link 无法自动推导（因为不是父子关系），必须显式写 filter。

## 五、完整场景配置

### 5.1 场景一：globalPickOption → globalPickDependency → globalPickDependencyDetail

```json
{
  "type": "cascadePanel",
  "props": {
    "layoutZone": "left",
    "levels": [
      { "metamodelApiKey": "globalPickOption", "filter": "entityApiKey IS NULL", "label": "选项集" },
      { "metamodelApiKey": "globalPickDependency", "label": "依赖定义" },
      { "metamodelApiKey": "globalPickDependencyDetail", "label": "依赖明细" }
    ]
  }
}
```

推导结果：
- 第 1 层：`entityApiKey IS NULL`（显式）
- 第 2 层：`entityApiKey = $level1.apiKey`（自动，p_meta_link refer=entityApiKey）
- 第 3 层：`dependencyApiKey = $level2.apiKey`（自动，p_meta_link refer=dependencyApiKey）

### 5.6 场景六：省市区级联（通过依赖明细）

```json
{
  "type": "cascadePanel",
  "props": {
    "layoutZone": "left",
    "levels": [
      { "metamodelApiKey": "globalPickOption", "filter": "entityApiKey = 'province'", "label": "省份" },
      { "metamodelApiKey": "globalPickDependencyDetail",
        "filter": "dependencyApiKey = 'provinceToCity' AND controlOptionApiKey = $prev.apiKey",
        "label": "省→市映射", "displayField": "dependentOptionApiKeys" },
      { "metamodelApiKey": "globalPickOption",
        "filter": "entityApiKey = 'city' AND apiKey IN $prev.dependentOptionApiKeys",
        "label": "城市" }
    ]
  }
}
```

或者更简洁的方式——直接用 `entityApiKey` 分层 + 依赖查询：

```json
{
  "type": "cascadePanel",
  "props": {
    "layoutZone": "left",
    "levels": [
      { "metamodelApiKey": "globalPickOption", "filter": "entityApiKey = 'province'", "label": "省份" },
      { "metamodelApiKey": "globalPickOption",
        "filter": "entityApiKey = 'city' AND apiKey IN $dep('provinceToCity', $prev.apiKey)",
        "label": "城市" },
      { "metamodelApiKey": "globalPickOption",
        "filter": "entityApiKey = 'district' AND apiKey IN $dep('cityToDistrict', $prev.apiKey)",
        "label": "区县" }
    ]
  }
}
```

`$dep(dependencyApiKey, controlApiKey)` 是语法糖，等价于：
```
查 globalPickDependencyDetail WHERE dependencyApiKey = 'provinceToCity' AND controlOptionApiKey = $prev.apiKey
返回 dependentOptionApiKeys 字段值（TEXT[] 数组）
```

## 六、表达式解析引擎

### 6.1 运行时解析流程

```
输入: filter = "entityApiKey = $prev.apiKey AND enableFlg = 1"
上下文: { prev: { apiKey: 'account', label: '客户', ... } }

Step 1: 词法分析 → tokens
  [field:entityApiKey] [op:=] [var:$prev.apiKey] [AND] [field:enableFlg] [op:=] [literal:1]

Step 2: 变量替换
  [field:entityApiKey] [op:=] [value:'account'] [AND] [field:enableFlg] [op:=] [value:1]

Step 3: 转为 FilterCondition[]
  [{ field: 'entityApiKey', op: 'eq', value: 'account' },
   { field: 'enableFlg', op: 'eq', value: '1' }]
  filterLogic: 'and'

Step 4: 传给现有的 applyFilters() 执行
```

### 6.2 $dep() 函数解析

```
输入: filter = "apiKey IN $dep('provinceToCity', $prev.apiKey)"
上下文: { prev: { apiKey: 'beijing' } }

Step 1: 解析 $dep('provinceToCity', 'beijing')
Step 2: 查 API: GET /meta/metadata?metamodelApiKey=globalPickDependencyDetail&entityApiKey=province
Step 3: 在结果中找 dependencyApiKey='provinceToCity' AND controlOptionApiKey='beijing'
Step 4: 取 dependentOptionApiKeys = ["beijingCity"]（TEXT[] 数组）
Step 5: 替换为 apiKey IN ('beijingCity')
```

### 6.3 与现有系统的兼容

| 现有配置 | 等价表达式 | 兼容方式 |
|:---------|:-----------|:---------|
| `depends: ['selected_entity_apiKey']` | `entityApiKey = $left.apiKey` | 向后兼容，不写 filter 时走旧逻辑 |
| `filterField: 'departApiKey'` | `departApiKey = $left.apiKey` | 向后兼容 |
| `filters: [{field:'entityApiKey', op:'eq', value:'province'}]` | `entityApiKey = 'province'` | filter 表达式编译为 FilterCondition[] |
| `chainQueryRef: 'provinceToCity'` | `apiKey IN $dep('provinceToCity', $prev.apiKey)` | chainQuery 是表达式 $dep() 的后端实现 |
| `filters: [{field:'deptParentApiKey', op:'eq', value:'$selectedDept'}]` | `deptParentApiKey = $prev.apiKey` | $动态值 是表达式的子集（适用于 dbc 列驱动的树形联动） |
