# 联动标识可配置化设计方案

## 一、现有能力深度分析

### 1.1 当前联动机制

```
PageContext.state (key-value 状态总线)
  ├── 左侧组件选中 → setState(stateKey, selectedApiKey)
  └── 右侧组件监听 → depends: [stateKey] → 值变化时重新加载
```

核心函数 `resolveStateKey(config)` 根据 DataSourceConfig 自动生成 stateKey：

| 数据源类型 | 自动生成的 stateKey | 示例 |
|:-----------|:-------------------|:-----|
| metaModel | `selected_metaModel_apiKey` | 固定 |
| metaItem | `selected_metaItem_apiKey` | 固定 |
| metadata(role) | `selected_role_apiKey` | 按 metamodelApiKey |
| metadata(globalPickOption) | `selected_globalPickOption_apiKey` | 按 metamodelApiKey |
| entityData(user) | `selected_user_id` | 按 entityApiKey |

### 1.2 当前局限性

#### 局限 1：单层联动，不支持多级级联

当前一个 splitPanel 内只有"左侧→右侧"一层联动。截图中的场景：

```
左侧: globalPickOption 列表（选中 province）
右侧: globalPickOption 子选项（应该只显示 province 下的城市）
      → 再选中城市后，显示该城市下的区县
```

这需要 **三级级联**：province → city → district，但当前 stateKey 只有一个 `selected_globalPickOption_apiKey`，无法区分"选中的是省份还是城市"。

#### 局限 2：同元模型多实例无法区分

同一个页面如果有两个 `metadata(globalPickOption)` 的列表面板，它们的 stateKey 都是 `selected_globalPickOption_apiKey`，会互相覆盖。

#### 局限 3：联动路径硬编码在 stateKey 命名规则中

`resolveStateKey` 的命名规则是写死的（`selected_{type}_apiKey`），无法表达：
- "这个列表选中后，应该用 `parent_metadata_api_key` 字段去过滤下一级"
- "这个列表选中后，应该用 `entity_api_key` 字段去过滤关联数据"

#### 局限 4：过滤条件只支持静态值

FilterCondition 的 value 是配置时写死的（如 `entityApiKey = province`），不支持"值来自上一级选中项的某个字段"。

#### 局限 5：depends 只能监听一个维度

右侧组件的 depends 是一个 stateKey 数组，但所有 depends 值变化都触发同一个 reload。无法表达"depends[0] 变化时重置 depends[1]"这种级联重置逻辑。

### 1.3 具体场景分析

#### 场景 A：省 → 市 → 区 三级联动

```
[省份列表] → 选中 guangdong → [chainQuery(provinceToCity)] → [城市列表] → 选中 guangZhou → [chainQuery(cityToDistrict)] → [区县列表]
```

省市区之间的关联**不是**通过 `parentMetadataApiKey` 直接关联，而是通过 `globalPickDependency` + `globalPickDependencyDetail` 依赖体系实现：

```
省份(guangdong)
  → 查 globalPickDependencyDetail WHERE dependencyApiKey='provinceToCity' AND controlOptionApiKey='guangdong'
  → 取 dependentOptionApiKeys = ["guangZhou","shenZhen",...] (TEXT[] 数组)
  → 查 globalPickOption WHERE entityApiKey='city' AND apiKey IN (...)
```

需要：
- 省份列表的 stateKey = `selectedProvince`
- 城市列表的 chainQueryRef = `provinceToCity`，chainQueryInputKey = `selectedProvince`
- 城市列表的 stateKey = `selectedCity`
- 区县列表的 chainQueryRef = `cityToDistrict`，chainQueryInputKey = `selectedCity`

**当前无法实现**：stateKey 自动生成为 `selected_globalPickOption_apiKey`，三个列表共享同一个 key；且缺少 chainQuery 机制。

#### 场景 B：实体 → 字段 → 选项值 三级联动

```
[实体列表] → 选中 account → [字段列表 filtered by entityApiKey=account] → 选中 industry → [选项值列表 filtered by itemApiKey=industry]
```

需要：
- 实体列表 stateKey = `selectedEntity`
- 字段列表 depends = `selectedEntity`，过滤 = `entityApiKey = ${selectedEntity}`
- 字段列表 stateKey = `selectedItem`
- 选项值列表 depends = `selectedItem`，过滤 = `itemApiKey = ${selectedItem}`

**当前部分支持**：entity→item 通过 `selected_entity_apiKey` 可以联动，但 item→pickOption 的第三级不支持。

#### 场景 C：角色 → 用户（跨模型联动）

```
[角色列表] → 选中 admin → [用户列表 filtered by roleApiKey=admin]
```

**当前支持**：通过 entityData + filterField 机制。但配置复杂，需要手动设置 filterField/filterFrom/depends。

## 二、设计方案：联动链（Linkage Chain）

### 2.1 核心概念

引入 **联动链（linkageChain）** 配置，替代当前的自动推导 stateKey + depends：

```typescript
/** 联动链节点 — 描述一个组件在联动链中的角色 */
interface LinkageNode {
  /** 联动链标识（同一条链上的组件共享同一个 chainId） */
  chainId: string;
  /** 在链中的层级（0=根，1=第二级，2=第三级...） */
  level: number;
  /** 选中后写入 PageContext 的 stateKey（覆盖自动生成） */
  stateKey?: string;
  /** 用上级选中值的哪个字段作为过滤值（默认 apiKey） */
  parentValueField?: string;
  /** 用本级数据的哪个字段匹配上级值（默认 parent_metadata_api_key） */
  filterField?: string;
  /** 上级选中值变化时，是否重置本级及下级的选中状态 */
  cascadeReset?: boolean;
}
```

### 2.2 DataSourceConfig 扩展

```typescript
interface DataSourceConfig {
  // ... 现有字段 ...
  
  /** 联动链配置（替代 stateKey + depends 的自动推导） */
  linkage?: LinkageNode;
}
```

### 2.3 三级联动配置示例

#### 省 → 市 → 区

> ⚠️ 省→市→区级联通过 globalPickDependency 依赖体系实现，不是 parentMetadataApiKey 直接过滤。
> 联动链方案对此场景应使用 chainQuery 而非 filterField。以下示例改用部门树场景演示联动链。

#### 部门 → 子部门 → 子子部门（树形联动链示例）

```json
// 顶级部门列表
{
  "type": "metadata",
  "metamodelApiKey": "department",
  "filters": [{ "field": "deptParentApiKey", "op": "eq", "value": null }],
  "linkage": {
    "chainId": "deptChain",
    "level": 0,
    "stateKey": "selectedDept"
  }
}

// 子部门列表
{
  "type": "metadata",
  "metamodelApiKey": "department",
  "linkage": {
    "chainId": "deptChain",
    "level": 1,
    "stateKey": "selectedSubDept",
    "parentValueField": "apiKey",
    "filterField": "deptParentApiKey",
    "cascadeReset": true
  }
}

// 子子部门列表
{
  "type": "metadata",
  "metamodelApiKey": "department",
  "linkage": {
    "chainId": "deptChain",
    "level": 2,
    "parentValueField": "apiKey",
    "filterField": "deptParentApiKey",
    "cascadeReset": true
  }
}
```

#### 实体 → 字段 → 选项值

```json
// 实体列表
{
  "type": "metadata",
  "metamodelApiKey": "entity",
  "linkage": { "chainId": "entityChain", "level": 0, "stateKey": "selectedEntity" }
}

// 字段列表
{
  "type": "metadata",
  "metamodelApiKey": "item",
  "linkage": {
    "chainId": "entityChain",
    "level": 1,
    "stateKey": "selectedItem",
    "parentValueField": "apiKey",
    "filterField": "entityApiKey",
    "cascadeReset": true
  }
}

// 选项值列表
{
  "type": "metadata",
  "metamodelApiKey": "pickOption",
  "linkage": {
    "chainId": "entityChain",
    "level": 2,
    "parentValueField": "apiKey",
    "filterField": "itemApiKey"
  }
}
```

### 2.4 运行时联动逻辑

```
useDataSource 改造：

1. 如果 config.linkage 存在：
   a. 从 PageContext.state 读取上级 stateKey 的值
   b. 上级 stateKey = chainId + '_level_' + (level-1)（或显式 stateKey）
   c. 用 parentValueField 从上级选中记录中取值
   d. 用 filterField 过滤本级数据
   e. 上级值变化时，cascadeReset=true 则清空本级及所有下级的选中状态

2. 如果 config.linkage 不存在：
   a. 走现有的 resolveStateKey + resolveDepends 逻辑（向后兼容）
```

### 2.5 属性面板 UI 改造

当前联动标识是只读展示（截图中红框部分）。改造为可配置：

```
联动配置
  ┌─────────────────────────────────┐
  │ 联动链: [regionChain    ▼]  新建 │  ← 下拉选择已有链或新建
  │ 层级:   [1 - 第二级     ▼]      │  ← 在链中的位置
  │ 上级取值字段: [apiKey    ▼]      │  ← parentValueField
  │ 本级过滤字段: [deptParentApiKey ▼] │  ← filterField
  │ ☑ 上级变化时重置选中              │  ← cascadeReset
  │                                   │
  │ 联动标识: selectedCity (自动)      │  ← 只读展示最终 stateKey
  └─────────────────────────────────┘
```

### 2.6 向后兼容

- `linkage` 字段可选，不配置时走现有自动推导逻辑
- 现有的 `stateKey`/`depends`/`filterField`/`filterFrom` 字段保留
- `linkage` 优先级高于旧字段，两者不冲突

## 三、实现优先级

### P0：解决当前截图中的问题（省→市→区）

不需要完整实现联动链，只需：
1. 让 `stateKey` 可在属性面板中手动编辑（当前是只读的）
2. 让 `depends` 可在属性面板中手动编辑
3. 支持 FilterCondition 的 value 引用 PageContext.state 中的动态值（如 `$selectedProvince`）

这三个改动就能让用户手动配置出三级联动，不需要 linkage 抽象。

### P1：联动链自动推导

基于 `parent_metadata_api_key` 字段自动识别层级关系：
- 如果数据源是 `globalPickOption` 且 `entityApiKey=city`，自动推导 depends = 上一个 `entityApiKey=province` 的组件
- 如果数据源是 `metadata(item)` 且页面中有 `metadata(entity)` 组件，自动建立联动

### P2：完整联动链 UI

实现 `LinkageNode` 配置 UI，支持可视化编排多级联动链。

## 四、P0 最小改动方案

### 4.1 stateKey 可编辑

PropertyPanel 中联动标识从只读改为可编辑输入框，用户可以自定义 stateKey。

### 4.2 FilterCondition 支持动态值

FilterCondition 的 value 支持 `$stateKey` 语法，运行时从 PageContext.state 中取值：

```typescript
// 配置
{ field: 'deptParentApiKey', op: 'eq', value: '$selectedDept' }

// 运行时解析
const resolvedValue = value.startsWith('$') ? pageState[value.slice(1)] : value;
```

### 4.3 useDataSource 改造

`applyFilters` 函数增加动态值解析：

```typescript
function resolveFilterValue(value: unknown, pageState: Record<string, unknown>): unknown {
  if (typeof value === 'string' && value.startsWith('$')) {
    return pageState[value.slice(1)] ?? '';
  }
  return value;
}
```

### 4.4 配置示例（P0 方案下的省→市→区）

> ⚠️ 省→市→区级联通过 globalPickDependency 依赖体系实现，不是 parentMetadataApiKey 直接关联。
> 正确的方式是使用 chainQuery 链式查询。

```
省份列表:
  数据源: metadata(globalPickOption), filter: entityApiKey=province
  联动标识: selectedProvince  ← 手动填写

城市列表:
  数据源: chainQueryRef=provinceToCity, chainQueryInputKey=selectedProvince
  联动标识: selectedCity  ← 手动填写

区县列表:
  数据源: chainQueryRef=cityToDistrict, chainQueryInputKey=selectedCity
```

这样不需要任何新抽象，只需要：
1. stateKey 输入框可编辑
2. FilterCondition value 支持 `$` 前缀引用 state
3. useDataSource 的 applyFilters 解析 `$` 引用
