# 链式查询模板设计方案

> 版本：1.0 | 日期：2026-04-23
> 目标：chainQuery 配置预定义为可复用模板，画布组件通过引用模板实现级联查询

---

## 一、核心思路

将 chainQuery 的 steps + target 配置抽象为**查询模板（chainQueryTemplate）**，作为元数据存储在 `p_common_metadata` 中。画布组件的数据源配置只需引用模板的 apiKey，不需要重复配置查询逻辑。

```
定义层（一次配置）：
  chainQueryTemplate: provinceToCity
    steps: [{ metamodelApiKey: globalPickDependencyDetail, matchField: controlOptionApiKey, ... }]
    target: { metamodelApiKey: globalPickOption, matchField: apiKey, matchMode: IN, ... }

使用层（画布组件引用）：
  dataSource: { chainQueryRef: 'provinceToCity', inputStateKey: 'selected_xxx' }
```

---

## 二、元模型设计

### 2.1 新增元模型：chainQueryTemplate

| 字段 | db_column | 类型 | 说明 |
|:-----|:----------|:-----|:-----|
| apiKey | api_key | 固定列 | 模板标识（如 provinceToCity） |
| label | label | 固定列 | 显示名（如 省份→城市） |
| description | description | 固定列 | 描述 |
| sourceMetamodelApiKey | dbc_varchar1 | VARCHAR | 输入来源元模型（如 globalPickOption） |
| targetMetamodelApiKey | dbc_varchar2 | VARCHAR | 输出目标元模型（如 globalPickOption） |
| queryConfig | dbc_textarea1 | TEXT | 完整的 steps + target JSON 配置 |
| enableFlg | dbc_smallint1 | SMALLINT | 是否启用 |

### 2.2 queryConfig JSON 结构

```json
{
  "steps": [
    {
      "metamodelApiKey": "globalPickDependencyDetail",
      "matchField": "controlOptionApiKey",
      "extraFilters": { "dependencyApiKey": "provinceToCity" },
      "outputField": "dependentOptionApiKeys",
      "outputFormat": "collect"
    }
  ],
  "target": {
    "metamodelApiKey": "globalPickOption",
    "matchField": "apiKey",
    "matchMode": "IN",
    "extraFilters": { "entityApiKey": "city" }
  }
}
```

### 2.3 预置模板数据

| apiKey | label | source | target | 说明 |
|:-------|:------|:-------|:-------|:-----|
| provinceToCity | 省份→城市 | globalPickOption | globalPickOption | 省→市级联 |
| cityToDistrict | 城市→区县 | globalPickOption | globalPickOption | 市→区级联 |
| entityToAllPickOptions | 实体→所有选项值 | entity | pickOption | 实体下所有字段的选项值 |
| entityToAllFormulas | 实体→所有计算公式 | entity | formulaCompute | 实体下所有字段的计算公式 |

## 三、使用方式

### 3.1 DataSourceConfig 扩展

```typescript
interface DataSourceConfig {
  // ... 现有字段 ...
  
  /** 引用预定义的链式查询模板 */
  chainQueryRef?: string;        // 模板 apiKey（如 'provinceToCity'）
  chainQueryInputKey?: string;   // 输入值来源的 stateKey
}
```

### 3.2 运行时执行

```
useDataSource 检测到 chainQueryRef:
  1. 从缓存中获取模板配置（启动时 useCanvasMetadata 已加载）
  2. 从 PageContext.state[chainQueryInputKey] 获取输入值
  3. 组装 ChainQueryRequest = { input, steps: 模板.steps, target: 模板.target }
  4. 调用 POST /meta/chain-query
  5. 返回结果
```

### 3.3 属性面板 UI

```
关联上级
  上级组件: [列表面板 - globalPickOption  ▼]
  
  关联方式:
  ┌─────────────────────────────────────┐
  │ ── 直接关联 ──                       │
  │ entityApiKey = 上级.apiKey（子数据） │
  │ apiKey = 上级.apiKey（同记录详情）   │
  │ ── 级联查询模板 ──                   │
  │ 省份→城市                            │  ← 从 chainQueryTemplate 加载
  │ 城市→区县                            │
  │ 实体→所有选项值                      │
  └─────────────────────────────────────┘
```

"级联查询模板"分组的选项从 `chainQueryTemplate` 元数据动态加载，按 `sourceMetamodelApiKey` 过滤（只显示上级元模型匹配的模板）。

## 四、管理界面

chainQueryTemplate 本身也是元数据，可以在画布中配置管理页面：

```
左侧: 列表面板 → chainQueryTemplate 列表
右侧: 详情 + queryConfig JSON 编辑器
```

未来可以做成可视化的 steps 编辑器（拖拽排序 + 每步下拉选择），但 P0 阶段用 JSON 编辑就够了。

## 五、实现步骤

| 步骤 | 内容 |
|:-----|:-----|
| 1 | init_local_dev.py 注册 chainQueryTemplate 元模型 + p_meta_item 字段 |
| 2 | init_local_dev.py 初始化 4 个预置模板数据 |
| 3 | useCanvasMetadata 启动时加载 chainQueryTemplate 列表 |
| 4 | ParentRefEditor 的"关联方式"下拉增加"级联查询模板"分组 |
| 5 | useDataSource 支持 chainQueryRef（从模板获取 steps/target） |
| 6 | DataSourceConfig 类型增加 chainQueryRef/chainQueryInputKey |

## 六、与现有机制的关系

| 配置方式 | 适用场景 | 存储 |
|:---------|:---------|:-----|
| parentRef (直接关联) | entity→item, 选项集→选项值 | DataSourceConfig.parentRef |
| chainQueryRef (模板引用) | 省→市, 实体→所有选项值 | DataSourceConfig.chainQueryRef → chainQueryTemplate 元数据 |
| chainQuery (内联配置) | 高级场景，模板不够用时 | DataSourceConfig.chainQuery（保留，兜底） |
| filter 表达式 | 高级用户手写 | DataSourceConfig.filter（保留，兜底） |

优先级：chainQueryRef > chainQuery > parentRef > filter > filters > depends
