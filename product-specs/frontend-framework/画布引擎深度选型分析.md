# 画布引擎深度选型分析

> 基于我们的元数据驱动 aPaaS 场景，深度对比 8 个候选项目

---

## 一、我们的核心需求

| 需求 | 优先级 | 说明 |
|:---|:---|:---|
| 元数据驱动 | P0 | 组件配置从 p_meta_item / p_common_metadata 动态生成，不是硬编码 |
| 画布区域定义 | P0 | 页面可划分多个区域，每个区域放不同组件 |
| 复合组件 | P0 | 容器组件内可嵌套子组件（如卡片内放表格+按钮） |
| 属性面板 | P0 | 点击组件右侧显示可编辑属性（字段选择、数据源、样式等） |
| 拖拽排序 | P1 | 组件可拖拽调整顺序 |
| JSON 配置存储 | P0 | 画布配置可序列化为 JSON 存入元数据 |
| React 19 兼容 | P0 | 必须兼容 React 19 |
| Ant Design 集成 | P1 | 最好能用 Ant Design 组件 |
| 轻量 | P1 | 不要引入过重的依赖 |

## 二、候选项目深度分析

### 2.1 Amis（百度，18.1k stars）⭐⭐⭐⭐⭐

**架构**：JSON Schema → 渲染引擎 → React 组件。一个 JSON 配置生成完整页面（表格、表单、CRUD、图表等）。

**核心优势**：
- **与我们的场景最契合** — Amis 本身就是"JSON 配置生成管理后台页面"，和我们的"元数据驱动页面"理念完全一致
- **内置 100+ 组件** — CRUD、Form、Table、Chart、Dialog、Drawer 等全部内置，不需要自己写
- **内置可视化编辑器** — amis-editor 提供完整的拖拽画布、属性面板、组件面板
- **数据源对接** — 内置 API 适配器，可以直接对接后端接口
- **嵌套布局** — 原生支持容器组件嵌套（Grid、HBox、Panel、Tabs 等）
- **条件显示/隐藏** — 内置表达式引擎，支持字段联动
- **国际化** — 内置多语言支持

**劣势**：
- 包体积大（~500KB+）
- 有自己的一套组件体系，不直接用 Ant Design
- React 19 兼容性需要验证
- 学习曲线中等

**与元数据对接**：
```json
// Amis JSON 配置（可从元数据动态生成）
{
  "type": "page",
  "title": "语言管理",
  "body": {
    "type": "crud",
    "api": "/meta/metadata/global-pick-options?globalPickApiKey=language",
    "columns": [
      {"name": "apiKey", "label": "值", "type": "text"},
      {"name": "label", "label": "标签", "type": "text"},
      {"name": "defaultFlg", "label": "默认", "type": "mapping", "map": {"1": "默认"}},
      {"name": "enableFlg", "label": "状态", "type": "mapping", "map": {"1": "启用", "0": "禁用"}}
    ],
    "itemActions": [
      {"type": "button", "label": "编辑", "actionType": "dialog"},
      {"type": "button", "label": "删除", "actionType": "ajax", "confirmText": "确定删除？"}
    ]
  }
}
```

**评分**：元数据契合度 ⭐⭐⭐⭐⭐ | 画布能力 ⭐⭐⭐⭐⭐ | 集成难度 ⭐⭐⭐（中等）

---

### 2.2 NocoBase（14k stars）⭐⭐⭐⭐

**架构**：数据模型驱动 + Schema 驱动 UI + 插件微内核。Collection（数据表）→ Block（UI 区块）→ Page（页面）。

**核心优势**：
- **数据模型驱动** — 和我们的元模型体系理念最接近（Collection ≈ Entity，Field ≈ Item）
- **Block 概念** — 页面由多个 Block 组成（表格 Block、表单 Block、图表 Block），与我们的 pageSection 概念一致
- **Schema 驱动** — UI 完全由 JSON Schema 描述，运行时可编辑
- **插件架构** — 所有功能都是插件，高度可扩展

**劣势**：
- 是一个完整的应用平台，不是可嵌入的组件库
- 引入 NocoBase 等于引入一整套技术栈（Node.js 后端 + Sequelize ORM）
- 与我们已有的 Java 后端体系冲突
- 不能只用前端部分

**评分**：元数据契合度 ⭐⭐⭐⭐⭐ | 画布能力 ⭐⭐⭐⭐ | 集成难度 ⭐（极难，架构冲突）

---

### 2.3 lowcode-engine（阿里，14.8k stars）⭐⭐⭐

**架构**：搭建协议 + 物料体系 + 渲染引擎 + 编辑器。

**核心优势**：
- 企业级低代码引擎，阿里内部大规模使用
- 完整的搭建协议（JSON Schema）
- 物料体系（组件注册、属性面板自动生成）
- 编辑器功能完整（拖拽、嵌套、属性编辑、大纲树）

**劣势**：
- 极其复杂，学习成本高
- 包体积大（500KB+）
- 协议与我们的元模型体系有冲突（需要适配层）
- 文档以中文为主，社区活跃度下降
- React 19 兼容性未知

**评分**：元数据契合度 ⭐⭐⭐ | 画布能力 ⭐⭐⭐⭐⭐ | 集成难度 ⭐⭐（难）

---

### 2.4 Puck（12.1k stars）⭐⭐⭐

**架构**：Config（组件定义）+ Data（页面数据）+ Editor（编辑器 UI）。

**核心优势**：
- 轻量（~50KB）
- API 简洁
- React 原生
- Slot API 支持嵌套

**劣势**：
- **不是为管理后台设计的** — 主要面向 CMS/营销页场景
- **没有内置数据组件** — 没有 Table、Form、CRUD，需要全部自己写
- **属性面板能力有限** — 只支持基础 field type，复杂配置（如列定义数组）需要自定义
- **嵌套组件 API 不稳定** — DropZone 已废弃改为 Slot，0.20 版本 API 变化大
- **实际集成中遇到多次白屏问题**

**评分**：元数据契合度 ⭐⭐ | 画布能力 ⭐⭐⭐ | 集成难度 ⭐⭐⭐⭐（简单但能力不足）

---

### 2.5 craft.js（7.8k stars）⭐⭐⭐

**架构**：React 组件即编辑器组件，通过 HOC 包装实现拖拽和编辑。

**核心优势**：
- 最高的定制自由度
- React 原生，组件即编辑器
- 嵌套支持好

**劣势**：
- 需要自建属性面板、工具栏、组件面板
- 没有内置任何业务组件
- 工作量大
- 维护活跃度一般

**评分**：元数据契合度 ⭐⭐ | 画布能力 ⭐⭐⭐⭐ | 集成难度 ⭐⭐（需要大量自建）

---

### 2.6 GrapesJS（23.9k stars）⭐⭐

**架构**：原生 JS 页面构建器，HTML/CSS 级别的编辑。

**核心优势**：
- 最成熟的页面构建器
- 支持 HTML/CSS 级别的自由编辑
- 丰富的插件生态

**劣势**：
- **不是 React 组件** — 原生 JS，需要包装
- **面向静态页面** — 不适合数据驱动的管理后台
- **没有数据绑定** — 不支持 API 对接、表格渲染等

**评分**：元数据契合度 ⭐ | 画布能力 ⭐⭐⭐⭐⭐ | 集成难度 ⭐（不适合）

---

### 2.7 Appsmith / ToolJet / Budibase ⭐⭐

**架构**：全栈低代码平台，前后端一体。

**共同劣势**：
- 是完整的应用平台，不是可嵌入的组件库
- 有自己的后端（Node.js），与我们的 Java 后端冲突
- 不能只用前端画布部分
- 引入等于替换整个技术栈

**评分**：不适合嵌入式集成

## 三、最终推荐

### 推荐方案：Amis + amis-editor

| 维度 | 理由 |
|:---|:---|
| **理念匹配** | Amis 就是"JSON 配置生成管理后台页面"，和我们的"元数据驱动"完全一致 |
| **开箱即用** | 内置 CRUD、Table、Form、Dialog、Drawer 等 100+ 组件，不需要自己写 |
| **画布编辑器** | amis-editor 提供完整的拖拽画布、属性面板、组件面板、大纲树 |
| **嵌套布局** | 原生支持 Grid、HBox、Panel、Tabs 等容器组件嵌套 |
| **数据源** | 内置 API 适配器，可直接对接我们的后端接口 |
| **JSON 存储** | 页面配置就是 JSON，可直接存入 pageSection 元数据 |
| **表达式引擎** | 内置条件显示/隐藏、字段联动、数据映射 |
| **生态成熟** | 百度内部大规模使用，18.1k stars，持续维护 |

### 集成方式

```
元数据层（p_meta_item / p_common_metadata）
    ↓ 动态生成
Amis JSON Schema（页面配置）
    ↓ 存储在 pageSection.componentProps
amis-editor（画布编辑器）← 管理员使用
    ↓ 保存
Amis Renderer（运行时渲染）← 普通用户使用
```

### 与 Puck 的对比

| 维度 | Puck | Amis |
|:---|:---|:---|
| 内置组件 | 0 个（全部自建） | 100+ 个（CRUD/Table/Form/Dialog...） |
| 属性面板 | 基础 field type | 完整的属性编辑器 |
| 嵌套布局 | Slot API（不稳定） | 原生支持（Grid/HBox/Panel/Tabs） |
| 数据绑定 | 无 | 内置 API 适配器 |
| 表达式引擎 | 无 | 内置条件/联动/映射 |
| 管理后台场景 | 不适合 | 专门为此设计 |
| 包体积 | ~50KB | ~500KB |
| 学习成本 | 低 | 中 |

### 实施路径

1. `npm install amis amis-editor` — 安装 Amis 和编辑器
2. 画布编辑器页面用 `amis-editor` 渲染
3. 运行时页面用 `amis` 的 `render` 函数渲染
4. 页面配置（Amis JSON Schema）存储在 `pageSection.componentProps` 中
5. 从元数据动态生成 Amis JSON Schema（字段列表、选项来源、校验规则等）

## 最终结论（2026-04-18 实际验证）

### Amis 方案验证失败

经过实际安装和代码级验证，Amis 整体方案在当前技术栈下不可用：

| 组件 | 版本 | 问题 | 严重程度 |
|:---|:---|:---|:---|
| amis-editor-core | 6.13.0 | 使用 `componentWillMount` / `componentWillReceiveProps`（React 19 已移除） | 致命 |
| amis-editor-core | 6.13.0 | 依赖 `mobx@4.15.7` + `mobx-react@6.3.1`，与项目 `mobx@6.15.0` 冲突 | 致命 |
| amis-editor | 6.13.0 | `dist/style.css` 不存在，无 CSS 文件 | 严重 |
| amis（渲染器） | 6.13.0 | 依赖 `mobx-react@6.3.1`（mobx 4 时代），React 19 下运行时崩溃 | 致命 |
| amis-core | 6.13.0 | 无废弃 API，peerDeps react>=16.8.6 | 兼容 |

核心矛盾：Amis 生态（editor + renderer）深度绑定 mobx 4 + mobx-react 6，而 React 19 要求 mobx 6 + mobx-react-lite 4。
除非 Amis 团队完成 React 19 适配（预计需要大版本升级），否则无法使用。

### 最终方案：自研轻量画布编辑器

基于 React 19 + Ant Design 6 + HTML5 Drag & Drop 自研，详见 `画布编辑器详细设计.md`。

优势：
- 零外部依赖冲突，完全掌控
- 与元数据体系深度集成（组件直接绑定 metamodelApiKey）
- 原子组件 + 复合组件 + 画布三层架构，可扩展
- JSON Schema 存储，与现有 pageSection 元数据无缝对接
