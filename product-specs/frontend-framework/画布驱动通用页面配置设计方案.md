# 画布驱动通用页面配置设计方案

> 版本：1.0 | 日期：2026-04-21
> 目标：将所有手写管理页面迁移为画布配置驱动，形成通用能力，客户可通过画布自助配置任意管理页面

---

## 一、核心问题

画布体系已跑通简单 CRUD（语言管理），但复杂页面存在能力缺口：

| 页面类型 | 代表 | 画布能否覆盖 | 缺口 |
|:---|:---|:---|:---|
| 简单 CRUD | 语言管理 | ✅ | — |
| 主从布局 | MetaModelManagementView | ❌ | 布局容器、主从联动、Tab |
| 树形管理 | 部门管理 | ❌ | 树联动、节点选中刷新 |
| 多实体聚合 | 元模型管理（4表联查） | ❌ | 跨元模型数据源 |
| 配置型 | 角色权限配置 | ❌ | 复杂交互、自定义数据源 |

## 三、整体架构设计

在现有画布体系上叠加三层能力：

```
第三层：页面模板（PageTemplate）
  masterDetail / treeDetail / simpleList / dashboard / configPanel

第二层：数据联动协议（PageContext + DataSource）
  DataSource 声明 / PageContext 发布-订阅 / SlotContext 注入

第一层：新增原子组件（8 个）
  masterDetail / listPanel / detailHeader / dynamicTabs
  relationNav / impactPanel / columnTable / metaTable

现有画布体系（52 原子 + 5 复合 + ActionContext + PageRuntime）
```

### 3.2 核心设计原则

1. **声明式数据源**：组件不直接调 API，而是声明 dataSource 类型和参数，运行时统一加载
2. **发布-订阅联动**：组件通过 PageContext 发布事件（如 `select`），其他组件订阅并响应
3. **插槽上下文注入**：容器组件（masterDetail、dynamicTabs）向子组件注入上下文数据
4. **渐进增强**：简单页面不需要联动协议，复杂页面按需启用
5. **向后兼容**：现有画布 JSON Schema 格式不变，新能力通过扩展 props 实现

## 四、数据联动协议设计（核心）

### 4.1 DataSource — 数据源声明

每个数据组件在 propSchema 中声明自己的数据需求：

```typescript
interface DataSourceConfig {
  /** 数据源类型 */
  type: 'metadata' | 'metaModel' | 'metaItem' | 'metaLink' | 'metaOption'
        | 'columnMapping' | 'custom';
  /** 元模型标识（metadata 类型必填） */
  metamodelApiKey?: string;
  /** 数据分组（metadata 类型可选） */
  entityApiKey?: string;
  /** 依赖的上下文变量（如 ${selectedApiKey}） */
  depends?: string[];
  /** 自定义 API 路径（custom 类型） */
  apiPath?: string;
  /** 是否懒加载（Tab 内容默认懒加载） */
  lazy?: boolean;
}
```

画布 JSON Schema 中的使用示例：

```json
{
  "type": "dataTable",
  "props": {
    "dataSource": {
      "type": "metaItem",
      "depends": ["selectedModelApiKey"]
    },
    "columns": ["apiKey", "label", "dbColumn", "requireFlg", "uniqueKeyFlg"]
  }
}
```

### 4.2 PageContext — 页面级状态总线

扩展现有 ActionContext，增加通用的页面状态管理：

```typescript
interface PageContextValue {
  /** 现有能力（保持不变） */
  dispatch: (payload: ActionPayload) => void;
  modal: ModalState | null;
  closeModal: () => void;
  refreshKey: number;

  /** 新增：页面状态存储（key-value） */
  state: Record<string, unknown>;
  setState: (key: string, value: unknown) => void;

  /** 新增：组件间事件（发布-订阅） */
  on: (event: string, handler: (data: unknown) => void) => () => void;
  emit: (event: string, data: unknown) => void;
}
```

组件间联动示例（左侧列表选中 → 右侧刷新）：

```
listPanel 组件：
  用户点击某行 → emit('select', { apiKey: 'entity' })
                → setState('selectedModelApiKey', 'entity')

dataTable 组件（右侧 Tab 内）：
  dataSource.depends = ['selectedModelApiKey']
  → 监听 state.selectedModelApiKey 变化
  → 自动重新加载数据
```

### 4.3 SlotContext — 容器→子组件的数据注入

容器组件（masterDetail、dynamicTabs）向子组件注入上下文：

```typescript
interface SlotContext {
  /** 容器注入的数据（如主从布局中，master 选中的记录） */
  parentData?: Record<string, unknown>;
  /** 容器注入的状态键（子组件可读取） */
  contextKeys?: string[];
  /** 所在插槽名称 */
  slotName: string;
}
```

运行时渲染时，容器组件将 SlotContext 合并到 RenderContext 中传递给子组件。

## 六、数据源类型扩展

### 6.1 数据源注册表

在 PageRuntime 中注册所有数据源类型，组件只需声明 dataSource.type，运行时自动调用对应 API：

```typescript
const DATA_SOURCE_REGISTRY: Record<string, DataSourceLoader> = {
  // 现有
  metadata:      (params) => listMetadata(params.metamodelApiKey, params.entityApiKey),

  // 新增
  metaModel:     ()       => listMetaModels(),
  metaItem:      (params) => listMetaItems(params.selectedModelApiKey),
  metaLink:      ()       => listMetaLinks(),
  metaOption:    (params) => listMetaOptions(params.selectedModelApiKey),
  columnMapping: (params) => getColumnMapping(params.selectedModelApiKey),

  // 未来扩展
  entity:        (params) => listEntities(params.entityApiKey),
  custom:        (params) => fetchCustomApi(params.apiPath, params),
};

interface DataSourceLoader {
  (params: Record<string, unknown>): Promise<unknown[]>;
}
```

### 6.2 依赖解析

当 dataSource.depends 中的 context key 变化时，自动重新加载数据：

```typescript
function useDataSource(config: DataSourceConfig, pageContext: PageContextValue) {
  const [data, setData] = useState<unknown[]>([]);
  const [loading, setLoading] = useState(false);

  // 收集依赖值
  const depValues = (config.depends ?? []).map(key => pageContext.state[key]);

  useEffect(() => {
    // 依赖值有 undefined → 不加载（等待上游组件设置）
    if (depValues.some(v => v === undefined)) { setData([]); return; }

    const loader = DATA_SOURCE_REGISTRY[config.type];
    if (!loader) return;

    // 构建参数：将 depends 的 key-value 合并到 config 参数中
    const params: Record<string, unknown> = { ...config };
    (config.depends ?? []).forEach((key, i) => { params[key] = depValues[i]; });

    setLoading(true);
    loader(params).then(setData).catch(() => setData([])).finally(() => setLoading(false));
  }, [config.type, ...depValues]);

  return { data, loading };
}
```

## 八、PageRuntime 改造

### 8.1 渲染流程升级

```
现有流程：
  pageApiKey → page + pageSection → CanvasSchema → 逐个渲染原子组件

升级后流程：
  pageApiKey → page + pageSection → CanvasSchema
    → 创建 PageContext（state + emit/on）
    → 递归渲染组件树
       → 遇到容器组件（masterDetail/dynamicTabs）→ 递归渲染 slot 子组件
       → 遇到数据组件 → 解析 dataSource → useDataSource 自动加载
       → 遇到交互组件 → 绑定 emit/setState
    → 组件间通过 PageContext 联动
```

### 8.2 递归渲染器

```typescript
function renderNode(node: CanvasNode, ctx: RenderContext): ReactNode {
  const atomDef = getAtomDef(node.type);
  if (!atomDef) return null;

  // 容器组件：递归渲染 slot
  if (node.props.masterSlot || node.props.detailSlot || node.props.tabs) {
    return atomDef.render({
      ...node.props,
      // 将 slot 中的 CanvasNode[] 转换为 ReactNode[]
      masterSlot: renderSlot(node.props.masterSlot as CanvasNode[], ctx),
      detailSlot: renderSlot(node.props.detailSlot as CanvasNode[], ctx),
      tabs: renderTabs(node.props.tabs as DynamicTabConfig[], ctx),
    }, ctx);
  }

  return atomDef.render(node.props, ctx);
}

function renderSlot(nodes: CanvasNode[] | undefined, ctx: RenderContext): ReactNode[] {
  if (!nodes || !Array.isArray(nodes)) return [];
  return nodes.map(n => <div key={n.id}>{renderNode(n, ctx)}</div>);
}
```

### 8.3 向后兼容

- 现有的简单画布页面（无 dataSource、无 slot）完全不受影响
- PageContext 的 state/emit/on 是新增字段，不影响现有 ActionContext
- 容器组件的 slot 渲染是新增逻辑，不影响现有原子组件的 render 函数

## 十、客户自助配置场景

### 10.1 场景一：客户新增"客户管理"主从页面

```

  apiKey: pageCustomerMaster
  pageName: 客户管理
  pageType: 1（全屏）
```

## 十一、CanvasNode 类型扩展（JSON Schema 变更）

### 11.1 现有 CanvasNode（不变）

```typescript
interface CanvasNode {
  id: string;
  type: string;
  props: Record<string, unknown>;
}
```

### 11.2 扩展：支持嵌套 slot

容器组件的 props 中可以包含 CanvasNode[]，形成树形结构：

```typescript
// masterDetail 的 props
{
  masterSlot: CanvasNode[];   // 左侧子组件列表
  detailSlot: CanvasNode[];   // 右侧子组件列表
}

// dynamicTabs 的 props
{
  tabs: Array<{
    key: string;
    label: string;
    slot: CanvasNode[];       // Tab 内容子组件列表
  }>;
}
```

### 11.3 PageSchema 不变

```typescript
interface PageSchema {
  version: string;
  components: CanvasNode[];   // 顶层组件列表（可包含容器组件）
}
```

顶层 components 中放一个 masterDetail，其 slot 中嵌套子组件，形成树形画布。

## 十三、风险与对策

| 风险 | 影响 | 对策 |
|:---|:---|:---|
| 嵌套 slot 导致画布编辑器复杂度飙升 | 编辑体验差 | Phase 4 专门优化，面包屑导航 + 插槽预览 |
| 数据联动性能（多组件同时刷新） | 页面卡顿 | useDataSource 内置 debounce + 缓存 |
| 容器组件 JSON Schema 过于复杂 | 客户配置困难 | 页面模板一键生成，客户只需微调 |
| 现有页面迁移回归 | 功能丢失 | Phase 5 逐页面对比验证，保留手写页面作为 fallback |

---

## 十四、技术决策总结

| 决策 | 选择 | 理由 |
|:---|:---|:---|
| 组件间通信 | PageContext 发布-订阅 | 松耦合，组件不需要知道彼此存在 |
| 数据加载 | 声明式 DataSource + useDataSource | 组件不直接调 API，运行时统一管理 |
| 嵌套布局 | slot 模式（CanvasNode[] 嵌套） | 与现有 CanvasNode 兼容，无需新数据结构 |
| 容器组件 | masterDetail 为核心 | 覆盖 80% 的复杂页面场景 |
| 迁移策略 | 渐进式（先新页面用画布，再迁移老页面） | 降低风险，保留 fallback |
