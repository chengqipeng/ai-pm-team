# 元数据驱动页面设计器 — 详细设计方案

> 版本：1.0 | 日期：2026-04-18
> 定位：基于元模型体系，实现所有后台管理页面的配置化生成
> 技术栈：React 19 + Ant Design 6 + 自研 SchemaRenderer

---

## 一、核心问题

当前 front-admin 的每个管理页面（用户管理、角色管理、部门管理等）都是手写 TSX，字段渲染逻辑散落在各页面中，存在大量重复代码。元模型已经定义了完整的字段类型、校验规则、选项来源、显示属性，但前端没有统一消费这些元数据。

目标：**一个实体 = 零代码页面**。只要在元模型中注册了 entity + item，前端自动生成完整的列表页、表单页、详情页。

---

## 二、框架选型分析

### 2.1 候选方案对比

| 方案 | 代表 | 优点 | 缺点 | 适合场景 |
|:---|:---|:---|:---|:---|
| 低代码平台 | Amis / LowCodeEngine | 开箱即用，可视化拖拽 | 黑盒、定制困难、与元模型体系耦合差 | 通用低代码，非元数据驱动 |
| JSON Schema 表单 | Formily / react-jsonschema-form | 表单能力强 | 仅覆盖表单，不覆盖列表/详情/画布 | 纯表单场景 |
| 自研 Schema Renderer | 当前方案 | 完全可控、与元模型一一绑定 | 需要自建组件库 | 元数据驱动的 aPaaS |
| 页面构建器 | GrapesJS / Craft.js | 可视化拖拽画布 | 偏静态页面，不适合数据驱动 | CMS / 营销页 |

### 2.2 结论：自研 SchemaRenderer + Ant Design 6

理由：
1. 元模型体系是自有的，字段类型（30+ 种 ItemTypeEnum）、选项来源（pickOption/globalPickOption/busiType）、校验规则（checkRule）、布局结构（fieldSet）都有独特的数据模型，第三方框架无法直接消费
2. 已有 React 19 + Ant Design 6 技术栈，组件库成熟
3. SchemaRenderer 已在前端框架设计方案中规划，本方案是其落地实现
4. 需要覆盖列表、表单、详情、画布四种页面类型，不仅仅是表单

## 三、架构设计

### 3.1 四层架构

```
┌─────────────────────────────────────────────────────┐
│ 第四层：画布（Canvas）                                │
│   PageCanvas → 多个 Section → 每个 Section 放一个     │
│   CompositeComponent 或 FieldComponent               │
├─────────────────────────────────────────────────────┤
│ 第三层：复合组件（CompositeComponent）                 │
│   EntityForm / EntityGrid / EntityDetail /            │
│   FilterPanel / ActionBar / StatCard                 │
├─────────────────────────────────────────────────────┤
│ 第二层：字段组件（FieldComponent）                     │
│   严格按 ItemTypeEnum 一一绑定，30+ 种                 │
│   TextInput / Select / MultiSelect / DatePicker /    │
│   BooleanRadio / RelationPicker / ImageUpload / ...  │
├─────────────────────────────────────────────────────┤
│ 第一层：元数据层（Metadata）                           │
│   MetaModel / MetaItem / PickOption /                │
│   GlobalPickOption / BusiType / FieldSet / CheckRule │
└─────────────────────────────────────────────────────┘
```

### 3.2 数据流

```
后端元数据 API
  ├── /metadata/items?entityApiKey=user          → 字段定义列表
  ├── /metadata/pick-options?entityApiKey=user    → 选项值
  ├── /metadata/busi-types?entityApiKey=user      → 业务类型
  ├── /metadata/global-pick-options?key=language   → 全局选项集
  ├── /metadata/field-sets?entityApiKey=user       → 字段分组
  └── /metadata/check-rules?entityApiKey=user      → 校验规则
        ↓
  MetadataStore（MobX）— 缓存 + 合并
        ↓
  SchemaResolver — 将元数据转换为 UI Schema
        ↓
  SchemaRenderer — 递归渲染 React 组件树
        ↓
  FieldComponent / CompositeComponent — 最终 UI
```

### 4.2 字段组件统一接口

```typescript
interface MetaFieldProps {
  /** 元数据字段定义 */
  item: XEntityItem
  /** 当前值 */
  value: unknown
  /** 值变更回调 */
  onChange: (value: unknown) => void
  /** 表单模式 */
  mode: 'create' | 'edit' | 'detail'
  /** 选项数据（pickOption / busiType / globalPickOption） */
  options?: OptionItem[]
  /** 是否只读（由 readonlyStatus + mode 计算） */
  readonly?: boolean
  /** 是否禁用（由表达式引擎计算） */
  disabled?: boolean
}

interface OptionItem {
  value: string
  label: string
  defaultFlg?: number
  enableFlg?: number
}
```

### 4.3 默认值统一机制

所有字段组件的默认值遵循统一规则（优先级从高到低）：

| 优先级 | 来源 | 适用类型 | 示例 |
|:---|:---|:---|:---|
| 1 | 编辑时已有数据 | 所有 | user.status = '1' |
| 2 | 选项列表 `defaultFlg=1` | SELECT / busiType / globalPickOption | internalUser, zhCn, asiaShanghai |
| 3 | 元数据 `defaultValue` | BOOLEAN / SELECT / NUMBER | status='1', isVirtual='0' |
| 4 | Common 级固定值 | 特定字段 | departApiKey='companyRoot' |

```typescript
function resolveDefaultValue(
  item: XEntityItem,
  options?: OptionItem[],
  existingValue?: string
): string | undefined {
  // 优先级 1：已有值
  if (existingValue) {
    // 检查值是否在选项列表中有效（防止旧值已废弃）
    if (options && options.length > 0) {
      if (options.some(o => o.value === existingValue)) return existingValue
      // 旧值无效，降级到 defaultFlg
    } else {
      return existingValue
    }
  }
  // 优先级 2：选项列表 defaultFlg=1
  if (options && options.length > 0) {
    const def = options.find(o => o.defaultFlg === 1 && o.enableFlg !== 0)
    if (def) return def.value
  }
  // 优先级 3：元数据 defaultValue
  if (item.defaultValue != null && item.defaultValue !== '') {
    return String(item.defaultValue)
  }
  return undefined
}
```

## 五、复合组件（CompositeComponent）

### 5.1 EntityForm — 实体表单

```typescript
interface EntityFormProps {
  entityApiKey: string
  mode: 'create' | 'edit'
  recordId?: string          // 编辑时传入
  fieldSets?: FieldSetConfig[] // 字段分组（来自 fieldSet 元数据）
  onSave: (data: Record<string, unknown>) => void
  onCancel: () => void
}
```

内部逻辑：
1. 加载 `listEntityItems(entityApiKey)` → 字段定义
2. 加载 `listPickOptions(entityApiKey)` → 选项值
3. 加载 `listBusiTypes(entityApiKey)` → 业务类型
4. 扫描 `referGlobalFlg=1` 的字段 → 加载 globalPickOption
5. 按 `fieldSet` 分组，按 `itemOrder` 排序
6. 对每个字段调用 `resolveFormComponent(itemType)` → 渲染对应 FieldComponent
7. 对每个字段调用 `resolveDefaultValue()` → 填充默认值
8. 提交时按 `checkRule` 校验

### 5.2 EntityGrid — 实体列表

```typescript
interface EntityGridProps {
  entityApiKey: string
  columns?: string[]          // 指定显示哪些字段（默认全部可见字段）
  actions?: GridAction[]      // 行操作按钮
  onRowClick?: (record: Record<string, unknown>) => void
  pageSize?: number
}
```

内部逻辑：
1. 加载字段定义 → 生成列配置
2. 按 `itemType` 选择列渲染器（文本截断、标签、日期格式化等）
3. 支持排序、筛选、分页
4. 选项类字段自动显示 label（而非 code/apiKey）

### 5.3 EntityDetail — 实体详情

```typescript
interface EntityDetailProps {
  entityApiKey: string
  recordId: string
  fieldSets?: FieldSetConfig[]
}
```

只读展示，按 fieldSet 分组，所有字段只读渲染。

### 5.4 FilterPanel — 筛选面板

根据字段类型自动生成筛选条件（文本→模糊搜索，选项→下拉，日期→范围，数字→区间）。

### 5.5 ActionBar — 操作栏

新建、批量删除、导入、导出等操作按钮，可配置。

## 七、元数据到 UI 的自动推导

### 7.1 SchemaResolver — 元数据 → UI Schema

```typescript
class SchemaResolver {
  /**
   * 从元数据自动生成页面 Schema。
   * 无需手写页面配置，只要有 entity + item 元数据就能生成完整页面。
   */
  static autoResolve(entityApiKey: string, metadata: EntityMetadata): PageCanvasConfig {
    const items = metadata.items.filter(isFieldVisible)
    const fieldSets = metadata.fieldSets ?? [{ label: '基本信息', fields: items.map(i => i.apiKey) }]

    return {
      apiKey: `auto_${entityApiKey}`,
      label: metadata.entity.label,
      entityApiKey,
      sections: [
        { key: 'filter', layout: 'full', component: { type: 'FilterPanel', props: { fields: items.filter(i => isFilterable(i)).map(i => i.apiKey) } } },
        { key: 'actions', layout: 'full', component: { type: 'ActionBar', props: { actions: ['create', 'batchDelete'] } } },
        { key: 'grid', layout: 'full', component: { type: 'EntityGrid', props: { columns: items.slice(0, 8).map(i => i.apiKey) } } },
      ]
    }
  }
}
```

### 7.2 自动推导规则

| 元数据属性 | 推导结果 |
|:---|:---|
| `itemType=2 (SELECT)` + `pickOption` 有数据 | 渲染 `<MetaSelect>`，选项从 pickOption 加载 |
| `itemType=2` + `referGlobalFlg=1` | 渲染 `<MetaSelect>`，选项从 globalPickOption 加载 |
| `itemType=31 (BOOLEAN)` | 渲染 `<MetaBooleanRadio>`，选项 0/1 |
| `itemType=3 (MULTI_SELECT)` | 渲染 `<MetaMultiSelect>`，多选 |
| `requireFlg=1` | label 后加红色 `*`，提交时校验非空 |
| `readonlyStatus=3` | 字段只读 |
| `creatable=0` + mode='create' | 新建时只读 |
| `updatable=0` + mode='edit' | 编辑时只读 |
| `defaultValue` 非空 | 新建时自动填充 |
| `itemOrder` | 字段排序 |
| `fieldSet` 有分组 | 表单按分组渲染 |
| `checkRule` 有规则 | 提交时执行校验 |

## 八、实现路径

### Phase 1：字段组件库（2 周）

- 实现 30+ 个 `<Meta*>` 字段组件，严格按 ItemTypeEnum 映射
- 统一 `MetaFieldProps` 接口
- 统一 `resolveDefaultValue` 默认值机制
- 统一选项加载（pickOption / globalPickOption / busiType）

### Phase 2：复合组件（3 周）

- `<EntityForm>` — 元数据驱动表单，替代所有手写表单
- `<EntityGrid>` — 元数据驱动列表，替代所有手写列表
- `<EntityDetail>` — 元数据驱动详情
- `<FilterPanel>` — 自动筛选面板

### Phase 3：画布 + 页面配置（2 周）

- `<PageCanvas>` — 画布渲染器
- 页面配置 JSON Schema 定义
- 页面配置存储（新增 `pageLayout` 元模型）
- 自动推导（无配置时从元数据自动生成）

### Phase 4：可视化设计器（4 周）

- 拖拽画布编辑器
- 区域（Section）拖拽排列
- 复合组件拖入区域
- 字段选择器（从元数据字段列表中勾选）
- 实时预览
- 配置导出/导入

## 十、技术决策总结

| 决策 | 选择 | 理由 |
|:---|:---|:---|
| 渲染引擎 | 自研 SchemaRenderer | 与元模型体系深度绑定，完全可控 |
| UI 组件库 | Ant Design 6 | 已有技术栈，组件丰富，企业级 |
| 状态管理 | MobX | 响应式自动追踪，适合表单场景 |
| 字段组件 | 自研 Meta* 系列 | 严格按 ItemTypeEnum 一一绑定 |
| 页面配置 | JSON Schema + 元模型存储 | 配置即生效，支持 Common/Tenant 双层 |
| 可视化设计器 | 自研拖拽画布 | 基于 dnd-kit，与画布 Schema 直接对接 |
