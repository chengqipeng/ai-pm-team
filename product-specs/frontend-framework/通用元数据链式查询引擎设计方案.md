# 通用元数据链式查询引擎设计方案

> 版本：1.0 | 日期：2026-04-23
> 目标：一个接口覆盖所有元模型间的关联数据查询，支持任意层级、任意数据格式

---

## 一、问题全景

### 1.1 查询链路的多样性

| 场景 | 链路 | 层级 |
|:-----|:-----|:-----|
| 实体→字段 | entity → item | 1 级 |
| 实体→字段→选项值 | entity → item → pickOption | 2 级 |
| 实体→字段→计算公式→公式引用 | entity → item → formulaCompute → formulaComputeItem | 3 级 |
| 选项集→依赖→明细→反查选项值 | globalPickOption → dependencyDetail → globalPickOption | 2 级 + 反查 |
| 元模型→字段→选项值 | metaModel → metaItem → metaOption | 2 级 |

### 1.2 数据格式的多样性

中间步骤产出的数据可能是：

| 格式 | 示例 | 来源 |
|:-----|:-----|:-----|
| 单值 | `"account"` | 普通字段（entityApiKey） |
| 文本数组 | `["guangZhou","shenZhen","zhuHai"]` | 多选字段（TEXT[] 数组），如 globalPickDependencyDetail.dependentOptionApiKeys |
| JSON 数组 | `["field1","field2"]` | 未来可能的配置字段 |
| JSON 对象 | `{"key":"value"}` | 未来可能的复杂配置 |
| 多条记录的字段值集合 | 查出 10 条记录，收集每条的 apiKey | 一对多关系 |

### 1.3 设计原则

1. **一个接口**：`/meta/chain-query`，覆盖所有场景
2. **声明式配置**：前端传入查询步骤描述，后端执行
3. **数据处理可配置**：每一步可以声明如何解析上一步的输出
4. **可注册扩展**：新的数据格式解析器可以注册，不改核心代码

## 二、接口设计

### 2.1 请求结构

```
POST /meta/chain-query
Content-Type: application/json

{
  "input": "guangdong",
  "steps": [ ... ],
  "target": { ... }
}
```

### 2.2 完整数据模型

```typescript
/** 链式查询请求 */
interface ChainQueryRequest {
  /** 初始输入值（通常是左侧选中记录的 apiKey） */
  input: string;
  
  /** 查询步骤（0~N 步，每步产出中间结果供下一步使用） */
  steps?: ChainQueryStep[];
  
  /** 最终目标查询（用 steps 的产出作为过滤条件） */
  target: ChainQueryTarget;
}

/** 中间查询步骤 */
interface ChainQueryStep {
  /** 查询哪个元模型 */
  metamodelApiKey: string;
  
  /** 匹配条件：用哪个字段匹配输入值 */
  matchField: string;
  
  /** 附加过滤条件（可选，如 dependencyApiKey='provinceToCity'） */
  extraFilters?: Record<string, string>;
  
  /** 从结果中提取哪个字段作为下一步的输入 */
  outputField: string;
  
  /**
   * 输出数据的解析方式：
   *   single    — 取第一条记录的 outputField 值（默认）
   *   collect   — 收集所有记录的 outputField 值，合并为列表。数组类型字段（TEXT[]）自动展开
   *   json      — 取第一条记录的 outputField 值，解析为 JSON 数组
   *   jsonPath  — 取第一条记录的 outputField 值，用 jsonPath 表达式提取
   */
  outputFormat?: 'single' | 'collect' | 'json' | 'jsonPath';
  
  /** jsonPath 模式的路径表达式 */
  jsonPath?: string;
}

/** 最终目标查询 */
interface ChainQueryTarget {
  /** 查询哪个元模型 */
  metamodelApiKey: string;
  
  /** 用哪个字段匹配 steps 的产出 */
  matchField: string;
  
  /**
   * 匹配模式：
   *   EQ  — 等于（steps 产出单值时）
   *   IN  — 在列表中（steps 产出列表时）
   */
  matchMode?: 'EQ' | 'IN';
  
  /** 附加过滤条件（如 entityApiKey='city'） */
  extraFilters?: Record<string, string>;
  
  /** 按哪个字段排序（可选） */
  orderBy?: string;
}
```

### 2.3 响应结构

```json
{
  "code": 200,
  "data": {
    "records": [ ... ],
    "total": 21,
    "stepOutputs": ["guangZhou,shenZhen,zhuHai,..."]
  }
}
```

`stepOutputs` 返回每一步的中间产出，方便调试和前端缓存。

```json
  "target": {
    "metamodelApiKey": "globalPickOption",
```

```json
      "metamodelApiKey": "globalPickDependencyDetail",

  "target": {
    "metamodelApiKey": "globalPickOption",
    "matchField": "apiKey",
    "matchMode": "IN",
    "extraFilters": { "entityApiKey": "district" }

```

### 3.3 实体→字段（0 步，直接查询）

```json
{
  "input": "account",
  "target": {
    "metamodelApiKey": "item",
    "matchField": "entityApiKey",
    "matchMode": "EQ"
  }
}
```

### 3.4 实体→字段→选项值（1 步 + 目标）

```json
{
  "input": "account",
  "steps": [
    {
      "metamodelApiKey": "item",
      "matchField": "entityApiKey",
      "outputField": "apiKey",
      "outputFormat": "collect"
    }
  ],
  "target": {
    "metamodelApiKey": "pickOption",
    "matchField": "itemApiKey",
    "matchMode": "IN"
  }
}
```

注意：这里 step 的 outputFormat=collect，意思是查出 account 的所有字段，收集每条的 apiKey，合并为列表 `["name","phone","email",...]`，然后 target 用 `itemApiKey IN (...)` 查出所有选项值。

### 3.5 实体→字段→计算公式→公式引用字段（2 步 + 目标）

```json
{
  "input": "account",
  "steps": [
    {
      "metamodelApiKey": "item",
      "matchField": "entityApiKey",
      "outputField": "apiKey",
      "outputFormat": "collect"
    },
    {
      "metamodelApiKey": "formulaCompute",
      "matchField": "itemApiKey",
      "outputField": "apiKey",
      "outputFormat": "collect"
    }
  ],
  "target": {
    "metamodelApiKey": "formulaComputeItem",
    "matchField": "computeApiKey",
    "matchMode": "IN"
  }
}
```

### 3.6 未来场景：JSON 格式数据

假设某个元模型的字段存储了 JSON 数组：

```json
{
  "input": "someConfig",
  "steps": [
    {
      "metamodelApiKey": "someMetamodel",
      "matchField": "apiKey",
      "outputField": "jsonConfig",
      "outputFormat": "json"
    }
  ],
  "target": {
    "metamodelApiKey": "anotherMetamodel",
    "matchField": "apiKey",
    "matchMode": "IN"
  }
}
```

`outputFormat: "json"` 会把 `jsonConfig` 字段的值解析为 JSON 数组 `["a","b","c"]`，作为下一步的输入。

## 四、后端实现

### 4.1 核心执行引擎

```java
@PostMapping("/meta/chain-query")
public ChainQueryResponse chainQuery(@RequestBody ChainQueryRequest request) {
    // 初始输入
    Object currentInput = request.getInput();
    List<String> stepOutputs = new ArrayList<>();
    
    // 逐步执行
    if (request.getSteps() != null) {
        for (ChainQueryStep step : request.getSteps()) {
            currentInput = executeStep(step, currentInput);
            stepOutputs.add(String.valueOf(currentInput));
        }
    }
    
    // 执行目标查询
    List<?> records = executeTarget(request.getTarget(), currentInput);
    
    return new ChainQueryResponse(records, records.size(), stepOutputs);
}
```

### 4.2 步骤执行

```java
private Object executeStep(ChainQueryStep step, Object input) {
    // 查询数据
    List<?> rows = queryByField(step.getMetamodelApiKey(), step.getMatchField(), 
                                input, step.getExtraFilters());
    
    if (rows.isEmpty()) return Collections.emptyList();
    
    // 按 outputFormat 解析
    switch (step.getOutputFormat()) {
        case "single":
            return getFieldValue(rows.get(0), step.getOutputField());
            
        case "collect":
            // 增强：数组类型字段（List<String>）自动展开后合并
            return rows.stream()
                .map(r -> getFieldValue(r, step.getOutputField()))
                .filter(Objects::nonNull)
                .flatMap(v -> {
                    if (v instanceof List<?> list) {
                        return list.stream().map(String::valueOf);
                    }
                    return Stream.of(String.valueOf(v));
                })
                .distinct()
                .collect(Collectors.toList());
            
        case "json":
            String jsonStr = String.valueOf(getFieldValue(rows.get(0), step.getOutputField()));
            return JSON.parseArray(jsonStr, String.class);
            
        case "jsonPath":
            // 预留：用 JsonPath 库提取
            return getFieldValue(rows.get(0), step.getOutputField());
            
        default:
            return getFieldValue(rows.get(0), step.getOutputField());
    }
}
```

### 4.3 目标查询执行

```java
private List<?> executeTarget(ChainQueryTarget target, Object input) {
    if ("IN".equals(target.getMatchMode())) {
        // input 是列表
        List<String> values = toStringList(input);
        List<?> all = queryAll(target.getMetamodelApiKey(), target.getExtraFilters());
        return all.stream()
            .filter(r -> values.contains(String.valueOf(getFieldValue(r, target.getMatchField()))))
            .collect(Collectors.toList());
    } else {
        // input 是单值
        return queryByField(target.getMetamodelApiKey(), target.getMatchField(), 
                           input, target.getExtraFilters());
    }
}
```

### 4.4 字段值获取（通用）

```java
/**
 * 从元数据记录中获取字段值。
 * 支持固定列（apiKey, entityApiKey 等）和 dbc 列映射的业务字段。
 */
private Object getFieldValue(Object record, String fieldName) {
    if (record instanceof GenericMetadata gm) {
        // GenericMetadata: 先查 fields Map，再查固定列
        Object val = gm.getFields().get(fieldName);
        if (val != null) return val;
    }
    // 反射获取 Entity 字段
    try {
        Field field = findField(record.getClass(), fieldName);
        if (field != null) {
            field.setAccessible(true);
            return field.get(record);
        }
    } catch (Exception e) {
        // ignore
    }
    return null;
}
```

### 5.3 属性面板配置 UI

用户看到的是友好的下拉选择，不是 JSON：

```
关联上级
  上级组件: [列表面板 - 全局选项集(province)  ▼]
  关联方式: [省份→城市（级联查询）            ▼]  ← 系统自动推导
```

系统根据选择自动生成 chainQuery 配置，用户不需要理解 steps/target 的概念。

### 5.4 关联方式的推导逻辑

```
1. 查 p_meta_link：上级元模型 → 当前元模型 是否有直接关系
   → 有：生成 parentRef（直接过滤）
   
2. 查 p_meta_link：上级元模型 → 中间元模型 → 当前元模型 是否有两级链路
   → 有：生成 chainQuery（两步查询）
   
3. 查 globalPickDependency：上级和当前是否同为 globalPickOption 且有依赖关系
   → 有：生成 chainQuery（级联查询）
   
4. 都没有：不提供关联选项
```

## 六、数据处理器注册机制

### 6.1 后端处理器接口

```java
public interface StepOutputProcessor {
    /** 处理器标识（对应 outputFormat 值） */
    String getFormat();
    
    /** 从查询结果中提取输出值 */
    Object process(List<?> rows, String outputField, Map<String, String> params);
}
```

### 6.2 内置处理器

| format | 处理逻辑 | 适用场景 |
|:-------|:---------|:---------|
| single | 取第一条记录的字段值 | 一对一关系 |
| collect | 收集所有记录的字段值，去重合并为列表。数组类型字段（TEXT[]）自动展开后合并 | 一对多关系、多选字段 |
| split | ~~取第一条记录的字段值，按分隔符拆分~~（**已废弃**，多选字段改为 TEXT[] 数组后直接用 collect） | — |
| json | 取第一条记录的字段值，解析为 JSON 数组 | JSON 格式存储 |
| jsonPath | 取第一条记录的字段值，用 JsonPath 提取 | 复杂 JSON 结构 |

### 6.3 扩展新处理器

```java
@Component
public class XmlOutputProcessor implements StepOutputProcessor {
    @Override
    public String getFormat() { return "xml"; }
    
    @Override
    public Object process(List<?> rows, String outputField, Map<String, String> params) {
        String xml = String.valueOf(getFieldValue(rows.get(0), outputField));
        // 解析 XML，提取目标值
        return XmlUtils.extractValues(xml, params.get("xpath"));
    }
}
```

注册后自动可用，不改核心代码。

## 七、与现有机制的关系

| 机制 | 适用场景 | 是否保留 |
|:-----|:---------|:---------|
| parentRef (type=direct) | 一级直接关系（entity→item） | ✅ 保留，最简单 |
| chainQuery | 多级关系 + 数据格式转换 | ✅ 新增，最通用 |
| depends + filterField | 跨模型联动（department→user） | ✅ 保留，向后兼容 |
| filter 表达式 | 高级用户手写 | ✅ 保留，兜底 |
| filters[] | 可视化条件编辑器 | ✅ 保留，简单过滤 |

优先级：chainQuery > parentRef > filter > filters > depends
