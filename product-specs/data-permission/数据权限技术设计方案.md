# aPaaS 数据权限技术设计方案

> 基于 paas-privilege-core 接口层 + paas-privilege-service 实现层的完整源码分析

---

## 一、文档概述

### 1.1 目标

本文档基于老系统 `paas-privilege-core`（接口层）和 `paas-privilege-service`（实现层，740 个 Java 文件，~113K 行代码）的深度分析，输出一份完整的数据权限技术设计方案，作为新系统 `paas-privilege-service`（Spring Cloud 架构）重构的设计蓝图。

### 1.2 系统定位

数据权限服务是 aPaaS 平台的核心安全基础设施，解决的核心问题是：**"谁能看到哪些数据"**。它是 CRM 系统中最复杂的服务之一，涉及共享规则引擎、权限反规范化、权限匹配链、V3 权限同步等多个子系统。

### 1.3 术语表

| 术语 | 英文 | 含义 |
|------|------|------|
| 数据权限 | Data Permission | 控制用户对业务数据记录的可见性和操作权限 |
| 共享规则 | Sharing Rule | 定义数据在用户/部门/公共组/区域间的自动共享策略 |
| 反规范化 | Denormalize | 将权限关系展开写入冗余表，加速运行时查询 |
| 权限匹配链 | MatchLink | 链式处理模式，逐级判断用户对数据的访问权限 |
| Share 表 | Share Table | 存储共享关系的数据表，每条记录表示一个"数据→主体"的权限授予 |
| 权限主体 | Subject | 被授予权限的对象：用户/部门/公共组/区域 |
| rowCause | Row Cause | 共享记录的来源类型标识 |
| 区域 | Territory | CRM 中按地理或业务维度划分的权限域 |
| 权限组 | Privilege Group | 用于权限计算的组织单元抽象 |

### 4.5 共享规则执行流程

```mermaid
sequenceDiagram
    participant Biz as 业务系统
    participant MQ as RocketMQ
    participant Consumer as DataChangeSharingRuleConsumer
    participant Engine as SharingRulePrivilegeOperate
    participant Meta as paas-metarepo-service
    participant DB as MySQL (share表)

    Biz->>MQ: 数据变更消息<br/>(创建/修改/删除/转移)
    MQ->>Consumer: 消费消息

    Consumer->>Consumer: 解析消息，提取entityId, dataIds

    Consumer->>Meta: 查询实体的共享规则列表
    Meta-->>Consumer: List~XObjectShareRule~

    loop 每条共享规则
        Consumer->>Engine: 执行规则

        alt 基于负责人 (owner)
            Engine->>DB: 查询数据负责人
            Engine->>Engine: 判断负责人是否属于来源组
            Engine->>Engine: 确定目标主体
        else 基于条件 (condition)
            Engine->>Engine: 解析条件表达式
            Engine->>DB: 按条件筛选匹配数据
            Engine->>Engine: 确定目标主体
        end

        alt 目标是用户
            Engine->>DB: INSERT INTO x_{entity}_share<br/>(userId=目标用户, rowCause=5)
        else 目标是部门
            Engine->>DB: INSERT INTO x_{entity}_share<br/>(groupId=部门ID, groupType=部门, rowCause=5)
        else 目标是公共组
            Engine->>DB: INSERT INTO x_{entity}_share<br/>(groupId=公共组ID, groupType=公共组, rowCause=5)
        else 目标是部门及下级
            Engine->>Engine: 展开部门树
            loop 每个子部门
                Engine->>DB: INSERT INTO x_{entity}_share
            end
        end
    end

    Consumer->>MQ: 发送反规范化消息
```

| 添加团队成员 | 创建团队成员share | 1 |
| 移除团队成员 | 删除团队成员share | 1 |

## 五、权限反规范化引擎（Denormalize Pipeline）

### 5.1 设计思想

权限反规范化的核心思想是：将复杂的"用户→组→数据"多级间接关系，展开为扁平化的直接关系表，使运行时权限查询从多表 JOIN 变为单表查询。

### 5.2 Pipeline 架构

```mermaid
flowchart LR

        I1["权限变更事件"]
        I2["组织变更事件"]
        I3["规则变更事件"]
    end

    subgraph Transform["@DataTransform<br/>数据转换 (18个实现)"]
        T1["用户→部门映射"]
        T2["部门→子部门展开"]
        T3["公共组→成员展开"]
        T4["区域→成员展开"]
        T5["职位→用户映射"]
        T6["...其他12个"]
    end

    subgraph Processor["@DataProcessor<br/>数据处理"]
        P1["GroupHier处理器<br/>组层级关系"]
        P2["GroupUser处理器<br/>组用户关系"]
        P3["ObjectsShare处理器<br/>对象共享关系"]
        P4["UserHier处理器<br/>用户层级关系"]
    end

    subgraph Operator["DataOperator<br/>数据操作"]
        O1["批量INSERT"]
        O2["批量DELETE"]
        O3["批量UPDATE"]

    Operator --> DB["反规范化表<br/>x_priv_group_user_dn<br/>x_priv_group_hier_dn"]
```

### 5.3 注解驱动的自动注册

## 六、权限匹配链（MatchLink）

### 6.1 设计模式

采用责任链模式（Chain of Responsibility），每个 Matcher 负责一种权限来源的判定，按优先级链式执行。

```mermaid
flowchart LR
    REQ["权限查询请求<br/>(userId, dataId, entityId)"] --> M1

    M1["AdminMatcher<br/>管理员判定"] -->|不匹配| M2
    M1 -->|匹配| RESULT

    M2["OwnerMatcher<br/>负责人判定"] -->|不匹配| M3
    M2 -->|匹配| RESULT

    M3["DepartMatcher<br/>部门判定"] -->|不匹配| M4
    M3 -->|匹配| RESULT

    M4["TeamMemberMatcher<br/>团队成员判定"] -->|不匹配| M5
    M4 -->|匹配| RESULT

    M5["PublicGroupMatcher<br/>公共组判定"] -->|不匹配| M6
    M5 -->|匹配| RESULT

    M6["ShareRuleMatcher<br/>共享规则判定"] -->|不匹配| M7
    M6 -->|匹配| RESULT

    M7["TerritoryMatcher<br/>区域判定"] -->|不匹配| DENY
    M7 -->|匹配| RESULT

    RESULT["返回权限结果<br/>(可见 + 访问级别)"]
    DENY["返回无权限"]
```

### 6.2 匹配逻辑伪代码

```java
public class PrivilegeMatchChain {
    private List<PrivilegeMatcher> matchers; // 按优先级排序

    public MatchResult match(Long userId, Long dataId, Long entityId) {
        for (PrivilegeMatcher matcher : matchers) {
            MatchResult result = matcher.match(userId, dataId, entityId);
            if (result.isMatched()) {
                return result; // 短路返回，第一个匹配即生效
            }
        }
        return MatchResult.DENIED;
    }
}
```

## 十四、错误处理与容错

### 14.1 异常处理策略

| 场景 | 策略 | 说明 |
|------|------|------|
| MQ消费失败 | 重试3次 + 死信队列 | RocketMQ 自带重试机制 |
| 共享规则执行超时 | 异步重试 + 告警 | 超过60s的规则执行记录告警 |
| ES同步失败 | 重试 + 降级到SQL查询 | ES不可用时降级 |
| 分布式锁获取失败 | 等待重试 + 超时放弃 | 最多等待30s |
| 反规范化失败 | MQ重试 + 修复消费者 | DenormalizeRepairConsumer 专门处理修复 |
| 数据迁移中断 | 消息缓冲 + 回放 | MqBlockService 保证消息不丢失 |

### 15.1 架构演进方向

| 关联方式 | ID关联 | api_key关联（遵循元数据规范） |

### 15.2 核心重构点

1. **统一 Share 表**：将 13 张标准实体 share 表合并为 1 张统一表，通过 `entity_api_key` 区分，消除代码重复
2. **消除 Dubbo 依赖**：所有 RPC 接口迁移为 Feign Client
3. **Pipeline 简化**：反规范化 Pipeline 从注解驱动改为配置驱动，降低复杂度
4. **ES 索引优化**：V3 ES 索引增加 `entity_api_key` 字段，支持按实体过滤
5. **MQ Topic 精简**：合并冗余 Topic，减少消费者数量
6. **权限匹配链标准化**：MatchLink 改为 Spring Bean 自动注册，支持动态扩展

## 十六、总结

本技术设计方案完整覆盖了 aPaaS 数据权限系统的所有核心模块：

1. **数据权限模型**：7 种权限来源（rowCause）+ Share Table 模式 + 分表策略
2. **共享规则引擎**：2 种规则类型 × 7 种主体类型 × 10 种条件操作符，MQ 异步执行
3. **权限反规范化**：注解驱动的 Pipeline（Input→Transform→Processor→Operator），18 个 Transform 实现
4. **权限匹配链**：责任链模式，7 级优先级判定
5. **V3 权限体系**：9 种事件类型 × 40+ 处理器，ES 索引加速查询，13 种 CheckType 校验
6. **MQ 消息体系**：15 个 Producer + 10+ Consumer，多优先级消费 + 隔离 Topic
7. **数据迁移保障**：三级阻塞策略 + 消息缓冲 + 回放机制
8. **并发与性能**：分布式锁 + 批量操作 + 缓存策略 + ES 查询优化
9. **新系统重构**：统一 Share 表 + api_key 关联 + Spring Cloud 架构 + MyBatis-Plus
