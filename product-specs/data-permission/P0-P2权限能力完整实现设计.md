## P0-1: 数据修改触发规则重算

### 问题

`EntityDataService.update()` 中调用了 `sharingRuleEngine.matchAndApply()`，但：
1. 传入的 `ownerId`/`departId` 来自 `fixedUpdates`，可能是 null（只修改了扩展字段时）
2. 没有字段变更检测 — 不知道哪些字段变了
3. 没有清理旧 share — 数据不再满足条件时旧 share 残留

### 改造方案

```java
// EntityDataService.update() 改造
public boolean update(..., Map<String, Object> oldData, Set<String> changedApiKeys) {
    // ... 现有逻辑 ...

    // 4. 负责人变更 → 更新 owner share
    if (updated && fixedUpdates != null && fixedUpdates.getOwnerId() != null) {
        dataShareWriteService.updateOwnerShare(...);
    }

    // 5. 共享规则重算（完整版）
    if (updated) {
        // 5a. 获取完整的当前数据（合并 fixedUpdates + extColumns + oldData）
        Map<String, Object> currentFields = new HashMap<>();
        if (oldData != null) currentFields.putAll(oldData);
        if (extColumns != null) currentFields.putAll(extColumns);
        if (fixedUpdates != null && fixedUpdates.getName() != null)
            currentFields.put("name", fixedUpdates.getName());

        // 5b. 获取当前 ownerId 和 departId（优先用新值，否则用旧值）
        Long currentOwnerId = fixedUpdates != null && fixedUpdates.getOwnerId() != null
                ? fixedUpdates.getOwnerId()
                : (oldData != null ? toLong(oldData.get("owner_id")) : null);
        Long currentDepartId = fixedUpdates != null && fixedUpdates.getDepartId() != null
                ? fixedUpdates.getDepartId()
                : (oldData != null ? toLong(oldData.get("depart_id")) : null);

        // 5c. 清除该数据的所有规则 share，然后重新匹配
        dataShareWriteService.cleanRuleShareByData(tenantId, entityApiKey, id);
        sharingRuleEngine.matchAndApply(tenantId, entityApiKey, id,
                currentOwnerId, currentDepartId, currentFields, operatorId);
    }
}
```

### 新增方法

`DataShareWriteService.cleanRuleShareByData()` — 清除指定数据的所有规则 share（cause=5）

### 改造文件

| 文件 | 改动 |
|:---|:---|
| `EntityDataService.java` | update 方法中补全规则重算逻辑 |
| `DataShareWriteService.java` | 新增 `cleanRuleShareByData()` |
| `EntityDataApiService.java` | update 接口传入 oldData |

## P1-1: 异步权限处理

### 方案

使用 Spring `@Async` + `ThreadPoolTaskExecutor`，不引入 RocketMQ。

### 配置

```java
@Configuration
@EnableAsync
public class AsyncPermissionConfig {
    @Bean("permissionExecutor")
    public Executor permissionExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(2);
        executor.setMaxPoolSize(5);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("perm-async-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}
```

### 异步化的操作

| 操作 | 当前 | 改后 |
|:---|:---|:---|
| 共享规则匹配（create 时） | 同步 | **异步** |
| 共享规则重算（update 时） | 同步 | **异步** |
| 共享规则变更全量重算 | 不支持 | **异步** |
| 部门变更缓存清除 | 同步 | 保持同步（快速操作） |

### 新增服务

### 实现

```java
@Service
public class PermissionTraceService {
    public PermissionTraceResult trace(long tenantId, String entityApiKey,
                                        long dataId, Long userId) {
        List<TraceReason> reasons = new ArrayList<>();

        // 1. 负责人检查
        TenantData data = entityDataService.getById(tenantId, entityApiKey, dataId);
        if (data != null && userId.equals(data.getOwnerId())) {
            reasons.add(TraceReason.owner());
        }

        // 2. 部门检查
        Long userDeptId = platformUserService.getUserDepartId(tenantId, userId);
        DataPermission config = configService.get(tenantId, entityApiKey);
        if (config != null && config.getDefaultAccess() > 0 && userDeptId != null
                && userDeptId.equals(data.getDepartId())) {
            reasons.add(TraceReason.department(getDeptName(userDeptId)));
        }

        // 3. 层级检查
        if (config != null && config.getHierarchyAccess() > 0 && userDeptId != null) {
            List<Long> descendants = departmentService.getDescendantIds(tenantId, userDeptId);
            if (descendants.contains(data.getDepartId())) {
                reasons.add(TraceReason.hierarchy(getDeptName(data.getDepartId())));
            }
        }

        // 4. share 表检查（团队成员/共享规则/共享组/区域/手动共享）
        List<DataShare> shares = shareQueryService.getSharesForData(tenantId, entityApiKey, dataId, userId);
        for (DataShare share : shares) {
            reasons.add(TraceReason.fromShare(share));
        }

        return new PermissionTraceResult(!reasons.isEmpty(), reasons);
    }
}
```

### 数据模型

```sql

-- 职位-用户关联

    id              BIGINT       NOT NULL,
    tenant_id       BIGINT       NOT NULL,
    position_id     BIGINT       NOT NULL,
    user_id         BIGINT       NOT NULL,
    delete_flg      SMALLINT     NOT NULL DEFAULT 0,
    created_at      BIGINT,
    created_by      BIGINT,
    updated_at      BIGINT,
    updated_by      BIGINT,
    PRIMARY KEY (id)
);
CREATE UNIQUE INDEX uk_pu_tid_pid_uid ON p_position_user (tenant_id, position_id, user_id);
```

### 权限逻辑

职位权限的核心是"上级职位的用户可以看到下级职位用户的数据"。实现方式：

1. 用户 A 在职位 P1，用户 B 在职位 P2（P1 的下级）
2. 用户 B 创建数据 D，D 的 owner_id = B
3. 查询时，用户 A 的权限主体展开包含 P1 及其下级职位 P2 的所有用户
4. 通过 share 表或 depart_id 匹配实现

简化方案：职位权限通过"职位→用户→部门"的间接关系实现，不新增独立的权限维度。上级职位的用户自动获得下级职位用户所在部门的数据可见权限。
