# 新老项目权限体系对比分析

---

## 一、架构对比

| 维度 | 老项目 (paas-privilege-service) | 新项目 (paas-platform-service) |
|:---|:---|:---|
| 代码规模 | 740 个 Java 文件，~113K 行 | ~15 个权限相关文件，~2K 行 |
| 技术栈 | Spring Boot + Dubbo + Hibernate JPA + RocketMQ + ES + Redisson | Spring Boot + Spring Cloud + MyBatis-Plus + Redis |
| 服务拆分 | 独立微服务，30+ Dubbo 接口，16+ Feign Client | 内嵌在 platform-service 中，无独立服务 |
| 数据存储 | MySQL (xsy_privilege) + Elasticsearch + Redis | PostgreSQL (paas_db) + Redis |
| 消息队列 | RocketMQ，15+ Producer，20+ Consumer | 无 MQ（同步调用） |
| 权限查询 | 反规范化表 + ES 索引，O(1) 查询 | 实时查询 share 表 + depart_id IN，O(N) |

## 二、功能模块对比

| 功能模块 | 老项目 | 新项目 | 差距 |
|:---|:---:|:---:|:---|
| **共享规则引擎** | ✅ 完整（~154K 行） | ✅ 基础（~800 行） | 老项目支持条件匹配、MQ 异步重算、批量处理；新项目仅同步匹配 |
| **权限反规范化** | ✅ Pipeline 模式（18 个 Transform） | ❌ 无 | 老项目预计算权限写入反规范化表；新项目实时查询 |
| **权限匹配链** | ✅ MatchLink 链式模式（~10K 行） | ❌ 无 | 老项目链式判断多维度权限；新项目简单 SQL 条件 |
| **V3 权限体系** | ✅ ES 索引 + 40+ 事件处理器 | ❌ 无 | 老项目用 ES 加速大数据量权限查询 |
| **部门权限** | ✅ 完整（含社交部门、维度部门） | ✅ 基础（同部门 + 下级） | 老项目支持多维度部门；新项目仅支持单一部门树 |
| **职位权限** | ✅ 完整（职位树 + 数据版本，~16K 行） | ❌ 无 | 老项目通过职位树继承权限 |
| **团队成员** | ✅ 完整（多优先级 MQ，~14K 行） | ❌ 无 | 老项目支持团队成员读/写/删除权限 |
| **区域权限** | ✅ 完整（Territory 层级，~12K 行） | ❌ 占位（TODO） | 老项目支持销售区域层级权限 |
| **共享组** | ✅ 完整（公共组 + 内部组 + 授权组） | ✅ 基础（仅公共组成员） | 老项目支持 3 种组类型；新项目仅 1 种 |
| **手动共享** | ✅ 完整（对象级共享/取消，~79K 行） | ✅ 基础（仅写入 share） | 老项目支持共享给用户/部门/职位/区域 |
| **权限分析** | ✅ 一致性/完整性/溯源分析 | ❌ 无 | 老项目可追溯"为什么用户能看到这条数据" |
| **MQ 异步处理** | ✅ 15+ Topic，多优先级消费 | ❌ 无 | 老项目异步处理大规模权限变更 |
| **租户隔离** | ✅ 隔离 Topic + 分布式锁 | ❌ 无 | 老项目大租户独立消费队列 |
| **数据权限配置** | ✅ 完整（OWD 模型） | ✅ 基础（defaultAccess + hierarchyAccess） | 基本对齐 |
| **admin 管理界面** | ✅ 完整（xsy-paas-ui-webadmin） | ✅ 基础（all.html 管理后台） | 新项目 UI 更现代但功能少 |

## 五、数据存储对比

### 5.1 Share 表

| 维度 | 老项目 | 新项目 |
|:---|:---|:---|
| 表结构 | 每个标准对象独立 share 表（13 张）+ 自定义对象共用表 | 统一 `p_data_share` 表，通过 entity_api_key 区分 |
| 分表策略 | 按对象类型分表 | 按 (tenant_id, entity_api_key) 哈希分片（1000 张） |
| 关联方式 | ID 关联（subject_id: Long） | apiKey 关联（subject_api_key: String） |
| 字段 | dataId, groupId, groupType, accessLevel, rowCause, scope, newGroupId | dataId, subjectApiKey, subjectType, accessLevel, shareCause, ruleApiKey |

### 5.2 反规范化表（老项目独有）

| 表 | 用途 |
|:---|:---|
| `p_priv_group_hier_dn` | 组层级反规范化（组→关联组→对象→规则→访问级别） |
| `p_priv_user_hier_dn` | 用户层级反规范化（用户→关联用户→原因） |
| `p_priv_group_user_dn` | 组-用户反规范化 |
| `p_priv_subject_subject` | 主体-主体关系 |
| `v3_p_priv_subject_share_data` | V3 主体共享数据（dataId→subjectId→权限） |

### 5.3 ES 索引（老项目独有）

```json
{
  "index": "p_data_share_index",
  "fields": ["tenantId", "entityApiKey", "dataId", "subjectId", "subjectType", "shareCause", "accessLevel"]
}
```

## 六、权限查询路径对比

### 老项目（三层查询）

```
用户请求
  → MatchLink 链式匹配
    → 1. 快速路径：管理员/全员可见 → 跳过
    → 2. 负责人匹配（owner_id）
    → 3. 反规范化表查询（p_priv_group_hier_dn + p_priv_user_hier_dn）
    → 4. V3 ES 查询（大数据量场景）
    → 5. share 表查询（兜底）
  → 返回权限级别（无/只读/读写/删除）
```

### 新项目（两阶段查询）

```
用户请求
  → DataPermissionFilter.buildCondition（阶段 1，分表上下文外）
    → 1. 快速路径：skipAll/内部调用/全员可见 → 跳过
    → 2. 展开用户权限主体（自己 + 部门 + 下级部门 + 共享组）
    → 3. 查 share 表可见数据 ID
    → 返回 PermissionCondition
  → PermissionCondition.applyTo（阶段 2，分表上下文内）
    → WHERE owner_id=? OR depart_id IN (?) OR id IN (share_ids)
```

## 七、性能对比

| 场景 | 老项目 | 新项目 | 差距 |
|:---|:---|:---|:---|
| 列表查询（小数据量） | ~5ms（反规范化表） | ~10-50ms（实时查 share 表） | 新项目慢 2-10x |
| 列表查询（大数据量 >10K） | ~10ms（ES 索引） | 降级（share 表 LIMIT 10000） | 新项目无法支撑 |
| 单条权限校验 | ~1ms（反规范化表） | ~5-20ms（查 share 表） | 新项目慢 5-20x |
| 权限变更生效 | 秒级（MQ 异步） | 即时（同步写入） | 新项目更快但阻塞主流程 |
| 大规模权限重算 | 分钟级（MQ 批量） | 不支持 | 新项目无批量重算能力 |
| 缓存策略 | Redis + Caffeine 二级缓存 | Redis 单级缓存（TTL 5-10 分钟） | 老项目缓存更精细 |

## 八、新项目缺失的关键场景

### 8.1 必须补齐（影响核心业务）

| 场景 | 影响 | 优先级 |
|:---|:---|:---:|
| 团队成员权限 | 销售协作场景无法支持 | P0 |
| 区域权限 | 销售区域管理无法支持 | P1 |
| 数据修改触发规则重算 | 字段变更后权限不更新 | P1 |
| 权限变更 MQ 异步化 | 大数据量场景阻塞主流程 | P1 |

### 8.2 建议补齐（提升体验）

| 场景 | 影响 | 优先级 |
|:---|:---|:---:|
| 职位权限 | 组织架构权限继承 | P2 |
| 权限分析/溯源 | 管理员无法排查权限问题 | P2 |
| ES 加速查询 | 大数据量场景性能 | P2 |
| 反规范化预计算 | 查询性能优化 | P3 |
| 助理代理权限 | 管理层代理场景 | P3 |
| 授权组/内部组 | 精细化组权限 | P3 |

### 8.3 可暂不实现

| 场景 | 原因 |
|:---|:---|
| 租户隔离 Topic | 当前租户规模不需要 |
| 多优先级 MQ 消费 | 当前数据量不需要 |
| 虚拟部门 | 新架构可通过共享组替代 |
| 社交部门/维度部门 | 新架构暂不需要 |

## 九、新项目的优势

| 优势 | 说明 |
|:---|:---|
| 架构简洁 | 15 个文件 vs 740 个文件，维护成本低 10x |
| 元数据驱动 | 权限配置通过元数据体系管理，与业务对象统一 |
| apiKey 关联 | 禁止 ID 关联，跨环境迁移一致性 |
| 统一 share 表 | 不再按对象类型分表，简化运维 |
| 上下文隔离 | PermissionCondition 不可变值对象，避免 ThreadLocal 污染 |
| 现代前端 | React 19 + Antd 6 + Vite 8，可编辑权限配置 |
| 实时生效 | 同步写入 share，无 MQ 延迟 |
| defaultAccess × hierarchyAccess 正交 | 比老项目的 OWD 模型更清晰 |

## 十、迁移建议

```
Phase 1（当前已完成）：
  ✅ 基础权限框架（share 表 + 权限过滤 + 缓存）
  ✅ 共享规则引擎（基于负责人 + 基于条件）
  ✅ 共享组（公共组成员管理）
  ✅ 部门权限（同部门 + 下级部门，正交配置）
  ✅ 数据权限配置（admin 可编辑）
  ✅ 部门 CRUD + 权限刷新

Phase 2（建议下一步）：
  → 团队成员权限（share_cause=1）
  → 区域权限（Territory 层级）
  → 数据修改触发规则重算（字段变更检测）
  → 权限变更 MQ 异步化

Phase 3（后续优化）：
  → 职位权限
  → 权限分析/溯源
  → ES 加速查询层
  → 反规范化预计算
```
