# 数据权限全局配置生效分析报告

> 分析 all.html（管理后台）中权限配置能否全局生效，以及前后端链路的断点

---

## 一、all.html 管理后台架构

`all.html` 是 aPaaS 管理后台的独立入口，通过 `admin-entry.tsx` → `AdminConsolePage` 渲染。

侧边栏"数据安全"分组包含 3 个权限管理模块：

| 菜单 | 组件 | 功能 | 状态 |
|:---|:---|:---|:---:|
| 共享规则 | `SharingRulesView` | 按实体查看/管理共享规则 + 条件 + 自动验证 | ⚠ 部分生效 |
| 公共组管理 | `PublicGroupsView` | 公共组列表 + 成员管理 + 验证 | ⚠ 部分生效 |
| 区域数据权限 | `PlaceholderView` | 占位页面 | ❌ 未实现 |

另外"组织与权限"分组中的 `RoleAuthorizationView`（角色权限配置）管理对象级 CRUD 权限和 `viewAll/modifyAll` 标记。

## 二、前后端 API 路径对照（核心断点分析）

### 2.1 前端 API 调用路径（permission.ts）

| 前端调用 | HTTP 方法 | URL | 用途 |
|:---|:---|:---|:---|
| `getDataPermission` | GET | `/permission/data-permission?entityApiKey=` | 查询实体权限配置 |
| `updateDataPermission` | POST | `/permission/data-permission` | 更新权限配置 |
| `listSharingRules` | GET | `/permission/sharing-rules?entityApiKey=` | 查询共享规则列表 |
| `listSharingRuleConditions` | GET | `/permission/sharing-rule-conditions?entityApiKey=&ruleApiKey=` | 查询规则条件 |
| `saveSharingRule` | POST | `/permission/sharing-rules` | 创建/更新规则 |
| `deleteSharingRule` | DELETE | `/permission/sharing-rules?entityApiKey=&apiKey=` | 删除规则 |
| `listPublicGroups` | GET | `/permission/public-groups` | 查询公共组列表 |
| `listPublicGroupMembers` | GET | `/permission/public-groups/members?groupId=` | 查询组成员 |
| `listRolePermissions` | GET | `/auth/roles/permissions?roleId=` | 查询角色权限 |
| `validateDataPermission` | GET | `/permission/validate?entityApiKey=` | 验证权限完整性 |

### 2.2 后端 API 实际路径

| 后端 Controller | RequestMapping | 实际路径 |
|:---|:---|:---|
| `SharingRuleApiService` | `/sharing-rule` | `/sharing-rule/{entityApiKey}` |
| `PublicGroupApiService` | `/public-group` | `/public-group/{groupApiKey}/members` |
| `EntityDataApiService` | `/entity/data` | `/entity/data/{entityApiKey}` |
| 无 | — | `/permission/data-permission` |
| 无 | — | `/permission/sharing-rules` |
| 无 | — | `/permission/public-groups` |
| 无 | — | `/permission/validate` |

### 2.3 🔴 关键断点：前后端 API 路径完全不匹配

| 问题 | 前端期望 | 后端实际 | 影响 |
|:---|:---|:---|:---|
| 共享规则列表 | `GET /permission/sharing-rules?entityApiKey=account` | `GET /sharing-rule/account` | ❌ 404，降级为 Mock 数据 |
| 规则条件 | `GET /permission/sharing-rule-conditions?entityApiKey=&ruleApiKey=` | `GET /sharing-rule/account/sr_xxx/conditions` | ❌ 404，降级为 Mock |
| 公共组列表 | `GET /permission/public-groups` | `GET /public-group/{groupApiKey}/members` | ❌ 404，降级为 Mock |
| 数据权限配置 | `GET /permission/data-permission?entityApiKey=` | 无对应 Controller | ❌ 404，降级为 Fallback 默认值 |
| 权限验证 | `GET /permission/validate?entityApiKey=` | 无对应 Controller | ❌ 404，返回空 |

**结论：前端所有 `/permission/*` 路径的请求全部 404，当前全部降级为前端内置的 Mock/Fallback 数据。管理后台看到的数据不是真实后端数据。**

## 三、前端 Fallback 机制分析

前端 `permission.ts` 对每个 API 都设计了 try-catch + fallback：

```typescript
// 典型模式
export async function listSharingRules(entityApiKey: string): Promise<SharingRule[]> {
  try {
    const { data } = await api.get('/permission/sharing-rules', { params: { entityApiKey } });
    if (Array.isArray(result) && result.length > 0) return result;
  } catch { /* endpoint may not exist */ }
  return MOCK_SHARING_RULES[entityApiKey] ?? [];  // ← 降级为 Mock
}
```

这意味着：
- 管理后台**能正常渲染**（不会白屏）
- 但展示的是**硬编码的 Mock 数据**，不是数据库中的真实配置
- 在管理后台做的任何修改操作（创建规则、添加成员）**不会持久化**
- 验证结果基于 Mock 数据，**不反映真实系统状态**

## 四、即使 API 对齐后，全局生效的链路分析

假设前后端 API 路径对齐，配置能否全局生效？

### 4.1 共享规则配置 → 全局生效链路

```
管理后台配置共享规则
  ↓ POST /sharing-rule/{entityApiKey}
SharingRuleService.create()
  ↓ 写入 p_tenant_sharing_rule（Tenant 级元数据）
  ↓
数据创建时自动触发
  EntityDataService.create()
    ↓ 步骤 5
  SharingRuleEngine.matchAndApply()
    ↓ mergeReadService.listMergedByEntityApiKey("sharingRule", entityApiKey)
    ↓ 读取 Common + Tenant 合并后的规则列表 ← ✅ 能读到新配置的规则
    ↓ 匹配成功 → DataShareWriteService.writeRuleShare()
    ↓ 写入 p_data_share 表
  ↓
查询时权限过滤
  DataPermissionFilter.buildCondition()
    ↓ DataShareQueryService.getVisibleDataIds()
    ↓ 查 p_data_share 表 ← ✅ 能查到规则写入的 share 记录
    ↓ PermissionCondition.applyTo(qw)
    ↓ WHERE ... OR id IN (visibleDataIds)
```

**结论：共享规则配置 → 全局生效链路是通的。** 前提是 API 路径对齐。

### 4.2 公共组成员配置 → 全局生效链路

```
管理后台添加公共组成员
  ↓ POST /public-group/{groupApiKey}/members
PublicGroupManageService.addMember()
  ↓ 写入 p_public_group_member 表
  ↓ 清除 Redis 缓存（user_pg: / pg_members: / user_subjects:）
  ↓
查询时权限展开
  UserSubjectService.getUserSubjects(tenantId, userId)
    ↓ PublicGroupMemberService.getUserGroupApiKeys()
    ↓ 查 p_public_group_member ← ✅ 能查到新添加的成员
    ↓ 返回 UserSubject.ofPublicGroup(pgApiKey)
  ↓
  DataShareQueryService.getVisibleDataIds(subjects)
    ↓ WHERE subject_type=2 AND subject_api_key IN ('pgApiKey') ← ✅ 能匹配
```

**结论：公共组成员配置 → 全局生效链路是通的。** 缓存清除机制已实现。

### 4.3 数据权限配置（DataPermission）→ 全局生效链路

```
管理后台修改 defaultAccess
  ↓ POST /permission/data-permission
  ↓ ❌ 无对应 Controller！
```

**结论：DataPermission 配置修改无法生效。** 缺少：
1. 后端 DataPermission CRUD Controller
2. `DataPermissionConfigService.evictCache()` 的触发入口

### 4.4 角色权限（viewAll/modifyAll）→ 全局生效链路

```
管理后台配置角色的 viewAll=1
  ↓ POST /auth/roles/permissions
  ↓ ❌ 后端 hasFullDataAccess() 始终返回 false（TODO 状态）
```

**结论：角色的全量数据访问权限无法生效。** `DataPermissionFilter.hasFullDataAccess()` 是硬编码 `return false`。

## 五、断点汇总

| # | 断点 | 严重程度 | 影响范围 | 修复方案 |
|:---:|:---|:---:|:---|:---|
| 1 | 前后端 API 路径不匹配（`/permission/*` vs `/sharing-rule/*`、`/public-group/*`） | 🔴 严重 | 所有权限管理功能 | 统一 API 路径，或前端适配后端路径 |
| 2 | 缺少 DataPermission CRUD Controller | 🔴 严重 | 数据权限配置无法修改 | 新增 `/permission/data-permission` 接口 |
| 3 | `hasFullDataAccess()` 硬编码 `return false` | 🟡 中等 | 管理员/总监无法看到全部数据 | 接入角色服务，读取 viewAll 标记 |
| 4 | 前端公共组数据模型不匹配（前端用 `groupId: number`，后端用 `groupApiKey: string`） | 🟡 中等 | 公共组成员查询 | 统一为 apiKey 关联 |
| 5 | 前端共享规则 CRUD 操作（创建/删除）的请求格式与后端不匹配 | 🟡 中等 | 规则管理操作 | 对齐请求体格式 |
| 6 | 区域数据权限完全未实现 | 🟠 低 | 区域权限 | 后续迭代 |
| 7 | 数据更新时共享规则重算依赖 oldData 传入，当前 update 调用链未传 | 🟡 中等 | 字段变更触发条件规则 | 在 update 前查询 oldData |

## 七、修复优先级建议

### P0 — 打通前后端 API（解决断点 #1）✅ 已修复

新增 `PermissionApiService.java`（`/permission` 前缀），统一对齐前端所有 API 路径：
- `GET /permission/data-permission` → DataPermissionConfigService
- `POST /permission/data-permission` → IMetadataMergeWriteService + 缓存清除
- `GET /permission/sharing-rules` → SharingRuleService.listByEntity
- `GET /permission/sharing-rule-conditions` → SharingRuleService.listConditions
- `POST /permission/sharing-rules` → SharingRuleService.create/update
- `DELETE /permission/sharing-rules` → SharingRuleService.delete
- `GET /permission/public-groups` → PublicGroupManageService 聚合
- `GET /permission/public-groups/members` → PublicGroupManageService.listMembers
- `GET /permission/validate` → 后端验证逻辑
- 前端零改动

### P1 — 补全 DataPermission CRUD ✅ 已修复

`PermissionApiService` 中已包含 `POST /permission/data-permission` 接口，
更新后自动调用 `DataPermissionConfigService.evictCache()` 清除缓存。

### P2 — 接入角色服务

- `hasFullDataAccess()` 读取角色的 `viewAll` 标记
- 需要 `RoleService` 或 Feign 调用角色服务

---

## 八、结论

**当前 all.html 管理后台的权限配置无法全局生效。** 根本原因是前后端 API 路径不匹配，前端所有 `/permission/*` 请求全部 404 后降级为 Mock 数据。后端的共享规则引擎、公共组成员展开、权限过滤等核心逻辑已实现且链路完整，但管理后台的配置操作无法到达后端。

打通 API 路径后，共享规则和公共组的"配置→执行→查询过滤"全链路可以全局生效。DataPermission 配置修改和管理员全量访问权限还需要补充对应的 Controller 和服务接入。
