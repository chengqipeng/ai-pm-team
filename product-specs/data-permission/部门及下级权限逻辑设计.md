# 部门及下级权限逻辑设计

> 参照 admin 后台现有模式（RoleAuthorizationView 矩阵 + SharingRulesView 配置卡片），设计部门层级访问的完整逻辑

---

## 一、现状问题

### 1.1 当前实现的缺陷

| 组件 | 问题 |
|:---|:---|
| `UserSubjectService.buildSubjects` | **无条件展开所有下级部门**，不区分 `hierarchyAccess` 配置 |
| `PermissionCondition.applyTo` | 只检查 `defaultAccess > 0` 决定是否加部门过滤，**完全忽略 `hierarchyAccess`** |
| `DataPermission.hierarchyAccess` | 字段存在但从未被读取，形同虚设 |
| admin 前端 | SharingRulesView 中以 Tag 展示 `hierarchyAccess`，但**不可编辑** |

### 1.2 导致的行为

- `defaultAccess=0`（私有）+ `hierarchyAccess=1`（只读）→ 上级**看不到**下级数据（因为 `defaultAccess=0` 时不加部门条件）
- `defaultAccess=1`（只读）+ `hierarchyAccess=0`（无）→ 上级**能看到**下级数据（因为 `buildSubjects` 无条件展开了下级部门）
- 两种情况都与配置语义不符

## 二、目标行为定义

### 2.1 权限矩阵（参照 Salesforce OWD 模型）

| defaultAccess | hierarchyAccess | 用户能看到的数据 |
|:---:|:---:|:---|
| 0（私有） | 0（无） | 仅自己负责的 + share 表授权的 |
| 0（私有） | 1（只读） | 仅自己负责的 + **下级部门**数据（只读）+ share 表授权的 |
| 0（私有） | 2（读写） | 仅自己负责的 + **下级部门**数据（读写）+ share 表授权的 |
| 1（只读） | 0（无） | 自己负责的 + **同部门**数据 |
| 1（只读） | 1（只读） | 自己负责的 + **同部门**数据 + **下级部门**数据 |
| 1（只读） | 2（读写） | 自己负责的 + **同部门**数据 + **下级部门**数据（读写） |
| 2（读写） | * | 全员可见，跳过过滤 |

关键规则：
- `defaultAccess` 控制**同部门**可见性
- `hierarchyAccess` 控制**下级部门**可见性（独立于 `defaultAccess`）
- 两者是**正交**的，可以独立配置
- `hierarchyAccess` 只影响**向下**看（上级看下级），不影响**向上**看（下级看上级）

### 2.2 SQL 生成示例

用户张伟（销售部 1001），销售部下有华东销售组（1007）和华南销售组（1008）：

```sql
-- defaultAccess=0, hierarchyAccess=0（纯私有）
WHERE owner_id = 张伟 OR id IN (share表授权)

-- defaultAccess=0, hierarchyAccess=1（私有+层级只读）
WHERE owner_id = 张伟 OR depart_api_key IN ('华东销售组', '华南销售组') OR id IN (share表授权)
-- 注意：不含 '销售部'（同部门），因为 defaultAccess=0

-- defaultAccess=1, hierarchyAccess=0（部门可见，无层级）
WHERE owner_id = 张伟 OR depart_api_key IN ('销售部') OR id IN (share表授权)
-- 注意：只含 '销售部'（同部门），不含下级

-- defaultAccess=1, hierarchyAccess=1（部门可见+层级只读）
WHERE owner_id = 张伟 OR depart_api_key IN ('销售部', '华东销售组', '华南销售组') OR id IN (share表授权)
-- 含同部门 + 所有下级部门
```

## 三、后端改造设计

### 3.1 UserSubjectService.buildSubjects 改造

当前：无条件展开所有下级部门
改后：只展开用户自己的部门，下级部门的展开交给 `PermissionCondition`

```java
private List<UserSubject> buildSubjects(long tenantId, Long userId) {
    List<UserSubject> subjects = new ArrayList<>();
    subjects.add(UserSubject.ofUser(userId, String.valueOf(userId)));

    try {
        Long deptId = platformUserService.getUserDepartId(tenantId, userId);
        if (deptId != null) {
            // 用户自己的部门（始终加入，用于 defaultAccess 同部门过滤）
            subjects.add(UserSubject.ofDepart(deptId, String.valueOf(deptId)));
        }
    } catch (Exception e) { ... }

    // 共享组
    try { ... } catch (Exception e) { ... }

    return subjects;
    // 注意：不再在这里展开下级部门
    // 下级部门的展开由 DataPermissionFilter.buildCondition 根据 hierarchyAccess 决定
}
```

### 3.2 DataPermissionFilter.buildCondition 改造

新增：根据 `hierarchyAccess` 展开下级部门，作为额外的 subject 传入 `PermissionCondition`

```java
public PermissionCondition buildCondition(long tenantId, String entityApiKey, Long userId) {
    // ... 快速路径短路 ...

    DataPermission config = configService.get(tenantId, entityApiKey);
    // ... defaultAccess=2 跳过 ...

    List<UserSubject> subjects = subjectService.getUserSubjects(tenantId, userId);

    // 新增：根据 hierarchyAccess 展开下级部门
    if (config.getHierarchyAccess() != null && config.getHierarchyAccess() > 0) {
        Long userDeptId = platformUserService.getUserDepartId(tenantId, userId);
        if (userDeptId != null) {
            List<Long> descendantIds = departmentService.getDescendantIds(tenantId, userDeptId);
            List<UserSubject> expanded = new ArrayList<>(subjects);
            for (Long childDeptId : descendantIds) {
                // 避免重复添加
                if (expanded.stream().noneMatch(s ->
                    s.getType() == UserSubject.SubjectType.DEPART &&
                    s.getSubjectId() != null && s.getSubjectId().equals(childDeptId))) {
                    expanded.add(UserSubject.ofDepart(childDeptId, String.valueOf(childDeptId)));
                }
            }
            subjects = expanded;
        }
    }

    List<Long> visibleDataIds = shareQueryService.getVisibleDataIds(...);
    return PermissionCondition.filterWith(userId, config, subjects, visibleDataIds);
}
```

### 3.3 PermissionCondition.applyTo 改造

当前：`defaultAccess > 0` 时加所有部门 subject
改后：分别处理同部门和下级部门

```java
public void applyTo(QueryWrapper<?> qw) {
    if (skip) return;

    qw.and(wrapper -> {
        // 条件 1：用户是负责人
        wrapper.eq("owner_id", userId);

        // 条件 2：同部门可见（defaultAccess > 0 时）
        // 条件 3：下级部门可见（hierarchyAccess > 0 时，下级部门已在 buildCondition 中展开到 subjects）
        // 两者合并：只要 defaultAccess > 0 或 hierarchyAccess > 0，就用 depart_api_key IN (...)
        if (subjects != null) {
            boolean needDeptFilter = (config.getDefaultAccess() != null && config.getDefaultAccess() > 0)
                    || (config.getHierarchyAccess() != null && config.getHierarchyAccess() > 0);
            if (needDeptFilter) {
                List<String> deptApiKeys = subjects.stream()
                        .filter(s -> s.getType() == UserSubject.SubjectType.DEPART)
                        .map(UserSubject::getSubjectApiKey)
                        .filter(id -> id != null)
                        .collect(Collectors.toList());
                if (!deptApiKeys.isEmpty()) {
                    wrapper.or().in("depart_api_key", deptApiKeys);
                }
            }
        }

        // 条件 4：share 表中的可见数据
        if (visibleDataIds != null && !visibleDataIds.isEmpty()) {
            wrapper.or().in("id", visibleDataIds);
        }
    });
}
```

### 3.4 关键设计决策

| 决策 | 选择 | 理由 |
|:---|:---|:---|
| 下级部门展开在哪里做？ | `DataPermissionFilter.buildCondition` | 需要读取 entity 级的 `hierarchyAccess` 配置，`buildSubjects` 是通用的不知道具体 entity |
| subjects 中如何区分同部门和下级部门？ | 不区分，统一用 `DEPART` 类型 | `applyTo` 中用 `depart_api_key IN (...)` 统一过滤，无需区分来源 |
| `defaultAccess=0` + `hierarchyAccess=1` 时同部门是否可见？ | 不可见 | `defaultAccess=0` 表示私有，同部门不可见；`hierarchyAccess=1` 只控制下级 |
| 缓存策略 | `user_subjects` 缓存不含下级部门，下级部门在 `buildCondition` 中实时展开 | 不同 entity 的 `hierarchyAccess` 可能不同，缓存不能包含 entity 级配置 |

### 4.1 admin 后台 — 数据权限配置编辑

参照 `RoleAuthorizationView` 的矩阵模式，在 `SharingRulesView` 的"数据权限配置"卡片中，将只读 Tag 改为可编辑的下拉选择器：

```
┌─────────────────────────────────────────────────────────────┐
│ 数据权限配置                                    [保存] [重置] │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  默认访问级别    [私有 ▾]     ← 下拉：私有/只读/读写          │
│  层级访问       [只读 ▾]     ← 下拉：无/只读/读写            │
│  负责人权限      [读写 ▾]     ← 下拉：只读/读写              │
│  团队成员权限    [无   ▾]     ← 下拉：无/只读/读写            │
│  允许手动共享    [✓]          ← 开关                         │
│  允许共享规则    [✓]          ← 开关                         │
│                                                             │

│  │ 共享规则：已启用                                          │ │
│  └──────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 权限效果预览逻辑

根据 `defaultAccess` 和 `hierarchyAccess` 的组合，生成人类可读的权限效果描述：

```typescript
function describePermEffect(dp: DataPermissionConfig): string[] {
  const effects: string[] = [];

  // 负责人
  effects.push(`负责人：${dp.ownerAccess === 2 ? '可读写' : '只读'}自己的数据`);

  // 同部门
  if (dp.defaultAccess === 0) effects.push('同部门：私有（不可见）');
  else if (dp.defaultAccess === 1) effects.push('同部门：只读');
  else effects.push('同部门：可读写（全员可见）');

  // 下级部门
  if (dp.hierarchyAccess === 0) effects.push('下级部门：不可见');
  else if (dp.hierarchyAccess === 1) effects.push('下级部门：只读');
  else effects.push('下级部门：可读写');

  // 共享
  if (dp.sharingRuleFlg) effects.push('共享规则：已启用');
  if (dp.sharingFlg) effects.push('手动共享：已启用');

  return effects;
}
```

### 4.3 前端 API 调用

使用已有的 `POST /permission/data-permission` 接口更新配置：

```typescript
await updateDataPermission({
  entityApiKey: 'opportunity',
  defaultAccess: 0,      // 私有
  hierarchyAccess: 1,    // 下级只读
  ownerAccess: 2,        // 负责人读写
  teamAccess: 0,         // 团队无
  sharingFlg: 1,         // 允许手动共享
  sharingRuleFlg: 1,     // 允许共享规则
});
```
