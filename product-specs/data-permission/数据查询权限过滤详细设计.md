# 数据查询权限过滤详细设计

> 基于现有 EntityDataService 查询链路，设计权限过滤的注入方案

---

## 一、现有查询链路分析

### 1.1 当前调用链

```
EntityDataApiService (Controller)
  │
  ├─ GET /{entityApiKey}          → listData()
  │     └─ EntityDataService.listPage(tenantId, entityApiKey, selectColumns, conditions, orderColumn, asc, page, size)
  │           └─ routeService.executeWithRoute(tenantId, entityApiKey, () -> {
  │                 QueryWrapper<TenantData> qw = new QueryWrapper<>();
  │                 qw.eq("entity_api_key", entityApiKey).eq("delete_flg", 0);
  │                 qw.条件/排序/分页...
  │                 dataDao.pageMaps(pageParam, qw);
  │               })
  │
  ├─ GET /{entityApiKey}/{id}     → getData()
  │     └─ EntityDataService.getById(tenantId, entityApiKey, id)
  │           └─ routeService.executeWithRoute(tenantId, entityApiKey,
  │                 () -> dataDao.getByIdAndTenant(id, tenantId))
  │
  └─ POST /batch-save             → batchSave()
        └─ EntityDataBatchService.batchSave(...)
              └─ batchGetByIds(tenantId, entityApiKey, ids)
                    └─ routeService.executeWithRoute(tenantId, entityApiKey, () -> {
                          QueryWrapper qw = ...;
                          dataDao.listMaps(qw);
                        })
```

### 1.2 当前问题

- 所有查询**无任何权限过滤**，任何用户可以看到同租户下所有数据
- `listPage()` 的 `conditions` 参数当前传 null，未被使用
- `getById()` 仅校验 tenantId，不校验用户是否有权查看该条数据
- API 层未传递 userId，Service 层无法获取当前操作用户

### 1.3 关键约束

- 查询走 `DynamicTableNameHolder` 路由到分片表 `p_tenant_data_{N}`
- share 表 `p_data_share_{M}` 路由到不同的分片表
- 两张表在不同的物理表中，**不能直接 JOIN**
- 必须采用"先查权限，再查数据"或"子查询"的方式

### 1.4 DynamicTableNameHolder 上下文污染问题（BUG 根因）

> **已发现线上 BUG**：`DataPermissionFilter.applyListFilter()` 在 `DynamicTableNameHolder.executeWithRoute()` 的 lambda 内部被调用时，
> 其内部的元数据查询（`DataPermissionConfigService.get()` → `MetadataMergeReadServiceImpl.getMetaModel()` → `MetaServiceImpl.getByApiKey()`）
> 会被 `DynamicTableNameInterceptor` 拦截，将元数据表名替换为当前 ThreadLocal 中的业务分片表名 `p_tenant_data_0`，导致：
>
> ```
> PSQLException: ERROR: relation "p_tenant_data_0" does not exist
> SQL: SELECT ... FROM p_tenant_data_0 WHERE delete_flg = 0 AND (api_key = ?) AND tenant_id = 292193
> ```
>
> **根因**：`DynamicTableNameInterceptor` 无差别替换所有经过 MyBatis 的 SQL 表名，元数据查询被嵌套在业务数据分表路由的 ThreadLocal 上下文中执行。
>
> **修复方案**：采用"上下文隔离"策略 —— 将所有元数据查询和权限预判移到 `executeWithRoute()` 之外，
> 在分表路由上下文建立之前完成权限条件的构建，分表上下文内只做纯条件注入，不再触发任何元数据查询。

---

## 二、设计方案

### 2.1 核心思路

采用**上下文隔离的两阶段查询**模式：

```
阶段 1：权限条件预构建（在 DynamicTableNameHolder 上下文之外）
  ├─ 管理员？ → 返回 SKIP 条件
  ├─ defaultAccess=2（全员读写）？ → 返回 SKIP 条件
  ├─ 查元数据配置（configService.get）  ← 此时无 ThreadLocal 污染
  ├─ 展开用户权限主体（subjectService）
  ├─ 查 share 表可见数据（shareQueryService，使用独立的分表路由上下文）
  └─ 返回 PermissionCondition 值对象（不可变，线程安全）

阶段 2：权限条件注入（在 DynamicTableNameHolder 上下文之内）
  └─ permissionCondition.applyTo(qw)  ← 纯内存操作，不触发任何 DB 查询
```

> 关键设计决策：阶段 1 的所有 DB 查询（元数据、share 表）都在分表路由上下文建立之前完成，
> 阶段 2 只做纯内存的 QueryWrapper 条件拼装，彻底避免 ThreadLocal 上下文污染。

### 2.2 架构分层

```
EntityDataApiService (Controller)
  │  ↓ 传递 userId
EntityDataService (Service)
  │
  │  ═══ 阶段 1：分表上下文之外 ═══
  │  ↓ 调用权限服务
DataPermissionFilter.buildCondition(tenantId, entityApiKey, userId)
  │  ├─ DataPermissionConfigService (读取 dataPermission 元数据)  ← 无 ThreadLocal 污染
  │  ├─ UserSubjectService (展开用户权限主体)
  │  └─ DataShareQueryService (查询 share 表，使用独立路由上下文)
  │  ↓ 返回 PermissionCondition（不可变值对象）
  │
  │  ═══ 阶段 2：分表上下文之内 ═══
  │  ↓ routeService.executeWithRoute(tenantId, entityApiKey, () -> {
  │       permissionCondition.applyTo(qw);  ← 纯内存操作
  │       dataDao.pageMaps(pageParam, qw);
  │     })
```

## 三、新增组件设计

### 3.1 PermissionCondition — 权限条件值对象（新增）

```java
/**
 * 权限过滤条件的不可变值对象。
 * 在 DynamicTableNameHolder 上下文之外构建，在上下文之内注入到 QueryWrapper。
 *
 * 设计要点：
 * - 不可变（immutable）：构建后不可修改，天然线程安全
 * - 无 DB 依赖：applyTo() 是纯内存操作，不触发任何数据库查询
 * - 可序列化：便于日志记录和调试
 */
public final class PermissionCondition {

    /** 是否跳过权限过滤（管理员/全员可见/内部调用） */
    private final boolean skip;
    /** 当前用户 ID（负责人判断） */
    private final Long userId;
    /** 权限配置 */
    private final DataPermission config;
    /** 用户权限主体列表（部门/公共组/区域） */
    private final List<UserSubject> subjects;
    /** share 表中可见的 data_id 集合（已预查询） */
    private final List<Long> visibleDataIds;
    /** 跳过原因（用于日志） */
    private final String skipReason;

    private PermissionCondition(boolean skip, Long userId, DataPermission config,
                                 List<UserSubject> subjects, List<Long> visibleDataIds,
                                 String skipReason) {
        this.skip = skip;
        this.userId = userId;
        this.config = config;
        this.subjects = subjects != null ? Collections.unmodifiableList(subjects) : null;
        this.visibleDataIds = visibleDataIds != null ? Collections.unmodifiableList(visibleDataIds) : null;
        this.skipReason = skipReason;
    }

    /** 创建"跳过过滤"的条件 */
    public static PermissionCondition skipWith(String reason) {
        return new PermissionCondition(true, null, null, null, null, reason);
    }

    /** 创建"需要过滤"的条件 */
    public static PermissionCondition filterWith(Long userId, DataPermission config,
                                                   List<UserSubject> subjects,
                                                   List<Long> visibleDataIds) {
        return new PermissionCondition(false, userId, config, subjects, visibleDataIds, null);
    }

    public boolean isSkip() { return skip; }
    public String getSkipReason() { return skipReason; }

    /**
     * 将权限条件注入到 QueryWrapper。
     * 此方法是纯内存操作，不触发任何 DB 查询，可安全在 DynamicTableNameHolder 上下文内调用。
     */
    public void applyTo(QueryWrapper<?> qw) {
        if (skip) return;

        qw.and(wrapper -> {
            // 条件 1：用户是负责人
            wrapper.eq("owner_id", userId);

            // 条件 2：用户所在部门（如果 defaultAccess > 0）
            if (config.getDefaultAccess() != null && config.getDefaultAccess() > 0) {
                for (UserSubject s : subjects) {
                    if (s.getType() == UserSubject.SubjectType.DEPART) {
                        wrapper.or().eq("depart_id", s.getSubjectApiKey());
                    }
                }
            }

            // 条件 3：share 表中的可见数据
            if (visibleDataIds != null && !visibleDataIds.isEmpty()) {
                wrapper.or().in("id", visibleDataIds);
            }
        });
    }

    @Override
    public String toString() {
        if (skip) return "PermissionCondition{SKIP, reason=" + skipReason + "}";
        return "PermissionCondition{userId=" + userId
                + ", subjects=" + (subjects != null ? subjects.size() : 0)
                + ", visibleDataIds=" + (visibleDataIds != null ? visibleDataIds.size() : 0) + "}";
    }
}
```

### 3.2 DataPermissionFilter — 权限过滤器（核心，重构）

```java
/**
 * 数据权限过滤器。
 *
 * 核心职责：在 DynamicTableNameHolder 上下文之外，预构建权限条件值对象 PermissionCondition。
 *
 * ═══ 上下文隔离设计 ═══
 * buildCondition() 方法内部会触发元数据查询（configService.get → MetaServiceImpl.getByApiKey）
 * 和 share 表查询（shareQueryService.getVisibleDataIds）。
 * 这些查询必须在 DynamicTableNameHolder 上下文建立之前完成，
 * 否则 DynamicTableNameInterceptor 会将元数据表名替换为业务分片表名，导致 SQL 错误。
 *
 * 调用方（EntityDataService）必须遵循以下模式：
 *   PermissionCondition cond = permissionFilter.buildCondition(...);  // 分表上下文之外
 *   routeService.executeWithRoute(tenantId, entityApiKey, () -> {
 *       cond.applyTo(qw);  // 分表上下文之内，纯内存操作
 *       ...
 *   });
 */
@Slf4j
@Service
@RequiredArgsConstructor
public class DataPermissionFilter {

    private final DataPermissionConfigService configService;
    private final UserSubjectService          subjectService;
    private final DataShareQueryService       shareQueryService;

    /**
     * 预构建权限条件（必须在 DynamicTableNameHolder 上下文之外调用）。
     *
     * @return 不可变的 PermissionCondition，可安全传入 lambda / 跨线程使用
     */
    public PermissionCondition buildCondition(long tenantId, String entityApiKey, Long userId) {
        // ── 快速路径短路 ──
        if (userId == null) {
            return PermissionCondition.skipWith("internal_call");
        }
        if (isAdmin(tenantId, userId)) {
            return PermissionCondition.skipWith("admin");
        }

        // ── 查元数据配置（此时无 ThreadLocal 污染） ──
        DataPermission config = configService.get(tenantId, entityApiKey);
        if (config == null) {
            return PermissionCondition.skipWith("no_config");
        }
        if (config.getDefaultAccess() != null && config.getDefaultAccess() == 2) {
            return PermissionCondition.skipWith("public_access");
        }

        // ── 展开用户权限主体 ──
        List<UserSubject> subjects = subjectService.getUserSubjects(tenantId, userId);

        // ── 查 share 表可见数据（使用独立的分表路由上下文） ──
        List<Long> visibleDataIds = shareQueryService.getVisibleDataIds(
                tenantId, entityApiKey, subjects);

        log.debug("[DataPermission] buildCondition: tenant={}, entity={}, user={}, subjects={}, visibleIds={}",
                tenantId, entityApiKey, userId,
                subjects != null ? subjects.size() : 0,
                visibleDataIds != null ? visibleDataIds.size() : 0);

        return PermissionCondition.filterWith(userId, config, subjects, visibleDataIds);
    }

    /**
     * 预构建单条数据的访问校验上下文（必须在 DynamicTableNameHolder 上下文之外调用）。
     *
     * @return AccessCheckContext，后续在分表上下文内通过 checkAccess() 校验
     */
    public AccessCheckContext buildAccessCheck(long tenantId, String entityApiKey, Long userId) {
        PermissionCondition cond = buildCondition(tenantId, entityApiKey, userId);
        if (cond.isSkip()) {
            return AccessCheckContext.grantAll();
        }

        DataPermission config = configService.get(tenantId, entityApiKey);
        List<UserSubject> subjects = subjectService.getUserSubjects(tenantId, userId);
        return AccessCheckContext.of(userId, config, subjects);
    }

    private boolean isAdmin(long tenantId, Long userId) {
        // TODO: 调用角色服务判断是否为管理员
        return false;
    }
}
```

### 3.3 AccessCheckContext — 单条数据访问校验上下文（新增）

```java
/**
 * 单条数据访问校验的预构建上下文。
 * 与 PermissionCondition 类似，在分表上下文之外构建，在上下文之内使用。
 */
public final class AccessCheckContext {

    private final boolean grantAll;
    private final Long userId;
    private final DataPermission config;
    private final List<UserSubject> subjects;

    private AccessCheckContext(boolean grantAll, Long userId,
                                DataPermission config, List<UserSubject> subjects) {
        this.grantAll = grantAll;
        this.userId = userId;
        this.config = config;
        this.subjects = subjects != null ? Collections.unmodifiableList(subjects) : null;
    }

    public static AccessCheckContext grantAll() {
        return new AccessCheckContext(true, null, null, null);
    }

    public static AccessCheckContext of(Long userId, DataPermission config, List<UserSubject> subjects) {
        return new AccessCheckContext(false, userId, config, subjects);
    }

    /**
     * 校验单条数据的访问权限（纯内存判断 + 可能查 share 表）。
     * 返回 null 表示无权访问，否则返回访问级别（1=只读, 2=读写）。
     *
     * 注意：如果需要查 share 表，shareQueryService 内部使用独立的分表路由上下文，
     * 不会受外层 DynamicTableNameHolder 影响。
     */
    public Integer checkAccess(Long dataId, Long ownerId,
                                DataShareQueryService shareQueryService,
                                long tenantId, String entityApiKey) {
        if (grantAll) return 2;

        // 快速路径：负责人
        if (ownerId != null && ownerId.equals(userId)) {
            return config.getOwnerAccess();
        }

        // 查 share 表（独立路由上下文）
        return shareQueryService.getMaxAccessLevel(tenantId, entityApiKey, dataId, subjects);
    }
}
```

### 3.6 DataShareQueryService — Share 表查询

```java
/**
 * 查询 p_data_share 表，获取用户可见的数据 ID 集合。
 * share 表有独立的分表路由（p_data_share_route）。
 */
@Service
@RequiredArgsConstructor
public class DataShareQueryService {

    private final ShareTableRouteService shareRouteService;
    private final DataShareDao           shareDao;

    /**
     * 查询用户可见的数据 ID 列表（用于列表查询的 IN 条件）。
     * 分页查询，避免一次加载过多数据。
     */
    public List<Long> getVisibleDataIds(long tenantId, String entityApiKey,
                                         List<UserSubject> subjects) {
        return shareRouteService.executeWithRoute(tenantId, entityApiKey, () -> {
            // SELECT DISTINCT data_id FROM p_data_share_{M}
            // WHERE tenant_id=? AND entity_api_key=? AND delete_flg=0
            //   AND (
            //     (subject_type=0 AND subject_api_key IN (?userApiKeys))
            //     OR (subject_type=1 AND subject_api_key IN (?deptApiKeys))
            //     OR (subject_type=2 AND subject_api_key IN (?pgApiKeys))
            //     OR (subject_type=3 AND subject_api_key IN (?terrApiKeys))
            //   )
            // LIMIT 10000
            return shareDao.selectVisibleDataIds(tenantId, entityApiKey, subjects);
        });
    }

    /**
     * 查询单条数据的最高访问级别（用于单条查询的权限校验）。

            // SELECT MAX(access_level) FROM p_data_share_{M}
            // WHERE tenant_id=? AND entity_api_key=? AND data_id=? AND delete_flg=0

// 改造后（上下文隔离模式）
public PageResult<Map<String, Object>> listPage(
        long tenantId, String entityApiKey,
        List<String> selectColumns, Map<String, Object> conditions,
        String orderColumn, boolean asc, int page, int size,
        Long userId) {

    // ═══ 阶段 1：在分表路由上下文之外，预构建权限条件 ═══
    // 此时 DynamicTableNameHolder 为空，元数据查询不会被拦截

    // ═══ 阶段 2：进入分表路由上下文，执行业务数据查询 ═══
    return routeService.executeWithRoute(tenantId, entityApiKey, () -> {
        QueryWrapper<TenantData> qw = new QueryWrapper<>();
        if (selectColumns != null && !selectColumns.isEmpty()) {
            qw.select(selectColumns.toArray(new String[0]));
        }
        qw.eq("entity_api_key", entityApiKey).eq("delete_flg", 0);

        // 权限条件注入（纯内存操作，不触发 DB 查询）
        permCondition.applyTo(qw);

        if (conditions != null) {
            conditions.forEach(qw::eq);
        }
        if (orderColumn != null) {
            qw.orderBy(true, asc, orderColumn);
        } else {
            qw.orderByDesc("updated_at");
        }
        Page<Map<String, Object>> pageParam = new Page<>(page, size);
        IPage<Map<String, Object>> mpResult = dataDao.pageMaps(pageParam, qw);
        return PageResult.of(mpResult);
    });
}

## 六、性能优化策略

### 6.1 快速路径短路

```
权限判定优先级（从快到慢，均在分表上下文之外完成）：
  1. userId == null（内部调用）     → PermissionCondition.skip("internal_call")，0ms
  2. isAdmin(userId)               → PermissionCondition.skip("admin")，~1ms（缓存）
  3. config == null                → PermissionCondition.skip("no_config")，~2ms（缓存）
  4. defaultAccess == 2（全员可见） → PermissionCondition.skip("public_access")，~2ms（缓存）
  5. 需要过滤                      → PermissionCondition.filterWith(...)，~5-50ms
     ├─ 展开用户权限主体            ~1ms（缓存）
     └─ 查 share 表可见数据         ~5-50ms（取决于数据量）
```

### 6.2 share 表查询优化

|------|------|

## 十、多线程安全性分析

### 10.1 TransmittableThreadLocal 上下文隔离验证

改造前的 BUG 根因是 `DynamicTableNameHolder`（ThreadLocal）在嵌套调用时发生上下文污染。
改造后通过"阶段分离"彻底消除了这个问题。

> **重要**：`DynamicTableNameHolder` 使用 `TransmittableThreadLocal`（TTL）替代原生 `ThreadLocal`。
> TTL 确保在线程池复用线程时上下文能正确传递和隔离，避免线程池场景下的值泄漏。
> 生产环境中所有提交到线程池的任务必须使用 `TtlRunnable`/`TtlCallable` 包装，
> 或使用 `TtlExecutors.getTtlExecutorService()` 包装线程池。

以下逐一验证各场景：

#### 场景 1：正常 HTTP 请求（单线程）

```
Thread: XNIO-1 task-4

① permissionFilter.buildCondition(tenantId, entityApiKey, userId)
   │  DynamicTableNameHolder.get() == null  ✅ 无污染（TTL 值为空）
   │  → configService.get()
   │     → MetaServiceImpl.getByApiKey()
   │        → SQL: SELECT ... FROM p_common_metadata WHERE ...  ✅ 正确表名
   │  → subjectService.getUserSubjects()
   │  → shareQueryService.getVisibleDataIds()
   │     → shareRouteService.executeWithRoute()  // 独立的 TTL 上下文
   │        DynamicTableNameHolder.set("p_data_share_17")
   │        → SQL: SELECT ... FROM p_data_share_17 WHERE ...  ✅ 正确表名
   │        DynamicTableNameHolder.clear()  // finally 清理
   │  返回 PermissionCondition（不可变值对象）

② routeService.executeWithRoute(tenantId, entityApiKey, () -> {
   │  DynamicTableNameHolder.set("p_tenant_data_42")
   │  → permCondition.applyTo(qw)  // 纯内存操作，不触发 DB 查询 ✅
   │  → SQL: SELECT ... FROM p_tenant_data_42 WHERE ...  ✅ 正确表名
   │  DynamicTableNameHolder.clear()  // finally 清理
   })
```

#### 场景 2：并发 HTTP 请求（多线程）

```
Thread-A: XNIO-1 task-4          Thread-B: XNIO-1 task-7
─────────────────────────         ─────────────────────────
① buildCondition(292193,          ① buildCondition(100001,
   "account", user1)                 "opportunity", user2)
   TTL-A: null ✅                    TTL-B: null ✅
   → configService.get()            → configService.get()
   → 返回 condA                     → 返回 condB

② executeWithRoute(292193,        ② executeWithRoute(100001,
   "account", () -> {                "opportunity", () -> {
     TTL-A:                            TTL-B:
       "p_tenant_data_42"               "p_tenant_data_88"
     condA.applyTo(qw) ✅              condB.applyTo(qw) ✅
     // 纯内存，不读 TTL               // 纯内存，不读 TTL
   })                                })
   TTL-A: null                       TTL-B: null
```

> TTL 是线程隔离的，Thread-A 和 Thread-B 各自持有独立的 TTL 副本，互不干扰。
> `PermissionCondition` 是不可变值对象，即使被多线程共享也不会有竞态条件。

#### 场景 3：share 表查询的嵌套 TTL

```
Thread-A:
① buildCondition()
   │  DynamicTableNameHolder.get() == null
   │
   ├─ shareQueryService.getVisibleDataIds()
   │     shareRouteService.executeWithRoute(tenantId, entityApiKey, () -> {
   │       DynamicTableNameHolder.set("p_data_share_17")  // 设置 share 表路由
   │       → SQL: SELECT ... FROM p_data_share_17 ...     ✅
   │       DynamicTableNameHolder.clear()                 // finally 清理
   │     })
   │
   │  DynamicTableNameHolder.get() == null  ✅ share 路由已清理
   │
② routeService.executeWithRoute()
   │  DynamicTableNameHolder.set("p_tenant_data_42")      // 设置业务数据路由
   │  → SQL: SELECT ... FROM p_tenant_data_42 ...         ✅
   │  DynamicTableNameHolder.clear()                      // finally 清理
```

> share 表查询使用独立的 `executeWithRoute` 调用，其 `finally` 块保证 TTL 被清理。
> 阶段 1 结束后 TTL 一定为 null，阶段 2 的 `set()` 不会受到残留值影响。

#### 场景 4：异常情况下的 TTL 清理

```
Thread-A:
① buildCondition()
   │  → configService.get() 抛出异常
   │  DynamicTableNameHolder.get() == null  ✅ 阶段 1 未设置过 TTL
   │  异常向上传播，不进入阶段 2

② buildCondition()
   │  → shareQueryService.getVisibleDataIds()
   │     shareRouteService.executeWithRoute(() -> {
   │       DynamicTableNameHolder.set("p_data_share_17")
   │       → SQL 执行抛出异常
   │       DynamicTableNameHolder.clear()  // finally 保证清理 ✅
   │     })
   │  异常向上传播，不进入阶段 2
```

> `DynamicTableNameHolder.executeWith()` 使用 try-finally 模式，即使 SQL 执行抛出异常，TTL 也会被清理。
> 阶段 1 中 `buildCondition()` 本身不设置 TTL，只有 share 表查询的内部 `executeWithRoute` 会临时设置并清理。

#### 场景 5：线程池复用线程（TTL 核心价值）

```
ThreadPool(size=2):

Task-1 提交到 Thread-A:
  │  TtlRunnable.capture() → 快照: {TABLE_NAME=null}
  │  Thread-A 执行:
  │    TtlRunnable.replay(快照) → 清除 Thread-A 残留值
  │    DynamicTableNameHolder.executeWith("p_tenant_data_42", ...)
  │    DynamicTableNameHolder.clear()  // finally
  │    TtlRunnable.restore(备份) → 恢复 Thread-A 原始状态

Task-2 提交到 Thread-A（复用同一线程）:
  │  TtlRunnable.capture() → 快照: {TABLE_NAME=null}
  │  Thread-A 执行:
  │    TtlRunnable.replay(快照) → 清除 Thread-A 残留值（即使 Task-1 有泄漏也会被清除）
  │    DynamicTableNameHolder.get() == null  ✅ 不会看到 Task-1 的残留值
  │    ...
```

> 原生 ThreadLocal 在线程池复用时，如果 Task-1 忘记清理，Task-2 会看到残留值。
> TTL 的 capture/replay/restore 机制确保每个任务开始时 TTL 状态与提交时一致，
> 任务结束后恢复线程原始状态，彻底避免线程池场景下的上下文泄漏。

### 10.2 不可变值对象的线程安全性

| 类 | 可变性 | 线程安全 | 说明 |
|---|---|---|---|
| `PermissionCondition` | 不可变 | ✅ | 所有字段 final，List 用 `Collections.unmodifiableList` 包装 |
| `AccessCheckContext` | 不可变 | ✅ | 所有字段 final，List 用 `Collections.unmodifiableList` 包装 |
| `DataPermission` | 需确认 | ⚠️ | 如果是 MyBatis 返回的 POJO，需确保 `buildCondition` 返回后不被修改 |
| `UserSubject` | 需确认 | ⚠️ | Lombok `@Data` 生成 setter，但包装在 unmodifiableList 中，元素本身不应被修改 |

> 建议：`DataPermission` 和 `UserSubject` 在传入 `PermissionCondition` 时做防御性拷贝，
> 或将它们改为 `@Value`（Lombok 不可变类）。

### 10.4 改造前后对比

| 维度 | 改造前（BUG） | 改造后（修复） |
|---|---|---|
| 上下文实现 | 原生 ThreadLocal | TransmittableThreadLocal（TTL） |
| 元数据查询位置 | `executeWithRoute` lambda 内部 | `executeWithRoute` 之前 |
| TTL 状态 | 元数据查询时 = "p_tenant_data_0" ❌ | 元数据查询时 = null ✅ |
| 权限条件传递方式 | 直接操作 QueryWrapper（有副作用） | 不可变值对象 PermissionCondition |

### 10.5 开发规范（防止回归）

为防止后续开发再次引入类似的上下文污染问题，制定以下规范：

1. **禁止使用原生 `ThreadLocal`，必须使用 `TransmittableThreadLocal`**
   - 所有需要线程上下文传递的场景，统一使用 `com.alibaba.ttl.TransmittableThreadLocal`
   - 原生 `ThreadLocal` 在线程池复用线程时会导致值泄漏或丢失

2. **线程池提交任务必须使用 TTL 包装**
   - `TtlRunnable.get(runnable)` 包装 Runnable
   - `TtlCallable.get(callable)` 包装 Callable
   - 或使用 `TtlExecutors.getTtlExecutorService(executor)` 包装整个线程池
   - 框架层的 `framework-threadpool` 模块应默认集成 TTL 包装

3. **禁止在 `executeWithRoute` lambda 内部调用元数据服务**
   - `MetaServiceImpl`、`DataPermissionConfigService`、`MetadataMergeReadServiceImpl` 等
   - 这些服务的 SQL 会被 `DynamicTableNameInterceptor` 拦截

4. **分表上下文内只允许以下操作**
   - 纯内存操作（QueryWrapper 条件拼装、数据转换）
   - 业务数据表的 CRUD（`dataDao.pageMaps`、`dataDao.insert` 等）
   - 这些操作的表名就是要被替换的目标表

5. **如需在分表上下文内查询其他表，必须使用独立的 `executeWithRoute`**
   - 如 share 表查询：`shareRouteService.executeWithRoute(...)` 会先 clear 再 set 新值
   - 但注意：嵌套的 `executeWithRoute` 会覆盖外层的 TTL 值，
     所以必须在外层 `executeWithRoute` 之前完成，或使用独立的路由服务

6. **Code Review 检查项**
   - 搜索 `executeWithRoute` lambda 内部的所有方法调用
   - 确认每个方法是否会触发非业务数据表的 SQL 查询
   - 如果会，必须将该查询移到 `executeWithRoute` 之外
   - 搜索 `new ThreadLocal`，确认没有遗漏的原生 ThreadLocal 使用
