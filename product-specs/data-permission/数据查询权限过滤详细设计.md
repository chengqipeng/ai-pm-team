# 数据查询权限过滤详细设计

> 基于现有 EntityDataService 查询链路，设计权限过滤的注入方案

---

## 一、现有查询链路分析

### 1.1 当前调用链

```
EntityDataApiService (Controller)
  │
  ├─ GET /{entityApiKey}          → listData()
  │     └─ EntityDataService.listPage(tenantId, entityApiKey, selectColumns, conditions, orderColumn, asc, page, size)
  │           └─ routeService.executeWithRoute(tenantId, entityApiKey, () -> {
  │                 QueryWrapper<TenantData> qw = new QueryWrapper<>();
  │                 qw.eq("entity_api_key", entityApiKey).eq("delete_flg", 0);
  │                 qw.条件/排序/分页...
  │                 dataDao.pageMaps(pageParam, qw);
  │               })
  │
  ├─ GET /{entityApiKey}/{id}     → getData()
  │     └─ EntityDataService.getById(tenantId, entityApiKey, id)
  │           └─ routeService.executeWithRoute(tenantId, entityApiKey,
  │                 () -> dataDao.getByIdAndTenant(id, tenantId))
  │
  └─ POST /batch-save             → batchSave()
        └─ EntityDataBatchService.batchSave(...)
              └─ batchGetByIds(tenantId, entityApiKey, ids)
                    └─ routeService.executeWithRoute(tenantId, entityApiKey, () -> {
                          QueryWrapper qw = ...;
                          dataDao.listMaps(qw);
                        })
```

### 1.2 当前问题

- 所有查询**无任何权限过滤**，任何用户可以看到同租户下所有数据
- `listPage()` 的 `conditions` 参数当前传 null，未被使用
- `getById()` 仅校验 tenantId，不校验用户是否有权查看该条数据
- API 层未传递 userId，Service 层无法获取当前操作用户

### 1.3 关键约束

- 查询走 `DynamicTableNameHolder` 路由到分片表 `p_tenant_data_{N}`
- share 表 `p_data_share_{M}` 路由到不同的分片表
- 两张表在不同的物理表中，**不能直接 JOIN**
- 必须采用"先查权限，再查数据"或"子查询"的方式

---

## 二、设计方案

### 2.1 核心思路

采用**两阶段查询**模式：

```
阶段 1：权限预判（快速路径短路）
  ├─ 管理员？ → 跳过权限过滤
  ├─ defaultAccess=2（全员读写）？ → 跳过权限过滤
  └─ 否 → 进入阶段 2

阶段 2：权限条件注入
  ├─ 小数据量（owner_id 直接过滤）→ 在 QueryWrapper 中追加 owner_id/depart_id 条件
  └─ 通用模式 → 先从 share 表查出可见 data_id 集合，再用 IN 条件过滤
```

### 2.2 架构分层

```
EntityDataApiService (Controller)
  │  ↓ 传递 userId
EntityDataService (Service)
  │  ↓ 调用权限服务
DataPermissionFilter (新增，权限过滤器)
  │  ├─ DataPermissionConfigService (读取 dataPermission 元数据)
  │  ├─ UserSubjectService (展开用户权限主体)
  │  └─ DataShareQueryService (查询 share 表)
  │  ↓ 返回过滤条件
QueryWrapper 注入权限条件
  │  ↓
dataDao.pageMaps() / dataDao.getMap()
```

## 三、新增组件设计

### 3.1 DataPermissionFilter — 权限过滤器（核心）

```java
/**
 * 数据权限过滤器。
 * 职责：根据当前用户和 entity 的权限配置，生成 QueryWrapper 的权限过滤条件。
 */
@Service
@RequiredArgsConstructor
public class DataPermissionFilter {

    private final DataPermissionConfigService configService;
    private final UserSubjectService          subjectService;
    private final DataShareQueryService       shareQueryService;

    /**
     * 判断是否需要权限过滤。
     * 返回 true 表示需要过滤，false 表示跳过（管理员或全员可见）。
     */
    public boolean needFilter(long tenantId, String entityApiKey, Long userId) {
        if (userId == null) return false;                    // 系统内部调用，不过滤
        if (isAdmin(tenantId, userId)) return false;         // 管理员，不过滤

        DataPermission config = configService.get(tenantId, entityApiKey);
        if (config == null) return false;                    // 无权限配置，不过滤
        if (config.getDefaultAccess() != null
            && config.getDefaultAccess() == 2) return false; // 全员读写，不过滤

        return true;
    }

    /**
     * 为列表查询注入权限条件。
     * 在 QueryWrapper 上追加 WHERE 条件，限制只返回用户有权查看的数据。
     */
    public void applyListFilter(QueryWrapper<?> qw,
                                 long tenantId, String entityApiKey, Long userId) {
        if (!needFilter(tenantId, entityApiKey, userId)) return;

        DataPermission config = configService.get(tenantId, entityApiKey);
        List<UserSubject> subjects = subjectService.getUserSubjects(tenantId, userId);

        // 构建权限条件：owner_id = userId OR data_id IN (share 表可见集合)
        qw.and(wrapper -> {
            // 条件 1：用户是负责人
            wrapper.eq("owner_id", userId);

            // 条件 2：用户所在部门（如果 defaultAccess > 0）
            if (config.getDefaultAccess() != null && config.getDefaultAccess() > 0) {
                for (UserSubject s : subjects) {
                    if (s.getType() == SubjectType.DEPART) {
                        wrapper.or().eq("depart_id", s.getSubjectApiKey());
                    }
                }
            }

            // 条件 3：share 表中的可见数据
            List<Long> visibleDataIds = shareQueryService.getVisibleDataIds(
                    tenantId, entityApiKey, subjects);
            if (visibleDataIds != null && !visibleDataIds.isEmpty()) {
                wrapper.or().in("id", visibleDataIds);
            }
        });
    }

    /**
     * 校验单条数据的访问权限。
     * 返回 null 表示无权访问，否则返回访问级别（1=只读, 2=读写）。
     */
    public Integer checkAccess(long tenantId, String entityApiKey,
                                Long userId, Long dataId, Long ownerId) {
        if (!needFilter(tenantId, entityApiKey, userId)) return 2; // 不需要过滤=读写

        // 快速路径：负责人
        if (ownerId != null && ownerId.equals(userId)) {
            DataPermission config = configService.get(tenantId, entityApiKey);
            return config.getOwnerAccess(); // 通常是 2（读写）
        }

        // 查 share 表
        List<UserSubject> subjects = subjectService.getUserSubjects(tenantId, userId);
        return shareQueryService.getMaxAccessLevel(tenantId, entityApiKey, dataId, subjects);
    }

    private boolean isAdmin(long tenantId, Long userId) {
        // TODO: 调用角色服务判断是否为管理员
        return false;
    }
}
```

### 3.4 DataShareQueryService — Share 表查询

```java
/**
 * 查询 p_data_share 表，获取用户可见的数据 ID 集合。
 * share 表有独立的分表路由（p_data_share_route）。
 */
@Service
@RequiredArgsConstructor
public class DataShareQueryService {

    private final ShareTableRouteService shareRouteService;
    private final DataShareDao           shareDao;

    /**
     * 查询用户可见的数据 ID 列表（用于列表查询的 IN 条件）。
     * 分页查询，避免一次加载过多数据。
     */
    public List<Long> getVisibleDataIds(long tenantId, String entityApiKey,
                                         List<UserSubject> subjects) {
        return shareRouteService.executeWithRoute(tenantId, entityApiKey, () -> {
            // SELECT DISTINCT data_id FROM p_data_share_{M}
            // WHERE tenant_id=? AND entity_api_key=? AND delete_flg=0
            //   AND (
            //     (subject_type=0 AND subject_api_key IN (?userApiKeys))
            //     OR (subject_type=1 AND subject_api_key IN (?deptApiKeys))
            //     OR (subject_type=2 AND subject_api_key IN (?pgApiKeys))
            //     OR (subject_type=3 AND subject_api_key IN (?terrApiKeys))
            //   )
            // LIMIT 10000
            return shareDao.selectVisibleDataIds(tenantId, entityApiKey, subjects);
        });
    }

    /**
     * 查询单条数据的最高访问级别（用于单条查询的权限校验）。
     */
    public Integer getMaxAccessLevel(long tenantId, String entityApiKey,
                                      Long dataId, List<UserSubject> subjects) {
        return shareRouteService.executeWithRoute(tenantId, entityApiKey, () -> {
            // SELECT MAX(access_level) FROM p_data_share_{M}
            // WHERE tenant_id=? AND entity_api_key=? AND data_id=? AND delete_flg=0
            //   AND (subject_type, subject_api_key) IN subjects
            return shareDao.selectMaxAccessLevel(tenantId, entityApiKey, dataId, subjects);
        });
    }
}
```

```java
// 改造前
public PageResult<Map<String, Object>> listPage(
        long tenantId, String entityApiKey,
        List<String> selectColumns, Map<String, Object> conditions,
        String orderColumn, boolean asc, int page, int size) {
    return routeService.executeWithRoute(tenantId, entityApiKey, () -> {
        QueryWrapper<TenantData> qw = new QueryWrapper<>();
        if (selectColumns != null && !selectColumns.isEmpty()) {
            qw.select(selectColumns.toArray(new String[0]));
        }
        qw.eq("entity_api_key", entityApiKey).eq("delete_flg", 0);
        if (conditions != null) {
            conditions.forEach(qw::eq);
        }
        if (orderColumn != null) {
            qw.orderBy(true, asc, orderColumn);
        } else {
            qw.orderByDesc("updated_at");
        }
        Page<Map<String, Object>> pageParam = new Page<>(page, size);
        IPage<Map<String, Object>> mpResult = dataDao.pageMaps(pageParam, qw);
        return PageResult.of(mpResult);
    });
}

// 改造后
public PageResult<Map<String, Object>> listPage(
        long tenantId, String entityApiKey,
        List<String> selectColumns, Map<String, Object> conditions,
        String orderColumn, boolean asc, int page, int size,
        Long userId) {                                          // ← 新增参数
    return routeService.executeWithRoute(tenantId, entityApiKey, () -> {
        QueryWrapper<TenantData> qw = new QueryWrapper<>();
        if (selectColumns != null && !selectColumns.isEmpty()) {
            qw.select(selectColumns.toArray(new String[0]));
        }
        qw.eq("entity_api_key", entityApiKey).eq("delete_flg", 0);

        // ═══ 权限过滤注入 ═══
        permissionFilter.applyListFilter(qw, tenantId, entityApiKey, userId);

        if (conditions != null) {
            conditions.forEach(qw::eq);
        }
        if (orderColumn != null) {
            qw.orderBy(true, asc, orderColumn);
        } else {
            qw.orderByDesc("updated_at");
        }
        Page<Map<String, Object>> pageParam = new Page<>(page, size);
        IPage<Map<String, Object>> mpResult = dataDao.pageMaps(pageParam, qw);
        return PageResult.of(mpResult);
    });
}

// 保留无 userId 的重载（内部调用，不过滤权限）
public PageResult<Map<String, Object>> listPage(

        List<String> selectColumns, Map<String, Object> conditions,

    return listPage(tenantId, entityApiKey, selectColumns, conditions,
```

```java
// 改造后：新增带权限校验的 getById
public TenantData getById(long tenantId, String entityApiKey, long id, Long userId) {
    TenantData record = getById(tenantId, entityApiKey, id);
    if (record == null) return null;

    // 权限校验
    Integer accessLevel = permissionFilter.checkAccess(
            tenantId, entityApiKey, userId, id, record.getOwnerId());
    if (accessLevel == null || accessLevel == 0) {
        return null;  // 无权访问，返回 null（或抛异常）
    }

    return record;
}
```

## 六、性能优化策略

### 6.1 快速路径短路

```
权限判定优先级（从快到慢）：
  1. userId == null（内部调用）     → 跳过，0ms
  2. isAdmin(userId)               → 跳过，~1ms（缓存）
  3. defaultAccess == 2（全员可见） → 跳过，~2ms（缓存）
  4. owner_id == userId（负责人）   → 仅追加 owner_id 条件，0ms
  5. 查 share 表                   → ~5-50ms（取决于数据量）
```

### 6.2 share 表查询优化

| 策略 | 说明 |
|------|------|
| 分页限制 | share 表查询 LIMIT 10000，超过则降级为仅 owner_id 过滤 |
| 索引覆盖 | `idx_ds_tid_eak_sub` 索引覆盖 (tenant_id, entity_api_key, subject_api_key, subject_type) |
| 结果缓存 | 对于频繁查询的用户+entity 组合，缓存可见 data_id 集合，TTL 30s |
| 异步预热 | 用户登录时异步预热权限主体缓存 |

### 6.3 大数据量降级

当 share 表返回的 data_id 超过 10000 条时，IN 条件性能下降。降级策略：

```
if visibleDataIds.size() > 10000:
    // 降级为仅 owner_id + depart_id 过滤
    // 放弃 share 表中的团队成员/公共组/共享规则/区域权限
    // 记录降级日志，后续通过 ES 优化
    qw.and(w -> w.eq("owner_id", userId)
                  .or().eq("depart_id", userDeptId));
```
