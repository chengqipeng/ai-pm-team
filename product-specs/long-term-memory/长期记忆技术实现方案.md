# AI Agent 长期记忆技术实现方案

> 参考来源：[OpenViking](https://github.com/volcengine/OpenViking) — 火山引擎开源的 AI Agent 上下文数据库
> 本方案在深度分析 OpenViking 的文件系统范式、分层上下文、目录递归检索、会话记忆提取等核心机制后，结合我们 aPaaS 平台的元数据驱动架构，设计了一套适配自身业务场景的长期记忆系统。

---

## 一、整体架构

### 1.1 设计理念

OpenViking 的核心洞察：传统 RAG 将上下文扁平化为向量碎片，丢失了层级结构和全局视图。OpenViking 用"文件系统范式"统一管理记忆、资源、技能三类上下文，通过 L0/L1/L2 三层渐进加载和目录递归检索，实现了 Token 消耗降低 83% 的同时任务完成率提升 49%。

我们的方案继承这一理念，但做了以下适配：
- 将文件系统范式映射为元数据驱动的记忆存储模型
- 将 8 类记忆分类体系与我们的业务对象模型对齐
- 增加记忆遗忘策略和反思修正机制（OpenViking 当前版本未深入实现）

### 1.3 记忆分类体系

借鉴 OpenViking 的 8 类记忆分类，结合我们的业务场景：

|:---|:---|:---|:---|

| preferences | user/memories/preferences/ | 用户偏好（按主题分文件） | 追加 |

| events | user/memories/events/ | 事件记录（决策、里程碑） | 只追加不更新 |

| skills | agent/memories/skills/ | 技能执行知识和工作流策略 | 可合并 |

### 1.4 三层信息模型（L0/L1/L2）

| 层级 | 名称 | Token 限制 | 用途 |
|:---|:---|:---|:---|
| L0 | 摘要 (Abstract) | ~100 tokens | 向量检索、快速过滤 |
| L1 | 概览 (Overview) | ~2k tokens | 重排序、内容导航、规划决策 |
| L2 | 详情 (Detail) | 无限制 | 完整内容、按需加载 |

## 二、记忆提取详细设计

### 2.1 提取触发机制

参考 OpenViking 的双阈值 Compact 上报机制，设计三种触发方式：

#### 2.1.1 会话压缩触发（自动）

```
会话消息流 ──→ 阈值检测 ──→ 触发提取
                  │
                  ├── 阈值1（50% 上下文窗口）：后台异步上报，不阻塞主流程
                  ├── 阈值2（70% 上下文窗口）：强制清理已上报消息
                  └── 会话结束（/new 或超时）：全量提取剩余消息
```

核心流程：
1. 当会话达到上下文窗口 50% 时，后台触发记忆上报
2. 记录上报消息范围 `(start_index, end_index)`，存入会话元数据
3. 上报完成后清理已上报消息；若达到 70% 阈值则强制清理
4. 会话结束时对未提取的消息执行最终提取

#### 2.1.2 主动记忆触发（工具调用）

Agent 通过 `commit_memory` 工具主动记录：

```json
{
  "tool": "commit_memory",
  "parameters": {
    "content": "用户偏好深色模式",
    "category": "preferences",
    "priority": "high",
    "tags": ["ui", "theme"]
  }
}
```

适用场景：
- 用户明确要求记住某事（"记住我喜欢..."）
- 重要决策节点
- 强偏好表达

#### 2.1.3 定时回扫触发（兜底）

对于未达到压缩阈值就中断的会话，设置定时任务（如每 30 分钟）扫描未提取的会话片段。

### 2.2 提取处理流水线

```
原始会话消息
    │
    ▼
┌──────────────────┐
│ 1. 消息预处理     │  过滤噪声（问候语、确认语）、合并连续同角色消息

│ 4. 去重与冲突检测  │  与已有记忆比对，检测矛盾和重复

┌──────────────────┐

### 2.3 8 类记忆提取 Prompt 设计

提取时使用结构化 Prompt，要求 LLM 从对话中按 8 个维度输出：

```
你是一个记忆提取专家。请从以下对话中提取长期记忆，按以下 8 个类别分类输出：

1. profile: 用户基本信息（姓名、角色、团队、技术栈等）
2. preferences: 用户偏好（编码风格、工具偏好、沟通方式等）
3. entities: 实体信息（提到的人、项目、系统、业务对象等）
4. events: 事件记录（做出的决策、达成的里程碑、遇到的问题等）
5. cases: 案例知识（成功/失败的解决方案、调试经验等）
6. patterns: 模式知识（反复出现的问题模式、最佳实践等）
7. tools: 工具知识（工具使用技巧、参数配置、常见错误等）
8. skills: 技能知识（工作流策略、执行步骤优化等）

输出格式：
{
  "memories": [
    {
      "category": "preferences",
      "content": "用户偏好使用 TypeScript 而非 JavaScript",
      "confidence": 0.95,
      "importance": "high",
      "source_turns": [3, 5],
      "tags": ["language", "typescript"]
    }
  ]
}

对话内容：
{conversation}
```

### 2.4 L0/L1/L2 生成机制

参考 OpenViking 的 SemanticProcessor 自底向上生成策略：

```
叶子节点记忆 → 生成 L0 摘要（~100 tokens）
                → 生成 L1 概览（~2k tokens，含导航指引）
                → L2 为原始完整内容

父目录 → 聚合子节点 L0 → 生成父级 L1
       → 生成父级 L0
```

示例：
```
user/memories/preferences/
├── .abstract.md          # L0: "用户偏好深色主题、TypeScript、简洁代码风格"
├── .overview.md          # L1: 详细偏好列表 + 各偏好的来源会话引用
├── coding_style.md       # L2: 完整的编码风格偏好记录
├── ui_preferences.md     # L2: 完整的 UI 偏好记录
└── communication.md      # L2: 完整的沟通偏好记录
```

### 2.5 去重与合并策略

| 记忆类别 | 去重策略 | 合并方式 |
|:---|:---|:---|
| profile | 语义相似度 > 0.9 视为重复 | 新值覆盖旧值，保留变更历史 |
| preferences | 同主题同维度视为重复 | 合并到同一文件，更新时间戳 |
| entities | 同实体名 + 语义相似 | 合并属性，追加新信息 |
| events | 时间 + 主题匹配 | 不合并，保留独立记录 |
| cases | 问题描述相似度 > 0.85 | 不合并，保留独立记录 |
| patterns | 模式描述相似度 > 0.8 | LLM 判断是否合并，合并时综合描述 |
| tools | 同工具 + 同知识点 | 合并更新，保留最新最佳实践 |
| skills | 同技能 + 同策略维度 | 合并更新 |

## 三、记忆召回详细设计

### 3.1 召回架构

参考 OpenViking 的三部分召回机制，设计四层召回体系：

```
┌─────────────────────────────────────────────────────────┐
│                    记忆召回四层体系                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Layer 1: 用户画像注入（会话开始时，一次性）                │
│  ────────────────────────────────────────                │

│                                                         │
│  Layer 3: Agent 主动召回（工具调用）                       │
│  ────────────────────────────────────────                │
│  • 多跳推理场景                                           │
│  • 时间范围查询                                           │
│  • 跨会话检索                                             │
│  • 目录语义探索                                           │
│                                                         │
│  Layer 4: 技能/工具记忆注入（拦截式）                      │
│  ────────────────────────────────────────                │
│  • 拦截 skill 文件读取，追加使用经验                       │
│  • 工具记忆注入 system prompt                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

在会话初始化时执行：

### 3.3 Layer 2: 每轮自动召回

参考 OpenViking 的目录递归检索策略（Directory Recursive Retrieval）：

```
用户消息 ──→ 意图检测 ──→ 查询构建 ──→ 目录递归检索 ──→ 重排序 ──→ 注入
```

#### 3.3.1 意图检测（轻量级）

```python
def should_skip_retrieval(message: str) -> bool:
    # 跳过问候语
    greetings = ["你好", "在吗", "hi", "hello", "hey"]
    if message.strip().lower() in greetings:
        return True
    # 跳过极短消息
    if len(message.strip()) <= 3:
        return True
    # 跳过纯确认
    confirmations = ["好的", "ok", "是的", "对", "嗯"]
    if message.strip().lower() in confirmations:
        return True
    return False
```

#### 3.3.2 查询构建

```python
def build_recall_query(session, current_message: str) -> str:
    # 使用最近 N 条用户消息拼接作为搜索查询
    recent_messages = session.get_recent_user_messages(n=5)
    query = "\n".join([m.content for m in recent_messages])
    return query
```

#### 3.3.3 目录递归检索（核心算法）

这是 OpenViking 最核心的检索创新。传统向量检索是"一次性搜索碎片"，目录递归检索是"先定位目录，再逐层深入"：

```
Step 1: 意图分析
  输入用户查询 → LLM 生成 0~5 个类型化子查询
  例："上周关于数据库优化的讨论" →
    - 时间查询: "上周的对话"
    - 主题查询: "数据库优化"
    - 事件查询: "讨论和决策"

Step 2: 初始定位（向量检索）
  对每个子查询执行向量检索 → 命中 L0 摘要 → 定位到高分目录
  例: 命中 user/memories/events/ 和 agent/memories/cases/

Step 3: 目录内精细探索
  在高分目录内执行二次检索（向量 + 关键词）
  更新候选集中的高分结果

Step 4: 递归下钻
  如果高分目录有子目录 → 递归执行 Step 3
  使用优先队列管理探索顺序，分数收敛时停止

Step 5: 结果聚合与重排序
  收集所有候选结果 → 标量过滤（时间、类别）→ 模型重排序 → 返回 Top-K
```

```python
def directory_recursive_retrieval(query: str, max_results: int = 5) -> list:
    # Step 1: 意图分析
    sub_queries = intent_analyzer.analyze(query)  # 生成 0~5 个子查询
    
    # Step 2: 初始向量检索，定位高分目录
    priority_queue = PriorityQueue()  # (score, uri)
    for sq in sub_queries:
        hits = vector_index.search(sq.embedding, top_k=20)
        for hit in hits:
            dir_uri = get_parent_directory(hit.uri)
            priority_queue.push(hit.score, dir_uri)
    
    # Step 3 & 4: 递归探索
    candidates = []
    visited = set()
    while not priority_queue.empty() and len(candidates) < max_results * 3:
        score, dir_uri = priority_queue.pop()
        if dir_uri in visited:
            continue
        visited.add(dir_uri)
        
        # 目录内精细检索
        dir_results = search_within_directory(dir_uri, sub_queries)
        candidates.extend(dir_results)
        
        # 子目录递归
        subdirs = memory_fs.ls(dir_uri, dirs_only=True)
        for subdir in subdirs:
            sub_score = estimate_relevance(subdir, sub_queries)
            if sub_score > RELEVANCE_THRESHOLD:
                priority_queue.push(sub_score, subdir)
    
    # Step 5: 重排序
    reranked = reranker.rerank(query, candidates, top_k=max_results)
    return reranked
```

#### 3.3.4 结果注入方式

参考 OpenViking RFC 的设计，以模拟 function call 结果注入（而非直接拼入 prompt）：

```
User: 如何优化数据库？

System: [Function Result] {
  "success": true,
  "memories": [
    {
      "uri": "user/memories/events/2026-03-28-db-optimization.md",
      "layer": "L1",
      "content": "3月28日讨论了索引优化策略，决定对高频查询表添加复合索引...",
      "relevance": 0.92
    },
    {
      "uri": "agent/memories/cases/mysql-slow-query-fix.md",
      "layer": "L1",
      "content": "MySQL 慢查询优化案例：通过 EXPLAIN 分析发现全表扫描...",
      "relevance": 0.87
    }
  ]
}

这种注入方式的优势：
- Agent 感知到搜索行为的存在，后续可主动发起类似搜索
- 查询关键词透明，Agent 可选择不同关键词重新搜索
- 不污染持久化的对话历史

### 3.6 召回性能优化

| 优化策略 | 说明 |
|:---|:---|
| L0 优先过滤 | 先用 L0 摘要做向量检索，命中后再加载 L1/L2 |
| 目录级缓存 | 高频访问目录的 L0/L1 缓存在内存中 |
| 异步预加载 | 会话开始时异步预加载用户高频记忆目录 |
| 查询缓存 | 相似查询（余弦相似度 > 0.95）复用缓存结果 |
| 分层 Token 预算 | L1 通常足够，仅确认需要时才加载 L2 |

## 四、记忆遗忘策略详细设计

> OpenViking 当前版本主要关注记忆的写入和检索，遗忘机制尚未深入实现。本节是我们的扩展设计。

### 4.1 设计原则

记忆遗忘不是简单删除，而是模拟人类记忆的自然衰减：
- 重要记忆不遗忘，低价值记忆逐渐淡化
- 频繁被召回的记忆强化，长期未访问的记忆衰减
- 矛盾的旧记忆被新记忆取代时标记为过时
- 遗忘是可逆的，"遗忘"的记忆降级而非删除

### 4.2 记忆生命周期状态机

```
                    ┌──────────┐
        创建 ──────→│  Active   │←──── 召回命中（强化）
                    └────┬─────┘
                         │ 衰减触发
                         ▼
                    ┌──────────┐
                    │  Fading  │←──── 偶尔命中（部分恢复）
                    └────┬─────┘
                         │ 持续未访问
                         ▼
                    ┌──────────┐
                    │ Dormant  │←──── 主动搜索命中（可恢复）
                    └────┬─────┘
                         │ 超过保留期
                         ▼
                    ┌──────────┐
                    │ Archived │      物理归档，不参与常规检索
                    └──────────┘
```

| 状态 | 检索可见性 | L0/L1/L2 可用性 | 恢复条件 |
|:---|:---|:---|:---|
| Active | 全部可见 | 全部可用 | — |
| Fading | 仅 L0 参与向量检索 | L0 可用，L1/L2 按需加载 | 被召回时恢复为 Active |
| Dormant | 不参与自动召回 | 仅主动搜索可触达 | 主动搜索命中时恢复为 Fading |
| Archived | 不可见 | 需手动恢复 | 管理员手动恢复 |

### 4.3 衰减评分模型

每条记忆维护一个综合分数 `memory_score`，由以下因子加权计算：

```python
def calculate_memory_score(memory) -> float:
    now = current_time()
    
    # 1. 时间衰减因子（艾宾浩斯遗忘曲线）
    days_since_creation = (now - memory.created_at).days
    days_since_last_access = (now - memory.last_accessed_at).days
    time_decay = math.exp(-0.1 * days_since_last_access)  # 指数衰减
    
    # 2. 访问频率因子
    access_frequency = memory.access_count / max(days_since_creation, 1)
    frequency_factor = min(1.0, access_frequency / FREQUENCY_BASELINE)
    
    # 3. 重要性因子（提取时的置信度 × 重要性标签）
    importance_weights = {"critical": 1.0, "high": 0.8, "medium": 0.5, "low": 0.2}
    importance_factor = memory.confidence * importance_weights[memory.importance]
    
    # 4. 关联强度因子（被其他记忆引用的次数）
    reference_factor = min(1.0, memory.reference_count / REFERENCE_BASELINE)
    
    # 5. 类别保护因子（某些类别天然更持久）
    category_weights = {
        "profile": 1.0,      # 用户画像几乎不遗忘
        "preferences": 0.9,  # 偏好较持久
        "entities": 0.7,     # 实体信息中等
        "events": 0.6,       # 事件随时间衰减
        "cases": 0.5,        # 案例中等衰减
        "patterns": 0.8,     # 模式较持久
        "tools": 0.7,        # 工具知识中等
        "skills": 0.8,       # 技能较持久
    }
    category_factor = category_weights.get(memory.category, 0.5)
    
    # 加权综合
    score = (
        0.30 * time_decay +
        0.20 * frequency_factor +
        0.25 * importance_factor +
        0.10 * reference_factor +
        0.15 * category_factor
    )
    return score
```

### 4.4 遗忘触发策略

#### 4.4.1 定期衰减扫描

```python
# 每日执行一次
def daily_decay_scan():
    all_memories = memory_store.get_all(status="active")
    for memory in all_memories:
        score = calculate_memory_score(memory)
        
        if score < FADING_THRESHOLD:      # 0.3
            memory.status = "fading"
            # 从 L1/L2 索引中移除，仅保留 L0
            vector_index.downgrade(memory.uri, keep_layers=["L0"])
            
        elif score < DORMANT_THRESHOLD:    # 0.1
            memory.status = "dormant"
            # 从所有索引中移除
            vector_index.remove(memory.uri)
```

#### 4.4.2 容量驱动遗忘

当记忆总量超过预设上限时，按分数从低到高淘汰：

```python
def capacity_driven_forget(max_memories: int):
    total = memory_store.count(status__in=["active", "fading"])
    if total <= max_memories:
        return
    
    overflow = total - max_memories
    # 按分数升序获取最低分记忆
    lowest = memory_store.get_lowest_scored(limit=overflow)
    for memory in lowest:
        if memory.status == "fading":
            memory.status = "dormant"
        elif memory.status == "active":
            memory.status = "fading"
```

#### 4.4.3 矛盾驱动遗忘

当新记忆与旧记忆矛盾时，旧记忆标记为过时：

```python
def contradiction_driven_forget(new_memory, conflicting_memories):
    for old_memory in conflicting_memories:
        old_memory.status = "archived"
        old_memory.archived_reason = "contradicted_by"
        old_memory.archived_by = new_memory.uri
        # 在新记忆中记录替代关系
        new_memory.replaces = old_memory.uri
```

### 4.5 遗忘保护机制

某些记忆不应被遗忘：

| 保护规则 | 说明 |
|:---|:---|
| 用户画像保护 | profile 类记忆 score 下限为 0.5，不会进入 dormant |
| 关键决策保护 | importance=critical 的记忆不参与自动遗忘 |
| 近期保护 | 7 天内创建的记忆不参与衰减扫描 |
| 高引用保护 | reference_count > 5 的记忆 score 下限为 0.4 |
| 用户标记保护 | 用户手动标记为"重要"的记忆永不遗忘 |

## 五、记忆反思修正机制

> 这是本方案相对于 OpenViking 的重要扩展。OpenViking 的记忆是"只写不改"或"简单合并"，缺乏对已有记忆的主动审视和修正能力。

### 5.1 设计理念

反思修正模拟人类的元认知能力：
- 定期审视已有记忆的准确性和一致性
- 发现矛盾时主动修正
- 根据新信息更新过时认知
- 从失败中提取教训，修正错误模式

### 5.2 反思触发时机

```
┌─────────────────────────────────────────────────────────┐
│                  反思触发时机                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 会话结束反思（每次会话 commit 后）                     │
│     → 检查本次会话是否产生了与已有记忆矛盾的信息            │
│                                                         │
│  2. 冲突检测反思（记忆写入时）                             │
│     → 新记忆与已有记忆语义冲突时触发                       │
│                                                         │
│  3. 定期全局反思（每周一次）                               │
│     → 全量扫描记忆一致性，合并碎片化记忆                    │
│                                                         │
│  4. 失败驱动反思（任务失败后）                             │
│     → 分析失败原因，检查是否因错误记忆导致                  │
│                                                         │
│  5. 用户反馈反思（用户纠正 Agent 时）                      │
│     → 用户说"不对"、"错了"、"我改主意了"时触发             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.3 会话结束反思

每次会话 commit 后，异步执行反思流程：

```python
async def post_session_reflection(session, extracted_memories):
    # 1. 收集本次会话提取的新记忆
    new_memories = extracted_memories
    
    # 2. 对每条新记忆，检索可能冲突的已有记忆
    for new_mem in new_memories:
        related = await vector_index.search(
            new_mem.embedding, 
            top_k=10,
            exclude_session=session.id
        )
        
        # 3. LLM 判断是否存在矛盾
        conflicts = await detect_conflicts(new_mem, related)
        
        for conflict in conflicts:
            # 4. 决定修正策略
            resolution = await resolve_conflict(new_mem, conflict)
            
            if resolution.action == "update_old":
                # 更新旧记忆
                await update_memory(conflict.old_memory, resolution.updated_content)
                await log_reflection(
                    type="conflict_resolution",
                    old=conflict.old_memory,
                    new=new_mem,
                    action="updated_old"
                )
            elif resolution.action == "archive_old":
                # 归档旧记忆
                await archive_memory(conflict.old_memory, reason=f"superseded by {new_mem.uri}")
            elif resolution.action == "keep_both":
                # 两者都保留，添加关联标注
                await link_memories(new_mem, conflict.old_memory, relation="related_view")
```

### 5.4 冲突检测算法

```python
async def detect_conflicts(new_memory, candidates) -> list:
    """使用 LLM 检测新记忆与候选记忆之间的矛盾"""
    
    conflicts = []
    # 先用语义相似度粗筛
    high_similarity = [c for c in candidates if c.similarity > 0.7]
    
    if not high_similarity:
        return conflicts
    
    # LLM 精细判断
    prompt = f"""
请判断以下新记忆与已有记忆之间是否存在矛盾或需要更新的关系。

新记忆：
{new_memory.content}

已有记忆：
{format_candidates(high_similarity)}

对每条已有记忆，判断：
1. "no_conflict" - 无矛盾，信息互补
2. "contradiction" - 直接矛盾（如偏好改变、事实更正）
3. "evolution" - 信息演进（旧信息过时但不矛盾）
4. "duplicate" - 重复信息

输出 JSON 格式：
[{{"memory_uri": "...", "relation": "...", "explanation": "..."}}]
"""
    result = await llm.generate(prompt)
    
    for item in parse_json(result):
        if item["relation"] in ["contradiction", "evolution"]:
            conflicts.append(ConflictInfo(
                old_memory=find_by_uri(high_similarity, item["memory_uri"]),
                relation=item["relation"],
                explanation=item["explanation"]
            ))
    
    return conflicts
```

### 5.5 冲突解决策略

```python
async def resolve_conflict(new_memory, conflict) -> Resolution:
    """决定如何解决记忆冲突"""
    
    prompt = f"""
检测到记忆冲突，请决定最佳解决方案。

新记忆（来自最近会话）：
{new_memory.content}
置信度: {new_memory.confidence}

旧记忆：
{conflict.old_memory.content}
创建时间: {conflict.old_memory.created_at}
访问次数: {conflict.old_memory.access_count}

冲突类型: {conflict.relation}
冲突说明: {conflict.explanation}

请选择解决方案：
1. "update_old" - 用新信息更新旧记忆（适用于偏好变更、信息更正）
   → 提供更新后的内容
2. "archive_old" - 归档旧记忆，保留新记忆（适用于完全替代）
3. "keep_both" - 两者都保留（适用于不同视角/时间点的有效信息）
4. "discard_new" - 丢弃新记忆（适用于新信息置信度低于旧信息）

输出 JSON：
{{"action": "...", "updated_content": "...(仅 update_old 时需要)", "reason": "..."}}
"""
    result = await llm.generate(prompt)
    return Resolution.from_json(parse_json(result))
```

### 5.6 定期全局反思

每周执行一次全局记忆审计：

```python
async def weekly_global_reflection():
    """全局记忆一致性审计"""

    for category in MEMORY_CATEGORIES:

        clusters = cluster_by_similarity(memories, threshold=0.8)

                merged = await merge_memories(cluster)
                if merged:
                    await write_memory(merged)
                    for old in cluster:
                        await archive_memory(old, reason="merged")
    
    # 2. 一致性检查
    # 检查 profile 与 preferences 是否一致

    preferences = memory_fs.ls("user/memories/preferences/", layer="L1")
    
    inconsistencies = await check_consistency(profile, preferences)
    for issue in inconsistencies:
        await create_reflection_task(issue)
    
    # 3. 过时检测
    # 标记超过 90 天未访问且低分的记忆
    stale = memory_store.get_all(

        score_below=FADING_THRESHOLD

### 5.7 失败驱动反思

当 Agent 任务失败时，分析是否因错误记忆导致：

```python
async def failure_driven_reflection(task, error):
    """任务失败后的记忆反思"""
    
    # 1. 收集本次任务中被召回的记忆
    recalled_memories = task.get_recalled_memories()
    
    # 2. LLM 分析失败原因
    prompt = f"""
任务执行失败，请分析是否与召回的记忆有关。

任务描述: {task.description}
错误信息: {error}
执行过程: {task.execution_log}

本次召回的记忆:
{format_memories(recalled_memories)}

请分析：
1. 失败是否与某条召回的记忆有关？
2. 如果有关，该记忆是否需要修正？
3. 是否需要新增记忆来避免未来类似失败？

输出 JSON：
{{
  "memory_related": true/false,
  "problematic_memories": [{{"uri": "...", "issue": "...", "fix": "..."}}],
  "new_memories_needed": [{{"category": "...", "content": "..."}}]
}}
"""
    analysis = await llm.generate(prompt)
    result = parse_json(analysis)
    
    # 3. 执行修正
    if result["memory_related"]:
        for pm in result["problematic_memories"]:
            memory = memory_store.get(pm["uri"])
            memory.confidence *= 0.5  # 降低置信度
            if pm["fix"]:
                await update_memory(memory, pm["fix"])
    
    # 4. 新增教训记忆
    for nm in result["new_memories_needed"]:
        await commit_memory(
            category=nm["category"],
            content=nm["content"],
            importance="high",
            source="failure_reflection"
        )
```
