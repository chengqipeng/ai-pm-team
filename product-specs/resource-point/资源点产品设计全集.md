# 第一部分：智能资源点计费体系（核心设计）

> 来源：Wiki - 智能资源点计费 (ID:148137766)

## 1. 业务背景

销售易智能产品体系深度依赖第三方能力支撑，包括大模型调用、OCR识别、语音转文字、网络检索、文档解析等。使用该类第三方能力需向供应商支付服务费用，因此销售易将第三方能力包装为可售卖的流量资源包，客户使用智能产品时需额外采购对应资源包。

当前流量资源包采用"按能力类型拆分"模式，即每种第三方资源对应专属资源包（如会话Token包、OCR识别包、文档解析包等），客户需单独采购、销售需拆分核算。

随着智能产品功能迭代（如多模态、OCR服务扩展），第三方能力接入种类持续增加，原有分散式流量计费模式已无法适配业务发展需求，亟需构建统一、高效、灵活的流量计费体系。

## 2. 业务现状

目前销售易已上线6类流量资源包，采用"单能力单包"模式：
- 会话Token
- 文档解析页数
- 语音识别时长
- 实时语音识别（大模型版）
- 网络检索次数
- 营销推荐次数

现状总结：流量资源按第三方能力拆分，计量单位不统一；销售需按场景匹配资源包，客户需逐一进行采购；新第三方能力上线需单独配置资源包并走上架流程。

## 3. 业务痛点

- **客户与销售侧**：采购报价效率低，认知成本高。销售与客户均无法清晰对应"业务场景-流量资源"。
- **业务侧**：资源管控弱，易引发业务风险。各类流量资源独立统计，任何一种资源用尽都会直接导致依赖该资源的业务无法运行。
- **运营与研发侧**：资源管理效率低，适配性差。新第三方能力上线需单独设计资源包、走CRM上架流程。

## 4. 业务目标

核心目标：构建"统一计量、便捷采购、透明管控、灵活适配"的流量计费体系。

## 5. 核心设计思路

- **统一流量单位**：将所有第三方资源封装为「智能资源点」，作为唯一采购、消耗、统计单位，且单个资源点对应固定人民币价值
- **便捷化**：简化客户采购决策、销售报价核算、运营资源配置流程
- **全场景适配**：覆盖所有智能产品场景，支持新第三方资源快速接入
- **可视化管理**：提供资源消耗看板、明细日志
- **弹性适配**：支持第三方价格动态调整，通过换算比例配置实现弹性映射

## 6. 资源点设计

- 基础单元：智能资源点
- 单位价值（暂定）：1智能资源点 = 0.01人民币
- 资源包和License包含的资源点数：销售售卖的单位
- 通过调整资源包的售价来平衡阶梯价格、公司利润和客户折扣
- 通过资源点 - 原子能力折算关系平衡原子能力价格波动

## 7. 功能设计

### 7.1 License操作说明（P0）

智能产品的License包装及单个License所含资源点数量，受市场行情、销售及营销策略等多重因素影响，会进行周期性或临时性调整。因此，销售运营人员在Console中操作License相关流程时，系统不自动变更租户的资源点数量，相关资源点调整需在Console的智能资源管理模块中手动完成。

### 7.2 Console充值（P0）

业务规则：销售运营人员可在Console后台为客户租户充值智能资源点。

列表页显示字段：
- 充值类型：License开通/License增购/租户续约/资源包增购
- 资源点数量：本次充值的资源点数量
- 剩余数量：本次充值的资源点剩余可用数量
- 到期日期：本次充值的资源点到期日期
- 充值时间、最后操作人、充值说明

充值页面交互：
- 租户ID：自动显示
- 租户名称：自动显示
- 当前可用资源点余量：根据租户ID查询
- 充值类型：单选必填（License开通/License增购/租户续约/资源包增购）
- 购买时长：License类充值时必填，支持两位小数，单位为月
- 资源点到期日期：License开通/增购时自动读取不可编辑；续约/资源包增购时手动填写
- 开通License：支持选择多个License，输入数量后自动计算资源点数
- 计算公式：License包含的资源点数 × License数量 × 购买时长
- 本次充值资源点数：自动汇总，不可编辑
- 充值后可用余量：当前可用余量 + 本次充值资源点数量

### 7.3 初始化（P0）

租户开通Neo Agent平台License时，不再赠送资源用量，也不赠送资源点。

### 7.4 运行时资源点控制（P0）

- 流量校验：运行时需先校验当前租户智能资源点是否有剩余
- 自动扣减逻辑：根据原始资源与资源点换算比例，将实际消耗换算为资源点，从租户剩余流量账户中扣减
- 扣减顺序：优先扣减到期时间最早的充值资源包
- 资源点为租户级共享
- 资源点不足提示：「资源点用量已达到系统最大上限，暂时无法处理您的请求，请联系系统管理员增购资源点，谢谢。」

使用流水核心字段：
- 来源类型：Agent、组件、资源变更等
- 资源点用量：换算后扣除的资源点数量
- 原始资源类型：会话Token、网络检索等
- 原始资源用量：原子级能力实际使用量
- 来源描述：使用说明

### 7.5 资源点与原始资源换算（P0）

- 换算比例配置：产研可在后台配置各类第三方能力与资源点的换算比例（支持动态调整）
- 比例生效规则：新比例仅对后续流量消耗生效，历史消耗记录仍按旧比例计算

### 7.6 资源到期（P0）

- 系统每天定时检查租户资源包是否到期
- 到期规则：充值流水记录的到期时间 = 昨天
- 到期后自动生成资源到期类型的使用流水，并扣减租户可用余额

### 7.7 系统资源用量（P0）

- 显示当前智能资源点总量和剩余量
- 历史资源（Token、网络检索次数等）余量不再显示
- 资源点的用量显示不依赖智能平台License，只要租户充值了资源点即显示

### 7.8 资源告警（P0）

- 资源用量告警的资源类型可选择智能资源点
- 资源告警默认数据自动初始化智能资源点告警提示

### 7.9 充值流水（P1）

- 系统资源用量页面新增资源充值历史Tab
- 充值历史列表显示：充值资源类型、充值类型、充值数量、剩余数量、到期时间、状态
- 资源与资源点比例展示

### 7.10 流量看板（P0）

- 流量看板可统计智能资源点消耗和原始资源消耗
- 最佳实践看板中增加对资源点消耗的统计，原始资源消耗的统计继续保留

### 7.11 沙盒资源调配（P0）

- 沙盒环境资源调配页面支持智能资源点调配
- 源环境扣减时优先扣减最先到期的资源包
- 若一个包不足以扣减，按照到期时间顺序依次扣减多个资源包
- 调配成功后在目标环境生成充值流水
- P1：沙盒删除时自动将沙盒中的资源点调配到生产环境

## 8. 日志设计

AI执行信息表核心字段：

| 字段 | 类型 | 描述 |
|:---|:---|:---|
| 来源类型 | 单选 | Agent、Prompt Template、MCP、组件、知识库同步、资源变更、其他 |
| 资源类型 | 单选 | 大模型调用、网络检索、录音转文本、文档解析、OCR-名片、OCR-通用文字识别 |
| 资源点用量 | 实数 | 精确到小数点后8位 |
| 资源点剩余量 | 实数 | 精确到小数点后8位 |
| 资源用量 | 整数 | 原始资源用量 |

## 9. 历史数据处理

### 处理范围
- 资源总量：将历史租户充值的资源转换成资源点
- 租户资源余量：将租户剩余的各种类型资源折算为资源点
- 历史流量日志不处理

### 折算规则
- 折算原则：折算前后租户可以执行业务的次数保持不变
- 折算公式：折算资源点数 = 剩余流量（原计量单位） × 该类型资源对应的资源点比例
- 折算后的资源点到期时间：与当前租户到期时间保持一致
- 历史资源折算后生成充值记录

# 第二部分：资源点元数据设计

> 来源：Wiki - 资源点元数据 (ID:149870730)

## 设计需求

1. **计费**：每种类型的每种资源都需要单独定价、折算为资源点
2. **客户接入自己资源**：在支持海外和私有化的场景下，各种资源都需要支持用户使用自己的资源
3. **各场景使用资源注册**：接入客户资源后需要支持某些场景替换资源

## 设计原则

1. 尽量复用大模型支持自定义时的模型
2. 资源点需要快速上线，设计上需要支持接入客户资源和资源注册

# 第五部分：资源点能力增强

> 来源：Wiki - 资源点能力增强 (ID:152080378)

## 1. 沙盒删除时资源点自动转移至生产环境

当用户执行删除沙盒操作时，自动将沙盒环境的全部剩余资源点转移至生产租户。资源点转移过程无需用户手动操作，系统自动完成。

- 在生产环境产生转入记录，资源点总量、可用量随之增加
- 在沙盒环境产生转出记录，资源点总量、可用量自动扣减
- 删除沙盒时的提示信息更新为：「确认要删除该沙盒吗？沙盒删除后将无法恢复，沙盒内剩余资源点将自动转移至生产环境。」

- 充值时间

- 充值数量
- 当前剩余数量
- 到期时间
- 状态：可用/已过期
- 列表按充值时间倒序显示

**Tab 2：消耗规则**
- 资源名称：资源与资源点比例表的类别列
- 消耗资源点：资源与资源点比例表的消耗资源点列

- 服务云产品开通License时，包含资源点规则为开通License包含XX资源点/租户/年

# 第六部分：Console资源点充值优化 — 新增服务类License

> 来源：Wiki - 资源点充值优化-PRD (ID:152099858) + TCHub - Console资源点充值-方案设计

## 一句话方案

在Console资源点充值页面的License下拉列表中新增4个服务类License（客服机器人Agent、智能质检Agent的基础版/旗舰版），每个License包含固定的月资源点数，充值交互和计算逻辑保持现状不变。

## 1. 现状与问题

Console充值页面已支持按License充值和按资源包充值两种模式，License下拉列表包含10个销售/渠道/分析/工单类License。客服机器人Agent和智能质检Agent已商业化，但其License未出现在充值页面的下拉列表中，运营无法通过标准流程充值。

## 2. 新增License配置

| License名称 | 包含资源点数（点/License/月） |
|:---|---:|
| 客服机器人Agent 基础版 | 8,333.33333 |
| 客服机器人Agent 旗舰版 | 16,666.66666 |
| 智能质检Agent 基础版 | 20,833.33333 |
| 智能质检Agent 旗舰版 | 41,666.66666 |

业务规则：
- BR-01：新增License与现有License使用完全相同的充值交互和计算逻辑
- BR-02：充值计算公式不变：包含资源点数 × 数量 × 购买时长(月)
- BR-03：计算结果如存在小数点，按个位向上取整（ceil）

## 3. 计算示例

开通1个客服机器人Agent 基础版，购买时长12个月：
```
充值资源点 = 8,333.33333 × 1 × 12 = 99,999.99996 → 向上取整 = 100,000
```

开通1个智能质检Agent 旗舰版，购买时长6个月：
```
充值资源点 = 41,666.66666 × 1 × 6 = 249,999.99996 → 向上取整 = 250,000
```

## 4. 验收标准

| 编号 | 验收项 | 验证方法 |
|:---|:---|:---|
| AC-01~04 | License下拉列表包含4个新增License | UI验证 |
| AC-05 | 客服机器人Agent 基础版 × 1 × 12月 = 100,000 | 计算验证 |
| AC-06 | 客服机器人Agent 旗舰版 × 1 × 12月 = 200,000 | 计算验证 |
| AC-07 | 智能质检Agent 基础版 × 1 × 12月 = 250,000 | 计算验证 |
| AC-08 | 智能质检Agent 旗舰版 × 1 × 12月 = 500,000 | 计算验证 |
| AC-09 | 现有License充值交互和计算逻辑不受影响 | 回归测试 |
| AC-10 | 资源包增购交互和计算逻辑不受影响 | 回归测试 |
| AC-11 | 充值成功后，租户资源点余额正确增加 | 数据验证 |

# 第七部分：Console资源点充值逻辑优化 — per_tenant计费模式

> 来源：TCHub - 产品-PRD：Console资源点充值逻辑优化 + 产品-方案设计

## 一句话方案

在Console资源点充值页面中新增"按租户"计费模式的交互支持，选择per_tenant类型License后隐藏数量输入框，系统自动按年资源点折算月资源点并向上取整。

## 1. 三种计费模式

| 计费模式 | billingMode | 适用License | 本次改动 |
|:---|:---|:---|:---|
| 按License（用户数） | per_license | 销售Agent、销售云AI智能旗舰版、渠道Agent、分析师Agent、工单Agent、智能坐席辅助 | 保持现状 |
| 按租户 | per_tenant | 客服机器人Agent 基础版/旗舰版、智能质检Agent 基础版/旗舰版 | **本次新增** |
| 按资源包 | per_pack | 资源包增购-2万点、资源包增购-2千点 | 保持现状 |

## 2. License元数据扩展

| 字段名 | apiName | 类型 | 说明 |
|:---|:---|:---|:---|
| 计费模式 | billingMode | enum(per_license, per_tenant, per_pack) | License元数据新增字段 |
| 月资源点数 | resourcePointsPerMonth | integer | per_license时必填 |
| 年资源点数 | resourcePointsPerYear | integer | per_tenant时必填 |
| 每包资源点数 | resourcePointsPerPack | integer | per_pack时必填 |

## 3. per_tenant交互规则（核心改造）

- BR-07：选择per_tenant类型License后，隐藏"数量"输入框
- BR-08：资源点描述文案显示为"包含 {resourcePointsPerYear} 资源点"
- BR-09：计算公式为 `ceil(resourcePointsPerYear / 12 × duration_months)`，向上取整
- BR-10：修改购买时长后，实时重新计算参考资源点数
- BR-11：同一License在同一次充值中仅允许添加一次
- BR-12：支持在同一次充值中混合添加per_license和per_tenant类型License

交互示意：

per_license（保持现状）：
```
开通License：  [销售Agent-基础版 v]  数量 [  10  ]  包含 500 资源点    参考资源点数：5,000
```

per_tenant（本次新增）：
```
开通License：  [客服机器人Agent 基础版 v]  包含 100,000 资源点    参考资源点数：66,667
```

## 4. 充值类型与计费模式的关系

| 充值类型 | 可选License的billingMode |
|:---|:---|
| License开通 | per_license + per_tenant |
| License增购 | per_license + per_tenant |
| 租户续约 | per_license + per_tenant |
| 资源包增购 | 不显示License下拉 |

## 5. 计算公式汇总

按License（per_license）：
```
充值资源点 = resourcePointsPerMonth × quantity × duration_months
```

按租户（per_tenant）：
```
充值资源点 = ceil(resourcePointsPerYear / 12 × duration_months)
```

按资源包（per_pack）：
```
充值资源点 = pack_quantity × resourcePointsPerPack
```

## 6. 业务场景

### 场景1：per_tenant License开通充值（8个月）
- 选择"客服机器人Agent 基础版"
- 隐藏数量输入框，显示"包含 100,000 资源点"
- 参考资源点数：ceil(100,000/12 × 8) = 66,667

### 场景2：per_tenant License 12个月充值
- 选择"智能质检Agent 基础版"
- 参考资源点数：ceil(250,000/12 × 12) = 250,000

### 场景3：混合per_license + per_tenant续约充值
- 销售Agent-旗舰版 × 3 × 12月 = 72,000
- 智能质检Agent 基础版 × 12月 = 250,000
- 合计：322,000

## 7. 错误处理

| 错误场景 | 处理策略 |
|:---|:---|
| License元数据缺少billingMode字段 | 默认按per_license处理 |
| per_tenant的resourcePointsPerYear为空或0 | 参考资源点数显示为0，禁用确定充值按钮 |
| 同一License重复添加 | 前端拦截，提示"该License已添加，不可重复选择" |

## 8. 验收标准

| 编号 | 验收项 |
|:---|:---|
| AC-01 | 选择per_tenant类型License后，数量输入框隐藏 |
| AC-02 | per_tenant类型显示"包含 X 资源点"文案 |
| AC-03 | 购买8个月客服机器人Agent 基础版，充值资源点 = 66,667 |
| AC-04 | 购买12个月客服机器人Agent 基础版，充值资源点 = 100,000 |
| AC-05 | 购买1个月客服机器人Agent 基础版，充值资源点 = 8,334 |
| AC-06 | 购买7个月智能质检Agent 旗舰版，充值资源点 = 291,667 |
| AC-07 | 混合添加per_license和per_tenant License，汇总正确 |
| AC-08 | 同一License不允许重复添加 |
| AC-09 | 购买时长变更后，per_tenant参考资源点数实时重新计算 |
| AC-10 | per_license类型交互和计算逻辑不受影响 |

# 第九部分：Console充值逻辑优化测试用例

> 来源：TCHub - 场景测试用例（35条）

### 用例统计

| 优先级 | 数量 | 说明 |
|:---|:---|:---|
| P0（功能测试） | 11 | 覆盖新增产品展示、小数资源点、BigDecimal精确运算、CEILING取整、前后端一致性、i18n |
| P1（边界测试） | 10 | 零值、极值、精度、混合计算、负数防御 |
| P1（异常测试） | 7 | 非法输入、并发、超时、i18n降级 |
| P2（回归测试） | 7 | 原有充值流程、消耗逻辑、历史记录、API兼容 |

### 重点关注

1. **CEILING取整策略**（P0-04/P0-07）：前后端一致性是关键验收点
2. **Double精度边界**（P1-07）：验证BigDecimal是否真正规避IEEE 754精度问题
3. **极大数量乘以小数资源点**（P1-05）：验证BigDecimal不溢出
4. **API向后兼容性**（P2-07）：Integer→Double类型变更对旧客户端的影响

### 风险点

1. 前后端取整时机差异：前端在汇总时Math.ceil，后端在写入时CEILING，可能产生±1差异
2. 历史数据兼容：resourcePoint从Integer改为Double，需确认数据库字段类型已同步变更
3. 下游消耗模块类型假设：如果下游模块仍假设resourcePoint为Integer，可能在类型转换时丢失精度
