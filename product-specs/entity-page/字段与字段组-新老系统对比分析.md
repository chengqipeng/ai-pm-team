## 一、字段管理（Item）对比

### 1.1 Admin CRUD 对比

| 能力 | 老系统 (XItemAction) | 新系统 (EntityItemsTab + ItemFormDialog) | 状态 |
|:-----|:---------------------|:----------------------------------------|:-----|
| 列表展示 | `item-list` → 按 entity 加载 + itemOrder 排序 + 系统/自定义区分 | EntityItemsTab 表格（#/名称/apiKey/类型/必填/状态/db_column/操作） | ✅ 对齐 |
| 新建弹框 | `popup-item-add` → 左侧字段类型选择器 + 右侧属性表单 + 全局选项集 + 布局分配 + 字段级权限 | ItemFormDialog → 左侧类型列表 + 右侧属性表单 | ⚠️ 差异见下 |
| 保存（新建） | `save-item` → itemTypeProperty 处理 + 动态默认值 + Markdown + 关联筛选 + 布局 + 权限 + pickOption + apiKey `__c` | batchSaveMetadata createMap.item + createMap.pickOption | ⚠️ 差异见下 |

### 1.3 字段删除引用检查（P0 差距）

老系统在删除字段前执行 6 种引用检查，新系统完全缺失：

| # | 检查项 | 老系统错误码 | 说明 | 新系统 |
|:--|:-------|:------------|:-----|:-------|
| 1 | 被查重规则引用 | `ERROR_ITEM_USED_BY_DUPLICATERULE` | 字段被查重规则的 criteria 引用 | ❌ 无 |
| 2 | 被计算公式引用 | `ERROR_ITEM_USED_BY_COMPUTEFORMULA` | 字段被公式计算的表达式引用 | ❌ 无 |
| 3 | 被汇总聚合引用 | `ERROR_ITEM_USED_BY_COMPUTEAGGREGATE` | 字段被汇总计算引用 | ❌ 无 |
| 4 | 被 JOIN 引用 | `ERROR_ITEM_USED_BY_JOIN` | 字段被引用类型字段引用 | ❌ 无 |
| 5 | 被关联筛选引用 | `ERROR_ITEM_USED_BY_REFERFILTER` | 字段被关联过滤条件引用 | ❌ 无 |
| 6 | 通用引用检查 | `ERROR_ITEM_IS_USED` | 其他引用（校验规则等） | ❌ 无 |

## 二、字段组（FieldSet）对比

### 2.1 Admin CRUD 对比

| 能力 | 老系统 (FieldSetAction) | 新系统 (EntityFieldSetsTab + FieldSetFormDialog) | 状态 |
|:-----|:------------------------|:------------------------------------------------|:-----|
| 列表展示 | `list` → 字段组列表 + 创建人/时间 | EntityFieldSetsTab 表格 | ✅ 对齐 |
| 新建 | `save-fieldset` → apiKey `__c` + 重复检查 + 成员 | FieldSetFormDialog → batchSaveMetadata | ⚠️ 差异见下 |

| 删除 | `delete` → xFieldSetAppService.deleteFieldSet（级联删除成员） | batchSaveMetadata deleteMap（软删除） | ⚠️ 差异 |
| 成员管理 | `list-all-items` → 左右穿梭框（已有/可选） + 排序 | 勾选列表 + 上下箭头排序 | ✅ 基本对齐 |

### 2.2 差异明细

| # | 差异点 | 老系统 | 新系统 | 优先级 |
|:--|:-------|:-------|:-------|:-------|
| FS-1 | apiKey 重复检查 | `isApiKeyExistFlg` 前置校验 | 依赖 batchSave 唯一约束 | P1 |
| FS-2 | 特殊字段组限制 | 产品 picker 字段集限制 ≤10 个字段 + 只允许特定类型 | 无限制 | P2 |
| FS-3 | 交易管理字段过滤 | O2C 实体的字段组只允许 INTEGER/REAL/PERCENTAGE 类型 | 无过滤 | P2 |
| FS-4 | 复制场景字段组 | `fieldsNotCopy` 字段组限制不能包含标准字段 | 无此逻辑 | P2 |
| FS-5 | 编辑模式成员 diff | 老系统直接替换全部成员 | 新系统有 TODO 注释，编辑模式成员变更未完成 | ❌ P0 |
| FS-6 | 字段过滤 | `isNeedFilter` 过滤不适合的字段（关联客户的 REFER/JOIN、联系人角色等） | 仅过滤 deleteFlg/enableFlg | ⚠️ P2 |
| FS-7 | 删除方式 | 物理删除（`deleteFieldSet`） | 软删除（`deleteFlg=1`） | ⚠️ 语义差异 |
| FS-8 | 成员排序 | `fieldSetItemOrder` 排序 + 前端拖拽 | 上下箭头调整顺序 | ✅ 对齐 |

### 2.3 编辑模式成员管理（P0 差距）

FieldSetFormDialog 中有明确的 TODO 注释：
```typescript
// TODO: 成员变更的 diff 计算（对比 selectedKeys 与数据库，生成 createMap + deleteMap）
```

老系统的做法是：编辑时直接传入完整的成员列表，后端 `updateFieldSet` 方法会：
1. 删除旧的所有 fieldSetItem
2. 创建新的 fieldSetItem 列表

新系统需要实现类似逻辑。

## 四、关键差距总结

### P0 必须修复

| # | 问题 | 影响 | 修复方案 |

| 1 | 字段删除无引用检查 | 删除被引用的字段会导致公式/查重/汇总等功能异常 | 后端 batchSave 的 deleteMap.item 前置校验：查询 duplicateRuleCriteria/formulaCompute/aggregationCompute/entityLink(JOIN)/referFilter/checkRule 中是否引用该字段 |
| 2 | 字段组编辑模式成员变更未实现 | 编辑字段组时无法修改成员 | 实现 diff 逻辑：对比 selectedKeys 与数据库，生成 deleteMap.fieldSetItem + createMap.fieldSetItem |

| 6 | apiKey 自动生成 | 根据 label 拼音自动生成 apiKey 建议值 |
| 7 | 关联筛选(referFilter) | REFER 类型字段的关联过滤条件配置 |

### P2 后续优化

## 五、设计体系一致性评估

### ✅ 一致的部分

1. **元数据存储**：字段通过 `p_common_metadata` / `p_tenant_item` 大宽表存储，dbc 列映射完整（已验证 21 个属性全部有映射）
2. **类型专属属性**：新系统已将 `itemTypeProperty` JSON 拆分为独立 dbc 列字段，比老系统更规范
3. **CRUD 统一入口**：新系统通过 `batchSaveMetadata` 统一处理，支持事务内多表操作
4. **apiKey `__c` 后缀**：自定义字段/字段组统一追加后缀
5. **字段组成员模型**：`FieldSetItem` 的 `fieldSetApiKey` + `itemApiKey` + `itemOrder` 与老系统一致
6. **选项值管理**：SELECT/MULTI_SELECT 的 pickOption CRUD 与老系统一致

### ⚠️ 需要注意的差异

1. **引用检查缺失**：这是最大的 P0 差距，老系统有 6 种引用检查，新系统完全没有
2. **字段类型覆盖**：新系统缺少 JOIN(26)/FORMULA(27)/LOCALIZATION(32)/MULTIREF(34)/MULTIVALUE_REFER(41) 共 5 种类型
3. **标准字段保护**：老系统严格区分标准/自定义字段的编辑权限，新系统未区分
4. **字段组编辑**：编辑模式的成员变更逻辑未完成（有 TODO 注释）
