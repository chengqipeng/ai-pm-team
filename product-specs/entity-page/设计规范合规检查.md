## 二、多选字段存储类型（重要）

### 规范要求
新系统多选字段（MULTI_SELECT=16）使用 **PG 原生数组类型 `VARCHAR(300)[]`**，存储在 `dbc_array` 列中。
详见 `product-specs/metadata-system/多选字段数组类型设计方案.md`。

### 当前问题
前端 `fieldRenderers.tsx` 中 MULTI_SELECT 的渲染逻辑：
```typescript
case ItemType.MULTI_SELECT:
case ItemType.MULTI_TAG: {
  const arr = Array.isArray(value) ? value : String(value).split(',')
  // ...
}
```

`String(value).split(',')` 是老系统的逗号分隔字符串处理方式。新系统后端返回的是 `List<String>`（JSON 数组），前端 Axios 拦截器转换后就是 `string[]`，不需要 split。

**修复**：
```typescript
case ItemType.MULTI_SELECT: {
  // 新系统后端返回 string[]（PG TEXT[] 数组），不需要 split
  const arr = Array.isArray(value) ? value : [String(value)]
  // ...
}
```

## 三、ID 关联 vs apiKey 关联

### 规范要求
> 元数据之间的关联禁止使用 ID（如 entity_id、item_id），统一使用 api_key 关联。

### 检查结果

| 位置 | 当前写法 | 是否合规 | 说明 |
|:-----|:---------|:---------|:-----|
| 详细设计 - 校验规则 `checkErrorItemApiKey` | apiKey 关联 | ✅ | 正确 |
| 详细设计 - 查重条件 `itemApiKey` | apiKey 关联 | ✅ | 正确 |
| 详细设计 - 字段组成员 `itemApiKey` | apiKey 关联 | ✅ | 正确 |
| 详细设计 - 查重条件 `ruleApiKey` | apiKey 关联 | ✅ | 正确 |
| 详细设计 - 字段组成员 `fieldSetApiKey` | apiKey 关联 | ✅ | 正确 |
| 前端 `BizDataRecord.id` | 使用 number id | ⚠️ | 业务数据用 id 是正确的（雪花 ID），元数据用 apiKey |
| 前端 `EntityFormDialog` record.id | 使用 number id | ⚠️ | 同上，业务数据 CRUD 用 id 正确 |
| 前端 `deleteBizData(entityApiKey, record.id)` | id 参数 | ✅ | 业务数据删除用 id 正确 |

**结论**：详细设计中的元数据关联全部使用 apiKey，合规。业务数据 CRUD 使用雪花 id，也是正确的。
