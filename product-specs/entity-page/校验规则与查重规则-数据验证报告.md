## 一、校验规则（checkRule）— 1 条

| 字段 | 值 |
|:-----|:---|
| entity | order（订单） |
| apiKey | lineItemCountValidationRule |
| label | 明细数量校验规则 |
| activeFlg | 1（激活） |
| checkFormula | `AND(ISNULL(order.originalOrderId.order.createdAt),order.lineItemCount<=0)` |
| checkErrorMsg | 至少添加一条明细数据 |
| checkErrorMsgKey | XdMDChkRulErr.order.lineItemCountValidationRule |
| checkAllItemsFlg | 0 |

### CheckRuleEngine 验证

**问题 1**：公式中使用了跨实体引用 `order.originalOrderId.order.createdAt`，这是老系统的关联字段引用语法。新系统的 `ComputeUtils.evaluate` 需要支持这种跨实体引用，否则会执行异常。

**当前引擎行为**：公式执行异常时 `catch(Exception)` 记录 warn 日志并跳过，不阻断业务。这是正确的容错策略。

**问题 2**：`checkFormula` 存储在 `dbc_textarea1`（不是 `dbc_varchar1`）。CheckRuleEngine 中通过 `mergeReadService.listMergedByEntityApiKey` 加载后，CommonMetadataConverter 会根据 p_meta_item 的 db_column 映射自动将 `dbc_textarea1` 映射为 `checkFormula` 字段。✅ 正确。

### DuplicateRuleEngine 验证

**问题 1：activeFlg 全部为 None**

所有查重规则的 `activeFlg`（dbc_smallint1）都是 None。引擎中检查 `activeFlg == 1`，None 不等于 1，所以**所有查重规则都不会执行**。

**修复**：引擎应该将 `activeFlg=None` 视为激活（老系统默认激活），或者在 init_local_dev.py 中修正数据。

**问题 2：entity_api_key 为 None 的疑似查重规则**

4 条疑似查重规则的 `entity_api_key` 为 None，引擎通过 `listMergedByEntityApiKey` 加载时不会返回这些记录。✅ 正确（疑似查重是独立功能，不在标准查重流程中）。

**问题 3：criteria 中的 itemApiKey 是字段 apiKey 还是字段 ID？**

数据显示 criteria 的 `itemApiKey`（dbc_varchar2）值为 `accountName`、`name`、`productName` 等，这些是字段的 apiKey。✅ 正确，与引擎中 `data.get(itemApiKey)` 的取值方式一致。

**问题 4：priceBookEntry 的查重条件**

priceBookEntry 有 2 个条件（priceBookId + productId），criteriaLogic = `1 AND 2`。引擎当前实现是所有条件 AND 连接，没有解析 criteriaLogic 表达式。对于 `1 AND 2` 这种简单情况是正确的，但如果有 `1 OR 2` 或 `1 AND (2 OR 3)` 则不支持。
