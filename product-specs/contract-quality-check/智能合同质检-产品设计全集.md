## 一、业务全景

> 来源：Wiki [智能合同质检](https://wiki.ingageapp.com/pages/viewpage.action?pageId=148113396)

### 1.1 业务背景与核心问题

在销售易 CRM 的销售管理流程中，合同签订是 L2C（Lead to Cash）流程的关键环节。当前合同签订流程存在以下核心痛点：

**合同草拟与谈判高度依赖人力**
- 销售代表起草合同时，依赖个人经验判断内容完整性
- 法务审核条款时，需要逐字逐句阅读长文本，效率低下
- 销售经理审批时，难以快速识别关键风险点

**合同内容质量难以保障**
- 合同多为长文本（PDF/Word 附件），关键要素容易遗漏（如服务期、验收标准、税率等）
- 合同金额、客户信息等与 CRM 中的商机、报价单、客户数据不一致
- 条款风险难以识别：偏向对方的条款、与标准模板的偏离、违约责任不明确等

**核心问题**：如何在合同签订前，通过 AI 能力自动检查合同的完整性、风险性和可执行性，确保合同内容准确、条款合规、义务可落地，从而减少返工、缩短审核时间、降低履约风险？

### 1.2 核心目标

| 目标 | 指标 |
|:---|:---|
| 降低返工 | 合同返工率降低 30% |
| 提升效率 | 法务审核时间缩短 40% |
| 管控风险 | 合同合规率提升至 95% 以上 |

### 1.3 适用角色

| 角色 | 使用频率 | 核心诉求 | 主要场景 |
|:---|:---|:---|:---|
| 销售代表 | 每天使用 | 提交审批前自检，避免低级错误 | 手动触发质检，修改合同后重新质检 |
| 审批经理 | 每天使用 | 审批时辅助决策，关注风险提示 | 查看质检报告，根据风险评分决策 |
| 法务/财务 | 每天使用 | 审核时快速定位，处理非标条款 | 查看风险条款列表，对比标准模板 |
| 管理员 | 按需使用 | 配置质检规则，适配企业需求 | 配置必填字段、审查规则、标准模板 |

### 1.4 业务流程（结合自用租户）

#### 核心审批流程（纸质合同）

- **起草阶段（销售）**：在报价单阶段确定主要要素，在订单阶段提交合同草本
- **初审（上级/商务）**：直属上级审批，流转到商务匹配订单信息与合同
- **法务审核（法务）**：核心风控节点，确认合同条款
- **用印环节（商务）**：法务审完后流转回商务盖章
- **寄出与回传**：盖章后寄给客户，客户盖章后寄回
- **归档验收（财务）**：验收寄回的纸质合同，确认无误后归档

#### 智能合同质检介入节点（建议至少3次）

| 质检次数 | 场景 | AI作用 |
|:---|:---|:---|
| 第一次 | 销售上传草稿时（自检） | 一致性比对（合同vs CRM数据）+ 完整性检查 |
| 第二次 | 法务/商务审核与用印前（风控） | 风险识别 + 一致性复核 |
| 第三次 | 合同回传与财务归档时（防篡改） | 防篡改比对 + 归档确认 |

| 产品线 | 版本 | license名称 | function | 控制范围 |
|:---|:---|:---|:---|:---|

### 1.6 范围界定

**P0范围**
- 内容完整性检查
- 数据一致性检查（合同 vs CRM数据）
- 风险条款识别
- 历史版本查看（最多保留10个版本）
- 质检结果导出（PDF、Word）
- 多套配置模板
- 质检规则配置

**P1范围**
- 审查规则模板（预置模板按行业+业务模式+合同角色分类）
- 标准模板对比
- 合同版本比对
- 履约义务拆解
- 一键生成任务
- 批量质检报表
- 风险预警

### 1.7 用户故事

**销售代表**
- 起草完合同后，系统自动检查是否遗漏关键字段
- 提交审批前，系统自动对比合同金额、客户信息与CRM数据
- 起草合同时，系统提示哪些条款偏向客户、与标准模板偏离较大

**审批经理**
- 审批时快速查看质检报告（完整性评分、风险等级、关键差异）
- 批量查看团队合同的质量分布（V2）

**法务/财务**
- 审核时系统自动标记高风险条款
- 快速知道合同内容是否完整，与CRM核心内容是否一致

**管理员**
- 定义必填字段、审查规则库、标准合同模板
- 选择预置合同类型模板快速应用审查规则

## 二、前台交互

> 来源：Wiki [智能合同质检](https://wiki.ingageapp.com/pages/viewpage.action?pageId=148113396)
> UE设计：https://www.figma.com/design/xqOeGpz6bjNVVOAvX7ibbI/

### 2.1 入口位置与操作

- **位置**：实体详情页增加"合同质检"组件，放在"AI组件"分组下
- **适用范围**：任何实体详情页布局中可添加
- **权限**：拥有"智能合同质检" function 的用户可见
- **默认状态**：首次质检展示默认文案 + 选择合同附件按钮；质检后展示质检结果列表

### 2.2 选择质检合同附件流程

1. 点击"选择合同附件"/"重新质检"按钮，弹出选择对话框
2. 系统列出当前实体记录上的所有PDF/Word格式附件（含附件类型字段和相关文档组件）
3. 每个附件显示：名称、位置、上传时间、上传人、文件大小、最新质检时间、质检模板
4. 用户单选一个附件（不支持多选）
5. 已质检过的附件显示提示："该附件已质检过，是否重新质检？"
6. 自动匹配质检模板（根据实体、业务类型、职能）
   - 只有一个可用模板：直接开始质检
   - 无可用模板：toast提醒"暂无可用质检模板，请联系管理员"
   - 多个可用模板：进入模板选择列表手动选择

### 2.3 质检流程

后台调用合同质检触发器，启动以下流程：
1. OCR 识别服务：将合同附件转换为纯文本
2. 合同内容解析器：从OCR全文中提取关键结构化信息
3. CRM 数据查询器：查询关联的opportunity、quotation、account数据（管理员权限，忽略数据权限）
4. 内容完整性检查器：检查合同要素是否完整
5. 数据一致性检查器：对比合同与CRM数据的一致性
6. 风险条款识别器：识别高风险条款
7. 质检结果存储器：存储质检结果到子实体
8. 质检报告生成器：生成前端展示的质检报告

**质检中前台展示**：loading效果，轮播展示阶段：
- 正在识别合同内容...
- 正在提炼关键要素...
- 正在检查内容完整性...
- 正在检查数据一致性...
- 正在识别风险条款...
- 正在进行最终评估...

### 2.4 质检结果卡片（核心界面）

| 模块 | 内容 |
|:---|:---|
| 顶部操作栏 | 左侧提示文案 + 右侧"重新质检"按钮 |
| 01 检测结论 | 质检附件名称（可预览）、质检时间、质检人、综合评分（完整性/一致性/风险）、结论summary、操作（导出报告/重新质检/查看历史版本） |
| 02 完整性检查 | 罗列缺失字段（仅label），可展开查看已填写字段 |
| 03 一致性检查 | 对比展示合同原文与CRM系统数据（字段名称、合同值、CRM值、修改建议） |
| 04 风险条款识别 | 按风险等级分色展示（高/中/低），每条含：风险级别、条款名称、原文条款位置、风险说明、修改建议。高风险默认展开，中低风险默认折叠 |

**导出报告**：支持PDF和Word两种格式
**历史版本**：同一附件最多展示最新5个质检结果

### 2.5 数据权限

- 选择附件：只能选择自己有查看权限的附件
- 查看质检结果：只展示自己有查看权限附件的质检结果

### 2.6 异常处理

- 网络错误："网络连接出现问题，请检查后重试"
- 服务异常："质检服务暂时不可用，请稍后重试"
- 连续3次失败："质检服务异常，请联系管理员"

## 三、后台配置

> 来源：Wiki [智能合同质检](https://wiki.ingageapp.com/pages/viewpage.action?pageId=148113396)
> UE设计：https://www.figma.com/design/xqOeGpz6bjNVVOAvX7ibbI/

### 3.1 菜单位置

路径：系统设置 > NeoAgent > 销售智能管理 > 智能合同质检
权限：当前用户有后台"销售智能管理"function，且租户有"销售经理Agent-智能合同质检function"授权

### 3.2 模板列表

支持创建多个合同质检模板，每个模板包含：
- 模板名称（可重复）
- API名称（不允许重复）
- 适用的实体
- 模板描述
- 模板状态：启用/禁用（创建后默认禁用）
- 创建人、最近修改人、最近修改时间
- 操作：复制、删除、启用/禁用、模板分配

**模板分配**：根据实体的业务类型和用户职能进行分配

### 3.3 模板创建表单

**第一步：基本信息**
- 模板名称（必填）
- 适用实体（必填）
- API名称（必填）
- 模板描述（非必填）
- 创建方式：
  1. 上传合同模板智能提取（默认）：上传PDF/Word文档，AI自动提炼关键要素和风险条款
  2. 从预置质检方案创建（P1）
  3. 从空白方案直接创建

**第二步：详细模板信息配置**

**完整性和一致性规则**（最多30个字段）：

| 字段 | 说明 |
|:---|:---|
| 合同关键要素名称 | 要素标签 |
| API名称 | 唯一标识 |
| 字段类型 | 文本/金额/日期/数字/数组 |
| 必填校验 | 默认必填 |
| 一致性校验 | 默认不校验 |
| CRM映射 | 一致性校验为"是"时必填，支持实体字段或查询记录 |
| 提取指引与逻辑 | AI提取指导 |

**风险条款配置**（最多30个规则）：

| 字段 | 说明 |
|:---|:---|
| 审查类型 | 分组聚合（如商业条款、法律责任、LTC合规等） |
| 规则名称 | 风险规则名称 |
| 风险等级 | 高/中/低 |
| 风险识别指引 | AI识别逻辑（定位范围+判定标准） |
| 修改建议 | 修改建议文本 |

## 四、核心Prompt模板

> 来源：Wiki [合同质检后台配置、核心Prompt、POC效果、成本预估](https://wiki.ingageapp.com/pages/viewpage.action?pageId=148121092)

### 4.1 通过合同模板提取配置内容

#### KeyElement - 提取关键要素

- **用途**：分析标准合同模板，识别所有"变量"字段，自动生成提取指引
- **输入变量**：合同模板原文 (standard_template)
- **核心逻辑**：
  1. 行业与标的物分析
  2. 识别显式变量（下划线、括号、留白）和隐式核心变量（合同总额、违约金比例等）
  3. 编写提取指引（严禁使用章节号，必须使用"语义锚点"定位）
  4. 定义字段属性（Field Label、Field Type）
- **输出格式**：JSON `extraction_config` 数组

#### RiskRuleGenerator - 风险规则提取

- **用途**：基于标准合同模板（安全基线），反向推导风险审查规则库
- **输入变量**：合同模板原文 (standard_template)
- **核心逻辑**：
  1. 全域基线扫描（付款时效、验收机制、违约责任、解约权）
  2. 反向推导通用风险（付款拖延、验收缺失、单方解约、违约金过低）
  3. 植入LTC流程合规规则（非标赠送、退款陷阱、高额违约金、关联交易）
  4. 编写识别指引（语义定位，禁止章节号）
  5. 风险定级（High/Medium/Low）
- **输出格式**：JSON `risk_rules` 数组

### 4.2 AI处理核心节点

#### ContractContentParser - AI提取结构化字段

- **用途**：从OCR全文中提取后台配置的关键字段
- **输入**：合同原文 + 质检规则配置
- **核心策略**：
  - 基础字段：关键词定位 + 金额清洗（移除¥/,/元）+ 日期统一（YYYY-MM-DD）
  - 数组字段：转换为"一句话摘要"格式，方便阅读
  - 空值处理：基础字段返回null，数组字段返回空数组
- **输出**：结构化JSON对象

#### CompletenessChecker - 内容完整性检查器

- **用途**：检查提取出的字段是否为空，计算完整性得分
- **输入**：合同提取数据 + 质检规则配置
- **输出**：completenessScore + missingFields列表

#### ConsistencyChecker - 数据一致性检查器

- **用途**：将合同提取数据与CRM数据进行比对
- **输入**：合同提取数据 + CRM系统数据 + 比对规则配置
- **核心策略**：
  - 数值比对：允许误差±1.0
  - 文本比对：忽略标点/空格/大小写/公司后缀
  - 数组比对（产品明细）：数量核对→模糊匹配→数值核对→LTO赠送检查
  - 数组比对（付款计划）：期数核对→金额核对→节点核对
- **评分**：Score = ((A - B) / A) * 100（A=总字段数，B=不一致数）
- **输出**：inconsistencies数组 + consistencyScore

#### RiskClauseIdentifier - 风险条款识别器

- **用途**：根据审查规则库对合同进行逐条审计
- **输入**：合同全文 + 审查规则库
- **核心策略**（三步走）：
  1. 规则类型解析：数值阈值型/行为禁止型/条款缺失型/条款篡改型
  2. 逻辑门判定：数值比对、合规与禁止性逻辑、完整性与存在性逻辑
  3. 输出生成：仅输出RISK条目，RiskReason必须包含"提取值 vs 规则值"对比证据
- **评分**：riskScore = 100 - (High×15 + Medium×5 + Low×2)
- **输出**：riskClauses数组 + riskScore

#### OverallScoreCalculator - 整体评分

- **用途**：汇总前三个节点结果，给出最终结论
- **权重**：完整性20% + 一致性30% + 风险50%
- **输出**：overallScore + summary（100字左右，按P0-P3优先级强调问题）

### 5.5 分数计算逻辑

```
integrity_score = 正确数 / 全部完整性字段数 × 100（向下取整）
consistency_score = 正确数 / 全部一致性字段数 × 100（向下取整）
risk_score = 100 - (15 × 高风险数) - (5 × 中风险数) - (2 × 低风险数)（扣到0为止）
score = integrity_score × 0.2 + consistency_score × 0.3 + risk_score × 0.5
```

### 7.2 异常处理

- 网络错误/模型调用失败：自动重试3次，指数退避（2s/5s/10s）

### 7.3 审计日志

记录内容：质检触发时间/人/方式、质检耗时、状态、失败原因、Prompt输入输出

| 风险 | 缓解措施 |
|:---|:---|
| OCR识别准确率不高 | 质检报告中提示"建议人工复核关键条款" |
| 合同全文过长导致超时 | 滑动窗口分段处理 + 并行调用AI节点 |
| AI模型调用失败 | 重试机制 + 兜底策略 |
| 质检准确率不高 | 持续优化Prompt模板，基于反馈迭代 |

## 十、Agent架构中的定位

> 来源：TCHub P0-PRD、P0-方案设计、产品设计概览

### 10.1 在Agent四层架构中的位置

合同质检在Agent架构重构（P0）中被定义为一个 **Skill**（从现有组件迁移升级）：

| 属性 | 值 |
|:---|:---|
| Skill名称 | 合同质检 |
| apiKey | contract_check |
| 来源 | 迁移自现有组件 → 升级为Skill |
| 核心能力 | 多维度合同质检 |
| 依赖Tools | query_crm |
| 期次 | ⏳ 第二期（V2.0，第7-8周） |
| 延后原因 | 后台配置引擎（质检规则、模板）较为复杂，且现有功能入口仍可正常使用 |

### 10.2 Skill升级后的增强能力

升级为Skill后具备Agentic能力：
- 质检结果可反思补充
- 可根据上下文调整执行策略
- 可组合多个Tool完成任务
- 后台配置引擎（模板配置、质检规则）通过内部接口读取，Skill在执行时获取配置数据

### 10.3 与其他模块的关系

- **P2 Agent自主执行**：合同质检暂不纳入自主触发（文件选择逻辑复杂，后续扩展）
- **P0 管理后台**：Skill列表中展示"后台配置"链接，点击跳转到合同质检模板列表页
- **预置Skill**：系统预置，不可删除，但可编辑instructions和禁用
