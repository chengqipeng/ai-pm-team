# Context Preparation - 上下文准备工作流

## 定位
这是产品设计流程的**独立前置工作**。在启动任何设计流程（需求澄清→方案设计→PRD）之前，先通过本工作流完成上下文准备，确保 `context/features/` 下的文档完整且时效性达标。

设计流程不再负责采集或生成上下文，只负责**加载和引用**已有上下文。

## 核心原则

### 产出物定义
本工作流产出两类文档，存放在 `context/features/` 下：

| 类型 | 命名规则 | 信息源 | 内容 |
|:---|:---|:---|:---|
| 产品设计文档 | `功能现状.md` 或 `功能总览.md` | Wiki、Figma、会议纪要 | 功能定位、业务场景、交互设计、业务规则、Limits |
| 代码分析文档 | `代码分析.md` | 源码仓库 repos/ | 数据模型、调用链、扩展点、技术约束 |

文件结构：
- 每个功能一个中文子目录：`context/features/{功能名称}/功能现状.md` + `代码分析.md`（如有）
- 内容较多时可拆分子文件（如提示词模板拆分为多个维度文件）

### 时效性规则
- 产品设计文档：无硬性过期，Wiki有更新时手动刷新
- 代码分析文档：**7天过期**，超期需刷新后才能用于方案设计
- 文件头部必须包含"最后更新"日期，用于时效判断

### 信息完整度要求
采集内容需达到"可支撑产品方案设计"的颗粒度：
- 完整的业务规则与边界条件
- 具体的交互文案与错误提示
- 字段级配置规则
- 版本演进时间线
- 业务场景的具体客户案例
- Limits与约束条件的具体数值
- Wiki评论中的产品反馈（标记为"待处理"）

### 信息不足时的处理
- Wiki页面内容为空或仅有图片 → 标注"该页面内容为图片/空页面，无法提取文本信息"
- 信息不完整 → 保留原文并标注"待补充"
- 不要臆造或推测信息源中没有的内容

## 执行流程

用户触发本工作流时，按以下步骤执行：

### Step 1：确认准备范围

向用户确认：
1. 要准备哪些功能模块的上下文？
2. 需要哪些信息源？（Wiki / 代码分析 / Figma / 全部）
3. 是新采集还是刷新已有文档？

如果用户已明确（如"帮我采集提示词模板的Wiki和代码"），直接执行，无需反复确认。

### Step 2：检查已有上下文状态

读取 `context/features/` 下已有文档，汇总状态：

| 功能模块 | 产品设计文档 | 状态 | 代码分析文档 | 状态 |
|:---|:---|:---|:---|:---|
| {模块名} | {路径} | ✅有效 / ⚠️待刷新 / ❌不存在 | {路径} | ✅有效(<7天) / ⚠️已过期 / ❌不存在 |

向用户报告状态，确认哪些需要采集/刷新。

### Step 3：执行采集（按信息源类型）

#### 3A. Wiki 采集
1. 确认Wiki页面范围（主页面 + 全部子页面）
2. 通过 `wiki_get_page_tree` 获取子页面完整列表
3. 通过 `wiki_batch_get_pages` 批量获取内容
4. 提取：正文、表格、列表项；图片标注"交互示意图，参见Wiki"
5. 评论区产品反馈单独记录
6. 组织为结构化文档，写入 `context/features/`

#### 3B. 代码分析
1. 确认代码仓库和模块范围（repos/ 下的哪些目录）
2. 使用 `#source-code-analyzer` skill 进行深度分析
3. 分析内容：
   - 核心数据模型：实体、字段、关系（与产品设计文档对比，标注差异）
   - 核心流程：从入口到数据层的调用链（Mermaid流程图或时序图）
   - 扩展点：接口、抽象类、配置项、Hook点
   - 技术约束：框架限制、性能瓶颈、已知TODO/FIXME
   - 关键文件索引：核心文件路径和职责
4. 结果写入 `context/features/{feature}/*-code.md` 或 `code-analysis.md`

#### 3C. Figma 设计稿
1. 用户提供 Figma 文件链接
2. 使用 Figma MCP 工具获取设计数据
3. 提取：页面结构、组件层级、交互标注、设计规范
4. 关键截图下载到本地
5. 设计信息整合到产品设计文档的对应章节

### Step 4：质量检查

- [ ] 所有目标页面/模块都已访问
- [ ] 关键业务规则未遗漏
- [ ] 版本演进信息完整
- [ ] Limits数值准确
- [ ] 代码分析文档包含数据模型和扩展点
- [ ] 文件头部日期正确

### Step 5：更新索引

1. 更新 `.kiro/steering/pm-context-product.md` 中的"功能现状文档索引"表格
2. 更新本文件末尾的"已采集功能索引"

## 输出规范

### 文件头部（必须）
```markdown
# {功能名称} - {文件主题}

> 最后更新：{YYYY-MM-DD}
> 数据来源：{信息源描述}

## Changelog
| 日期 | 变更内容 |
|:---|:---|
| {日期} | {变更说明} |
```

### 代码分析文档头部（必须包含日期，用于7天过期判断）
```markdown
# {功能名称} - 代码实现分析

> 最后更新：{YYYY-MM-DD}
> 代码仓库：repos/{repo-name}
> 分析范围：{涉及的模块/目录}

## Changelog
| 日期 | 变更内容 |
|:---|:---|
| {日期} | {变更说明} |
```

### 内容格式要求
- 使用中文
- 表格优先于纯文本列表
- 业务规则逐条列出，不合并为段落
- 代码示例保留完整
- 版本信息标注在对应功能点旁边

## tchub 文档管理

### tchub 概念模型
- **Project（项目）**：顶层组织单元，如"Neo AI 平台"
- **Feature（功能）**：项目下的功能模块，如"提示词模板管理"
- **Workstream（工作流）**：功能下的具体工作项，如"支持多模态大模型"
- **Document（文档）**：挂在工作流下的产品文档

### 上传范围约定
上传 tchub 时，**只上传 `product-specs/{project-name}/` 根目录下的文档**，`archive/` 子目录下的历史版本不上传：

| 路径 | 是否上传 |
|:---|:---|
| `product-specs/{project}/产品-需求澄清.md` | ✅ 上传 |
| `product-specs/{project}/产品-方案设计.md` | ✅ 上传 |
| `product-specs/{project}/产品-PRD.md` | ✅ 上传 |
| `product-specs/{project}/archive/*.md` | ❌ 不上传，仅本地留存 |

## pm-context-product.md 维护

### 维护对象
文件：`.kiro/steering/pm-context-product.md`（auto-included，所有对话都会加载）

### 路线图状态更新
当功能状态变化时，更新"进行中 & 规划中"表格：
- 新功能开始 → 添加行，状态 🔄进行中
- 功能上线 → 从表格移除
- 新规划 → 添加行，状态 📋规划中

### 功能文档索引更新
当 `context/features/` 下有新文档时，更新"功能现状文档索引"表格：
- 新增采集 → 添加行
- 文档更新 → 更新"最后更新"日期
- 文档路径变更 → 更新路径

### 维护原则
- 只修改路线图和文档索引部分，不动其他内容
- 其他部分的修改需用户明确要求
