# pm-context-product 生成器

## 定位
通用辅助工具，帮助任何产品的 PM 生成适配自身产品的 `.kiro/steering/pm-context-product.md`。
该文件是 auto-included steering，每次对话自动注入，直接决定 Kiro 对产品的理解深度。内容太泛则方案缺乏针对性，内容偏离则基于错误假设做设计，内容缺失则无法理解技术约束。

---

## 执行流程

### Step 1：收集产品定位信息

向用户确认以下信息。如果用户消息中已提供，直接使用，不反复确认。

**必填项：**

| 维度 | 问题 |
|:---|:---|
| 产品名称 | 产品/平台叫什么？ |
| 产品定位 | 一句话描述产品是什么、解决什么问题 |
| 核心价值 | 产品带来的核心价值是什么（从 X 到 Y 的转变） |
| 负责模块 | PM 负责哪些具体模块/服务/功能域 |
| 目标用户 | 产品的核心用户角色有哪些 |

**选填项（有则更好）：**

| 维度 | 问题 |
|:---|:---|
| 核心优势 | 产品的 2-3 个差异化优势 |
| 技术栈 | 主要技术选型（后端/前端/中间件/数据库） |
| 代码仓库 | `repos/` 下的哪些目录包含相关代码 |
| 设计文档 | 已有的架构/设计文档路径 |
| 上下游依赖 | 与哪些其他模块/团队有依赖关系 |
| 行业/领域 | 产品所在的行业或业务领域 |

### Step 2：分析代码与文档

根据用户提供的模块信息，自动执行以下分析（有对应资源时才执行，没有则跳过）：

| 分析项 | 方法 | 提取内容 |
|:---|:---|:---|
| 代码仓库结构 | 扫描 `repos/` 下对应目录 | 微服务拆分、模块职责、包结构 |
| 设计文档 | 读取 README.md、*-DESIGN.md、*-SPEC.md 等 | 架构设计、设计决策、规范约束 |
| 数据模型 | 从代码或文档中提取 | 核心实体、字段、关联关系 |
| API 接口 | 从 Controller/API 定义中提取 | 对外 API 列表、参数、返回值 |
| 已有 context | 检查 `context/features/` | 已采集的功能文档 |

**分析原则：**
- 信息优先级：设计文档 > 代码结构 > README > 用户口述
- 有明确文档的内容直接引用，不臆造
- 没有代码仓库时，完全基于用户提供的信息生成

### Step 3：识别产品类型，确定章节结构

根据收集到的信息，判断产品属于哪种类型（可以是多种的组合），然后从章节库中选择适用章节。

**产品类型识别（用于选择条件章节）：**

| 类型标签 | 判断依据 |
|:---|:---|
| `平台型` | 产品包含多个微服务/模块，提供基础能力供上层使用 |
| `数据驱动型` | 产品核心是数据模型/元数据/Schema 定义 |
| `AI/智能型` | 产品涉及大模型调用、Agent、Prompt 管理 |
| `业务应用型` | 产品面向具体业务场景（CRM/ERP/电商等） |
| `前端/体验型` | 产品核心是 UI 框架、组件库、设计器、布局引擎 |
| `基础设施型` | 产品是框架、中间件、DevOps 工具等底层基础设施 |
| `集成/连接型` | 产品核心是对接外部系统、数据同步、API 网关 |

**章节库：**

必选章节（所有产品）：

| 章节 | 内容要求 |
|:---|:---|
| 产品定位 | 2-3 句话，说清"是什么、做什么、核心价值" |
| 核心优势 | 2-3 条，每条一句话 + 简要展开 |
| 全局基线约束 | 10-15 条具体可执行的硬约束（命名规范、编码规范、安全要求、性能要求等） |
| 进行中 & 规划中 | 表格，当前迭代方向和状态 |
| 功能现状文档索引 | 指向 `context/features/` 下已有文档 |

条件章节（根据产品类型标签选择）：

| 章节 | 触发条件 | 内容要求 |
|:---|:---|:---|
| 架构分层/数据架构 | `数据驱动型` 或 `平台型` | 核心分层、表结构/存储策略，不超过 2 个表格 |
| 微服务/模块矩阵 | `平台型` | 服务列表表格，每行一个服务，含职责说明 |
| AI/智能能力 | `AI/智能型` | Agent/模型/Prompt 等能力列表，调用约束 |
| 业务场景矩阵 | `业务应用型` | 按角色或场景分类的功能列表 |
| 技术栈 | `平台型` 或 `基础设施型` 或用户明确提供 | 一个表格，按层级列出 |
| 数据模型 | 产品有核心业务实体 | 一个表格，只列核心实体和关键字段 |
| 接口规格 | 产品对外暴露 API 且 API 设计是核心关注点 | 核心 API 列表 |
| 前端/组件体系 | `前端/体验型` | 组件库、设计规范、布局体系 |
| 集成/协议 | `集成/连接型` | 对接系统列表、协议规范、数据流 |
| 设计参考 | 有详细设计文档可引用 | 文档路径索引，指向 repos/ 或 context/ 下的具体文件 |

### Step 4：生成 pm-context-product.md

**文件头部（固定格式）：**
```markdown
---
inclusion: auto
description: {产品名称}产品定位、{2-3个核心关键词}与全局基线约束
---
```

**生成原则：**

| 原则 | 说明 |
|:---|:---|
| 简洁优先 | 每个章节用最少文字传达最多信息。此文件每次对话注入，太长浪费 token |
| 表格优先 | 能用表格的不用段落 |
| 引用优先 | 详细内容通过"设计参考"章节指向具体文档路径，不在此文件中展开全部细节 |
| 准确优先 | 所有具体数据（表名、字段名、API 路径、数量等）必须来自代码或文档，不编造 |
| 约束可执行 | 全局基线约束必须具体，如"字段命名统一 camelCase"，不写"注意命名规范"这种空话 |

**长度控制：**

| 章节 | 目标长度 |
|:---|:---|
| 产品定位 | 2-3 句话 |
| 核心优势 | 3 条，每条 1-2 句 |
| 架构/数据体系 | 不超过 2 个表格 + 1 个层级关系图 |
| 微服务/模块矩阵 | 1 个表格 |
| 技术栈 | 1 个表格 |
| 数据模型 | 1 个表格，只列核心实体 |
| 全局基线约束 | 10-15 条 |
| 进行中 & 规划中 | 1 个表格 |
| 总文件长度 | 150-250 行 markdown |

### Step 5：确认与写入

生成后向用户确认：
1. 产品定位描述是否准确
2. 核心优势是否体现差异化
3. 全局约束是否有遗漏或需要调整
4. 是否需要增减章节

用户确认后写入 `.kiro/steering/pm-context-product.md`。

如果用户说"直接生成"或"不用确认"，跳过确认直接写入。

## 全局基线约束的编写指南

全局基线约束是 pm-context-product.md 中最重要的章节之一，它直接约束 Kiro 在所有设计中的行为。

**好的约束（具体、可执行、可验证）：**
- 字段 apiName 统一 camelCase 英文，展示名用简短中文
- 接口响应时间 <3s（P95）
- 所有 API 返回统一 Result 包装，错误码 6 位数字
- 数据库表名统一 snake_case，前缀 `t_`
- 敏感字段（手机号、身份证）存储时加密，展示时脱敏

**差的约束（模糊、不可执行）：**
- 注意性能（多快？什么场景？）
- 遵循安全规范（什么规范？）
- 代码要规范（什么规范？）

**约束来源优先级：**
1. 代码中已有的规范（从代码或设计文档中提取的实际约束）
2. 用户明确提出的约束
3. 行业通用最佳实践（仅在前两者不足时补充）

## 质量检查

生成完成后，逐项检查：

- [ ] 产品定位一句话能说清"是什么、做什么"
- [ ] 核心优势有差异化，不是泛泛而谈
- [ ] 所有具体数据来自代码或文档，没有编造
- [ ] 全局约束每条都具体可执行
- [ ] 总长度在 150-250 行之间
- [ ] 文件头部 `inclusion: auto` 且 description 准确
- [ ] 进行中/规划中的方向与实际一致
- [ ] 条件章节的选择与产品类型匹配（没有多余章节，也没有遗漏关键章节）
- [ ] 设计参考章节（如有）指向了正确的文档路径

## 维护说明

pm-context-product.md 生成后并非一成不变：

| 场景 | 更新方式 |
|:---|:---|
| 产品定位调整 | 重新运行本工具，或手动修改产品定位和核心优势 |
| 新增模块/服务 | 在微服务矩阵或模块列表中添加行 |
| 功能上线 | 从"进行中"移除 |
| 新增全局约束 | 在全局基线约束中添加条目 |
| 功能文档采集 | 通过 `#pm-context-harvest` 更新功能现状文档索引 |
| PM 换负责模块 | 重新运行本工具生成新的产品上下文 |
