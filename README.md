# 产品设计工作流

基于 Kiro IDE 的 AI 产品设计框架。三个阶段，三个文档，从需求到研发可执行的 PRD。

## 它能做什么

把产品设计过程拆成三步，每步产出一个文档，Kiro 自动衔接：

| 阶段 | 产出 | 回答的问题 | 读者 |
|:---|:---|:---|:---|
| 需求澄清 | 产品-需求澄清.md | 做不做？ | PM、业务方、决策者 |
| 方案设计 | 产品-方案设计.md | 怎么做？ | PM、技术负责人、架构师 |
| PRD 撰写 | 产品-PRD.md | 做成什么样？ | 设计师、开发、测试、业务方 |

每个文档有明确的读者和目的，后一个不重复前一个的内容。需求澄清阶段会识别需求类型和特征标签，自动驱动后续文档的章节裁剪——不相关的章节不写。

## 快速开始

### 1. 环境准备

1. 安装 [Kiro IDE](https://kiro.dev)，推荐模型 Sonnet 4.6 或 Opus
2. 用 Kiro 打开本工作区

### 2. 填写产品上下文（必做）

编辑 `.kiro/steering/pm-context-product.md`，填写你负责的产品信息：
- 产品定位、核心能力
- 数据模型、全局约束
- 进行中和规划中的方向

这个文件会自动注入每次对话，让 Kiro 理解你的产品。

### 3. 准备功能上下文（按需）

如果需求涉及已有功能的优化，先准备功能现状文档：

```
#pm-context-harvest
帮我采集 {功能名称} 的 Wiki 文档
```

文档会存到 `context/features/{功能名称}/功能现状.md`。

### 4. 开始设计

```
帮我做 {项目名} 的需求澄清

需求描述：{一句话描述需求}
需求来源：{来源}
相关功能：context/features/{功能名称}/功能现状.md
```

完成后 Kiro 会自动提醒进入下一阶段，说"进入下一阶段"即可继续。

小需求可以一步到位：用 `#pm-0-full-pipeline` 一次性生成 3 个文档。但大多数情况建议分阶段走，每步检查调整。

### 5. 可选配置

- **Wiki MCP**：编辑 `.kiro/settings/mcp.json`，填写 Wiki 凭证，方案设计阶段可自动搜索 Wiki
- **源码分析**：把代码仓库 clone 到 `repos/` 下，方案设计阶段会自动分析相关代码

## 工作区结构

```
.
├── .kiro/                # Kiro 配置
│   ├── steering/         # 流程模板和工具（核心）
│   ├── hooks/            # 阶段自动衔接
│   ├── skills/           # 源码分析技能
│   └── settings/         # MCP 配置
├── context/
│   ├── features/         # 功能上下文（产品设计 + 代码分析）
│   └── projects/         # 项目上下文
├── product-specs/{project-name}/  # 产品设计输出
│   ├── 产品-需求澄清.md
│   ├── 产品-方案设计.md
│   └── 产品-PRD.md
├── repos/                # 代码仓库（git clone，用于源码分析）
├── tools/                # MCP 工具脚本
└── README.md
```

### context/features/ 文件分类

| 文件 | 视角 | 来源 | 说明 |
|:---|:---|:---|:---|
| `功能现状.md` | 产品设计 | Wiki 采集 | 功能定位、业务规则、交互设计 |
| `代码分析.md` | 代码实现 | 源码分析 | 数据模型、调用链、扩展点、技术约束 |

两类文档互补：产品设计文档描述"设计意图"，代码分析文档描述"实际实现"。方案设计阶段会同时读取两者。

## 三阶段详解

### 阶段 1：需求澄清（做不做？）

Kiro 会帮你：
- 澄清核心痛点并量化
- 明确目标用户和使用场景
- 识别涉及的现有模块
- 完成四维价值评估（业务/用户/技术/战略，总分 20 分）
- 识别需求类型（9 选 1）和特征标签（多选）

特征标签驱动后续文档的章节裁剪——不相关的章节不写。评分 <10 分会建议放弃。

### 阶段 2：方案设计（怎么做？）

合并了"调研分析"和"方案设计"，一个文档讲清楚"基于现状，我们怎么做"。Kiro 会自动：
- 读取 `context/features/` 下的现状文档和代码分析
- 搜索 Wiki 补充信息（需配置 Wiki MCP）
- 进行竞品调研（可说"跳过竞品调研"）

信息可信度排序：代码实现 > context 文档 > Wiki > 竞品信息

### 阶段 3：PRD 撰写（做成什么样？）

输出单文件 PRD，开发能做技术设计，测试能写用例。末尾有审核清单，审核通过即完成产品设计交付。

## 下游流程

| 下游 | 主要输入 | 说明 |
|:---|:---|:---|
| UI/UX 设计 | PRD | 用户流程表格和业务规则定义交互 |
| 技术方案 | PRD + 方案设计 | PRD 定义接口规格，方案设计指导技术选型 |
| 开发 | PRD | 按模块开发，业务规则即代码逻辑 |
| 测试 | PRD | 验收标准直接转测试用例 |
| 业务验收 | PRD + 需求澄清 | 需求澄清确认需求没跑偏 |

## FAQ

| 问题 | 解答 |
|:---|:---|
| 小需求要走完整流程吗？ | 不用，`#pm-0-full-pipeline` 一次性生成 3 个文档 |
| 大需求和小需求的区别？ | 产出物一样，区别是分阶段走可以中间停下来检查调整 |
| Kiro 不理解产品上下文？ | 检查 `.kiro/steering/pm-context-product.md` 是否完整 |
| PRD 有问题需要改？ | 小问题直接改，大问题回阶段 2 重新设计 |
| PRD 之后谁做 Spec？ | 研发团队自行完成，PM 专注产品设计 |
| 想快速出 Demo？ | 用 `#pm-tool-prototype` |
| 怎么让 Kiro 读 Wiki？ | 配置 `.kiro/settings/mcp.json` 中的 wiki server |
| 怎么让 Kiro 分析代码？ | 把代码仓库 clone 到 `repos/` 下，阶段 2 会自动分析 |
| 怎么上传文档到 tchub？ | 用 `#pm-context-harvest`，说"上传到 tchub" |

## 归档约定

当项目发生大版本迭代或方向性调整时，将文档快照归档到 `归档/` 目录：

```
归档/{项目名}-V{版本}-{日期YYMMDD}/
```

- 当前版本永远在 `product-specs/{project-name}/` 下
- 归档仅本地留存，不上传 tchub
- 小改动不需要归档，直接在 product-specs 下迭代
